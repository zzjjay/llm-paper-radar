# рҹ“Ў LLM Paper Radar

> Daily, automated digest of LLM compression and inference-optimization papers.

A small pipeline that fetches papers from arXiv + HF Daily + OpenReview + watched authors, kills obvious off-topic locally with a keyword prefilter, scores the rest with Claude Sonnet 4.6 against a two-axis rubric (topic relevance Г— practicality), tags each survivor with one of eight topic buckets (PTQ / Low-bit / QAT / KV cache / Pruning & distillation / Diffusion / Trending / Survey & methodology), and renders two views: a compact **table-only README** for skimming and a **per-day detail page** with summaries, "why this paper" rationale, and related/compared methods. No numeric threshold вҖ” anything not hard-gated surfaces, with per-bucket caps controlling digest length. A single cron job keeps it running.

## рҹ§® How papers are scored

Two stages: a cheap local **keyword prefilter**, then **Claude Sonnet 4.6** scoring with [`prompts/relevance.md`](prompts/relevance.md).

### Stage 1 вҖ” keyword prefilter (local, free)

Configured under `filter.prefilter` in [`config.yaml`](config.yaml). Each paper's title + abstract is matched against two word-boundary-aware lists:
- **whitelist** (e.g. `PTQ`, `AWQ`, `MXFP4`, `W4A4`, `BitNet`, `KV cache quantization`, вҖҰ)
- **blacklist** (e.g. `image classification`, `ImageNet`, `federated learning`, `RAG`, вҖҰ)

If a paper has **zero whitelist hits AND вүҘ `max_blacklist_hits` (default 2) blacklist hits**, it's hard-gated locally with no LLM call. Word boundaries are enforced so `QuIP` doesn't match inside `equipping` and `MIT` doesn't match inside `Amit`.

This is conservative on purpose вҖ” anything ambiguous goes to the LLM. Add patterns over time based on what you see in rejected.jsonl.

### Stage 2 вҖ” Sonnet two-axis rubric

Sonnet returns a structured JSON breakdown which the orchestrator combines into a 0вҖ“10 composite.

| axis | range | what it captures |
|---|---|---|
| `topic_relevance` | 0вҖ“5 | How squarely the paper sits in LLM compression. 5 = core PTQ/QAT/KV/low-bit/pruning_distill/diffusion work with proper accuracy benchmarks. 3 = compression-adjacent (sparse attention coupled with quant). 0 = unrelated. |
| `practicality` | 0вҖ“5 | Algorithm simplicity + clear OR plausible inference perf + low calibration cost + small GPU memory footprint. 5 = AWQ-style "few lines, big speedup, data-free." Already-shipped numbers are a bonus, not a requirement вҖ” a credible deployment path is enough. 0 = complex, intractable calibration, no perf story. |

`relevance_score = topic_relevance + practicality` (so 0вҖ“10). **There is no numeric threshold** вҖ” every paper with `hard_gate=false` is surfaced. Per-bucket caps below control digest length.

### Hard gate

`hard_gate = true` zeros both axes. Triggered when any of:

- **Out of scope**: unrelated topic (RAG, agents, alignment, pure training, multimodal-w/o-compression), pure speculative decoding (coupled spec + quant stays in), pure review-article survey with no new measurement, or no fit to any of the eight buckets
- **`practicality = 2`** вҖ” only 1 favorable signal; deployment cost outweighs gain
- **No benchmark AND composite вүӨ 7**: `accuracy_benchmarks вҲҲ {none, unknown}` AND `topic_relevance + practicality вүӨ 7` вҖ” unless the paper compares against an established baseline (GPTQ, AWQ, SmoothQuant, QuaRot, BitNet, KIVI, вҖҰ) or both axes вүҘ 4
- **Scale**: largest model < 1B (BERT-base, GPT-2-small); stricter for `ptq` only вҖ” < 7B вҶ’ hard_gate (gaps at 1B routinely flip at 7B+; modern LLM family + unknown size = default trust)
- **Unstructured sparsity** needing novel GPU kernels not in shipping stacks (vLLM / SGLang); N:M, MoE expert pruning, layer drop stay in scope

### Topic buckets and per-bucket caps

Eight LLM-pickable buckets вҖ” seven strict compression buckets plus `trending` as a soft catch-all for compression-adjacent decoding-acceleration work. No `other` вҖ” papers that fit none get hard-gated. Caps in [`config.yaml`](config.yaml) вҶ’ `render.topic_caps`:

| bucket | cap | what goes here |
|---|---|---|
| **`ptq`** | 8 | Post-training quantization (W-only / W+A / KV-quant when PTQ is the primary contribution), bit-width вүҘ 3. GPTQ, AWQ, SmoothQuant, QuaRot, SpinQuant, MXFP4, NVFP4 |
| **`low_bits`** | 5 | Sub-3-bit (вүӨ 2-bit) quantization, regardless of training method. BitNet b1.58, AQLM, VPTQ, QuIP#, ternary, binary |
| **`qat`** | 5 | Quantization-aware training or PTQ + full-network fine-tune, bit-width вүҘ 3. LLM-QAT, EfficientQAT, PB-LLM |
| **`kv_cache`** | 5 | KV cache compression where layout / eviction is the main contribution. KIVI, KVQuant, H2O, StreamingLLM |
| **`pruning_distill`** | 3 | Pruning / sparsity / distillation with a deployable kernel path (N:M, MoE expert pruning, layer drop, SFT KD). Wanda, SparseGPT, Sheared LLaMA, MiniLLM |
| **`diffusion`** | 3 | **Quantization only** on diffusion or flow-matching backbones. Step-distillation / pruning / sampling-trajectory / guidance methods вҶ’ hard_gate. Q-Diffusion, SVDQuant, DiRotQ |
| **`trending`** | 3 | Compression-adjacent decoding-acceleration without a direct compression algorithm вҖ” parallel/dual-view drafters, spec-decoding frameworks. Use sparingly; routine EAGLE/Medusa still hard_gate. Orthrus, DFlash, PagedAttention (via milestone override) |
| **`survey`** | 3 | Compression-method head-to-head OR compression-actionable measurement (empirical PTQ comparisons, outlier-bottleneck studies, calibration-data studies, eval methodology). Excludes infrastructure/energy benchmarks, downstream-property studies (bias/trust/CL), pure theory/position papers. Pure review surveys still hard_gate. |

**Tie-breaks**
- Bit-width wins: 2-bit PTQ вҶ’ `low_bits` (not `ptq`); 1.58-bit pretrained вҶ’ `low_bits` (not `qat`)
- Algorithm beats survey: new loss / rotation / datatype / calibration recipe вҶ’ bucket by the algorithm even if the paper also benchmarks broadly. `survey` is only when measurement/methodology is *itself* the contribution.

**Caps scope:** the README compact table lists every surviving paper (no cap). Caps only gate which papers get a full detail block on the per-day digest page.

**Milestone override:** papers in [`seeds.yaml`](seeds.yaml) under `category: trending` (vLLM, FlashAttention v1/v2/v3, EAGLE / Medusa / Lookahead, SGLang) get force-routed to the `trending` bucket вҖ” bypassing any LLM `hard_gate` вҖ” when HF trending rank вүӨ 20 OR `code_meta.stars` вүҘ 5000. Lets landmark serving / attention / spec-decoding papers surface even though they're "no new compression algorithm." See [`pipeline/filter.py`](pipeline/filter.py).

**Watched authors:** `sources.arxiv_authors.authors` in [`config.yaml`](config.yaml) (Dan Alistarh / IST Austria, Song Han / MIT HAN Lab, Qualcomm AI Research) gets a dedicated **рҹ‘Ө Watched authors** block on each detail page that bypasses caps and shows *all* of their papers. The compact table still applies `hard_gate`, so off-topic watched-author work stays detail-page-only.

### Table ordering

Bucket first (BUCKET_ORDER above), composite within bucket:

```
composite   = relevance_score Г— 30 + heat_score
heat_score  = trending_bonus + hf_daily_upvotes + star_bonus
trending_bonus = 100 / hf_daily_rank             (rank 1..30, else 0)
star_bonus     = min(log(github_stars + 1) Г— 3, 25)
```

`relevance_score` (0вҖ“10) dominates вҖ” a 9/10 paper outweighs ~270 HF upvotes вҖ” but heat lets a viral paper (HF Daily #1 = +100) jump ahead of a same-bucket peer with marginally higher relevance. `star_bonus` reads HF's `githubStars` field (no API call); ~1k stars вүҲ +21, capped at 25 so framework repos can't outweigh relevance, and missing-repo = 0 is never penalized. Tune the weights via `RELEVANCE_WEIGHT` / `STAR_WEIGHT` / `STAR_BONUS_CAP` in [`pipeline/render.py`](pipeline/render.py).

Each stage is independently runnable from the CLI. Every command defaults to **today only** вҖ” add `--backfill-days N` to process today + N days back (e.g. `--backfill-days 6` for a 7-day batch). Windowed sources (`arxiv_authors`, `openreview`) take `--window-days` instead and fetch the whole window in one call.

```bash

```

`scripts/daily.sh` chains all of the above end-to-end, runs `scripts/snapshot.sh` to archive the rendered paper list under `snapshots/`, then `git commit && git push` if anything changed. `PAPER_RIVER_MAX=N` caps how many paper-river `.org` files get generated per run (default: no cap).

### рҹҢҠ Paper River (optional)

For each surfaced paper, `scripts/daily.sh` invokes the [`ljg-paper-river`](https://github.com/lijigang/ljg-skills) Claude Code skill to produce a deep-lineage analysis вҖ” "еҖ’иҜ»жі•": recursively trace 5 layers of intellectual ancestry, then walk forward Feynman-style вҖ” saved as `paper-river/<acronym>-<arxiv-id>.org` and auto-translated to an `_en.org` sibling. Render links each digest to its own language. Disable with `PAPER_RIVER_SKIP=1`; render works fine without any `.org` file.

Install the skill once:

```
/plugin marketplace add lijigang/ljg-skills
/plugin install ljg-skills@ljg-skills
```
