* 问题之河

把一百万个 token 喂给大语言模型，如何让 GPU 内存不爆掉——同时还要精度不掉、速度不降？

KV Cache 是推理时的记忆本：模型每见一个 token，就要把对应的 key 和 value 向量记下来，后面生成每个词都要翻一遍这本账。本子的大小和序列长度成正比，100K token 就要几十 GB，1M token 就要几百 GB。一块最好的 H800 也就 80GB。

怎么让这本账又薄又不失真，是这条研究线要解决的问题。研究者们前赴后继，每个人都找到了一把剪刀，每把剪刀能剪的方向不一样，最后的问题是：怎么同时从两个方向剪，还不把书剪烂？

* 演化叙事

** 2023.06 | H2O: 注意力是幂律分布，那就只留重要的 token

*** 它看到了什么问题

推理服务商们遇到了一个死局：上下文越长，内存越贵，但你不能提前知道哪个 token 后面还有用，所以只好全部留着。一块 80GB 的 A100，装上 7B 模型的权重之后，能给 KV Cache 的空间只剩下二三十 GB，大概能存 5 万个 token 的历史。超过这个量，不是报错就是换页到 CPU，速度立刻垮掉。

没有人研究过「到底哪些 token 真的在被用」。

*** 它的解法

H2O 的人做了一件很朴素的事：把注意力矩阵拿出来盯着看，发现注意力分数的分布是幂律的——小部分 token 拿走了大部分的注意力权重，其他 token 几乎被忽视。这些「重击者（Heavy Hitters）」token 不是随机的，往往对应语义关键词和频繁共现的词。

于是 H2O 提出了一个动态淘汰赛：维护固定大小的 KV Cache，每当要写入新 token 时，把累积注意力分数最低的那个踢出去。谁被用得最少，谁出局。

这是*序列维度*上的压缩：不改变每个向量的精度，只决定保留哪些 token 的向量。

*** 为什么这招管用（以及哪里埋了新坑）

有用，因为注意力确实极度稀疏——在 OPT 和 LLaMA 等模型上超过 95% 的注意力权重集中在 5% 的 token 上。H2O 把 KV Cache 压到原来的 5%，精度几乎不变，对比 DeepSpeed 和 HuggingFace Accelerate 吞吐量提升 29 倍。

但坑也在这里。序列维度的压缩有一个硬下限：你总要留住若干 token，否则模型完全失忆。随着 context 从 10 万膨胀到 100 万，即便只保留 1% 的 token，绝对数量仍然是 1 万个，内存压力没有本质缓解。更麻烦的是，淘汰策略依赖累积注意力分数，在流式推理场景下计算代价高，而且还有一个反常识的现象：初始 token 不管语义多平淡，注意力分数总是高得离谱——为什么？

** 2023.09 | StreamingLLM: 初始 token 是注意力的「汇点」，不能踢

*** 它看到了什么问题

滑动窗口注意力（只保留最近 N 个 token）是最自然的流式压缩方案，但它有一个奇怪的断崖：一旦序列长度超过窗口大小，精度就急剧崩塌——不是平滑退化，是直接断掉。为什么？

而且 H2O 也注意到了一件事：初始 token 往往排在「重击者」名单的最前面，即便它们是标点符号或无意义的开头词。这不合逻辑。

*** 它的解法

MIT 的研究者找到了答案。在 autoregressive 语言模型里，初始 token 对所有后续 token 都可见，训练久了，模型把「剩余注意力」集中倾倒在这些位置上——就像把零钱丢进一个公共的罐子。这些 token 是「注意力汇（Attention Sink）」，它们承载的是注意力分布的归一化需要，不是真实的语义依赖。

一旦滑动窗口把这些锚点踢掉，注意力分布就失去了归一化的接地点，整个 softmax 变形，精度雪崩。

StreamingLLM 的方案因此极简：永远保留初始 4 个 sink token 的 KV + 最近 N 个 token 的 KV。这样就能无限流式生成，无需 fine-tuning，LLaMA-2 在 4M token 下依然稳定。

*** 为什么这招管用（以及哪里埋了新坑）

有用，因为它解释了之前的反常现象，并给出了简洁的解法。流式场景下比滑动窗口重算快 22 倍。

但 StreamingLLM 是一个*固定策略*：哪些 token 留下，规则写死了，不根据内容调整。它解决了「不崩」的问题，没有解决「高压缩率下还能保精度」的问题。更重要的是，H2O 和 StreamingLLM 都只动了序列维度——决定留哪些 token，但每个保留下来的 token 的 key/value 向量，还是原原本本的 16-bit 浮点数存着。那这些向量本身能压吗？

** 2024.02 | KIVI: 换方向——不减 token 数量，压每个向量的精度

*** 它看到了什么问题

序列维度已经被 H2O 和 StreamingLLM 啃了一遍。KIVI 的人换了一个角度：KV Cache 里存的是大量浮点向量，16-bit 精度是必要的吗？也许我们可以用更少的 bit 存同样的信息。

但量化 KV Cache 有一个之前没人仔细研究过的问题：key 向量和 value 向量的数字分布长什么样？通道之间均匀吗？token 之间均匀吗？不了解分布就贸然量化，误差会集中在哪里？

*** 它的解法

KIVI 的人系统地研究了 LLaMA、Falcon、Mistral 等模型的 KV Cache 分布，发现了一个规律：key cache 有固定通道的异常值——少数几个特定维度（channel）的数值极大，其余维度相对平滑。

这个规律直接告诉你量化方案应该长什么样：key cache 按通道分组量化（per-channel），因为异常值锁定在固定通道，把每组通道的量化范围分别确定，就能精确捕获这些峰值。value cache 没有这个通道异常值问题，按 token 分组量化（per-token）更合适。

这种非对称设计（key 按通道，value 按 token）+ 2-bit 量化，就是 KIVI 的核心。额外保留少量全精度近期 token 作为缓冲。

这是*通道维度*上的压缩：不管保留哪些 token，把每个保留下来的向量本身的比特数从 16 压到 2。

*** 为什么这招管用（以及哪里埋了新坑）

有用，而且效果好得出乎意料：2-bit 量化下 LLaMA 精度几乎不变，同样内存下批次大小扩大 4 倍，吞吐量提升 2.35~3.47 倍。KIVI 开辟了一条和序列维度正交的路。

但正交也意味着局限：KIVI 把所有历史 token 都量化后保留，完全没有 token 淘汰。在 1M token 的场景下，即便每个向量只用 2-bit，累积起来的体积依然巨大：1M token × 每层若干向量 × 2-bit，乘下来还是几十 GB。2-bit 已是量化的激进下限，再低精度就真的撑不住了。

序列和通道两个方向都有上限。那么能不能同时用两条路？

** 2024.10 | ShadowKV: 两条路一起走，但策略还是全局统一的

*** 它看到了什么问题

到 2024 年底，有人开始认真考虑把序列压缩和通道压缩结合起来。ShadowKV 就是第一批认真做这件事的工作之一。

它的具体问题是：在超长 context（128K~1M token）下，单独的序列压缩或通道压缩都不够用了，但朴素地把两种方法叠在一起又会造成精度的双重损失。还有一个系统级问题：GPU 内存紧张，但 CPU 有大量富余内存，如何把两者结合起来？

*** 它的解法

ShadowKV 在两个维度上分别动手：

通道维度——对 pre-RoPE 的 key cache 做 SVD 低秩分解。发现在同一序列内部，pre-RoPE key 的主导奇异向量高度一致，可以用低秩表示压掉大量维度，只保留低秩投影。

序列维度——把 key cache 按块（chunk）聚合成地标（landmark），解码时查询与地标的余弦相似度，只选出少量真正相关的 KV 对做注意力计算。

同时把 value cache 整体卸载到 CPU，解码时按需搬回 GPU。

*** 为什么这招管用（以及哪里埋了新坑）

有用：在 A100 上实现了 3 倍吞吐量提升，支持 6 倍更大的批次，在多种长上下文基准上精度接近全 KV Cache。

但 ShadowKV 的「两维」是*串行的*，而且是全局统一的策略：SVD 压缩比例固定，稀疏 token 比例固定，对所有 segment 一视同仁。

这里有一个它没有发现的深层问题：KV Cache 里不同的 segment 并不均匀。有些 segment 的 token 高度集中在少数关键位置（适合序列压缩），有些 segment 的向量在通道维度上分布平滑（适合量化）。用一个全局比例强行覆盖所有 segment，必然在某些地方过度压缩掉了有效信息，在某些地方又保留了无用冗余。这就是 MosaicKV 进攻的开口。

** 2026.07 | MosaicKV: 每个 segment 按自己的特点选压缩策略

*** 它看到了什么问题

MosaicKV 的出发点是一个实证观察：KV Cache 内部各个 segment 在序列维度和通道维度上的重要性分布是不均匀的。

以往所有方法——不管是纯序列、纯通道，还是 ShadowKV 的串行叠加——都用一套全局参数统治所有 segment。这就好像给所有人定一个统一的饮食配方，忽略了每个人的代谢差异。

另一个没被充分利用的资源：GPU 在解码阶段是 memory-bound 的（内存带宽是瓶颈，计算单元有大量空闲），CPU 也有富余内存。这些资源可以用来异步管理压缩 KV Cache，而不是白白浪费。

*** 它的解法

MosaicKV 把整个 KV Cache 拆成若干 segment，对每个 segment 独立评估：

- 序列维度重要性：哪些 token 的注意力权重集中（保留它们，淘汰其余）
- 通道维度重要性：向量的方差分布如何（方差集中则更适合量化，否则不适合）

然后为每个 segment 动态选择最优的压缩组合：序列维度压多少、通道维度压多少，两者的比例由 segment 内的数据特点决定，而非一个全局超参数。

同时利用 GPU/CPU 的闲置资源做异步压缩-解压，把额外开销藏入解码流水线的空隙里。

*** 为什么这招管用

在 H800 上：注意力计算加速 16 倍，解码延迟降低 4.8 倍，吞吐量提升 7.3 倍，内存减少 3 倍，LongBench 和 RULER 精度损失平均 1.76%。

有效，因为它找对了抽象层次：不是全局统一策略，不是两个独立策略的简单叠加，而是在 segment 粒度上做联合优化——让每个 segment 穿合脚的鞋。

* 前沿延伸

MosaicKV 发表于 2026 年 7 月，目前尚无直接引用它的后续论文。

在同一方向上，有两篇同期或稍早值得对读的工作：

*EVICPRESS*（arXiv:2512.14946，2025.12）同样尝试联合压缩和 token 淘汰，提出统一 utility 函数权衡量化质量与淘汰收益，实现 2.19 倍 TTFT 加速。它以单请求延迟为优化目标，MosaicKV 以批量吞吐为优化目标，两者角度不同，可互补。

*KVComp*（arXiv:2509.00579，2025.09）在 prefill 阶段做 2D blockwise 有损压缩，结合量化与 Huffman 编码，压缩率提升 83%。压缩发生在存储层；MosaicKV 的压缩发生在注意力计算路径上，两者可以叠加。

截至目前，MosaicKV 是这条「动态 2D per-segment 压缩」路线的最新进展。

* 洞见

读完这五层，你能看到一个在算法领域反复上演的模式：*先找新维度，在这个维度做到极致，然后发现两个维度正交，然后发现简单叠加不够，最后在更细粒度上做联合优化*。

序列维度的人做了两年，通道维度的人做了一年，然后有人把两边叠起来，最后有人说「你们叠的粒度不对，应该在 segment 级别叠」。这不是偶然——任何单维度方法都会撞上内在上限，逼你找正交方向；正交方向的粗暴组合又会遇到策略不匹配，逼你做更细粒度的联合优化。

这条线背后真正发生的认知转变是：从「把 KV Cache 当成均匀的整体来处理」到「承认 KV Cache 内部是非均匀的，要分区域差异化对待」。MosaicKV 的核心贡献不是某种具体的压缩算法，而是这个抽象层次的转变。

*下一步最可能的方向*：硬件原生 2D 表示。今天的 MosaicKV 还需要在注意力计算前解压，这个开销被藏进流水线，但没有真正消除。如果能设计让 GPU 直接在压缩格式上执行注意力矩阵乘法的专用算子，才是这条演化线的终点。

* 启发

这条线最有启发的不是它的答案，是它的提问方式。

H2O 的人没有问「怎么优化 KV Cache」，而是先去看了注意力分数的分布长什么样。KIVI 的人没有直接套现有量化方案，而是先去看了 key 和 value 向量的数字分布有什么结构。MosaicKV 的人没有直接合并两种压缩，而是先问了「KV Cache 内部各 segment 是不是均匀的」。

*观察驱动提问*，而不是方法驱动套用。

这意味着：任何你手头正在「全局统一对待」的数据结构，都值得花时间问一句「它内部是均匀的吗」。大概率不是，大概率有人没仔细看过，那就是可以下手的地方。
