* 问题之河

LLM 推理时为什么越来越慢、越来越占显存？答案藏在一个叫 *KV cache* 的东西里——每生成一个 token，模型都要把之前所有 token 的 key 和 value 存下来，免得每次都重算。上下文一长，几个 GB 的显存就被它吃掉了。

所以一群人开始想：能不能把这个 cache 压一压？比如本来每个数用 16 位浮点存，能不能压到 4 位、3 位、甚至 2 位还不丢精度？

这条路一走就是四五年，每一代都在解决上一代留下的"小辫子"：先是发现了 *outlier*（少数几个通道数值大得离谱，把整个量化区间搞坏），然后想办法把 outlier 隔离开；再后来发现"隔离"还是太粗暴，干脆用随机旋转把整个向量"打散"成各向同性；可是打散完了又冒出一个新问题——注意力分数算的是内积，光把 MSE 压低没用，内积要无偏才行。这一篇 eOptShrinkQ 站出来说：你们前面这些"补丁"都是没看见问题的根——KV 向量里本来就藏着一块共享的低秩结构，把它先剥出来，剩下的残差自然就是各向同性的，前面那些补丁就都不需要了。

这个故事真正有意思的地方是：剥结构用的工具——*最优奇异值收缩*——根本不是从 LLM 圈里长出来的，而是从一个看上去八竿子打不着的领域来的：从产妇腹部信号里抽取胎儿心电图。

* 演化叙事

** 2022 | LLM.int8(): 发现了 outlier 这个鬼

*** 它看到了什么问题

大家原本以为：把模型权重从 FP16 压到 INT8 不就是个简单的线性映射吗？Dettmers 一跑大模型就发现，事情没那么简单。Transformer 的激活值里有少量"outlier 通道"——大部分维度数值都在 ±2 以内，但有几个维度能飙到 ±60。你拿这种东西做均匀量化，量化区间被这几个 outlier 撑得巨大，剩下 99% 的"正常值"全部挤在零点附近，相当于直接归零了。

*** 它的解法

把 outlier 隔离出来。具体做法：检测哪些通道是 outlier 通道，这些通道走 FP16 原样保留；其他通道走 INT8。本质上是"承认现实——有的数就是大，别强行塞进窄区间"。

*** 为什么这招管用（以及哪里埋了新坑）

这个工作把"outlier 是 LLM 量化的核心障碍"这件事钉死了。但它的方案是"硬隔离"——你得动态决定哪些走 FP16，存储和计算都变复杂。新坑就是：能不能不隔离，而是用更聪明的量化方式让 outlier 自己变得不极端？

** 2024 | KIVI: 把 K 和 V 分开治

*** 它看到了什么问题

LLM.int8() 处理的是权重和激活。但 KV cache 是一种特殊的"激活"——它会随着 token 数累积，时间一长就主导显存。Liu 等人仔细看了一下 KV cache 的统计结构，发现一个不对称：*K 的 outlier 集中在固定通道上*（某些维度天生就大），而 *V 的 outlier 分布在 token 维度上*（某些 token 整个都大）。

如果你像 LLM.int8() 那样一刀切，K 和 V 用同一种策略，那肯定有一个会被治得难受。

*** 它的解法

按结构对症下药：K 按通道量化（per-channel），让每个通道有自己的 scale；V 按 token 量化（per-token），让每个 token 有自己的 scale。这样 outlier 都被各自的 scale "局部消化"掉了。最终配置是 K 用 2 bits，V 用 5 bits。

*** 为什么这招管用（以及哪里埋了新坑）

它揭示了 KV cache 不是同质的——key 和 value 有完全不同的统计结构。但坑也很明显：per-channel 和 per-token 都是"分组—算 scale—量化"这种启发式拼接，遇到更极端的 outlier（比如几十个通道都很大），单一 scale 还是顶不住。能不能从根本上消除"outlier 通道"这个东西？

** 2024 | KVQuant: 在量化之前先做手脚

*** 它看到了什么问题

KIVI 是"既然你大，我给你单独一个 scale"。但 Hooper 团队想得更进一步：能不能在量化之前先做点什么，让你不再"大"？

*** 它的解法

两手：第一，*预旋转*——在通道维度上乘一个固定旋转矩阵，让 outlier 通道的"能量"分散到多个通道里去，没有一个通道再特别突出。第二，*非均匀数据类型*——既然数据分布不是均匀的，那量化的码本也不该均匀，把分辨率分配到密度高的区域。

*** 为什么这招管用（以及哪里埋了新坑）

它已经在朝"治本"的方向走——通过旋转抹平 outlier。但这个旋转还是手工挑的固定变换。问题是：每个 head、每个 layer 的统计结构都不一样，一个固定旋转能压得住所有情况吗？而且非均匀数据类型增加了硬件开销。如果能找到一种统一的、理论最优的、不挑分布的处理方式就好了。

** 2025 | TurboQuant: 用随机旋转一招治百病

*** 它看到了什么问题

KVQuant 用的是数据相关的旋转，依赖你估准了分布。Zandieh 团队的想法更激进：随机旋转。

理由很数学：在高维空间里，一个随机正交旋转能"几乎确定地"把任何向量打散成各向同性——也就是说，旋转之后的每个坐标都近似服从同一个均值方差的高斯分布。一旦如此，每个坐标就可以用标准的 Lloyd-Max 量化器单独压，而 Lloyd-Max 在高斯分布下是渐近最优的。

*** 它的解法

每个 KV 向量进来 → 乘一个随机正交旋转 → 每个坐标走 Lloyd-Max 标量量化 → 存下来。解压的时候再把旋转转回去。这套流程在 MSE 意义上达到了已知方法的近最优率。

但还有一个新问题：注意力分数算的是 q 和 k 的内积。MSE 最优不代表 *内积无偏*。如果你的量化让内积平均比真实值偏大或偏小，attention pattern 就跑偏了。TurboQuant 用一种叫 *QJL* 的技巧来修正内积偏差——但这个修正要花掉每坐标一个比特。

3.5 比特/坐标，TurboQuant 在 LongBench 上追平了 FP16。这是一个很硬的 baseline。

*** 为什么这招管用（以及哪里埋了新坑）

随机旋转 + Lloyd-Max 这个组合是"理论最优"的——*前提是*你的输入向量真的是各向同性的（也就是没有方向偏好）。但 KV cache 真是这样吗？

eOptShrinkQ 的回答是：不是。

** 2026.04 | eOptShrinkQ: 你的"各向同性"是错的

*** 它看到了什么问题

Su 仔细看了一下：在同一个 attention head 里，连续 n 个 token 的 key 向量排成一个 n × d 矩阵 S~，*这个矩阵不是各向同性的*。相邻 token 共享上下文（同一段话题、同一段语境），所以它们的 key 向量之间有强烈的共享结构。

数学上，S~ 可以分解成 *S + Z*：
- *S* 是低秩部分——所有 token 共享的"上下文方向"
- *Z* 是全秩残差——每个 token 真正独有的内容

这恰好是随机矩阵理论里研究了几十年的 *spiked model*（带尖峰的随机矩阵模型）。

现在再看 TurboQuant 的假设：它假设每个 key 是从一个各向同性分布里抽出来的。但实际上每个 key 里都掺杂了一个"指向共享上下文方向"的偏置。这个偏置就是 QJL 不得不修正的内积偏差的根源——TurboQuant 是在"治症"，砸 1 bit 来修这个偏差，而不是消除产生偏差的原因。

*** 它的解法

把治症改成治本：在量化 *之前*，先用最优奇异值收缩（eOptShrink）把低秩的共享结构 S 单独剥出来存（很省，因为 rank 很小），把残差 Z 喂给 TurboQuant。

关键点：
1. 残差 Z 在数学上 *真的* 满足 thin shell 和 delocalization——也就是说它 *真的* 各向同性。TurboQuant 在这种纯净输入上才真正达到它的理论最优。
2. 不需要 outlier handling（共享结构剥掉了，剩下的没有 outlier）。
3. 不需要 QJL 内积偏差修正（残差的内积偏差从理论上就 ≈ 0）。
4. 省下来的比特全部用在重建质量上。

而剥结构用的 eOptShrink 不是普通 SVD 截断——它来自作者自己 2025 年的一篇统计学论文，原本是用来从孕妇腹部信号里抽取胎儿心电图的（妈妈的心电是"信号 S"，胎儿心电是"残差 Z"，要把 Z 抽干净）。eOptShrink 的好处是它能自动判断该截几个奇异值（靠 BBP 相变阈值），不需要手调 rank，也不需要知道噪声的协方差结构。

*** 为什么这招管用（以及哪里埋了新坑）

数字：~2.2 比特/坐标的 eOptShrinkQ 打败了 3.0 比特/坐标的 TurboQuant。多针检索任务上，2.2 比特的 eOptShrinkQ 甚至 *超过了* FP16（解释：剥掉冗余共享结构反而是一种正则化）。

新坑：
1. 需要块处理——必须凑齐 128 个 token 才能压一次。生成阶段要有 FP16 buffer。
2. 总结类任务上略有下降——因为"共享结构"对这类任务其实是有用信号，被剥掉了一些。
3. 整个流水线建立在"spiked model 适用于真实 LLM"这一假设上。论文做了实证验证，但泛化到所有架构（如非 RoPE、其他位置编码）的程度未知。

* 前沿延伸

这篇是 2026 年 4 月刚挂出来的，截至 2026 年 5 月还没有公开的后续工作直接基于它。可以预期的方向：

1. *和 token 驱逐方法组合*（SnapKV、PyramidKV）——eOptShrinkQ 压缩"每 token 占多少位"，token eviction 压缩"留多少个 token"，两者正交。
2. *从离线 BBP 阈值到在线自适应*——当前是按块（n=128）静态做 SVD，能不能在 streaming 场景下做增量 eOptShrink（类似 OjaKV 用 Oja's rule 那样）。
3. *跨 head / 跨 layer 共享低秩 basis*——xKV 已经发现层间奇异向量是对齐的，结合 eOptShrink 的最优收缩可能进一步省存储。
4. *把同样的思路搬到激活量化和权重量化*——SVDQuant 是固定 rank 的前身，eOptShrink 提供自动 rank 选择，应该能直接替换上去。

* 洞见

把整条线看完，背后真正在发生的不是"压缩率从 4-bit 一路降到 2-bit"——而是 *看待 KV cache 的视角* 在变：

1. *从"数值现象"到"几何结构"。* 早期工作（LLM.int8()、KIVI、KVQuant）把 KV cache 看成一堆数，问题是"这些数的分布有 outlier"。TurboQuant 开始把它看成几何对象——单个向量在高维球面上的位置。eOptShrinkQ 又升了一级——把它看成 *矩阵*，研究的是 n 个 token 的向量排在一起时整体的奇异值结构。视角越往上抽，能用的数学工具越深。

2. *从"对症"到"治本"再到"换问题"。* 治症（隔离 outlier）→ 治本（旋转抹平）→ 干脆改写问题（先剥结构再量化纯净残差）。每一代都在问"上一代修补的那个东西的根源是什么"。

3. *跨界移植是最大的杠杆。* eOptShrinkQ 的关键创新不是发明了新算法——eOptShrink 早就在 2025 年为胎儿心电图写好了。真正的工作是 *看出* KV cache 的数学结构和胎儿心电信号的数学结构是同一个 spiked model。这种"两个领域共享同一个数学骨架"的洞察，往往比领域内逐步优化的价值大得多。

下一步最可能的方向：把 spiked model + 最优收缩这套工具推广到 *权重量化* 和 *训练动力学分析*。如果 KV cache 是 spiked，那 weight gradient 大概率也是；如果 weight gradient 是，那 optimizer state 压缩也能用这套——这是一个潜在的"工具一旦造出来到处都能用"的局面。

* 启发

1. *找问题时，看上一代的"补丁"。* TurboQuant 用 QJL 修内积偏差，看上去是个细节优化——但补丁本身就是线索：偏差从哪来？是不是某个假设错了？eOptShrinkQ 就是死磕这一行字诞生的。每次看到"为了 X 我们又加了 Y"，问一句"为什么不直接消除 X 的源头"。

2. *跨领域 PhD 是结构性优势。* Su 是应用数学背景，做随机矩阵理论和生医信号。这两块在 LLM 圈里都偏冷门——可正因此，看 KV cache 时眼里多了一层别人没有的滤镜。如果你做某个工程领域时同时在啃一门看似无关的数学/物理，别急着切走——交叉点出现的时候，那个"看似无关"会变成你的护城河。

3. *理论保证是 paper 的卖点，不是落地的卖点。* BBP 相变保证自动定秩、内积偏差证明 ≈ 0、delocalization 保证近最优——三个"理论保证"很漂亮。但真正让审稿人和读者点头的还是后面那几张表：2.2 bit > TurboQuant 3.0 bit。理论先把读者带进来，实验把读者按在椅子上。两个都要，缺一不可。

4. *诚实标注的"局限"是一种信号。* 论文最后说"总结类任务略有下降"——这种主动认账，往往说明作者真的把局限想清楚了，而不是把它们藏起来。读 paper 时这种段落要重点看，那是真正定义"该用什么、不该用什么"的地方。
