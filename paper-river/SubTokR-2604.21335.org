* 问题之河

KV cache 太大，是大模型推理的命脉问题。一个 70B 模型跑 32K 上下文，KV cache 能吃掉几十 GB 显存——比模型权重还重。问题是：到底能扔掉什么？

这条河上每一代人都在问同一件事："哪些东西其实根本不需要存？" 但每一代人切的"块"越来越细：先是按 *头* (head) 切，再是按 *词* (token) 切，再是按 *页* (page) 切，最后切到 *词内部* 去——一个词不必整块保留或整块丢弃，可以只保留它的一半、四分之一。

这条河难在哪里？三件事缠在一起：

- *正确性*：扔错了东西，模型马上变蠢
- *效率*：决定"扔什么"本身不能太贵，否则得不偿失
- *泛化*：决定要扔什么的时候，往往还不知道用户后面要问什么

一代代人就是在这三者之间往返腾挪，把"压缩的颗粒度"一点一点磨细。

* 演化叙事

** 2017 | Vaswani et al. - Attention Is All You Need: 把循环网络扔掉，全靠注意力

*** 它看到了什么问题

在它之前，RNN/LSTM 是序列建模的标配。但 RNN 有个死结：第 100 个词必须等前 99 个词都算完才能开始。GPU 这种"一万个核并行"的硬件根本喂不饱它。

*** 它的解法

干脆扔掉时间维度的依赖。每个词都跟所有词直接算"关注度"，一次性算完。这就是注意力机制。

*** 为什么这招管用（以及哪里埋了新坑）

并行度上去了，效果还更好。但埋下一个雷：每个词都要存自己的 Key 和 Value，给后面的词查询用。这就是 *KV cache*。当上下文涨到 32K、128K，这个 cache 比模型本身还大。

后面所有压缩工作，都在拆这个雷。

** 2019 | MQA - Multi-Query Attention: 先从最粗的"头"开始砍

*** 它看到了什么问题

Transformer 里每一层有很多个"头"（比如 32 个）。每个头都要存自己的一套 K 和 V。Shazeer 一看：这 32 套 K/V 之间真的有那么大差别吗？或者说，给推理速度让点路，能不能让它们共享？

*** 它的解法

简单粗暴：所有 32 个头共用同一份 K 和 V。只有 Query 各算各的。KV cache 直接缩小 32 倍。

*** 为什么这招管用（以及哪里埋了新坑）

KV 读写带宽是推理的真正瓶颈，特别是 decode 阶段。共享 KV 等于把瓶颈拓宽。但代价是模型质量掉了——32 个头本来各看各的，现在被迫共用同一份资料。

留下的坑：能不能不那么极端？

** 2023 | GQA - Grouped-Query Attention: 头不必全共享，分组就行

*** 它看到了什么问题

MQA 太一刀切。32 个头共享 1 份 KV 损失太大。但 32 头各存各的又浪费。中间没有挡。

*** 它的解法

把 32 个头分成 4 组，每组 8 个头共享一份 KV。质量接近原版，KV cache 缩小 8 倍。

*** 为什么这招管用（以及哪里埋了新坑）

它给"共享"这件事加了一个旋钮。Llama2、Mistral 等大模型默认就是 GQA。

但"头"这层已经挤干了。再往下压，就得换轴了——不是按头切，而是按 *词* 切：能不能干脆不存那些"没用"的 token？

** 2024 | StreamingLLM - Attention Sinks: 大部分 token 其实可以扔

*** 它看到了什么问题

它做了个朴素观察：长上下文里，模型最后一层的注意力，其实绝大部分压在"最开始几个 token" + "最近几十个 token"上。中间几千个 token 收到的注意力几乎为零。

那存它们干嘛？

*** 它的解法

只保留 *最近窗口* 里的 token + 开头几个 *sink token*（即被反复关注的开头几个 token，像句号一样起锚定作用）。中间的全扔掉。

*** 为什么这招管用（以及哪里埋了新坑）

这是 *query-independent* 路线的第一次大胜——不需要知道用户要问什么，光靠位置启发式就能砍掉 90% 以上的 KV cache。流式推理变得可行。

但它太天真了——中间有些 token 其实很重要（比如某个关键名字、某个事实）。被无差别丢掉就再也找不回来。

** 2023 | H2O - Heavy-Hitter Oracle: 按"贡献度"挑词，不靠位置

*** 它看到了什么问题

StreamingLLM 的窗口策略对"叙事性"长文还行，对需要回忆具体事实的任务就不行——重要 token 不一定在窗口里。

*** 它的解法

不再按位置，而是按"历史累计注意力分数"打分。哪些 token 历史上被反复关注，就是"重锤"（heavy hitter），就保留；哪些总是被无视，就丢。

*** 为什么这招管用（以及哪里埋了新坑）

这是一次"精准化"——从位置启发式升级到 *数据驱动* 启发式。比 StreamingLLM 在 QA 类任务上稳得多。

但它仍然是 *query-independent* 的：用历史平均分数决定保留谁，无视"用户*这次*问什么"。问 A 跟问 B 用的 KV cache 是一样的——但显然不应该这样。

** 2024 | Quest - Query-Aware Sparse Retrieval: 让 query 决定取什么

*** 它看到了什么问题

H2O 选出来的"重锤" token，对今天这个 query 不一定有用。今天我问"主角的名字是谁"，那段战斗场面再"重要"也是噪声。

*** 它的解法

把 KV cache 按 *页* 分块，每页存它的 K 的统计摘要（max/min）。当 query 到来，用这个摘要快速估算"哪些页可能含相关内容"，只把那几页加载进来算 attention。

这就是 *query-aware* 路线的开端。

*** 为什么这招管用（以及哪里埋了新坑）

省下的不只是显存，还有计算——只有相关页参与 attention。长上下文推理一下子可行了。

但它有两个坑：
1. *颗粒度还是粗*：页一般 16 或 64 个 token，整页要么全要要么全不要
2. *被动响应*：必须等 query 真到了才能选；想做"先压缩、再服务多个未知 query"的场景就没法用

** 2025 | Expected Attention: 不等 query，先估计"未来 query 会怎么看"

*** 它看到了什么问题

Quest 在 query 到达后才挑页。但很多场景下，我们想 *提前* 把 KV 压缩存起来（比如 KV cache 共享、跨请求复用）——这时候根本不知道未来 query 是什么。

*** 它的解法

不存所有信息，而是估计"未来 query 的概率分布"，用这个分布算每个 token 的 *期望注意力分数*。期望分数高的 token 保留，低的丢。

这等于把"query-aware"从被动变成主动——不需要真 query 也能做出 query-aware 的决策。

*** 为什么这招管用（以及哪里埋了新坑）

这是这条河上最强的 token 级方法。本文实验里也把它作为最硬的对照。

但它还是有个固化假设：*一个 token 要么全留要么全扔*。一个被判为"中等重要"的 token，按 Expected Attention 的逻辑大概率被丢掉——但万一它里面只有 *一部分维度* 真的关键呢？

下一步该往哪走？继续在 token 这层做不动了。要么换轴：往 token 内部去。

** 2026 | Sub-Token Routing (本文): 一个 token 内部还能再切

*** 它看到了什么问题

它做了一个反直觉的观察：在固定 KV 总预算下，被保留的 token 实际上 *内部分布极不均匀*——很多 token 只保留了一部分 value 维度，很多 token 被完全丢弃，少数被完全保留（论文 Figure 1）。

也就是说，"一个 token 全部保留 vs 完全丢弃"这种二选一，从来就不是真实情况。模型自己已经在做"部分保留"了——只是过去没人显式建模这件事。

*** 它的解法

把每个 token 的 value 向量切成 S 份（比如 4 份，称为 value-group）。引入两条路径：

1. *Query-independent 路线*：训练一个路由器，决定每个 token 内部哪几个 value-group 该保留；同时配套训练一个"重建器"补全被丢的部分
2. *Query-aware 路线*：用一个 predictor 在 query-aware 模式下，把全局预算（比如总共保留 1.0 个 token 的量）分配到所有 (token, value-group) 对上——重要 token 分得多份，不重要的一份不给

它和 token 级方法 *不是替代*，而是 *互补*：先用 token 级方法（如 Expected Attention）决定哪些 token 整体上更值得关注，再用 sub-token routing 决定保留下来的每个 token 内部怎么压。

为了把同样思路放进 LoRA 微调，它把 LoRA 的低秩更新也按内部子空间切，称为 *routed subspace LoRA*——和 MoE-LoRA 等"在多个 LoRA 专家之间路由"不同，它在 *单个 LoRA 内部的子空间之间* 路由。同一个"更细颗粒度路由"哲学，同时落到适配和压缩两侧。

*** 为什么这招管用（以及哪里埋了新坑）

直觉：value 向量里不同维度承载的语义不同——有些是"这个词是动词还是名词"，有些是"它指的具体事物"，有些是"它的情感色彩"。对当前 query 来说，可能只有一类有用。强行整块保留或丢弃是浪费。

实验显示：单独用 sub-token routing 跟最强的 token 级方法 Expected Attention 持平；但 *组合起来* 在更激进的压缩比下（比如总预算砍到 25%）几乎不掉精度——这是单 token 级方法做不到的。

留下的新坑：
- *训练成本*：需要训路由器和重建器（query-independent 模式）或 predictor（query-aware 模式）
- *硬件友好度*：sub-token 粒度的稀疏对当前 GPU 内核未必友好，需要专门内核才能拿到理论速度
- *轴还能再细吗*：value 内部已经切到 4 份。再切下去（比如按维度逐个挑），收益和复杂度划不划算？

* 前沿延伸

截至本文成稿（2026 年 5 月 v2），这是这条河目前已知的最细颗粒度方案。是否有更进一步的工作（比如把同样思路推到 *key* 路径、或推到 *每个 head 内部的子向量*），暂时未见明确公开论文。

可能的几个走向（推测，*非确认*）：

- *Key 路径上的对称工作*：本文只动 value，留下 key 不变。但 key 也是 KV cache 的一半。能不能在 key 上做类似的 sub-token 路由？要难一些，因为 key 直接决定"被检索到与否"
- *硬件协同设计*：sub-token 稀疏对现有 attention 内核（FlashAttention 系列）不友好，可能催生专门的 "sub-token aware" 内核
- *与量化的结合*：sub-token routing 决定"留哪几份"，量化决定"留下来的存几比特"。两条压缩轴正交，组合空间大

* 洞见

这条河上最根本的一件事在悄悄变：*"什么算一个不可分的信息单元"这个定义本身一直在缩小*。

- 2017 注意力诞生时，一个 token 就是天然原子
- MQA/GQA 把"头"从原子降级成可拆分单位
- StreamingLLM/H2O/Quest 把"token"从必须存的单位降级成"可丢可留的单位"
- Sub-Token Routing 走到极致——*一个 token 内部本身也不再是原子*

每一次原子边界的"打破"，都对应一次新的压缩自由度被解锁。这其实是个深刻的认知转变：

我们以前默认 transformer 的内部结构（头、token、value 向量）是"语义不可分的"。但越来越多的证据表明：这些边界都是 *工程上方便定义* 的，不是 *信息论上真有意义* 的。模型内部其实没有"一个 token"这种东西——只有一团分散在多个维度上的活跃模式。

如果这个判断对，那么下一步必然是 *基于实际信息分布的动态颗粒度*——不再人工设定切几份，而是让模型自己学出"对这个 query、这个上下文，最合适的切法"。

另一件值得注意的事：*query-aware 不一定要等 query*。Expected Attention 用"未来 query 的概率分布"把 query-aware 从被动变成主动。这个思路本身就是个范式转变，sub-token routing 的 query-aware predictor 实质也走了同样的路。

未来可能的方向：不光是"哪些信息要留"，连"留下来的信息要怎么 *组织*"都让 query 分布来决定。

* 启发

对做研究和工程的启发：

- *"轴"比"算法"更值钱*：MQA → GQA 是同一根轴上的小步；GQA → StreamingLLM 是换轴。一个领域真正卡住的时候，往往是"轴"用尽了；新轴一开，整片空间打开。看本文怎么把"压缩颗粒度"这根轴往细里推一格，整条线就豁然开朗
- *被默认为原子的东西，往往不是真原子*：每次有人去捅一下"这个看起来不可分的东西真的不能分吗"，都会有新发现。下次卡住时，先列出所有"被我默认为单位"的东西，挨个问"它真的不能再切吗"
- *互补 > 替代*：本文最强的实验结果不是 sub-token routing 一打十，而是它和 Expected Attention 组合起来。新方法不一定要打败旧方法，找到一个"和旧方法正交"的位置往往更值钱
- *做诊断再做方案*：本文 Figure 1 用一个简单的统计图（被保留 token 的内部分布直方图）就把"为什么需要 sub-token routing"说清楚了。先用数据证明问题真存在，再上方案——比直接卷新方法可信十倍

落点：如果你在做 LLM 推理优化，可以立刻问三件事——
1. 我现在压缩的"单位"是什么？还能更细吗？
2. 我的决策是 query-independent 还是 query-aware？能否把后者从"等 query 来"变成"估 query 的分布"？
3. 我的方法是替代别人，还是能找到一个互补的位置？
