* 故事开始：一个内存带宽的危机（2017 → 2019）

** 2017 年的世界

Transformer 刚出来的时候，attention 长这样：每个 token 有一个 query，所有 token 有 keys 和 values。Q 跟 K 算相似度（"我该看谁"），然后用 attention 权重去聚合 V（"我从他们那里拿到什么信息"）。

为了让模型能从不同角度看信息，Vaswani 设计了多头：把 Q、K、V 各切成 h 份，每份独立算一遍，最后拼起来。这就是 *MHA (Multi-Head Attention)*。每个头有自己的一份 K、V。

训练的时候没问题。整个序列同时处理，GPU 把所有 K、V 一起算出来，矩阵乘法跑满显卡。

** 推理时的崩溃

但是用模型生成文字的时候，是一个 token 一个 token 出的。每生成新 token，都要把前面所有 token 的 K、V 翻出来跟新 query 算一遍。

这时候 Shazeer 注意到一件事：GPU 算得快得很，但每生成一个 token，都要把巨大的 K、V 张量从显存里搬到计算单元里。瓶颈不是算力，是 *内存带宽*。

就像一个厨师做菜超快，但每次都要跑去仓库取食材——做菜不慢，跑腿慢死了。

** MQA 的暴力解（2019）

Shazeer 的方案干脆利落：*所有头共用一份 K 和 V*。Query 还是多头，但 keys 和 values 只留一组。

这叫 *MQA (Multi-Query Attention)*。K、V 张量瞬间小了 h 倍，搬运成本暴跌，推理快了几倍。

代价：模型质量掉了一些。多头的好处本来就是从不同角度看信息，现在所有头看到的 K、V 都一样，相当于把一群侦探变成一个侦探——快是快，但少看到了东西。

* 中间地带的探索（2019 → 2023）

** GQA 看到了什么

四年后，Ainslie 团队接过这个问题。MQA 太极端：从 "每头一份" 直接砍到 "全模型一份"。中间能不能有妥协？

他们的方案是 *GQA (Grouped-Query Attention)*：把 query 头分成几组，每组共享一份 K、V。比如 32 个 query 头分 8 组，就有 8 份 K、V——比 MHA 的 32 份少 4 倍，比 MQA 的 1 份多 8 倍。

调一个数（组数），就能在速度和质量之间滑。LLaMA-2、Mistral 都是用 GQA。

** GQA 还没解决的事

GQA 漂亮，但有个隐藏的限制：*它必须在预训练时就决定*。你训练时定了 8 组，推理时就是 8 组。已经训好的 MHA 模型，你不能直接套上 GQA——结构对不上。

要嘛从头训，要嘛花大代价转换（"uptraining"，需要 5% 的原训练成本）。这个限制后面会反复出现。

* 把 K 和 V 一起压（2023 → 2024）

** MLA 的视角

DeepSeek-V2 的团队从另一个角度切入。前面几篇都是在 "几个头共用" 上做文章。但他们问：能不能直接 *把 K 和 V 都压成一个更小的潜变量*？

这就是 *MLA (Multi-Head Latent Attention)*。具体做法：训练一个降维矩阵把 K、V 投影到一个低维 latent c 上，cache 里只存 c。需要算 attention 时，再用另一个升维矩阵从 c 恢复出 K、V。

结果震撼：DeepSeek-V2 比 DeepSeek 67B 的 KV cache *小了 93.3%*。

** MLA 留下的两个坑

MLA 强，但和 GQA 同一个毛病：*必须在预训练时就设计进去*。所有那些已经训好的 LLaMA、Mistral，没法直接装上 MLA。要用就得重新训一遍——或者花巨大代价做转换（MHA2MLA 这种工作）。

更深的一个问题没被点破：*MLA 把 K 和 V 当一回事压*。它们一起降维，一起升维。但它们做的事真的一样吗？

* 转折点：keys 和 values 不是一回事（2026）

** 一个被所有人忽略的不对称

Yao 等人接过这条线，提出一个朴素到让人意外的观察。

Attention 里 Q、K、V 这三个东西，按惯例维度全一样（都是 d_model）。但它们做的事根本不一样：

- *Q 和 K 做的是"选择"*——算相似度排个名，决定我该看谁。这本质是个排序问题。
- *V 做的是"传递"*——选完之后把信息搬过来。这是个表示问题。

排序需要多大维度？根据 Johnson-Lindenstrauss 引理，*要在 N 个 token 中区分出谁是谁，只需要 O(log N) 维*。表示需要多大维度？至少要够装下要传递的全部信息。

也就是说，K 用的维度跟 V 一样大，*几乎全是浪费*。

** 真正的不对称还有第二层

K 必须存在 cache 里——每个历史 token 的 K 都要记住。Q 不用——它每步都重新算。

所以压缩成本如果能 *只压在 Q 这边*，cache 就完全免单。

** Factored Keys 的招

把 key 投影矩阵 W_K 做 SVD：W_K ≈ A · B，其中 A 是 d×r，B 是 r×d，r 远小于 d。

然后：
1. *新的 key 矩阵就是 A*。算 key 时用 W_K' = A，输出维度从 d 变成 r。cache 里只存 r 维的 keys。
2. *B 被吸收进 query*。新的 W_Q' = W_Q · B^T。query 维度从 d 变成 r。但 query 不进 cache，没成本。
3. *attention 分数完全不变*：q · A · B · x = q · W_K · x，等式严格成立。

换句话说，*同样的注意力计算，只是 cache 里的 K 小了 4 倍*。

** 为什么这事别人没做

前面几条路都在 "几个头共用" 或 "K 和 V 一起压" 上想。Yao 看到的关键差异是：*K 和 V 在 cache 经济学上根本不对称*。K 累积存储是负担，Q 每步重算是免费的。

而且 SVD 是数学恒等式，不需要重训、不需要校准数据、不需要改架构。任何已经训好的模型，拿出来跑一遍 SVD，cache 就小了。

可选地再做一个轻量微调（QK 部分 3 个 epoch，<1% 训练数据），就能在更激进的压缩比下保住质量。

** 实测

- GPT-2、Mistral-7B：*key cache 省 75%，质量掉约 2%*
- 7B 从头训：r = d/4 的版本 perplexity 9.24，full attention 9.25——*完全打平，参数还少 12%，训练还快 8%*
- 跟 GQA、量化叠加：*总压缩比可达 16 倍*

* 这条线背后真正在发生什么

表面上是 KV cache 越压越小，从无压缩到 16 倍压缩。但更深层的认知转变是这样的：

*第一阶段* (MHA → MQA → GQA)：把多个头看作"可以合并的副本"。问题被理解为 *冗余*。

*第二阶段* (GQA → MLA)：把 K、V 看作"一个联合的状态"，可以整体降维。问题被理解为 *联合可压缩性*。

*第三阶段* (MLA → FactoredKeys)：把 K、V 看作 *功能完全不同的两个东西*。问题不再是"怎么把它压小"，而是 *"它本来就不应该那么大"*。Key 的维度是历史遗留的设计巧合，跟它该承担的功能（排序选择）完全不匹配。

这是从 "工程优化" 到 *"重新认识 attention 的内部分工"* 的转变。一旦你看清 K 和 V 做的是两件不同的事，所有过去的对称设计都显得别扭。

* 下一步往哪走

如果这个认识是对的——*Q/K 是排序，V 是传递*——那同样的道理可能在 attention 之外别处也成立：

- *V 还能不能动*？目前 V 维度保持不变，但它本身的"必要信息量"也可以问。比如不同位置的 V 是不是有低秩结构。
- *Q 维度*？Q 不进 cache 所以省了没意义，但是不是可以让 Q 在不同层有不同维度？
- *Selection 和 Transfer 完全解耦*？现在它们还是共享底层 hidden state，但概念上可以拆成两个独立的子网络。

更激进的一步：如果 selection 真的只需要 O(log N) 维，那么 *attention 整个 K 通道是不是可以做成一种学到的哈希*——用极低维向量做近似最近邻检索，跟传统的稠密 attention 彻底分家？

* 诚实标注

- *确认的引用链*：FactoredKeys 在论文里明确把 MQA、GQA、MLA 列为对比基线。MLA 改进 GQA，GQA 改进 MQA，MQA 改进 MHA——这条链是公开且无争议的。
- *推测的部分*：MLA 是否"明确批判"GQA 必须预训练时设计，是从 DeepSeek-V2 的设计动机推断的，论文摘要本身没有点名。
- *没追的分支*：Loki、SVD-LLM、Linformer 等也是相关工作，但它们不在主演化线上（属于平行的 sparse attention 或激活压缩思路），所以没纳入主叙事。
- *后续工作*：截至 2026-05-29，没有看到明确批判 FactoredKeys 的后续论文。可能是太新，也可能这条思路还在被消化。
