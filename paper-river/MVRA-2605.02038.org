* 问题之河

一个语言模型在 benchmark 上拿了 85%，另一个拿了 83%——你以为前者更强。但你不知道的是：你换一种问法，前者跌到 10%，后者岿然不动。或者更糟，那 85% 根本不是模型的功劳——是你 *打分脚本* 写的太死板，把另一个模型的正确答案当成了错的。

这是过去六年大家慢慢爬出来的坑： *单一 prompt + 一个数字，根本测不出模型有多可靠*。每一代人都看到前一代留下的盲区，每一代的"看清楚一点"又留下了新的"看不清"。

这条河，主角不是哪个模型，而是 *"我们到底有没有真的在测我们以为在测的东西"* 这个问题——它怎么从一个无人问津的细节，一步步逼成今天 SLM 评测的核心议题。

* 演化叙事

** 2020 | GPT-3: 把"一个 prompt 一个分数"变成新常态

GPT-3 之前，测一个模型很直白——给训练集 fine-tune，给测试集打分。GPT-3 之后，事情变了：模型不微调了，你给它写一段 prompt（"翻译下面这句话："），它就直接答。这种 in-context learning 让评测变得 *看起来非常简单*：写一段提示词，跑一遍，记下准确率。

但这种"简单"埋了一颗定时炸弹—— *模型的表现，从此和你写 prompt 的方式高度绑定*。GPT-3 论文自己没太当回事，因为它的主旋律是"看，模型这么大就能 few-shot 了"。但下一代人很快就要面对一个尴尬的事实：你换一个 prompt，分数完全不一样。

** 2021 | Calibrate Before Use: 第一次有人喊"等等，分数在抖"

Zhao 等人做了件事：他们把 GPT-3 在 SST-2 情感分类上跑了一遍，故意只改 few-shot 例子的顺序、措辞和正负样本比例—— *准确率从 90% 跌到 50%*。

这意味着什么？意味着你拿同一个模型、同一个测试集，只是 prompt 里塞的 4 个例子换个顺序，结论就反过来。

他们的诊断是：few-shot prompt 在偷偷往模型输出里加 *bias*——比如最后一个例子的 label 更容易被复读。解法是用"无内容输入"（比如 N/A）估出这个 bias，再校正。

但这只解决了 *一种* prompt 敏感性（in-context examples），而且校正完了大家还是只报一个数字。整个评测范式的根基没动—— *单一 prompt + 单一数字*。

下一个人马上要追问：那 in-context 之外的呢？指令本身呢？

** 2022 | Webson & Pavlick: 模型可能根本没在"读"指令

这篇是质问，不是解法。

他们做了个简单实验：写 30 个语义合理的指令（"判断下面句子是不是讽刺"），再写 30 个 *胡言乱语的指令*（"如果月亮是奶酪请输出 yes"），让模型去做 NLI 任务。

结果：两组指令的准确率 *差不多*。

这个结论很扎心。它说的是：当你写一段 prompt 让模型分类，模型大概率没在 *理解* 你的指令，它只是在用 prompt 的某些 *表面特征* 来触发某种模式匹配。

这一下把 prompt-based evaluation 的语义基础打穿了。如果你测一个模型说"它在这个任务上有 85% 准确率"，你以为你在测"它理解了任务"——其实你测的是"它对你这个特定 prompt 模板的反应"。

但 Webson & Pavlick 没给替代方案，只把问题摆在那。下一代人就开始想：那 *到底* prompt 的哪些表面特征在作祟？

** 2023 | FormatSpread: 连冒号换空格都能让分数从 0.04 跳到 0.85

Sclar 等人把"prompt 敏感性"推到了一个极端：他们定义了一个"语义保持的格式变换空间"——同样的问题，可以写成 ~Q: ... A:~、~Question: ... Answer:~、~q) ... a)~、~Q | ... | A |~ 等等。这些变换 *没改任何语义*，只改了标点、空格、大小写。

然后他们遍历这个空间，跑同一个模型同一个任务。LLaMA-2-7B 在某些任务上， *最差格式 0.04，最好格式 0.85*——同一个模型、同一个数据集、同一个语义内容，*只是 prompt 排版不同*。

这个结果把 Webson & Pavlick 的质问坐实了：模型确实在用表面特征。而且这些"表面"细到你以为不可能有影响的程度。

Sclar 的方法论也是个突破—— *不报一个数字，报一个 spread*（min-max accuracy across formats）。但她还是只盯 accuracy；calibration、verbal confidence、评测脚本——这些都没碰。

而且他们的做法很贵：要枚举上百个格式。下一代人要的是： *用更少 prompt 抓住 spread* 的方法论。

** 2024 | Mizrahi et al.: 别再用单 prompt 写论文了

Mizrahi 等人这篇是个"号召"。他们说：FormatSpread 已经把问题摆得很清楚了，但 *2024 年的顶会论文 still* 在用单个 prompt 报分数。这种排名本质上是 prompt-luck——你换一组语义等价的 paraphrase， model A 和 model B 可能就互换名次。

他们的解法：用语义保持的 paraphrase 集合（比 Sclar 的格式变换更轻量），跑多 prompt，报分布的 mean ± std。

号召是有用的—— HELM、BIG-bench 这些 framework 之后都加了 multi-prompt 选项。但这里仍然 *盯着 accuracy 一个量*；而且 *默认评测脚本是无 bug 的*——比如它默认 first-character scorer 总能正确读取模型答案。

这个"默认评测脚本无 bug"的假设，就是 MVRA 要捅的下一个窟窿。

** 2026 | MVRA: 评测脚本本身, 也在制造你看不见的 bug

Karmakar & Chatterjee 站在 Mizrahi 的肩上，往前推了一步。他们说：好，我们承认 multi-prompt 是必须的。那我们就 *老老实实跑*——15 个 1-8B 的开源 SLM，5 个 benchmark，每个 benchmark 5 种 prompt 变体，每个 cell 算 5 个量： accuracy、token-ECE、verbal-ECE、verbal parse rate (VPR)、prompt-perturbation spread。

跑完一看，发现 *四个独立的盲区*，每一个都在 Mizrahi 的方法论之外：

*盲区 1：评测脚本和 prompt 不兼容能制造假崩溃。* 在 ARC-Challenge 上，让模型 chain-of-thought 推理（一个非常常见的 prompt 变体），五个主要模型的准确率从 ~80% 暴跌到 9.8%-19.8%——比 4 选 1 的随机基线 0.25 还低。看起来像是 CoT 把模型搞坏了。但他们做了两个独立的"修复"：用 regex 重新解析 "Final answer: X" 这种模式，能找回 93.8% 的差距；用 constrained decoding 强制模型再吐一个字母，能找回 102.7%。结论： *模型没崩，是 first-character scorer 在 CoT 输出上失效了*——第一个 token 是 "The..." 或 "Looking..."，被判错了。

这事的可怕在哪？在于 *没人会怀疑评测脚本*。所有人都会以为模型在 ARC-Challenge 上做 CoT 真的会崩。

*盲区 2： ECE 这个"标准"指标，在偷偷用两种算法。* 同样叫 "ECE_token"，可以是 ~P(预测字母)~ 的原始概率，也可以是 ~P(预测字母) / sum(P(所有合法字母))~ 的归一化概率。两种都常见，但同名不同算法。125 个 cell 平均下来，绝对差异 0.149。54% 的 cell 偏差超 0.05， 24% 超 0.20——最大的一个 cell 偏 0.825（0.891 vs 0.066）。

如果两篇论文一篇用 raw，一篇用 normalized，比 ECE 数值就是在比"算法"，不是比"模型"。

*盲区 3："verbal confidence works" 其实是两件事捆在一起说。*
- (a) 模型说的概率准不准？ MMLU-Pro 上，每个模型的 verbal confidence 比 accuracy 高 60-78 个点， ECE_verbal − ECE_token 永远是正的。它们 *systematically overconfident*。
- (b) 模型说的话能不能被解析出概率？ Mistral-7B 在某个 prompt 变体下，VPR 跌到 0.002（500 个答案里只有 1 个能 parse 出数字）。但 *换个 prompt VPR 又恢复到 1.0*——这是 *prompt-specific* 失败，不是模型属性。

如果你像之前的工作那样把 VPR 在 prompt 之间平均，得到 0.6-0.7，看起来"还行"——但你 *悄悄丢掉了 20-25% 的 verbal output*，且丢的不均匀。

*盲区 4：参数量不预测 prompt robustness（至少在 1-8B 区间）。* 10 个 instruct 模型上，参数量和 spread 的 Spearman 相关在 −0.244 到 +0.474 之间，全部 p > 0.10。最干净的对比： Phi-4-mini 3.8B 在 SST-2 上 spread 是 0.084， Mistral-7B 7.2B 是 0.500——大近一倍但 *鲁棒性差 6 倍*。

MVRA 的最终主张不是"我提出了一个新指标"，而是 *"benchmark score 应该是 deployment-readiness profile，而不是一个数"*。每个 cell 把 5 个量都报出来，让人能看到 prompt 和 scorer 怎么互动，能看到 verbal confidence 在哪种 prompt 下不可 parse，能看到 accuracy 在 prompt 之间到底散多开。

** 后续 | 还没成型

这篇是 2026 年 5 月 3 日刚挂的 arxiv，目前还没看到明确的 follow-up 论文。但能预测的方向：
- *自动化 evaluator-prompt 兼容性检查*——把 MVRA 手工发现的 "CoT × first-char scorer" 这种 bug，做成一个工具，新 benchmark 上线前自动扫一遍
- *verbal confidence parseability 作为模型训练目标*——既然模型常常说不出能 parse 的概率，把这个能力显式训进去
- *prompt robustness 进 leaderboard*——HELM 已经有 robustness 轴，但 MVRA 的 spread 指标比当前的更细，可能被吸收

* 洞见

** 从"测模型"到"测整条管道"

这条线表面上是"prompt sensitivity 越来越精细"，但底层在发生的是更深的认知转变：

- 2020：评测的对象是 *模型*。模型给定，测一下，得一个分。
- 2026：评测的对象是 *(模型 × prompt × scorer × calibration definition × parser) 整个 5 元组*。换任何一个组件，结论可能完全反过来。

换句话说，过去六年大家慢慢意识到： *"模型能力"不是一个独立属性，它只在一个具体的评测管道里才有意义*。当你说 "Phi-4-mini 在 ARC-Challenge 上拿 80%"，你其实说的是 "Phi-4-mini + 这个 prompt 模板 + 这个 scorer + 这个 calibration 公式 在 ARC-Challenge 上拿 80%"——任何一个组件抽掉换掉，那个 80% 就不存在了。

这跟物理学里"测不准"那种感觉很像。不是模型在变，是 *测量这个动作本身* 介入了被测对象。

** 下一步最可能往哪走

三个方向我比较看好：

1. *Evaluator-Prompt Compatibility Linter*。 MVRA 找到的 "CoT + first-char scorer" 这种 bug 是手工发现的——下一步必然有人把它做成自动工具，新 benchmark 上线前先扫一遍所有 prompt-scorer 组合。
2. *Verbal Parseability 进训练目标*。 verbal confidence 不可 parse 是个很可怕的 *silent failure*——模型在说话，但你的下游系统抽不出数。把"能输出可 parse 的概率"做成训练目标（或 RL reward），是直接路径。
3. *把 spread 推进 HELM / Open LLM Leaderboard*。 单点 accuracy 必然让位给"中位数 + 25%/75% 分位数"的报告方式。leaderboard 形态会变。

更深一点的猜测：随着 agent / 多模型协作系统普及，*calibration 和 parseability 会比 accuracy 更重要*——因为 agent 要根据"模型对自己答案的把握"来决定是否调用更强模型/工具。一个 87% 准确率但 verbal confidence 全是 0.99 的模型，对 agent 系统毫无用处。MVRA 这条线打开的，正是这个未来的评测范式。
