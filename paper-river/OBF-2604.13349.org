* 一句话开场

多智能体之间到底应该传什么？这条线从"传一段话"一路演化到"传整块脑袋的状态"，再演化到"只传脑袋里最有用的那一小块"。OBF 这篇论文站在这条线的最末端，说了一句反直觉的话：传得少，可能传得更好。

* 第 1 站 ~ 起点：多智能体最自然的沟通方式 —— 说人话

把多个大模型拼成一个系统这件事最早怎么做？答案最朴素：让它们互相说话。

*CAMEL* (Li 等, 2023b) 和 *MetaGPT* (Hong 等, 2024) 是这一路的代表。给每个 agent 派个角色，比如一个当产品经理一个当工程师，然后让它们用自然语言对话。一来一回，复杂任务拆解成多人协作。这一思路被反复验证有效：写代码、做数学、互相 review，多人协作的版本经常打过单人。

但这里有个看不见的坑：*把脑袋里的东西说出来这件事本身就是有损的*。

举个例子。一个 agent 在想"这道题可能是 A 也可能是 B，但我倾向 A，因为 X，不过 X 不太靠谱"。当它把这段思考压成一句话发出去，它会写成"我认为答案是 A"。后面所有的犹豫、备选、依据强弱，全部丢了。下一个 agent 接到的是一个确定的句子，而不是一团带不确定性的内心戏。

这就是后人对这一路的批判：自然语言通讯把模型的内部状态强行压成了离散 token，*把模糊的东西变确定了*。

* 第 2 站 ~ 不如直接传内部状态：embedding 派、latent 派

如果说人话有损，那就别说人话。

2024-2025 一批工作转向"传更原始的东西"：

- *CIPHER* (Pham 等, 2024)：agent 之间不传 token，传"期望输出的 embedding"。相当于把"我想说的话"留在向量空间里，不再 round-trip 到文本。
- *COCONUT* (Hao 等, 2025)：直接在 latent 空间里推理，连中间步骤都不解码出来。
- *StateDelta* (Tang 等, 2025)：除了语言，再附加 hidden state 的变化量。

这一波的共同想法：*别让信息穿过 token 这道窄门*。穿过窄门就被压扁；不穿就保得住。

但这条线还有最后一步可以走：embedding 和 hidden state 都是"半成品"的内部状态。模型内部最完整的"思考痕迹"其实是 KV cache —— 注意力机制里存下来的、覆盖每一层每一个 token 的键值对。这才是模型真正"记住"了什么的全部物理底子。

* 第 3 站 ~ 极致版：把整块 KV cache 搬过去

*LatentMAS* (Zou 等, 2025) 把这一逻辑推到底：agent A 算完之后，把它的整块 KV cache 直接交给 agent B。B 不是从头读 A 的输出，是接着 A 的"脑内状态"继续算。

这种沟通是迄今最高保真的：A 的每一层每一个 token 上的所有注意力痕迹，B 全拿到。完全没有"说出口就被压扁"这件事。

但代价立刻浮现：*KV cache 体量极大*。一个不长的 prompt，对一个中等大小的模型来说，KV cache 可能上百 MB 到上 GB。多 agent、多轮、长 prompt 一叠加，传输和存储成本爆炸。如果 LatentMAS 想真用起来，必须解决这个 KV 太大的问题。

至此，一个新问题被推到台前：*如何压缩 agent 间传输的 KV*。

* 第 4 站 ~ 并行的另一条线：单 agent 内省 KV 的所有招数

巧合的是，"KV 太大"这个问题，单 agent 长上下文那一边早就在打了，已经积累了一套工具。

第一拨：*选择性保留*。

- *StreamingLLM* (Xiao 等, 2024)：保留最开头的几个 attention sink token + 最近的一段。中间老的全扔。
- *H2O* (Zhang 等, 2023)：按"历史累计注意力权重"打分，留下分高的（heavy hitter），扔掉分低的。
- *Scissorhands* (Liu 等, 2023b)：类似思路，按命中频率淘汰。

这一拨的共同假设：*只有一小部分 token 真的被反复关注，其它的扔掉对生成质量影响不大*。

但扔掉就是扔掉，被扔的那些 token 的信息彻底没了。于是第二拨出现：*补偿被淘汰的*。

- *RazorAttention* (Tang 等, 2024)：把扔掉的 context 总结成几个"补偿 token"塞回去。
- *CaM* (Zhang 等, 2024)、*WeightedKV* (Yuan 等, 2025)：不是把被扔的丢掉，而是合并进留下来的 KV 里。

这一拨的想法：硬扔太残忍，至少让被扔的留个影。

听上去这套工具直接可以拿来给 LatentMAS 用了。但 OBF 的作者指出：*不能直接拿*。

* 第 5 站 ~ 两条线合流：OBF 看到了什么别人没看到的

OBF 这篇论文的关键洞察，其实是一句质问：

#+begin_quote
H2O、StreamingLLM 这些 KV 压缩，是为"同一个模型继续往下生成"设计的。
判断"哪个 token 重要"的标准，是它对*本模型自己*下一步注意力的贡献。
但在 LatentMAS 里，留下的 KV 是给*另一个 agent* 用的。
对自己有用的，未必对别人有用。
#+end_quote

换句话说，所有现有的 KV 压缩方法，标准都搞错了对象。

由此推出 OBF 的两个动作：

** 5.1 重新定位问题：从"压缩"变成"通信"

OBF 把这件事重新框架为通信问题：*在固定的传输预算下，留下的 KV 状态对下游 agent 最有用即可*，至于这些 KV 对发送方自己还有没有用，无所谓。

这一框架的转变本身就是贡献：在 LatentMAS 出现之前，KV 压缩不会被当成通信问题来研究。

** 5.2 具体技术：Orthogonal Backfill (OBF)

OBF 想解决的是硬淘汰带来的信息损失。直观地说：

1. 像 H2O 一样选出一批"留下的 KV"和一批"要扔的 KV"
2. 把"要扔的 KV"投影到"留下的 KV"无法表达的那个方向上 —— 也就是它们的*正交残差*
3. 对这个残差做低秩近似，提取出最主要的几个方向
4. 把这些方向当成一个修正量，加回到"留下的 KV"的 values 里

类比一下：你要打包旅行，箱子装不下所有东西。常规做法是把不重要的东西扔了。OBF 的做法是：把扔掉的东西里"留下的东西无法替代"的那一部分萃出来，缝在留下的衣服上一起带走。

*为什么是正交残差*：因为留下的 KV 已经覆盖了一部分信息方向。被扔的 KV 里和"留下的"重复的那部分，没必要再带。真正需要带走的，是"留下的"完全无法表达的那个方向 —— 也就是正交方向上的信息。

** 5.3 实验

九个 benchmark，覆盖数学推理、代码、知识 QA。

- 通信成本降低 79.8% ~ 89.4%
- 性能持平甚至超过全量 KV relay
- OBF 在 9 个里 7 个拿到最好成绩

*7 个拿最好*这件事比"持平"更重要。它说明：传得更少不仅没变差，反而变好了。

* 第 6 站 ~ 这条线背后真正在发生什么

把这五站连起来读，会发现表面是"压缩技术演化"，骨子里发生的是另一件事：

*对"沟通"这件事的本质理解，一步步从"传东西"变成"对方能用得上"*。

| 阶段        | 沟通的本质                  | 评价指标         |
|-------------+-----------------------------+------------------|
| CAMEL/MetaGPT | "我说了什么"                | 信息完整度         |
| LatentMAS   | "我把全部状态都交出去"        | 保真度          |
| OBF         | "你拿到的东西对你下一步有用"  | 下游可用性          |

最有意思的反转是 OBF 这一站给出的：*更多的信息不等于更好的沟通*。这跟人类世界的经验高度一致 —— 一个好老师不会把所有知道的都倒给学生，而是挑那些学生当下能接住的。

* 第 7 站 ~ 下一步可能往哪走

顺着这条线再往前推一两步：

1. *训练感知的压缩*。OBF 是 training-free 的。下一步自然是让下游 agent 在训练时就学会"喜欢"哪些上游 KV 信号，让压缩和接收方协同优化。
2. *按任务自适应的预算*。九个 benchmark 上同一个压缩率，但数学题和检索题对 KV 的依赖差异很大。下一步会出现按 task / per query 动态调整保留预算的方案。
3. *多跳 agent 链上的累积损失*。论文只在单跳传输上验证。如果 agent A → B → C → D，每跳都压一次，损失会累积吗？这是个尚未碰的问题。
4. *把"对方能用得上"形式化*。OBF 用正交残差是一个工程直觉，理论上"对下游最有用"的最优压缩长什么样，仍是开放问题。可能要走信息瓶颈 (Information Bottleneck) 那一路的理论。
