* 问题之河

大模型一次问答要烧多少钱？业内有个朴素直觉：把输入的 prompt 砍短，token 少了，钱自然就少了。于是过去几年涌出一票「prompt 压缩」方法——把 1000 词的 prompt 压成 50 词喂给模型，号称 20× 提效。

但这条路有个魔鬼藏在背面：你砍的是「输入」的钱，模型「输出」却可能变长。一个被压懵的模型可能本来该输出 20 个 token，结果絮絮叨叨吐出 800 个。输出 token 一般比输入贵 3 倍，于是「省钱」反而成了「烧钱」。

这条研究线的核心问题就是：*prompt 压缩到底是不是真的省钱？省的是哪一头？什么时候省、什么时候不省？*

这问题难在哪？难在它没法只看一个 benchmark 给答案——同一种压缩方法在 MBPP 上让模型炸出 56 倍的输出，在 HumanEval 上却反而让输出缩短了。看哪个数据集，决定了你的结论是「压缩万岁」还是「压缩有毒」。一代又一代研究者，都在更聪明地决定「哪些 token 该保留」，却很少有人回头问：保留得不对的时候，模型究竟会做出什么反应。

* 演化叙事

** 2023 | Selective Context: prompt 里有冗余，按信息量裁

*** 它看到了什么问题

在 Selective Context 之前，业内对 prompt 的态度是「全文照发」。一个 RAG 系统检索回 5 段文档，全塞给 LLM。一个 few-shot prompt 有 10 个示例，全塞给 LLM。算账时大家盯着 prompt 写得好不好，没人问「这 1000 个 token 是不是每个都值得花那 0.001 美元」。

但只要看一眼 prompt 就知道，里面充满了"嗯"、"那个"、"也就是说"这种垃圾词。问题是：怎么知道哪些是垃圾？人工去删？1000 个 prompt 删不动。

*** 它的解法

Li 等人想到一个朴素的招：让一个小语言模型（base LM）扫一遍 prompt，给每个词、每个短语、每句话算一个「自信息」（self-information）——本质上就是「这个 token 出现得意外不意外」。模型已经能猜到的词（"the"、"is"、"because"）保留下来没什么价值，删掉对理解几乎没损失；那些模型猜不到的词（专有名词、数字、关键动词）才是真信息，必须保留。

按这个 score 从低到高扔，扔到 50% 还剩下的，就是「精华」。在摘要、QA、对话三种任务上，他们砍掉一半 prompt，BERTScore 只降 0.023。

*** 为什么这招管用（以及哪里埋了新坑）

管用是因为它抓住了一个本质洞察——*不是所有 token 对模型理解都同等重要*。这成为整条研究线的根基。

但坑在哪？Selective Context 是「逐 token 看局部困惑度」，它不知道这个 prompt 整体被压到什么程度，也不知道 prompt 里有几个段落、每段应该分多少预算。它给每个 token 打分时是孤立的，砍多了某一段、留多了另一段，整体未必协调。

** 2023 | LLMLingua: 全局预算 + 迭代压缩

*** 它看到了什么问题

Selective Context 单兵作战，没有全局视野。比如一个 prompt 里包含「指令 + 示例 + 用户问题」三段，指令最关键、示例次之、用户问题不能动。Selective Context 不区分这些段，可能把示例压得很狠却保住了一堆「the」「is」。

更尖锐的是：当你想要 5× 甚至 20× 压缩比时，一刀切的「按 score 排序砍」会砍到关键 token。

*** 它的解法

Jiang 等人在 LLMLingua 里加了两层：
1. *Budget controller*——先按 prompt 各段的重要性分配压缩预算（指令几乎不压、示例可以压一半、问题保留），从全局视角下手。
2. *Iterative token-level compression*——在每段内部，用 GPT-2 / LLaMA-7B 算困惑度，但不是一次性算完，而是迭代地删、重算、再删，因为删一个 token 会改变其他 token 的困惑度。

他们用这套方法在 GSM8K、BBH、ShareGPT 上做到 20× 压缩，几乎不掉点。

*** 为什么这招管用（以及哪里埋了新坑）

管用，因为它从「token 视角」升到了「段落+token 双视角」。但新坑也跟着出现：

- 长上下文场景（比如 RAG 检索回 50 段文档）下，预算控制器不知道哪段和当前 query 相关。
- 困惑度计算是 unidirectional 的（基于因果 LM），看不到后文。
- 困惑度本身和「压缩」这个目标其实不直接对应——一个低困惑度的 token 不一定可以删。

** 2024 | LongLLMLingua: 让压缩「知道问题」

*** 它看到了什么问题

把 LLMLingua 扔进 RAG 系统就露馅了：检索回来的 50 段文档里只有 2 段和用户 query 相关，但 LLMLingua 不知道哪 2 段。它会均匀地压缩所有段落，结果把那 2 段关键证据连同 48 段无关内容一起压扁，模型看到的还是一堆"摘要味"的噪声。

而且长上下文还有个臭名昭著的「lost in the middle」问题——模型对中段信息几乎视而不见。压缩如果不感知位置，把关键信息留在中段等于没留。

*** 它的解法

LongLLMLingua 干了两件事：

1. *Question-aware 评分*——计算每段文档与 query 的「对比困惑度」（有 query 时段落困惑度 vs 无 query 时困惑度的差），差越大说明这段越和 query 相关。
2. *重新排序*——压缩之后把最相关的段挪到开头或结尾，避开 lost-in-the-middle 区域。

NaturalQuestions 上提升 21.4% 性能，同时 4× 减少 token。

*** 为什么这招管用（以及哪里埋了新坑）

它把「通用压缩」升级成「任务感知压缩」。但有两个根本限制没解：
- 还是靠困惑度，单向、和压缩目标不对齐。
- 每次压缩都要跑一遍 GPT-2/LLaMA 推理，慢。

** 2024 | LLMLingua-2: 把压缩当成分类问题

*** 它看到了什么问题

Pan 等人盯着一个尴尬事实：困惑度根本不是为「压缩」设计的指标。它是「这个 token 出现的意外程度」，和「这个 token 能不能删而不影响下游」是两码事。一个高困惑度的 token 可能是错别字（该删），一个低困惑度的 token 可能是关键连词（不该删）。

而且 LLMLingua 系列每次压缩都要做一次困惑度计算，慢，且因为是 causal LM，看不到双向上下文。

*** 它的解法

LLMLingua-2 的关键转身：*把压缩重新定义为一个二分类问题*——对每个 token 预测「保留」还是「丢弃」。

但训练这个分类器需要标签。怎么来？*让 GPT-4 教*。他们用 MeetingBank 对话作为语料，让 GPT-4 做"摘要压缩"，把 GPT-4 的删保决策当作 ground truth，蒸馏出一个 Transformer encoder 分类器（双向，比 causal LM 看得全）。

结果：跨域泛化更稳，比 LLMLingua 快 3–6 倍，质量更好。

*** 为什么这招管用（以及哪里埋了新坑）

管用因为它对齐了「压缩目标」和「训练信号」——你想压缩，那就直接学压缩决策，别绕道困惑度。

但新坑藏在另一个维度上：*整条压缩研究线，从 Selective Context 到 LLMLingua-2，全都在比拼"压缩比 vs 任务质量"两个指标*。没人盯着「模型被压懵之后会输出什么」。大家默认：质量没掉 → 万事大吉。

直到 Johnson 跳出来戳穿这个默认。

** 2026 | Johnson Article 4: 「压缩悖论」

*** 它看到了什么问题

Johnson 在 2026 年的 Article 4 抛出一个让全场尴尬的实测：

用最简单的 first-N-words 压缩（压缩到 30%），喂给 DeepSeek-Chat。原本输出 21 token 就能答完的题，压缩后输出膨胀到 798 token——*38× 输出爆炸*。考虑到输出 token 比输入贵约 3 倍，这次「压缩」反而把总成本推高了。

他把这叫「compression paradox」：压缩输入 → 节省失败，因为输出爆了。

*** 它的解法

他没提新算法，他换了个评测视角：*别只算输入省了多少，要算端到端总成本*。这是一个 framing 上的贡献——逼着整个领域承认「压缩比」不是最终指标。

*** 为什么这招管用（以及哪里埋了新坑）

它管用因为 framing 改变了大家的算账方式。但这个工作埋了一个大坑：他的实验有 73% 是 MBPP 类型的 prompt。当别人想复现时——比如 Johnson 自己——换到 HumanEval 上，结果直接反转：压缩后输出不仅没爆，反而缩短到 0.55×。

同一个研究者，同一种压缩方法，不同 benchmark，方向相反的结论。这就是本篇论文要解的悬案。

** 2026 | Compression Method Matters（本篇）: 元凶是 prompt 结构

*** 它看到了什么问题

Johnson 自己想复现 Article 4，复现失败。一开始他以为是统计噪声——38× 和 0.55× 实在差太远，不可能是抽样误差。

他重新看两个 benchmark 的 prompt 结构，发现一个根本差异：

- *MBPP* 是模板化的："Write a Python function to solve this task. Only provide the code. / Task: <真正的任务描述>. / Code:"——*关键任务描述在中段*。first-N-words 压缩砍到 30% 时，前面那段套话保住了，真正的任务描述和 "Code:" 提示符全没了。模型看到一段废话开头、不知道要干嘛，于是开始按模板瞎编。
- *HumanEval* 是 code-first：函数签名 + type hint 出现在第一个 token。压到 30% 函数签名还在，模型照样能正常 autocomplete。

也就是说：*之前所有人争论的"压缩到底有没有用"，其实是在争论"任务关键信息到底有没有被压掉"——但这事儿没人量化过*。

*** 它的解法

Johnson 引入两个东西：

1. *Instruction Survival Probability (Ψ)*——把 prompt 切成若干「instruction segments」（指令段），每段有重要性权重，压缩后每段要么完整保留要么完全丢失（first-N-words 下是 0/1 函数）。Ψ 就是加权幸存率。
   - MBPP 在 r=0.3 时 Ψ≈0.15（任务指令死了）
   - HumanEval 在 r=0.3 时 Ψ≈0.72（函数签名活着）
   - GSM8K 在 r=0.3 时 Ψ≈0.41（推理步骤分散）

2. *Verbose Compensation Hypothesis*——模型在 Ψ 低于某阈值 τ 时会切换到「啰嗦补偿」模式，输出长度逼近 token 上限；高于 τ 时则只有线性下降。这是一个相变假设，解释了为什么 56× 爆炸和 5× 增长是同一个机制的两种状态。

3. *Compression Robustness Index (CRI)*——一个跨 benchmark 的指标，要求评测必须报告多个 benchmark 上的鲁棒性，避免单 benchmark 误导。

他用 5400 次 API 调用、跨 3 个 benchmark、3 个模型（DeepSeek、GPT-4o-mini、Mistral-Large）实证：MBPP 上 DeepSeek 输出 56× 爆炸（Ψ≈0.15），HumanEval 上只有 5×（Ψ≈0.72）。完美对应 Ψ 假设。

*** 为什么这招管用（以及哪里埋了新坑）

管用因为它指出了一个在过去两年所有压缩论文里都被忽略的变量：*prompt 结构不是配菜，是主菜*。LLMLingua 系列拼命研究"删哪些 token"，却没人问"删之后任务指令还在不在"。

留下的新问题：
- Ψ 当前是对 first-N-words 这种粗暴压缩定义的 0/1 函数，对 LLMLingua 这类语义感知压缩怎么扩展？
- "verbose compensation" 是模型 RLHF 训练的副作用还是更普遍的现象？不同 RLHF 方法（GRPO vs PPO vs DPO）下表现不一样，但没给出机制解释。
- 既然知道 prompt 结构决定一切，能不能反过来设计 "compression-aware prompt format"？

* 前沿延伸

截至 2026 年 5 月，这是这条线最新的转折。可以预期的下一步：

1. *结构感知的压缩算法*——LLMLingua-3 或类似工作可能会引入「instruction segment 识别」作为第一步，在压缩时强制保护高权重段。
2. *Compression-aware benchmark*——既有 benchmark（MBPP、HumanEval）可能被重新设计成「压缩鲁棒」版本，把任务指令前置、加冗余 marker。
3. *Output-aware compression metric*——评测标准从「压缩比 + 准确率」扩展到「总 token 成本 + 任务质量」三元组，CRI 可能成为标配。

* 洞见

这条演化线表面看是「越来越聪明的压缩算法」——从启发式打分，到全局预算，到任务感知，到训练对齐。但拉远看，*真正的转变发生在 2026 年*：研究问题从「输入侧」跳到了「输入-输出闭环」。

前面四代工作（Selective Context → LLMLingua-2）都在回答一个隐含问题——*「假设模型输出行为不变，怎么压缩输入最聪明？」* 大家心里默认模型是个温顺的黑盒，你给它什么它就照办。

Johnson 的两篇工作戳穿了这个假设。模型不是黑盒，是个 RLHF 训出来的「情绪化乐队」——你给它残缺指令，它会用过去训练时学到的「不知道该干啥就长篇大论」的策略来填空。压缩输入会触发模型的行为切换，这才是真正的成本来源。

更深的一层：*所有评测体系都隐含一个 prompt 结构分布*。MBPP 主导和 HumanEval 主导给出方向相反的结论，意味着我们过去两年关于「LLM 压缩是否可行」的所有结论，都依赖于"用哪个 benchmark 评"这个隐藏变量。Ψ 不只是一个指标，它是对「benchmark 公正性」本身的质疑。

下一步最可能的方向不是「更聪明的压缩」，而是「输出感知 + 结构感知的联合优化」——在压缩时同时建模 Ψ（输入完整性）和 verbose compensation（输出反应），把整个推理链路当一个整体优化。

* 启发

这条线对「怎么做研究」的启发，比对压缩算法本身的启发更深：

1. *指标错了，努力全废*。前四代工作都在比拼「压缩比 vs 准确率」，没人盯着「输出长度」和「总成本」。一个被忽视的指标可能让整个领域系统性地错估了自己的进展。当你做出新方法时，先问：我用的指标是不是默认了某个不该默认的东西？

2. *复现失败是金矿*。Johnson 是因为复现自己之前的工作失败、被迫去解释这个矛盾，才挖出了 Ψ 这个概念。如果他把 0.55× 当成"统计噪声"扫到地毯下，整篇论文就不存在。学术不诚实最大的代价不是骗了别人，是错过了真正的科学问题。

3. *benchmark 不是中立的*。MBPP 和 HumanEval 看起来都在测「LLM 写 Python」，但 prompt 结构的差异让它们对压缩有完全不同的敏感性。你做任何 LLM 评测，都应该问：这个 benchmark 的 prompt 结构如何？它和我的部署场景匹配吗？

落点：这意味着你可以——下次看到一个 LLM 优化技术声称"5× 提速"或"20× 压缩"时，先问三个问题：(1) 这是在什么 benchmark 上测的？(2) 输出长度有没有变？(3) 端到端总成本是多少？这三问能筛掉一半"看起来很美"的优化方案。
