* 问题之河

一台没有显卡的电脑，能跑得动一个 70 亿参数的大模型吗？

这个问题听起来不性感，但它卡住了一代人。GPU 又贵又耗电又难买，可大模型几乎天生为 GPU 而生——它的算子、它的库、它的优化经验全在那边。把模型搬到 CPU 上，立刻撞到一堵墙：内存带宽。CPU 的 DRAM 带宽只有 200 GB/s 量级，而每生成一个 token，都要把整个模型的权重从内存里搬一遍。模型越大，搬运越慢，token 蹦得越慢。

于是十年来一群人沿着同一条线一路往下挖：能不能把权重压得更小？小一倍，搬运快一倍。从 16 位浮点压到 8 位，再压到 4 位，再压到 2 位，再压到 1 位。每压一次，质量都掉一截，于是又得想办法把质量补回来。压、补、再压、再补——这条线就是这么一节一节长出来的。

但压缩还不是终点。真正的目标更狠：不只是把权重存得小，而是让计算本身变简单。如果一个权重只可能是 {-1, 0, +1} 三个值，那"乘法"就消失了——它要么加上 x，要么减去 x，要么什么都不做。整个矩阵乘法从浮点乘法器变成了一堆条件加法。这是一个完全不同的硬件世界。

FairyFuse 就站在这条线的最末端。它说：算法那一头已经被前人解决了（Fairy2i 证明了 2-bit 三值模型质量不掉），系统这一头还没人真把"无乘法"做出来——大家嘴上说 ternary，背地里还在做乘法。它把 CPU 的 AVX-512 指令集摸了一遍，写出了一个真正零乘法的内核，1.24 倍超过 llama.cpp Q4_K_M。

* 演化叙事

** 2015 | BinaryConnect: 神经网络的权重能不能只用 +1 和 -1?

*** 它看到了什么问题

2015 年深度学习刚起飞，大家都在拼模型多大、显卡多猛。Courbariaux 反着问：一个权重为什么要占 32 位？里面的有效信息真有这么多吗？乘法器是 GPU 上最贵的电路，能不能干脆把乘法换成加减?

*** 它的解法

训练时做两本账：一本是浮点的"影子权重"，正常做反向传播更新；另一本是从影子权重 sign() 出来的二值权重 {+1, -1}，专门用来正向算 loss。推理时只留二值权重，所有的"乘以权重"都变成"加上激活"或"减去激活"。

*** 为什么这招管用（以及哪里埋了新坑）

直觉是：神经网络的鲁棒性很高，权重的"方向"比"幅值"更重要。只保留方向，居然在 MNIST、CIFAR 上能跑得跟全精度差不多。

但坑在于：作者只在小数据上验证了。ImageNet 这种"真实世界"基准上能不能站得住，没人知道。而且二值化把权重的"量级"信息全丢了——一个原本很大的权重和很小的权重，被强行拉到同一个值，这在大模型上一定会出事。

** 2016 | XNOR-Net: 给二值权重戴一顶"量级"的帽子

*** 它看到了什么问题

XNOR-Net 接过 BinaryConnect 的接力棒，把它扔到 ImageNet 上一跑——直接崩了。原因很明显：二值化把每个卷积核都压成同样的 {+1, -1} 矩阵，但不同卷积核的"输出强度"本来差很多，现在全被拍平了。

*** 它的解法

每个卷积核额外存一个浮点缩放因子 α。计算时先做 binary 卷积（用 XNOR 加 popcount，比真乘法快 58 倍），最后再乘上 α 恢复量级。激活也一起二值化。

*** 为什么这招管用（以及哪里埋了新坑）

加一个 α 其实是个聪明的妥协：保留"方向用 binary，量级用 scalar"的分工。这套思想后来一路传到所有低 bit 量化方法里——AWQ、GPTQ、SmoothQuant，本质上都是在做"per-group scale + low-bit weight"。

但二值化本身还是太狠。神经网络里其实很多权重应该是 0——它就是不重要，不该参与计算。强行把它压成 +1 或 -1，等于在每一层都注入噪音。CNN 上勉强能扛，更深的网络扛不住。

** 2016 | Ternary Weight Networks: 让权重学会"沉默"

*** 它看到了什么问题

Li 等人盯着 XNOR-Net 的失败模式想：二值化错在哪？错在它假设每个权重都"必须"贡献。如果一个权重的最优值就是接近 0，二值化只能把它扔到 +1 或 -1，制造系统性偏差。

*** 它的解法

加一个值：0。权重的取值集合从 {+1, -1} 变成 {-1, 0, +1}。每权重的存储从 1 bit 涨到 2 bit（log2(3) 上取整），但表达力一下子翻了几倍。训练时设一个阈值 Δ，绝对值小于 Δ 的权重直接归零，其余按符号取 ±1。

*** 为什么这招管用（以及哪里埋了新坑）

三值的妙处在于它复刻了"稀疏 + 符号"两件事。神经网络本来就稀疏——大部分权重接近 0；三值化把这件事显式地表达了出来。计算时遇到 0 直接跳过，连加法都省了；遇到 ±1 才做一个加或减。乘法彻底消失。

但 2016 年没人敢把这套做到 LLM 上。当时的"大模型"也就是 ResNet、VGG。Transformer 都还没出生。三值化在 CNN 上能用，是因为 CNN 的卷积核小、层浅、过参数化。LLM 几十亿参数、深百层，三值化会不会崩，没人验证过。这一等就是 8 年。

** 2024 | BitNet b1.58: 1.58 比特的大语言模型时代

*** 它看到了什么问题

8 年里 LLM 量化的主流方法一直在 4-bit、8-bit 上磨：GPTQ、AWQ、SmoothQuant，都是后训练量化，输入一个 FP16 模型，输出一个 4-bit 模型。但没人敢碰 2-bit 以下——大家默认"会崩"。BitNet 的人反问：我们怎么知道会崩？没人真试过。

*** 它的解法

不做后训练。从零训练一个权重全是 {-1, 0, +1} 的 LLM，规模拉到 3B 参数。每个权重 log2(3) ≈ 1.58 bit。训练用的是和 BitNet b1（纯二值）一样的 STE（直通估计器）反传技巧——前向用三值，反向当浮点。

*** 为什么这招管用（以及哪里埋了新坑）

惊人的发现是：从头训练的三值 LLM，质量真的能追上同规模 FP16 模型。这给整个领域释放了一个明确信号——三值不是质量天花板，2-bit 时代真的可以来。

但 BitNet 也留下了一个巨大的坑：必须从头训练。当时业界已经堆积了无数 FP16 的 Llama、Qwen、Mistral checkpoint，每个都烧掉几百万美金。BitNet 说"你想要 ternary？那就重新训。"没人愿意。

** 2025 | Fairy2i: 把 FP16 模型后训练成复数三值

*** 它看到了什么问题

Fairy2i 的人盯着 BitNet 的痛点：能不能不重训，把现成的 FP16 模型"压"成 ternary？传统后训练量化在 2-bit 上为什么崩？因为信息丢失太多，校准几下根本补不回来。

*** 它的解法

两个动作。第一，搬到复数域：权重不再取 {-1, 0, +1}，而是取 {1, -1, i, -i}。这看起来只是符号游戏——但每个复数权重的实部和虚部各自都是 ternary {-1, 0, +1}，存储还是 2-bit。妙在训练时"相位"提供了额外的优化空间，loss landscape 更平滑。第二，叫 PhaseQuant 的量化感知微调：只跑几个 epoch 的校准训练，让 FP16 权重慢慢迁移到最近的 {1, -1, i, -i}。

复数线性层不是简单的 y = Wx，而是 widely-linear：y = Ux + W·conj(x)，多了一个共轭项。展开到实数运算后，每个复数层变成 8 个独立的实数 GEMV。

*** 为什么这招管用（以及哪里埋了新坑）

广受关注的复数神经网络（Trabelsi 2018）和信号处理里的 widely-linear 表示（Picinbono 1995）证明了，复数表示在表达力上严格强于实数。Fairy2i 借这个老结论给低 bit 量化"借了一只手"——通过相位维度补回 ternary 损失的信息。

但坑在于：算法是算法，系统是系统。Fairy2i 的参考实现是 PyTorch 写的研究原型，每个 ternary 权重在运行时被先 dequantize 成 FP16，再调标准的 cuBLAS 或 MKL 做 FMA（fused multiply-add）。说好的"无乘法"，最后还是在乘。论文里 16 倍压缩的承诺，硬件上根本没兑现。

并行支线上：T-MAC 和 BitNet.cpp 试图绕过 dequantize，用 SIMD 查表（VPSHUFB）把"权重 × 激活"做成 LUT 查找。但查表本身是 L1 cache 的瓶颈，查表项占用的片上空间和激活数据抢带宽，宽 SIMD 单元（512-bit）的吞吐反而上不去。

** 2026 | FairyFuse: 真正把乘法从 CPU 上抹掉

*** 它看到了什么问题

FairyFuse 把前人留下的坑看得很清楚：

1. 算法侧：Fairy2i 已经把"权重只可能是 ±1, ±i"做到了——也就是说，每个乘法在数学上根本不需要存在。
2. 系统侧：现有的所有运行时（Fairy2i 的 PyTorch、llama.cpp 的 Q4_K_M、T-MAC 的 LUT）都在以不同方式"假装"自己利用了 ternary 结构，实际上 inner loop 里要么有 FP multiply，要么有 cache-thrashing 的查表。

*** 它的解法

四件事拼成一个内核：

1. *Bit decode（位解码）*：16 个三值权重打包进 32 位整数。用 BMI2 指令 _pext_u32 一对，单周期就能解出"哪些位是 +1"和"哪些位是 -1"两个掩码。
2. *Masked add/sub（带掩码的加减）*：拿到掩码后，直接用 AVX-512 的 vaddps / vsubps 的掩码版本——掩码为 1 的通道做累加，为 0 的通道跳过。完全没有浮点乘。
3. *8-GEMV 融合*：Fairy2i 的 widely-linear 层展开后是 8 个 GEMV。如果一个个跑，激活会被反复加载、中间结果会被反复溢出到 DRAM——带宽全废。FairyFuse 把 8 个 GEMV 塞进同一个循环：解码一次掩码 8 个 GEMV 都用（O1），激活只加载一次共享（O2），共轭通过预存的符号翻转表达（O3），所有中间累加全留在寄存器里（O4）。
4. *Roofline 论证*：FP32 GEMV 的算术强度 (AI) 是 0.25，深度内存受限；ternary 打包让 AI 飙到 8，正好踩在 CPU 的"ridge point"上——内存和算力同时打满。这在带宽富裕的 GPU 上根本看不到收益（GPU 的带宽屋顶高得多），所以 ternary 反而让 CPU 成了它的"天命平台"。

*** 为什么这招管用（以及哪里埋了新坑）

汇编级验证了 inner loop 零浮点乘法指令。Llama-7B 在单颗 Intel Xeon 8558P 上跑出 32.4 tok/s，比 llama.cpp Q4_K_M 快 1.24 倍，且 WikiText-2 困惑度只比 FP16 高 0.05（5.52 vs 5.47），下游任务平均准确率 66.0%。

埋下的新坑：FairyFuse 只跑 7B 规模，且只在 Xeon 8558P 这一种 CPU 上测过。70B 怎么办？AMD Zen 5、ARM SVE、苹果 M 系列又怎么办？widely-linear 这件事本身只适用于 Fairy2i 这种复数 ternary 模型，BitNet b1.58 那种纯实数 ternary 复用不了 8-GEMV 融合，需要单独写。再者，KV cache 的存储和注意力计算还是 FP16，整个 pipeline 还没全部"无乘法化"。

* 前沿延伸

截至 2026 年 4 月，FairyFuse 是这条线的最新进展。可以预见接下来几个方向会冒出来：

1. *硬件平台扩展*：把同样的"masked add/sub + 8-GEMV 融合"模式移植到 ARM SVE2 和 Apple Neural Engine。苹果的 M 系列 CPU 带宽更窄但 SIMD 宽，可能更受益。
2. *注意力也无乘法化*：KV cache 用 ternary 表示，注意力的 Q·K 内积也尝试用 masked add/sub 实现。这是 BitNet 团队的下一步重点。
3. *Mobile / Edge 部署*：手机 SoC 的 CPU 带宽更窄，按 Roofline 论证，ternary 在手机上的相对收益会比 Xeon 上更大。
4. *训练侧融合*：FairyFuse 只优化了推理。PhaseQuant 校准训练能不能也跑在 ternary 内核上，让"训练 + 推理"同构？

* 洞见

读完这条线，最大的发现不是某一个算法的精妙，而是整条线背后有一个反复出现的模式——

*算法和系统之间永远有一个 gap，谁把 gap 关上谁就吃下整代红利。*

每隔几年这个故事就会重演一次：

- BinaryConnect 论文里说"乘法可以变成加法"——但当时的 GPU kernel 没有人为它写。
- XNOR-Net 论文里说"我们的卷积比传统快 58 倍"——这是个理论数字，真正的硬件加速器晚了好几年才出现。
- BitNet b1.58 说"1.58 bit LLM 质量能追上 FP16"——但运行时 bitnet.cpp 用的是 LUT，离硬件理论极限差很远。
- Fairy2i 说"复数 ternary 后训练就能拿到 2-bit 质量"——但 PyTorch 路径里全是 FP16 dequant。
- FairyFuse 终于在 CPU 上把承诺兑现了——但 GPU 上、ARM 上、手机上还没有。

每一篇"算法论文"都在画饼，每一篇"系统论文"都在啃饼。算法侧总是跑在前面，先证明"理论上可行"；系统侧总是慢半拍，把"理论可行"翻译成"工程可用"。两者之间的时间差，从几个月到几年不等——这个时间差，就是你的窗口。

第二个发现更深一层：*硬件的瓶颈决定了哪个量化方案能赢。*

为什么 2024 年才轮到 ternary LLM 大火？不是因为算法 2024 年才发明（2016 年就有了 TWN）。是因为 LLM 推理是 memory-bound 的，而 ternary 是直接砍内存的工具。Roofline 模型里，ternary 把算术强度从 0.25 推到 8，正好踩在 CPU 的 ridge point 上——但在 GPU 上几乎没有收益。

这暗示一个反直觉的判断：*在 GPU 主导的时代，ternary 不是主流；在 CPU 边缘部署崛起的时代，ternary 才显出价值。* 这一波 LLM 下放到端侧的浪潮，会把 BitNet/Fairy2i 这种 ternary 路线推到从未有过的中心位置。

第三个发现是研究路线的循环结构：*极端 -> 妥协 -> 在妥协里再次极端。*

1 bit (BinaryConnect) -> 1+scale (XNOR) -> 1.58 bit (TWN) -> 1.58 LLM (BitNet) -> 2 bit 复数 (Fairy2i) -> CPU 极致内核 (FairyFuse)。

每一代都不是"更激进"，而是"在前一代妥协的地方再激进一次"。BinaryConnect 太激进，被 XNOR 加 scale 救回来；XNOR 还是太激进，被 TWN 加 0 值救回来；TWN 没在 LLM 上验证，被 BitNet 救回来；BitNet 要从头训，被 Fairy2i 用复数救回来；Fairy2i 没真实现，被 FairyFuse 救回来。

这暗示下一篇会是谁。下一篇大概率不是"更激进的位数"（已经到 ternary 极限了），而是"在 FairyFuse 留下的妥协处再激进一次"——比如把它的 CPU 优化推广到 ARM 和手机端，或者把注意力机制也无乘法化。

* 启发

这条线对"怎么做研究"有三件具体启发：

1. *算法和系统的 gap 是富矿*。每次看到一篇算法论文里写"理论上可以快 N 倍"——去看它的代码。多半运行时根本没拿到那个 N 倍。把它真的拿到，就是一篇好的系统论文。FairyFuse 就是这么写出来的：Fairy2i 论文摆在那里两年了，没人去填那个坑。

2. *硬件特性决定算法走向，反之亦然*。研究低 bit 量化，先去画 Roofline。Roofline 告诉你哪种压缩方案在哪种硬件上能赢、能赢多少。FairyFuse 的核心论证不是算法多妙，而是 "ternary 压缩的收益曲线在 CPU 上正好越过 ridge point"——这个洞察值整篇论文。

3. *沉寂期是机会期*。TWN 在 2016 年就发明了，但在 LLM 上沉寂 8 年。这 8 年里 LLM 起飞了、CPU 边缘部署变重要了、内存带宽成了主瓶颈——所有的外部条件都在累积。一个老方法在新场景下的复兴往往比新方法更高效，因为老方法已经被验证过，缺的只是新场景的对齐。FairyFuse 也是同样故事：BMI2 (_pext_u32) 是 2013 年的指令，AVX-512 是 2016 年的指令，掩码加减是 2018 年就稳定的东西——它们都不新，只是没人把它们和 ternary LLM 这件 2024 年才成熟的事拧到一起。

落点在"能用"：这意味着你可以——

- 拿任何一篇你读到的算法论文，先去看它的 GitHub 仓库。如果仓库里全是 PyTorch baseline，没有硬件级的内核，那就是你的机会。
- 拿任何一个你熟悉的硬件平台（手机 NPU、苹果 M 芯片、AMD Zen），画一个 Roofline 图，看看 ternary、INT4、INT8 各自的 AI 落在哪里。Ridge point 的位置告诉你哪个 bit 宽在这个平台上最值得做。
- 不要盯着"最新的"算法 chase。盯着 5-8 年前的"被遗忘的好想法"，看哪些在今天的新场景下重新值钱了——这条路的命中率比追新论文高得多。
