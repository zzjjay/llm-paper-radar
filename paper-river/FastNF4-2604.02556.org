* 问题之河

一个简单的问题：你把模型压成 4 位整数省了显存，那 GPU 跑的时候到底快不快？

答案出乎意料地难。GPU 的算力都长在 16 位浮点的 Tensor Core 上，不认识 4 位。所以每次乘矩阵，你都得先把 4 位"解压"回 16 位。十年来量化研究的暗线，就是这场永远的猫鼠游戏——压缩一边省了显存，解压一边又把时间还回去。这条河的每一代人，都在找"省下的钱"到底从哪个洞漏掉的。

到了 2026 年，大模型已经塞不进单卡，4 位量化几乎是部署的默认。NF4（NormalFloat-4）配合 BitsAndBytes 库成了 HuggingFace 生态的事实标准。但当 Qwen3-32B 在 A100 上推理时，光"解压"这一步就吃掉端到端 30–40% 的延迟。本篇论文要做的事极其朴素——别动算法，别动精度，就把那张 16 项的查找表搬进 shared memory，问题立刻少掉一半。

这条河有趣的地方在于：每一代解决方案都被下一代证明"还有更便宜的办法"。本篇是目前为止最便宜的一招。

* 演化叙事

** 2018 | Jacob et al.: 让 INT8 推理真正落地

*** 它看到了什么问题

2018 年之前，神经网络一直是 FP32 训练 FP32 推理。手机和嵌入式设备根本跑不动。已经有人尝试过把权重压到 8 位整数，但效果很差——精度掉一大截，或者只能在玩具模型上有效。问题在于：训练时模型见到的是连续的浮点世界，推理时把它扔进离散的整数格子里，模型从未为这个失真做过准备。

*** 它的解法

两件事捆在一起做。第一，定义一个干净的"仿射映射"——浮点 = (整数 - 零点) × scale，把整个推理图重写成纯 INT8 算术。第二，提出 *quantization-aware training (QAT)*：训练时就把这个失真模拟进去，让模型主动学会在 INT8 下也活得很好。

可以这样想：以前是把一个不会游泳的人推进泳池；他们的办法是先在浴缸里教会他游泳，再推下去。

*** 为什么这招管用（以及哪里埋了新坑）

这套方案让 MobileNet 这类 CNN 真正能在手机上 INT8 推理，并且精度只掉不到 1%。它定义了"量化"在工业界的最小可用样貌——integer-only inference + QAT。

但它的世界只到 CNN 为止。坑在三处：(1) 它假设你能重新训练，对动辄上千 GPU 天的 LLM 不现实；(2) 量化粒度是 per-tensor 或 per-channel，对分布极不均匀的权重处理不好；(3) 它没遇到 Transformer 里那种"几个维度爆炸大、其他维度很小"的激活分布。

下一代人首先要解决的，就是这第三个坑——LLM 来了。

** 2022-Aug | LLM.int8(): 离群激活逼出混合精度

*** 它看到了什么问题

2022 年，OPT-175B 这类百亿级 LLM 出来了，Dettmers 一帮人直接把 Jacob 那套 INT8 量化扔上去，结果模型精度直接崩盘。诡异的是，模型越大崩得越狠——这和 CNN 时代的经验完全相反。

他们花了几周时间显微镜级地分析，发现了一件反直觉的事：当模型尺寸跨过约 6.7B 参数时，Transformer 的激活值里会突然冒出一小撮"离群特征"——某几个隐藏维度的值能比其他维度大 20–100 倍。这些离群维度只占 0.1%–1%，但承担了模型大部分的有效计算。把它们和普通维度一起塞进 INT8 的小格子，全部细节就被磨平了。

*** 它的解法

很务实：既然只有 1% 的列出问题，就 *混合精度* ——99% 的列照常走 INT8 矩阵乘，剩下 1% 的离群列单独拿出来用 FP16 算，最后两部分加起来。

类比：超市收银。99% 的顾客走自助快速通道（INT8，便宜快），剩下买茅台的 VIP 大客户走人工柜台（FP16，慢但准确）。

*** 为什么这招管用（以及哪里埋了新坑）

它第一次让百亿级 LLM 能在消费级硬件上跑起来，而且精度几乎无损。HuggingFace 直接把它集成进 `transformers`，BitsAndBytes 这个名字开始进入主流视野。

但它有两个明显的坑。第一，*8 位还是太大*——OPT-175B 量化到 INT8 还是要 175GB 显存，单卡装不下。第二，*离群分离逻辑本身很贵*：每次前向都要扫一遍激活找离群列，再做两次矩阵乘，再合并。它的设计目标是"能跑"，不是"跑得快"。

要进一步省显存，必须打到 4 位以下。但 4 位的"格子"小到只剩 16 个——离群分离这招不够用了。

** 2022-Oct | GPTQ: 二阶信息让 4 位也能保精度

*** 它看到了什么问题

LLM.int8() 教会大家：直接量化会出问题，是因为 *分布尾部* 那些大值。Dettmers 的做法是把它们绕过去（用 FP16 算）。但 Frantar 这帮人在 IST Austria 想得更狠——你能不能 *把尾部也量化* ，只是更聪明地选量化点？

直接四舍五入到 4 位的 16 个格子里，离群权重一定被压扁，精度必崩。但如果你知道"哪些权重对最终输出影响最大"，就可以让校准过程偏向保护它们。这个信息藏在二阶导数（Hessian）里——它告诉你每个权重的微小扰动会让 loss 变化多少。

*** 它的解法

把 1990 年代的 Optimal Brain Quantization 拿出来——基于 Hessian 的逐权重量化——再做工程加速。核心步骤：对每一层，按列处理；每量化一列，就用 Hessian 的逆把"量化误差"重新分配到剩下未量化的列上，让它们去补偿。

可以这样想：分蛋糕。如果你必须把蛋糕切成 16 份不均等的块，普通做法是机械等距切。GPTQ 的做法是——先切一刀，看看吃这一块的人有什么反应；如果他不满意，就把剩下的蛋糕切得偏向他的口味来补偿。一路切下来，整体满意度最高。

工程上把这套校准做到 OPT-175B 4 小时跑完，4 位精度几乎不输 FP16。

*** 为什么这招管用（以及哪里埋了新坑）

GPTQ 第一次证明 *4 位 LLM 也可以无损* ，没有 LLM.int8() 那种"必须留 1% FP16"的混合精度负担。整张权重矩阵都是 INT4，存储省到极致。

但它埋了两个坑：(1) 校准依赖一小批校准数据，跨数据集泛化偶尔翻车；(2) 它只管"怎么把 4 位算出来"，没管"4 位怎么算"。推理时还得 dequant 回 FP16，因为 Tensor Core 不认 INT4。这个 dequant 怎么搞，下一代人来填。

** 2023-May | QLoRA / NF4: 让 4 位变得"自然"

*** 它看到了什么问题

GPTQ 用统一的 INT4 网格——16 个等距格子——来量化。但 Dettmers 注意到一件事：训练好的 LLM 权重并不是均匀分布的，而是 *接近标准正态分布* 。如果你用等距格子去量化一个正态分布，中间堆积的大部分权重共享很少几个格子，两端稀疏的权重却各占一个格子。这是浪费。

更狠的是 GPTQ 那套校准。它需要校准数据集、需要 4 小时跑、需要懂 Hessian。对一个想 fine-tune 自己模型的研究生来说，门槛太高。

*** 它的解法

两条腿走。第一，*NormalFloat-4 (NF4)*：不用等距格子，直接按标准正态分布的 16 个等概率分位数挑量化点——格子在中间密、两边疏。这样每个格子里落的权重数量大致相等，信息熵接近最大。无需校准，信息论上最优。

第二，把这套东西打包成 *BitsAndBytes* 库 + *QLoRA* 训练框架：基础模型用 NF4 4 位存着，LoRA 适配器用 16 位训。65B 模型从 780GB 显存压到 48GB，单张消费级 GPU 就能 fine-tune。

类比：以前是用尺子等距画刻度（INT4），现在是看人群分布画刻度——人多的地方刻度密，人少的地方刻度疏（NF4）。地铁站台标志位都画在人多的地方，效率自然高。

*** 为什么这招管用（以及哪里埋了新坑）

NF4 让 4 位量化彻底民主化。无需校准、无需懂二阶导、HuggingFace 一行 `load_in_4bit=True` 就能用。它成了 2023–2026 年间事实上的标准。

但它留下了一个具体得让人头疼的坑——dequant kernel 写得很烂。

具体烂在哪？(1) 把 4 位还原到 FP16 需要查 NF4 的 16 项表。BitsAndBytes 的实现把这个表存在 *全局内存* ，每个线程的每次 dequant 都要重新从全局内存读。Ampere 上全局内存访问要 290 cycle。(2) 索引逻辑用了一个 *4 层条件树* （先看高位 0 还是 1，再看次高位…），每个权重要 7 条带分支的指令。GPU 的 32 个线程在一个 warp 里走不同分支会发生 *warp divergence* ——本来可以并行的 32 个线程被串行化。

NF4 算法本身没问题。问题是它的 dequant kernel 还停留在"能跑"的状态。这正是本篇论文要解决的事。

** 2026-Apr | FastNF4: 64 字节的 shared memory 改写一切

*** 它看到了什么问题

QLoRA/BitsAndBytes 火了三年，几乎所有用 HuggingFace 部署 4 位 LLM 的人都在用同一份 dequant kernel。Qi 这帮人在 USC 跑 profiler 看：Qwen3-32B 推理时间里，dequant 占 30–40%。72.4% 的 matmul 时间花在 dequant 上，真正的乘法只占小头。这个 kernel 是 *彻底的 memory-bound* ——GPU 算力等着内存。

更刺眼的是，NF4 的查找表只有 16 个值，FP32 存储一共才 64 字节。Ampere 每个 SM 配的 shared memory 有 100KB 量级。这张 64 字节的表硬是被放在 290 cycle 的全局内存里，每个 thread 重复读，循环上亿次。

*** 它的解法

朴素到让人发笑：把那 64 字节搬进 shared memory。

具体做法只有几步：(1) 每个 thread block 启动时，让 thread 0 单线程把 16 项表拷到 shared memory，64 字节顺序拷贝，几乎瞬时完成。(2) 一道 `__syncthreads()` 屏障，等所有线程看到完整的表。(3) 之后整个 block 的所有 dequant 操作都用 shared memory 查表——shared memory 一次访问 19–23 cycle，比全局内存的 290 cycle 快 12–15 倍。

第二刀同样朴素：把 4 层条件树砍成 2 条位运算指令。原本是"先比较高位、再比较次高位…"7 条带分支指令；现在直接 `>> 4` 拿高 nibble，`& 0x0F` 拿低 nibble，2 条指令、零分支。Warp 里 32 个线程走完全相同的代码路径，divergence 消失。

类比：原本图书馆每个读者借书都要去仓库取一遍（全局内存）；新方法是开门时让管理员把这 16 本热门书摆到前台（shared memory），之后所有读者直接在前台拿。每天上百万次借阅，差距是 12–15 倍。

*** 为什么这招管用（以及哪里埋了新坑）

效果：kernel 级 2.0–2.2x 加速，端到端 1.07–1.54x 加速，模型越大收益越大（Llama3.3-70B 在 batch 2 时 1.54x）。代价：64 字节 shared memory，零算法改动，零精度损失，逐 token 输出与原版 bit-exact。

更妙的是 *零侵入性* ——它就是直接替换 BitsAndBytes 里 `kDequantizeBlockwise` 一个函数，HuggingFace Transformers 那边一行代码都不用改。

但它依然留下了坑：(1) 这只是 *dequant 单步* 的优化。理想方案是 *融合 dequant + matmul* ，让 4 位直接喂进 Tensor Core 等价的计算，根本不产生 FP16 中间结果。(2) 它专攻 Ampere；Hopper / Blackwell 已经支持原生 FP4/INT4 Tensor Core，整个 dequant 步骤可能根本不需要。这套方案的命也就 1–2 代 GPU。

* 前沿延伸

** Marlin / Flute / QuTLASS: 干脆把 dequant 融进 matmul

FastNF4 的思路是"独立优化 dequant"。同一时期还有另一派——以 Marlin (Frantar 2024) 为代表——主张 dequant 不该独立存在。

他们的做法：写一个 *融合的 INT4×FP16 GEMM kernel* ，dequant 发生在 Tensor Core 寄存器层、和乘加重叠执行，根本不产生独立的 FP16 中间矩阵。Marlin 在 batch 较小时能达到接近 FP16 的算术强度，端到端 2–3x 加速。

差异：Marlin 一派性能上限更高，但代价是 *必须改算子接口* ——不再是"先 dequant，再走 cuBLAS GEMM"，而是一个新的 fused op。HuggingFace 集成它需要新的 quantized linear API。

FastNF4 选了另一条路：宁可慢一点，也要保持 BitsAndBytes 的即插即用。这是工程权衡，不是技术败北。两条路都活着，给不同用户。

** Hopper / Blackwell 原生支持 FP4

H100 起，Tensor Core 开始原生支持 FP8；B100 进一步原生支持 FP4。理论上 dequant 这个步骤未来会消失——4 位数据直接喂进 Tensor Core 算。

但 A100 在数据中心和大学集群里至少还要再服役 5 年。FastNF4 这类"老硬件极限优化"的窗口还很长。

* 洞见

读完这条河，看到几件事，越往后越清晰：

*第一，量化研究的真正主线不是"压缩率"，而是"压缩率 ÷ 解压代价"。* 从 2018 到 2026，每一代论文表面上在卷"更低的位数"——8 位、4 位、3 位、2 位——但暗线是"省下来的显存 vs 还回去的计算"。LLM.int8() 用 99/1 混合精度还回去了一部分；GPTQ 用校准时间还回去；QLoRA 用 dequant 时间还回去；FastNF4 终于开始 *讨回*  这部分时间。

*第二，"看似简单的工程优化"往往因为太基础而被晾了多年。* NF4 的查找表只有 64 字节，搬进 shared memory 在 CUDA 优化教材里是第一章的内容。但 BitsAndBytes 这个被几百万人用的库，硬是带着这个 bug 跑了三年。为什么？因为算法人写库的时候追求"能跑、对、可读"，性能优化的最后一公里需要系统人下场。这条河的有趣之处在于，*越成熟的领域，越容易出现"显而易见但没人做"的机会* 。

*第三，"零侵入性"是一个被低估的工程美德。* FastNF4 没改算法、没改 API、没改任何用户代码，只是一个函数级替换。这意味着所有用 BitsAndBytes 的 HuggingFace 用户明天 `pip install --upgrade` 就能拿到 1.5x 加速。同期 Marlin 性能更高但需要重写算子接口——它的传播会慢得多。在已经形成事实标准的生态里，"小改大用"比"全新方案"更值钱。

*第四，硬件代际更替会把整条研究线腰斩。* B100 原生支持 FP4，意味着 FastNF4 优化的整个目标——"消除 dequant 开销"——在新硬件上根本不存在。但这并不否定本篇的价值，反而印证了系统研究的本质：你优化的不是永恒的问题，而是"当前硬件代际下的瓶颈"。每一代硬件都会重新洗牌"哪里是 bottleneck"。研究者要做的是 *快速识别当代瓶颈、用最小代价吃掉它，然后准备好被下一代硬件淘汰* 。

*下一步会怎么走？* 短期内（A100/H100 主导的未来 2–3 年），dequant + matmul 融合路线（Marlin 系）会持续蚕食 BitsAndBytes 的份额，除非 BitsAndBytes 自己也走融合路线。中期，随着 B100/MI400 普及，"软件 dequant 优化"作为一个研究方向会逐步消失，转移到 *训练-时-量化感知* 和 *跨设备一致性* 这类更高层的问题。FastNF4 是这条河流的"最后一公里"工作之一。

* 启发

*对做研究* ：去找那些"被生态绑死的烂实现"——成熟领域里跑了几年没人动的关键函数，往往是 1.5x–3x 加速的金矿。它们没人动不是因为难，是因为算法人没动力、系统人不知道。这种工作发不了顶会但极其有用，且通常很快被几百万人用上。

*对工程权衡* ：在已经形成事实标准的生态里（HuggingFace、PyTorch、BitsAndBytes），"小改大用"的杠杆远大于"另起炉灶"。同样 1.5x 加速，零 API 改动比新 API 的传播速度快一个数量级。

*对硬件代际* ：你优化的瓶颈是当代硬件造的，不是永恒真理。在动手前先想：这个优化在下一代硬件上还有意义吗？如果答案是"两年后就过时"，要么压缩研发周期赶在窗口期发出去，要么把这工作的真正价值定位在 *理论洞察* 而非 *性能数字* 。

*对 4 位 LLM 部署* ：如果你正在用 BitsAndBytes 跑 4-bit 推理，盯着这个 PR 合入主干（或自己 cherry-pick 到 BnB 0.47.0+），可以白嫖端到端 1.07–1.54x 加速。模型越大、batch 越小、A100 越多，收益越明显。如果你已经在用 Marlin 或 AWQ kernel，本工作对你没用。
