* 问题之河

代码大模型现在大到什么程度？GPT-3 175B 要 325 GB 显存、五张 A100 才能装下；CodeGen 16B 也要 72 GB。训练一次 GPT-3 排放的 CO2 比一架 SF→NY 喷气机还多 3 倍。而且大头不在训练—— *推理* 占了部署期 80~90% 的总能耗，Google 自己的算下来推理吃了 ML 总能耗的 60%。

于是"把这些大模型压小"成了 SE/NLP 这几年最热闹的赛道。压缩技术其实都成熟了——蒸馏、剪枝、量化、NAS——一个个都有十年家底。但你真去看现状：

- 几乎所有 SE 压缩工作都 *只针对 encoder-only 模型* （BERT 那一支），最多做做 clone detection；
- 一篇论文 *只用一种技术* ——要么蒸馏，要么剪枝，要么量化；
- *没有人系统地问过：当你把多种技术拼起来时，先后顺序重不重要？*

CTT 的入口就在这里：它不发明新的压缩算法， *它问的是"压缩管道本身该怎么排"* ——并把这条问题接上一个更大的故事：能不能像经济学的"碳税"那样，对模型架构的浪费罚款、对部署友好的形态奖励？

这条研究线的脊柱，倒回去看其实是一群人在同一个问题上一步一步把范围撑开： *从一种模型到三种、从一种技术到四种、从只看准确率到看准确率+延迟+内存+CO2* 。

* 演化叙事

** 2019 | DistilBERT: 蒸馏第一次成为"压缩主路"

*** 它看到了什么问题

2019 年的 BERT-base 已经是 110M 参数、420MB；BERT-large 是 340M、1.3GB。在云端跑 demo 当然没问题，但所有 *on-device、IDE 插件、移动端* 的场景都被堵死了。当时主流的"压缩"思路是剪枝和量化，但都偏工程：剪掉的连接是不是真的没用？量化到 INT8 之后掉精度怎么办？没人有干净的答案。

更深的问题是：BERT 这么大，是因为它真的需要那么多参数，还是它在训练目标（MLM + NSP）上有冗余？没人系统答过。

*** 它的解法

DistilBERT 的人提出：既然 BERT 在很多任务上明显"游刃有余"，那让一个小学生模型去 *模仿* 它的输出分布——不只是模仿 hard label，而是模仿 teacher 的整个 softmax 分布（soft labels）。这就是 Hinton 2015 年提出的 KD，但 DistilBERT 是第一个把它当作"压缩 BERT 的主路"。

实操上：去掉 BERT 一半的 Transformer 层，再用三个损失训练—— MLM 损失（保留语言能力）+ KD 损失（模仿 teacher）+ cosine 损失（让学生的 hidden state 和 teacher 对齐）。结果：模型缩到 60%，速度快 60%，下游任务性能保留 97%。

*** 这招管用为什么，又埋下了什么坑

它管用的根本理由是—— *soft label 里携带的信息比 hard label 多得多* 。teacher 说"这句话 65% 像积极、30% 像中性、5% 像消极"，学生学到的就是整个语义几何，不只是一个标签。

但 DistilBERT 留了两个大坑：

*第一* ，它是 *通用 NLP 模型* ，不是代码模型。代码的 token 分布、语法、上下文结构和自然语言差别很大，你能不能照搬这一套，没人测过。

*第二* ，它就一个技术——KD。没和剪枝、量化、NAS 联动。剪枝去掉的是参数，量化压的是精度，NAS 改的是架构， *四个杠杆其实在拉同一辆车，但没人合起来推过* 。

** 2022 | Compressor: 把代码模型压到 3MB

*** 它看到了什么问题

到 2022 年，CodeBERT、GraphCodeBERT 这些代码大模型已经是 SE 标配，但它们和 BERT 一样大——CodeBERT 是 500MB。开发者要把 *vulnerability detection* 这种功能塞进 IDE 插件、CI 节点，500MB 完全不可行。

Compressor 的作者 Jieke Shi 团队跑去试 DistilBERT 那一套，发现两个问题： *第一* ，学生模型结构怎么选？DistilBERT 是手工选的"减半层数"，对代码任务不一定最优； *第二* ，KD 单独用 *能压到 30MB 已经是极限* ，离插件场景的 1~3MB 还差一个量级。

*** 它的解法

Compressor 把 DistilBERT 的 KD 套子扩了两步：

1. *用遗传算法搜索学生架构* ——别手工定层数和宽度，让进化搜索去找；
2. *搜索的目标函数是 "模型大小 + 任务损失" 的多目标 Pareto 前沿* ——你要 3MB 的也能拿到，要 10MB 性能更好的也能拿到。

结果：CodeBERT 从 500MB 压到 3MB（160x 压缩），在 vul detection、clone detection 等二分类任务上掉点很有限。

*** 这招管用为什么，又埋下了什么坑

它的真正贡献不是技术新——遗传搜索 + KD 都是现成的——而是 *第一次把 "代码模型可以压到 3MB" 这件事钉死了* 。后面所有 SE 压缩工作都拿这个 3MB 当地板。

但留下的坑相当明显：

- *任务太窄* ：只在二分类任务上验证（vul detection、clone detection），代码生成、摘要这些 seq2seq/autoregressive 的任务根本没碰；
- *架构太窄* ：只压 encoder-only 模型，BART/T5 这种 encoder-decoder、GPT 这种 decoder-only 完全没动；
- *没量过能耗* ：模型小了，但小模型推理一次到底省了多少电、少排了多少 CO2，没人算；
- *遗传搜索慢* ：跑一次完整搜索要好几天 GPU，工程友好性差。

** 2024 | Avatar: 第一次把"碳排放"摆上桌

*** 它看到了什么问题

同一拨人（Jieke Shi 团队）2024 年回来了。他们看到 Compressor 留下的两个最痛的坑：搜索慢、能耗没量。

更关键的认知转变是—— *研究界开始集体意识到 "模型小" 和 "模型绿" 不是一回事* 。一个小模型如果架构选得烂，跑起来反而可能比大模型更费电（cache miss、kernel 调度差等）。所以要把 *延迟、内存、CO2* 直接做成优化目标，而不是只看"参数量"。

*** 它的解法

Avatar 干了三件事：

1. *把遗传搜索换成 SMT (满足性模理论) 引导的搜索* ——这是个可解释的精确搜索器，能用约束（"延迟不超过 X、内存不超过 Y"）直接圈出可行域，不用瞎跑；
2. *把 KD 和 quantization 串起来* ：先 KD 得到一个小学生，再 INT8 量化把它再压一截；
3. *第一次把 CO2 emission 当成报告指标* ：用 CodeCarbon 库测了推理碳排，让 "greener language models for code" 这个口号有了数字。

结果在 clone detection 上：模型压到 3MB 的同时， *推理时 CO2 比未压缩模型低一个量级* 。

*** 这招管用为什么，又埋下了什么坑

Avatar 是第一次让 *"绿"* 这个词在 SE 压缩里有了具体含义—— 不是抽象的环保口号，而是 grams CO2 per inference 这种可测的数字。它也是第一次说 *组合两种压缩技术能比单用一种好* 。

但 Avatar 也没回答 CTT 后来要追问的两个问题：

- *组合顺序重不重要？* Avatar 是 KD → quantize，没试过反过来。它默认这个顺序就是对的；
- *为什么停在两种技术上？* 剪枝、NAS 都没引入。这事就更没说"四种技术该怎么排"。

而且——它还是只做 encoder-only。

** 2025 | ALPINE: 把语言依赖砍掉，但放弃了 KD

*** 它看到了什么问题

ALPINE 的 Saad 团队从另一个角度切：Avatar 路线虽然能压能减碳，但它的 KD 阶段要 *完整 finetune 一个学生模型* ——对 CodeBERT 这种 500MB 模型还能凑合，对 1B+ 的代码 LLM（CodeGen 16B、StarCoder）来说，光把 teacher 装进显存做 KD 都困难，更别说 finetune 学生。

另一个观察：现有方法都和具体语言绑死（你为 Python 训了一个学生，到 Java 上就废了），但代码模型本来就是 *跨语言* 的，能不能做一个"语言无关"的剪枝策略？

*** 它的解法

ALPINE 的解法是 *绕开 KD、走纯剪枝 + 后量化* ：

1. *adaptive pruning* —— 用统计指标自动识别哪些 attention head、哪些 FFN 维度对所有语言都不重要，剪掉它们；
2. *low-overhead quantization 插件* ——剪完之后用 8-bit 量化，不需要重训。

结果：58.1% 内存节省，跨多种语言保持性能，不用 KD finetune。

*** 这招管用为什么，又埋下了什么坑

它管用的核心理由是—— *KD 是把性能从老师转移到学生，但老师本身就贵；如果你能直接在老师身上动刀（剪+量化），就根本不需要学生这个中间环节* 。这对超大模型的工程现实很重要。

但坑也很大：

- *仍是 encoder-only* ——这条线五年了，每个人都只压 encoder-only；
- *放弃 KD 等于放弃了"性能修复"机制* ：剪 + 量化都是"减法"，掉精度后没退路；
- *没碰 NAS* ：架构选择层面完全不动；
- *仍没回答 "顺序"* ：剪和量化的先后没有系统验证。

到这里整条线的形状已经很清楚了： *五年了，大家都在加技术或换技术，但没有人系统地验证 "技术怎么排" 这个 meta-question* 。而且一直陷在 encoder-only 的舒适区里。

** 2026.04 | CTT: 给压缩管道排一个有原则的顺序

*** 它看到了什么问题

CTT 团队（Alam 等）一上来就把前人挨个挑了一遍 ([43] = ALPINE, [49] = Avatar, [50] = Compressor)：

#+begin_quote
"Unlike prior pipelines [43, 49, 50], which are limited to single architectures and do not examine how ordering impacts results under deployment budgets, CTT provides a unified framework..."
#+end_quote

他们看到的具体问题：

1. *架构覆盖问题* ：五年来的 SE 压缩工作几乎全部只在 encoder-only 上跑。但当今的代码 LLM 早就分三家——encoder-only（clone detection）、encoder-decoder（summarization）、decoder-only（generation），三种架构的推理瓶颈完全不同（attention 主导 vs cross-attention 主导 vs autoregressive 串行）。压一种证明不了对另两种有效；
2. *组合顺序问题* ：NAS、pruning、quantization、KD 四种技术过去最多被两两组合用，但 *先做 NAS 再剪 vs 先剪再 NAS、先量化再蒸馏 vs 先蒸馏再量化* ，到底哪个顺序好？没人系统跑过。这是 CTT 最关键的 claim—— "ordering is not incidental, it's a key optimization factor"；
3. *budget 缺失* ：现有工作都是 "压完看效果"，没有 "在 latency ≤ X、memory ≤ Y 的硬约束下，找最优压缩组合" 的预算视角；
4. *经济学比喻缺位* ：压缩技术明明是 "用一点性能换工程优势"，但从来没人用经济学的 *trade-off* 语言把它讲清楚。

*** 它的解法

CTT 把整个解法用"碳税"这个比喻锚定：

#+begin_example
税 (tax)                              退税 (rebate)
+-----------------------+              +-----------------+
| Stage 1: NAS + 剪枝    |              | Stage 3: KD     |
|   税基 = 架构冗余      | -----> ----> | 用 teacher 软标签 |
|   税率 = 性能折损       |              | 把扣掉的性能      |
|                       |              | 找补回来          |
+-----------------------+              +-----------------+
| Stage 2: Quantization |
|   税基 = 数值精度       |
|   税率 = 量化误差       |
+-----------------------+
#+end_example

具体执行：

1. *Stage 1 - 结构税* ：先用 NAS 在 hidden size / FFD / 层数 / head 数四个维度搜索一个紧凑的学生骨架，再用 structured pruning 把搜出来的骨架里剩余的冗余 layer/head 剪掉。NAS 太粗 + pruning 太散，组合起来正合适；
2. *Stage 2 - 精度税* ：在 KD 之前做 quantization-aware initialization——把学生先量化到 INT8，再让它在量化噪声下被蒸馏。这个顺序很关键： *先量化再蒸馏* 让学生从一开始就学会在量化噪声下保持性能，而不是蒸馏完再量化、再发现"哎呀掉了 5 个点"；
3. *Stage 3 - 退税* ：用 teacher 的 soft label 把前两步扣掉的性能找回来。

整条管道一个核心 claim： *NAS → pruning → quantization → KD 这个顺序是最优的* 。论文用消融实验证明：换任何其他顺序（KD 在前、量化在前、NAS 在后）效果都会下降。

*** 这招管用为什么，又埋下了什么坑

它管用的真正理由不是任何单一技术的创新——四个组件都是现成的——而是 *第一次把 "压缩管道的顺序设计" 当成一等问题来研究* 。这个研究问题之前一直在水面下，CTT 把它抬上来，并给出了一个跨三种架构、三种任务都站得住的答案。

经济学的"税 + 退税"比喻不是花活：它真的解释了为什么 *先扣再补* 比 *边扣边补* 好——蒸馏作为退税，需要前面有足够明确的"税基"（紧凑架构 + 低精度），它才能针对性地补；如果你边压边蒸馏，老师不知道学生最终要在多紧的约束下跑，soft label 的信号就不够精准。

实验上最戳的几个数：
- *encoder-only (clone detection)* ：8-10x 加速，~98% 准确率保留；
- *encoder-decoder (summarization)* ：3x 加速，~89% BLEU/ROUGE 保留；
- *decoder-only (generation)* ：4-7x 加速，91% textual metric / 68% pass@1 保留；
- *最高 49x 内存压缩、81% CO2 削减* ；
- *CodeT5 压缩后比 LLM.int8() 量化的版本快 73x、省 87x 内存、CO2 少 46x* ——这是 "光量化不够、必须连结构一起压" 的强证据。

但 CTT 也明白地承认了几个未填的坑：

- *只到 1.5B* ：再往上（10B+）显存装不下学生 + 老师同框，scale-up 还需要工程突破；
- *task-specific student* ：没做"一个学生模型同时会三种 SE 任务"的 generalist 蒸馏；
- *quantizer 是 INT8 固定* ：GPTQ/AWQ 等更现代的量化器虽然作者声明 "可插拔"，但没真换上试；
- *ordering 的最优性是经验性的* ：消融实验证明了 NAS→pruning→quantization→KD 在他们测的设置下最好，但没有理论证明它在所有场景下都最好。

* 前沿延伸

CTT 是 2026 年 4 月底出来的，到今天（2026-05-28）只有一个月， *目前还没有明确批判并改进 CTT 的后续工作* 。但有几条平行的、互补或对照的方向值得标记：

** 同代 | 量化器派: LLM.int8() / AWQ / GPTQ

CTT 自己在论文里把它们当成 *压缩 pipeline 的反面教材* ——意思是："如果你只做 quantization 不动结构，看起来便宜，但会输给系统化的 pipeline。" 具体证据：

- *LLM.int8() 量化的 CodeT5* ：比 CTT 学生慢 73x、内存多 87x、CO2 多 46x；
- *AWQ 量化的 Qwen Coder 2.5 1.5B-instruct* ：比 CTT 学生大 5x、内存多 4x、CO2 多 3x。

CTT 的态度是： *量化器是"组件"，pipeline 是"骨架"。组件可以替换（GPTQ 可以替 INT8），但骨架的"先结构再精度再退税"这个范式不能变。*

** 同代 | PEFT 派: LoRA / Adapter / BitFit / Prefix-Tuning

CTT 明确划清了边界：

#+begin_quote
"PEFT methods still keep the entire base model in memory at inference time, so the overall latency, peak VRAM, and energy use remain essentially unchanged."
#+end_quote

也就是说 PEFT 解决的是 *训练成本* 问题，不解决 *推理部署* 问题。CTT 解决的是后者。两者正交。

** 可能的后续方向（推测）

基于 CTT 自己列的局限性，最可能的下一步是：

1. *Scale to 10B+* ：怎么在不能把老师塞进显存的情况下做 KD？distributed teacher 或离线 logit caching；
2. *Generalist student* ：一个 1B 学生同时会 clone detection + summarization + generation，让 CTT 从 task-specific 走向 task-agnostic；
3. *自动化顺序搜索* ：把"NAS→pruning→quantization→KD"这个固定顺序变成可学习的，让模型按 task/budget 自己选最优 pipeline；
4. *和现代量化器深度耦合* ：把 GPTQ/AWQ 真的串进 Stage 2，而不只是声明"可插拔"；
5. *训练侧的碳税* ：CTT 目前只税推理，对 NAS 阶段巨大的搜索成本没追问。下一步可能是 *全生命周期* 的碳税。

* 洞见

这条演化线表面是"代码模型压缩技术演进史"，往深里看是另一件事—— *一个领域如何从 "技术堆叠" 走向 "技术编排"* 。

具体看五代的认知转变：

1. *DistilBERT* （2019）：发现 *KD 可以作为压缩的主路* （之前都是剪枝/量化）；
2. *Compressor* （2022）：发现 *学生架构应该被搜索* （之前是手工设计）；
3. *Avatar* （2024）：发现 *能耗/CO2 应该是一等指标* （之前只看准确率）；
4. *ALPINE* （2025）：发现 *KD 不是必须的，剪枝+量化也可以走通* （之前默认 KD）；
5. *CTT* （2026）：发现 *组合顺序本身就是一个独立的优化变量* （之前默认任何顺序都行）。

每一代都打破了上一代默认假设的"理所当然"。所以下一步该往哪走？最可能的猜测是—— *"压缩管道本身"会被自动化* 。CTT 把"先结构再精度再退税"这个顺序固化下来了，但下一代工作很可能把它变成 *learnable* ：让模型按 task、按 deployment budget、按 hardware 自己决定 pipeline 的拓扑。这相当于从 "AutoML for student architecture" 升级到 "AutoML for compression pipeline"。

另一个明显方向是 *把碳税框架做成端到端可微* ：CTT 现在的"税率"（性能折损）和"税基"（架构冗余）都是人定的，理论上可以让一个上层 controller 学着调它们， 让整个 pipeline 在 "性能 - 碳 - 延迟" 三维空间里自动寻优。

* 启发

对"怎么找问题"的启发非常具体：

- *被所有人共享的默认设置就是金矿* 。DistilBERT 之后五年，所有压缩工作都默认"组合多种技术时顺序不重要"。CTT 没发明新技术，就把这个默认假设审视了一遍，发现它是错的——结果发了 FSE。 *一个被领域共享的默认假设，往往就是下一代工作的入口* 。

- *换"问题的范围"比换"问题的答案"贡献大* 。这条线的真正进步不在于"我又压了多少 MB"，而在于一次次的 *范围扩张* ：从通用 NLP 到代码、从二分类到生成、从单技术到组合、从单架构到三架构、从准确率到碳排放。 *研究的真正进步常常是"我们能问的问题变大了"，而不是"我们的数字更好了"。*

- *给老套技术换个比喻能开新地* 。NAS、剪枝、量化、KD 没一个是新的，但 CTT 用"碳税 + 退税"这个经济学比喻把它们重新串了一次， *比喻让一个杂乱的工程组合变成了一个有原则的方法论* 。这是写作技巧也是研究技巧——找到一个能压住所有组件的统一隐喻，论文的层级就抬上去了。

- *永远问 "这个领域舒适在哪里"* 。SE 压缩五年都缩在 encoder-only 是非常可疑的——不是因为别人懒，而是因为 encoder-only 最好测、数据集最齐、baseline 最全。一个领域的"舒适区"就是下一代工作要拓展的方向。

这意味着你可以——挑一个被工程社区当成"理所当然"的环节（管道顺序、初始化策略、数据顺序、retry 逻辑……），把它从 implicit 变成 explicit ， *用消融实验证明它其实重要* 。一旦你证明了"它不是细节而是核心"，你就给整个领域加了一个新的优化维度。

* 诚实标注

- CTT → ALPINE [43] / Avatar [49] / Compressor [50] 这条引用链是 *论文 related work + introduction 里明确点名批判的* ，原文："Unlike prior pipelines [43, 49, 50], which are limited to single architectures and do not examine how ordering impacts results"。这是直接证据，不是推测。
- ALPINE → Avatar → Compressor 这条链是 *推测但合理* ：三者都是 SE 代码模型压缩的核心工作，Avatar 和 Compressor 是同一作者 Jieke Shi 团队的两代工作（明确递进关系），ALPINE 在 related work 里继续比较了 Avatar；但 CTT 没有重建这条内部演化的因果关系，是我根据时间线和技术承接关系串起来的。
- Compressor → DistilBERT 是 *推测* ：Compressor 没有直接引用 DistilBERT 作为方法基础，但它使用 KD 的方式（学生模仿 teacher 的 soft label）就是 DistilBERT 那一脉。把它作为"代码模型 KD 压缩的起点"是合理简化，但不是严格的"批判改进"关系。
- CTT 论文实际有 71 篇引用，本文挑了 *最直接的批判改进链* （ALPINE/Avatar/Compressor）作为主线，舍弃了其他支线（如 BERT 时代的 Q8BERT/DoReFa-Net 量化、Lottery Ticket Hypothesis 等剪枝理论）。
- LLM.int8() / GPTQ / AWQ / LoRA 是 *同代对照* 不是后续，CTT 用它们作为消融对比或边界划分。
- *目前 (2026-05-28) 没有找到明确批判改进 CTT 的后续工作* ——CTT 才发布一个月。"前沿延伸"一节是基于 CTT 自陈局限性的合理外推，不是已经发生的研究。
