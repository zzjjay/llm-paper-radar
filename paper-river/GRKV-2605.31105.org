* 问题之河

一句话：大模型读一本长篇小说时，它得把读过的每一句话都记在显存里，越读占用越多，最后撑爆。能不能让它"记住要点、忘掉废话"，还不掉智商？

这条河流的源头是一个很物理的约束。Transformer 每生成一个新词，都要回头看前面所有词的"键"（Key）和"值"（Value），这堆东西叫 KV 缓存。它的大小随上下文长度线性增长——上下文翻倍，显存翻倍。一个 30B 的模型、批量 128、序列长 1024，光 KV 缓存就要吃掉约 180GB 显存，比模型权重本身还大。

难就难在三个地方互相打架：

1. *不能全留* —— 全留就是线性爆炸，长上下文根本跑不起来。
2. *不能瞎扔* —— 注意力是"不可逆"的：今天看着没用的词扔了，明天的问题正好要用它，信息就永久丢了，模型开始一本正经地胡说八道。
3. *不能加训* —— 模型已经训好了，理想的压缩方法最好"即插即用"（training-free），不重新训练就能上。

于是一代一代人前赴后继，都在回答同一个问题的不同版本："*在固定的显存预算下，怎么决定留什么、扔什么、怎么扔，才能让模型表现掉得最少？*" 这条河从"按位置留"流到"按重要性扔"，再流到"扔之前先合并"，最后流到本文 GRKV 的"全局回归式合并"。

* 演化叙事

** 2023 | H2O & StreamingLLM: 同一个发现，两种活法

*** 它看到了什么问题

最朴素的想法是把 KV 缓存全留着，但前面算过——线性爆炸，长文本直接跑不动。第一个补丁是"滑动窗口"：只留最近的 N 个词。StreamingLLM 发现这招会 *灾难性崩溃*：一旦文本超过窗口、最开头那几个词被挤出去，困惑度（perplexity）瞬间爆炸。

为什么？因为 softmax 强制注意力权重加起来等于 1，模型学会了把"多余的、没处放的"注意力统统倾倒到最开头那几个词上——它们对每个后续位置都可见，天然成了一个"泄洪口"。把这个泄洪口扔了，整个注意力分布就失稳了。

H2O 看到的是另一面：注意力高度集中，一小撮词（它叫"重锤" Heavy Hitter）占据了绝大部分注意力质量，去掉它们模型立刻变蠢。

*** 它的解法

两家给出了 *philosophy 相反* 的答案，这也是这条河最初的分叉：

- *StreamingLLM —— 按位置静态留*。永远留住开头那几个词（它叫"注意力沉降" attention sink，发现留 4 个就够），再加一个最近词的滑动窗口。选谁完全看位置，不看内容。
- *H2O —— 按重要性动态扔*。预算分两半：一半给最近的词，一半给"累积注意力分数"最高的重锤。它把驱逐建模成一个动态子模优化问题，证明贪心算法接近最优。

可以这样理解二者的关系：把 StreamingLLM 的"留开头+留最近"中的"留开头"换成"留累积注意力最高的那些"，你大致就得到了 H2O。一个赌"位置稳定"，一个赌"重要性可测"。

*** 为什么这招管用（以及哪里埋了新坑）

它们俩共同坐实了一个经验事实——注意力是稀疏的、可压缩的——这是整条河的地基。但 H2O 埋了个关键的坑：它是 *按单个 token 打散了打分* 的。一个完整的语义片段（比如一个电话号码、一个人名、一句引文），注意力往往只集中在它的 *开头几个 token* 上。H2O 留住这几个领头的，把后续的接续部分当成低分给扔了——*把一个完整片段拦腰斩断*。

后果很具体：解码时模型"copy 了电话号码的国家区号，然后把剩下的位数给编了"。留对了一半，等于没留。这个"片段被撕碎"的坑，就是下一篇 SnapKV 的起点。

** 2024.04 | SnapKV: 别按 token 选，按"片段"选

*** 它看到了什么问题

SnapKV 盯住了 H2O 的两处软肋。其一就是上面那个 *片段碎片化*：只留 top 注意力的零散 token，"只留下了细节的碎片，丢掉了信息的完整性"。其二是 H2O 的打分 *query 无关*——它用的是所有 query 平均下来的累积注意力，是一个"和当前问题无关的扁平分数"。但真正重要的 token 是随着指令/问题变化的。

*** 它的解法

SnapKV 有一个漂亮的观察："模型在你开口之前，就已经知道你要找什么了"。具体做法两步：

1. *观察窗口（query-aware）*：取 prompt 末尾的一小段（比如 16~32 个 token）当"观察窗口"，用这段的 query 去看前面所有 key，把注意力权重投票累加，给每个前文位置打分。理由是：这段末尾窗口的注意力模式，和模型真正开始生成时的注意力模式 *高度一致*。所以 prompt 自己就泄露了"哪些位置将来重要"。
2. *池化保片段（span-based）*：投完票，不取零散的 top-k，而是沿序列方向做一维 *最大池化*（核大小 5~7），让每个高分锚点把 *相邻的一串 token* 一起拽进保留集。于是留下来的是一个个连续的"片段团"，不是孤立的尖峰。

把 H2O 连续变形成 SnapKV：先把"累积注意力打分"换成"末尾窗口投票"（从 query 无关变 query 相关），再在选择结果上套一层池化（从 token 级变片段级），你就得到了 SnapKV。

*** 为什么这招管用（以及哪里埋了新坑）

池化逼着"整个相邻片段一起搬家"，电话号码不会再被拦腰斩断——这也是为什么 SnapKV 在检索类任务上提升最大，1024 的缓存预算能打赢 H2O 4096 的预算。从此 *片段式保留*（span-based retention）成了现代主流。

（一个诚实标注：原始 SnapKV 论文里并没有"span-based retention"这个词，它说的是"clustering / pooling / 保留邻居 token"。"片段式保留"是后续论文——包括 GRKV——对这套池化机制的 *回溯性命名*，是公允的概括，但不是 SnapKV 自己的术语。）

它留下的坑：SnapKV 给 *每一层分配同样大* 的缓存预算。这正是 PyramidKV 要捅的地方。

** 2024.06 | PyramidKV: 每层该分多少缓存，不该一刀切

*** 它看到了什么问题

SnapKV（以及 H2O、StreamingLLM）都给所有层 *一样大* 的预算。PyramidKV 做了注意力分析后发现：注意力模式 *逐层不同*——低层注意力宽而散，中层逐渐聚焦到特定上下文，高层高度集中到极少数关键 token（即"大激活/注意力沉降"）。一刀切的预算，在某些层浪费、在另一些层饿死。

*** 它的解法

它把这个观察叫"金字塔式信息漏斗"（Pyramidal Information Funneling），解法两步：

1. *逐层不等分配（新东西）*：固定总预算，但 *低层多给、高层逐层递减*，形成一个金字塔。作者称这是第一个"逐层缓存量不同"的压缩方法。
2. *层内沿用 SnapKV*：每层内部仍用 SnapKV 那套（留最近窗口 + 按注意力选前文）。

变形视角：保持 SnapKV 的层内选择不动，只把"每层等预算"换成"金字塔预算"，就是 PyramidKV。

*** 为什么这招管用（以及哪里埋了新坑）

它用约 12% 的缓存就能逼近全缓存表现，极端情况下（0.7% 缓存）某些任务能反超基线 20 分。但它的金字塔形状是 *手工设定的固定单调衰减*，未必是最优分配——后续 Ada-KV、ZigZagKV 等就转向"按误差/方差自适应分配"。Ada-KV（head-wise 分配）、CriticalKV（扰动约束）都属于这一支的细化，也都是 GRKV 兼容并测试的"片段式驱逐"骨架。

但更大的转折在另一个方向：到此为止，所有方法的动作都是"*扔*"。下一代人开始问：扔掉的 token 真的一点用都没有吗？

** 2024~2025 | CaM / KVMerger / D2O / AsymKV: 别扔，"合并"进去

*** 它们看到了什么问题

这一代共享一个洞察，针对所有"驱逐"方法（H2O、SnapKV……）的根本毛病：*驱逐是永久删除*。一个 token 现在分数低，不代表它对将来的某个 query 没用；一旦删了，信息永久消失，压缩率越高，输出扰动、上下文丢失、幻觉越严重。CaM 一句话点破："基于注意力分数的驱逐，无论你怎么选，都必然带来输出扰动。"

于是核心动作从"扔"变成"合"：被驱逐的 token 还带着信息，*把它折叠进留下来的 token 里*，而不是直接丢。

*** 它们的解法（四种"合"法）

- *CaM*（ICML 2024）：把"将被驱逐"的缓存，按其注意力显著度加权采样，*合并进保留的局部/最近 token*。即插即用，套在 H2O/StreamingLLM 上就能用。
- *KVMerger*（2024）：发现序列内 *相邻 key 的余弦相似度很高*，于是把相邻、相似度超阈值的 key 贪心聚成"合并集"，再用高斯核加权把一簇压成一个"伪 token"。
- *D2O*（ICLR 2025）：两个层级。层级——按注意力密度动态分配（浅层密、给更多）；token 级——保留一个相似度阈值去"召回"已被丢弃的 token，决定是否把它合并回相似的保留 token。
- *AsymKV*（NeurIPS 2025）：发现 *相邻 key 同质、相邻 value 异质*。于是对相邻两个位置 (m, m+1) 做局部合并：key 用类牛顿步合成一个最优向量，value 在 key 被强行变相等后直接 *求和*（配一个基数向量 C 修正 softmax 分母）。核心直觉是"把信息损失从异质的 value 转嫁到同质的 key 上"。

*** 为什么这招管用（以及哪里埋了新坑）

合并确实比硬扔保住了更多信息。但 GRKV 一针见血地指出了这一代的 *共同结构性缺陷*：它们用的全是 *局部启发式*——CaM 灌进"局部"token，KVMerger/AsymKV 合并"相邻"token，D2O 召回进"相似的保留"token。在现代"片段式保留"的格局下，留下来的 token 大多是片段的边界，于是大量被驱逐的 token 像漏斗一样 *全灌进少数几个"片段边界载体 token"*。

载体被灌爆的后果是 *过度合并*：往一个 token 里塞太多东西，它原本的语义被"抹糊甚至擦除"。GRKV 给出的逐个诊断：

- CaM "偶有提升，但常常拉低平均分"——增益不稳定。
- KVMerger "把合并集中在片段边界附近"——因为它按相邻聚类。
- D2O "在片段式保留下经常退化"，检索任务上过度合并。
- AsymKV "拉低性能且开销高"：用不了 FlashAttention-2（基数向量重写了 softmax 分母，是非标准注意力，这点已从原文证实），GRKV 报称 16K 下约 15 倍延迟（*此具体倍数是 GRKV 自己的测量，AsymKV 原文未给出，标注为推测一致*）。

更深一层：这些合并方法大多是为 *老式 token 级保留* 设计的，碰上现代片段式保留水土不服。这就是 GRKV 的起点——载体太少（只有边界）、合并太局部（只看邻居）、目标太模糊（没有一个统一的优化目标）。

* 前沿延伸

** 2026.05 | GRKV: 把合并变成一个全局优化问题

*** 它看到了什么问题

承接上文：前人合并法的三宗罪——载体池太小（信息全挤到少数边界 token）、规则太局部（启发式，只看相邻/局部相似）、缺乏统一目标（没人直接去优化"压缩后到底偏离完整缓存多少"）。

*** 它的解法

GRKV 把目标 *显式写成一个全局优化问题*：让"压缩后缓存算出的注意力输出"尽量逼近"完整缓存算出的注意力输出"，最小化二者的差异。这是第一个把"压缩 vs 完整 的注意力输出差异"当成 *原则性目标函数* 来建模的合并方法。

具体机制：用 *岭回归*（ridge regression，带正则的最小二乘）来求解合并。

1. *所有保留 token 都当载体*（不只是边界），把载体池一下子撑大，从根上缓解"灌爆少数载体"的过度合并。
2. *岭正则项* 防止信息过度平滑、防止载体被抹糊。
3. 交替进行：value 的闭式更新 + key 的线性化更新，并用一个"代理 query 窗口"当作未来未知 query 的近似（这一步思想上承袭自 SnapKV 的观察窗口）。

变形视角：把 KVMerger 的"相邻 key 聚类合并"换成"对全体保留 token 做一次带正则的全局回归拟合"，把"局部高斯核权重"换成"岭回归闭式解给出的全局权重"，你就从局部启发式跨到了 GRKV 的全局回归。

*** 为什么这招管用（以及哪里埋了新坑）

它在 LongBench 16 个任务、RULER 13 个任务上，配合片段式驱逐，号称是 *唯一一个能整体提升表现* 的合并方法，且开销极小。模型覆盖 Mistral-7B-Instruct-v0.3、Llama-3.1-8B-Instruct、Qwen3-14B。

截至目前（2026-05），这是这条线的最新进展。可预见的新坑：岭回归虽是闭式但仍要解线性系统，全局合并的计算/数值稳定性在超长上下文（128K+）下能否保持"极小开销"，以及"代理 query 窗口"对真实未来 query 的近似在多跳推理任务上是否够用，都还没有定论。

认知主线: "留什么"(位置 vs 重要性) -> "怎么留整"(片段) -> "每层留多少"(分配)
          -> "扔的怎么救"(合并) -> "怎么合得不抹糊"(全局优化 + 扩大载体池)
#+end_example

* 洞见

读完整条线，表面是技术迭代，底层其实是 *三次认知重心的迁移*：

1. *从"哪个 token 重要"到"哪段信息完整"*。H2O 问"留哪些点"，SnapKV 改问"留哪些段"。一旦你承认信息是 *成片* 而非成点存在的，"按 token 打分"这个出发点本身就错了。这是从原子论到结构论的转变。

2. *从"删除"到"守恒"*。驱逐法默认"低分=无用=可删"；合并法拒绝这个等式，坚持"信息守恒"——没用的不是 token，是它 *当前的存放方式*。问题从"留谁扔谁"变成了"怎么把要扔的折叠进要留的"。这是从离散选择到连续重分配的转变。

3. *从"启发式"到"目标函数"*。CaM/KVMerger/D2O/AsymKV 都在"凭直觉设计合并规则"，GRKV 的真正突破不是岭回归这个工具，而是它 *第一个敢于直接写下"我到底想最小化什么"*——压缩输出与完整输出的差异。一旦目标显式化，载体该多大、权重该怎么分，就从"拍脑袋"变成"解出来"。这是从工程 trick 到优化问题的转变。

还有一条暗线值得注意：*基础设施会反向定义算法的生死*。AsymKV 因为用不了 FlashAttention-2 而被判"开销过高"——一个数学上漂亮的方法，输给了"能不能跑在现代 attention kernel 上"这个工程现实。GRKV 反复强调"minimal overhead / 可用 FlashAttention"，说明这条河现在不只比谁掉分少，还比谁 *和现有推理栈兼容*。

下一步最可能往哪走？目标函数已经被 GRKV 立起来了，接下来大概率是：(a) 把"代理 query 窗口"这个近似换成更可靠的未来 query 估计；(b) 把全局回归从训练-free 推向"轻量校准"，用极少量数据让合并矩阵更准；(c) 把合并与量化（另一条压缩支流）统一进同一个目标函数。

* 启发

对"怎么做研究、怎么找问题"，这条河给了几个能直接用的招：

- *当一个领域里所有人都在调同一类 trick 时，往后退一步去找那个没人写下来的目标函数*。GRKV 的杀招不是更聪明的启发式，而是把大家心照不宣的目标显式化。这意味着你可以：在任何"启发式泛滥"的方向上，问一句"我们到底想最小化什么"，往往就能开出一篇新论文。
- *别只盯着"扔什么"，去质疑那个动作本身*。从驱逐到合并，是把"删除"这个默认动作换成了"重分配"。这意味着你可以：拿到一个领域，先列出大家默认在做的"那个动作"，然后问"这个动作是不是问题的根源"。
- *上一代的"解法"就是下一代的"问题清单"*。每篇论文留下的坑（SnapKV 的等预算、合并法的载体灌爆）都精准变成了下一篇的入口。这意味着你可以：读论文时专门收集"它的局限/future work"，那是最廉价的选题来源。
- *先确认基础设施能不能容下你的算法*。AsymKV 的教训。这意味着你可以：在动手前先问"这东西能不能跑在 FlashAttention / vLLM 上"，兼容性不是收尾工作，是设计约束。
