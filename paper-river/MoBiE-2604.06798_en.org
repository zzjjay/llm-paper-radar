* One-line positioning

Take the "binarized large model" compression toolkit from dense LLMs and transplant it onto MoE LLMs.
Once transplanted, it turns out not to travel well — so three surgical fixes are made: share low-rank components across experts, inject global loss gradients into the Hessian, and constrain routing drift via null-space projection.

* The evolutionary story: starting from "how do we flatten a model"

** Layer 0: OBS / OBD — you can't prune blindly; check the second-order derivatives

This actually started in 1989/1993, originally for neural network pruning.
The problem was refreshingly simple: the model is too large to run — can we zero out some weights?
Naive answer: delete the ones with the smallest absolute value.
LeCun and colleagues said no — looking only at weight magnitude ignores how much the loss would actually move if you removed that weight.
The ones you should really remove are those whose removal barely changes the loss at all.

So how do you know whether removing a weight will shake the loss? Expand the loss in a Taylor series around the current weights; the second-order term contains the Hessian.
The Hessian tells you each weight's "sensitivity": a large diagonal entry means sensitive — do not touch.
*This idea of using second-order derivatives to determine weight importance* is the common ancestor of every low-bit quantization method that follows.

The problem it left open: the Hessian is huge — a single layer requires an O(d^2) matrix. Can you actually use it?

** Layer 1: GPTQ — making the ancestor's trick work on LLMs

By 2022, LLMs had taken off, with GPT-3-scale models running into hundreds of billions of parameters.
Full precision was infeasible; quantization to 4bit or lower was needed.
The mainstream approach was round-to-nearest (RTN): quantize each weight independently to the nearest grid point.
Simple, but quantization errors accumulate and benchmark scores drop noticeably.

The problem GPTQ identified: quantizing each weight independently is equivalent to assuming the weights are uncorrelated — but of course they are correlated within a layer.
*The fix*: treat each layer as a "column-wise reconstruction" problem — when quantizing column j, hold the already-quantized columns fixed and let the not-yet-quantized columns compensate; the objective is to minimize the mean squared output error of the whole layer.
What guides the compensation? The Hessian.
This is essentially OBS's second-order expansion transplanted to the layer level of an LLM, yielding a single-pass, layer-by-layer PTQ scheme.

GPTQ solved the problem of making 4bit usable. But it left a new headache: what about going lower — 2bit? 1bit?
Quantization error explodes because the gap between grid points becomes as large as the weights themselves.

** Layer 2: PB-LLM — don't push everything to 1bit; leave an escape hatch

By 2023, people were attempting 1bit quantization.
The difficulty is immediate: 1bit offers only two choices, ±α; almost every weight's true value gets collapsed into one of two numbers, and the information loss is enormous.
The model simply breaks.

The problem PB-LLM identified: not all weights are equally important.
The Hessian from GPTQ had already told you — some weights make the loss spike when perturbed; others barely matter.
*The fix*: split the weights into two pools. The important (salient) minority retains 4bit or 8bit; the large remaining majority is compressed to 1bit.
Hence "partially binarized." The average bit-width stays very low, but the critical weights are protected.

The problem it left open: this is fundamentally a mixed-precision scheme — raising the 1bit fraction saves more memory but costs more accuracy. Can you go fully 1bit without excessive degradation?

** Layer 3: BiLLM — push the "split into two pools" idea all the way

Early-2024's BiLLM pushed this to the edge of *pure binarization* (all weights at 1bit).
The problem it saw in PB-LLM: keeping some weights at higher precision is essentially admitting defeat, and ultimate memory savings are never fully realized.

*The fix*: BiLLM also classifies weights, but after classification both groups are kept at 1bit — just with different scales and different binarization strategies:
- Salient weights (the important minority): a finer "residual binarization" — fit once with 1bit, then fit the residual again with 1bit (effectively two stacked 1bit codebooks, though storage is still counted as 1bit)
- Non-salient weights: straightforward group-wise 1bit + scale

The importance criterion still inherits the Hessian from GPTQ.
In essence, BiLLM pushed the OBS → GPTQ → PB-LLM lineage to the lowest usable point: fully 1bit but still functional.

The problem it left open: BiLLM was designed for dense models. Every "salient / non-salient" decision is made independently per weight matrix.
What happens when you encounter MoE?

** Layer 4: ARB-LLM and kin — iteratively refine the error one more time

Work from the second half of 2024 through 2025 (ARB-LLM, BinaryMoS, I-LLM, etc.) mostly adds "error refinement" on top of the BiLLM framework:
quantize once, treat the residual as a new problem and quantize again; or alternately update the scale and binary codes so that both sides converge toward the optimum together.
The underlying idea is the same: 1bit has too little expressive capacity, so use multi-step approximation-and-correction to squeeze out smaller errors.

The problem it left open: all of these methods still view a single dense matrix — their lens is "weight importance" and "reconstruction error."
MoE models introduce two entirely new concerns: first, redundancy across experts (you don't need to compress every expert independently); second, experts are selected by a router — a small perturbation to expert weights changes their outputs, which changes the router's "signal," so a token that used to go to expert 3 may now go to expert 7 after quantization.
Neither of these was ever addressed by the dense-quantization lineage.

** Layer 5 (target paper): MoBiE — three MoE-specific surgical fixes

The problems MoBiE identifies are stated clearly in the abstract and map exactly onto the two dense-era blind spots above:

*Problem 1: redundancy across experts*
A single MoE layer can have 8, 64, or 128 experts, each an independent FFN.
The dense playbook quantizes each expert independently — but they share a substantial low-frequency structure.
*MoBiE's fix*: stack all experts in a layer and perform a *joint SVD*, extracting the shared low-rank component separately (kept at higher precision) and binarizing only the high-rank, expert-specific residual.
Share first, individualize later — the total bit-width stays the same but expressivity improves.

*Problem 2: importance estimation is task-agnostic*
The Hessian in GPTQ / BiLLM is computed from "layer inputs" on calibration data — it is local and disconnected from the end task's loss.
This is tolerable for dense models but not for MoE: different experts are activated on different tokens, and each expert's actual contribution to the final loss varies enormously. A purely local Hessian cannot rank them accurately.
*MoBiE's fix*: inject *global loss gradients* into the Hessian importance score — combining local sensitivity with each weight's true end-to-end influence on the loss.

*Problem 3: quantization causes router drift*
The MoE router is a small linear layer followed by a softmax that decides which expert each token visits.
When expert weights are quantized, expert outputs change, the router's "signal" changes, and routing assignments shift — this "routing drift" is invisible to a plain reconstruction-error objective.
*MoBiE's fix*: add a constraint that forces quantization error to lie in the *null space of the input* — in plain terms, require that quantization error land in directions the router cannot see.
The router depends on certain principal directions; as long as the error is confined to the null space, the router's output is approximately unchanged.

* The real thread running through this evolution

On the surface, this lineage is about "compression ratio converging toward 1bit."
But at a deeper level the evolution is really about *continuously changing the coordinate system through which "information" is viewed*:

- OBS era: coordinate system = individual weight — is this weight important?
- GPTQ era: coordinate system = a layer's output space — how large is this layer's output error?
- PB-LLM / BiLLM era: coordinate system = a binary partition of weights — which weights deserve special treatment?
- ARB era: coordinate system = residual space — can the error be peeled away layer by layer through iteration?
- MoBiE era: coordinate system shifts to *"the subspace the router cares about vs. the null space the router ignores"*

At each transition, *what matters is not the new method itself but the space in which "importance" is newly defined*.
MoBiE's genuinely novel contribution is not that it uses SVD or adds gradients — those are tools.
What is new is that it is the first method to acknowledge: *in MoE, the quantization objective cannot be only "do the weights / outputs look similar?" — it must also preserve "does the routing look similar?"*

* Where the next step leads

If the main thread above holds, the natural directions forward are:

1. *Activations need the same treatment* — MoBiE's abstract addresses only weight binarization. MoE activations likewise have a "router-relevant" subspace and a "router-irrelevant" null space. Activation quantization should follow the same null-space route.
2. *The router itself should participate in the quantization objective* — the current framing is "after quantization, don't let the router drift." The next step should be "let the router co-optimize during quantization," placing a differentiable surrogate for the router (e.g., a Gumbel-Softmax formulation) directly into the loss.
3. *Expert-level mixed precision* — joint SVD reveals that experts share redundant structure. The next natural idea is assigning different bit-widths based on each expert's activation frequency and importance: high-frequency experts stay at 2bit, long-tail experts are pushed to 1bit or lower.
