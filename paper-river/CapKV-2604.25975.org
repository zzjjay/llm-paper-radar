* 问题之河

大模型推理时，每生成一个新词，都要回头看一遍之前所有词的"笔记"——这堆笔记就是 KV cache。上下文一长，笔记就堆成山，显存先撑爆。

于是有了一个看似简单的问题：*留谁，扔谁？*

难就难在：你不知道下一秒模型会问什么。扔掉一个看起来"不重要"的 token，可能恰好是后面那道数学题的关键中间步。十年间一代一代人想了无数招——看注意力分数、看向量长度、看几何多样性、预测未来注意力——每招都管用，但每招都"差一口气"。没人能说清：到底什么叫"重要"？

CapKV 的答案是：这些招其实都在做同一件事的不同近似——*最大化保留信息容量*。差别只是：以前是凭直觉摸到的，现在能写成一个闭式公式。

* 演化叙事

** 2000 | Information Bottleneck: 给"该记什么"找个数学定义

*** 它看到了什么问题

二十多年前，机器学习里就有一个根本困惑：当你被迫压缩信息时——比如把一张图压成一个特征向量、把一段话压成一个 embedding——*该扔什么*？

直觉答案是"扔不重要的"。但什么叫"重要"？没人能给个不依赖具体任务的定义。

*** 它的解法

Tishby 等人借了通信领域的工具，写了一个简单方程：找一个中间表示 T，让它从输入 X 中只留下对预测目标 Y 有用的信息——具体地说，让 I(X;T) 尽量小（压缩得狠），同时让 I(T;Y) 尽量大（任务相关性高）。两者之间用一个 β 调权重。

这就是 *信息瓶颈* (Information Bottleneck)。一句话：*压缩的极限不是"扔少"，而是"扔无关"*。

*** 为什么这招管用（以及哪里埋了新坑）

它把"重要"这个含糊词翻译成了互信息——一个能算的东西。任何"选子集"的问题，原则上都能套这个框架。

坑在哪：互信息在高维实数空间里算不了闭式解，工程上得用变分近似、神经网络拟合，复杂且贵。所以二十多年里它主要活在理论分析里，工程上人们另起炉灶。

KV cache 驱逐就是另起炉灶的一个典型——直到本文才把它接回来。

** 2023 | H2O: 第一个把 KV 驱逐做成实用工具

*** 它看到了什么问题

GPT-4 出来后，大家开始追长上下文。问题立刻出现：上下文从 2K 涨到 32K，KV cache 占显存涨了 16 倍。一张 A100 80G 跑 13B 模型，prompt 一长就 OOM。

但当时没有任何系统的"驱逐"方法。要么全留（撑爆），要么 FIFO 砍最早的（傻），要么 sliding window（丢全局信息）。

*** 它的解法

Zhang 等人盯着 attention map 看，发现一个反直觉的现象：*大部分 token 在大部分时间几乎不被 attend，只有少数 "重击者" (Heavy Hitters) 反复被 attend*。

那好办：累计每个 token 历史上被 attend 的总分数，留分数最高的前 B 个，其余全扔。这就是 H2O（Heavy-Hitter Oracle）。

*** 为什么这招管用（以及哪里埋了新坑）

管用是因为 attention 本身就稀疏——少数 token 承担大部分语义负载，其余多是"填充"。删它们对输出几乎无损。

坑在哪：*累计分数对"刚发生的事"反应迟钝*。比如 prompt 末尾突然给了关键指令——"用中文回答"——但这个指令 token 的累计分数还来不及涨高，被 H2O 当成不重要扔掉。结果模型答了英文。

这就把球传给了下一篇。

** 2024 | SnapKV: 别看一辈子, 只看"刚才"

*** 它看到了什么问题

SnapKV 团队注意到一个更精细的模式：模型在 *prompt 处理完的瞬间*，已经"知道"自己后面要 attend 哪些 token——这个信息全写在 prompt 最后一小段的 attention pattern 里。

H2O 用累计分数，把这个最关键的"末窗口信号"和远古的历史噪声平均掉了。

*** 它的解法

只看 prompt 最后一个固定窗口（比如最后 32 个 token）的 attention 分布，对每个历史 token 计算它被这个窗口 attend 的频率。频率高的留下。

一句话：*把"它历来重要吗"换成"它对'马上要问的问题'重要吗"*。

*** 为什么这招管用（以及哪里埋了新坑）

实测涨点很猛——同样的预算下，长上下文 QA 准确率明显好过 H2O。证明了"近期信号"比"累计信号"更接近真正的 query 相关性。

坑在哪：它仍然*只用注意力分数*这一个信号源。注意力分数本身有噪声（softmax 容易被尾部 outlier 拉偏），而且依赖 query——可未来的 query 你看不见。如果窗口选得不好，照样错杀。

更隐蔽的问题：被选中的 top-B token 之间可能 *高度冗余*——都是同义的、同位的、表达同一个意思的。冗余多样性差，等价于浪费预算。

** 2024-25 | Knorm / KeyDiff / Expected Attention: 不看注意力, 看几何

*** 它看到了什么问题

注意力分数贵（要算 softmax），噪、依赖 query。一群人转身去问：能不能 *只看 key 向量本身的几何*，不算 attention？

更深的问题：之前的方法都把每个 token 单独打分、独立比较。但 cache 的价值是 *集合的价值*——选出来的 B 个 token 互相之间应该 "*覆盖面广*"，而不是扎堆在同一个语义角落。

*** 它的解法

三条岔路同时被探出来：

- *Knorm* (Devoto et al., 2024)：直接用 key 向量的 ℓ₂ 范数当重要性。理由：范数大的 key 更容易在 softmax 里胜出。一个除法、一个排序——极便宜。
- *KeyDiff* (Park et al., 2025)：贪心选一组 key，让它们两两之间的差异（distance）尽量大。要的是"语义多样性"，避免冗余。
- *Expected Attention* (Devoto et al., 2025)：建模 *未来 query 的分布*，估算每个 key 的期望注意力——绕过"未来 query 看不见"的死结。

*** 为什么这招管用（以及哪里埋了新坑）

这一族证明了一件事：*在 KV 驱逐里，"多样性"和"几何"和"注意力分数"是平起平坐的信号*。某些场景下，几何信号甚至更稳。

坑在哪——也是 CapKV 切入的口子：

每个方法都对了一部分，但 *没人能说清它们为什么对*，更没人能说哪个最优。Knorm 看尺度、KeyDiff 看差异、EA 看 query 期望、SnapKV 看注意力——*它们到底在追同一个东西吗*？还是各追各的？

如果是同一个东西，那个东西是什么？如果不是，为什么在不同 benchmark 上各自能赢一些？

这就是 CapKV 要回答的。

** 2026 | CapKV: 给整个家族写一份"母公式"

*** 它看到了什么问题

读完上面五年的论文，作者得到一个判断：*整个 KV 驱逐领域处于"前牛顿状态"*——各种局部规律都被观察到了，但缺一个统一的力学定律。

每个工程师下次设计新方法，都得靠直觉再猜一个 heuristic，跑实验、调超参、刷 benchmark。没有原则，只有 trick。

*** 它的解法

把 KV cache 建模成 *一组线性通信信道*：每个 key-value 对就是一个信道，把未来 query 携带的信息"传输"到 attention 输出。在 *线性-高斯近似* 下（attention 不是线性高斯的，但这是一个能算的代用品），可以写出一个闭式互信息：

"留下的子集 Z_C 对未来 query q 的互信息" = 某个 log-determinant 表达式。

这就是 *容量目标*。

然后展示一件惊人的事：

- 把容量目标 *一阶 Taylor 展开*，落到 key 范数上 —— 你重新发现了 *Knorm*。
- 引入 *二阶项*，落到 key 之间的相关性 —— 你重新发现了 *KeyDiff* 那种多样性。
- 把 query 不当随机变量、当观察值代入 —— 你重新发现了 *SnapKV/EA* 那种 query-aware 方法。

整个家族在母公式下原形毕露：它们是 *同一个互信息目标的不同近似*。

最后，CapKV 不再做近似，直接拿一个标准工具——*统计杠杆分数* (statistical leverage score)——来贪心最大化那个 log-determinant。一次扫描就能给出选哪些 token。

*** 为什么这招管用（以及哪里埋了新坑）

管用，是因为它*终于知道自己在优化什么*。所有之前的方法都在隐含地优化容量，但因为不知道目标，所以总优化得不彻底——要么只用一阶项（Knorm），要么忽略 query 维度（KeyDiff），要么忽略多样性（SnapKV）。CapKV 把这些维度合一。

实验上，它在 LongBench、reasoning benchmark 上全面超过前述方法。更重要的是，作者展示了*容量分数本身*就和 downstream 性能强相关——容量是个能预测性能的标尺，不只是个内部目标。

坑在哪：

- 线性-高斯假设是个 *作图工具* 而非真模型。真 attention 是非线性的、softmax 重尾。统一框架在这个近似下成立，更精确的近似还能解锁什么？
- 杠杆分数仍是 *one-shot* 静态选取。decoding 推进时 cache 的"最优子集"会动态变化，但本文还在用一次性预算分配。
- 信道独立性假设过强——真 cache 里 key 之间有强相关，这正是 H2O 漏选、SnapKV 重选的根因。

* 前沿延伸

CapKV 是 2026 年 4 月发布的，到本文写作时还没看到明确建立在它之上的后续论文。但可以预判几个方向：

- *动态容量*：把 one-shot 杠杆分数换成 decoding 步推进的滚动重选，配合 attention sink 或 streaming 模式。
- *非线性容量*：放弃线性-高斯，用基于 softmax 真分布的容量估计（可能借助 kernel 方法或低秩注意力分解）。
- *跨头/跨层联合*：当前框架仍在 single-head 单层上推导。多头多层联合容量是显然的下一步，但组合爆炸。
- *容量-精度联合压缩*：和 quantization (KIVI, KVQuant) 联合优化——同一个 IB 框架天然能容纳 bit-width 维度。

更广的方向：CapKV 把 KV 驱逐拉进 IB 框架的方式，也许能复制到 *speculative decoding 的 token 选择*、*MoE 的 expert routing*、*long-context retrieval 的 chunk selection*。三者本质都是"在预算约束下选保留谁"。

* 洞见

读这条线，表面上是"KV cache 压缩越做越好"，骨子里有三层更深的转变。

*第一层：从"独立打分"到"集合优化"*。H2O/SnapKV 时代，每个 token 单独算个分数，按分数 top-B。直到 KeyDiff 才意识到"两个分数高但语义重复的 token，不如一个分数稍低但补足覆盖的 token"。CapKV 把这个观察形式化了。这是 *从打分到选集* 的范式跃迁——和推荐系统从 pointwise 走到 listwise 几乎是同一种觉醒。

*第二层：从"实证有效"到"知道为什么有效"*。整个 KV 驱逐领域过去五年是 trick 堆叠：观察现象→提出 heuristic→刷 benchmark→出新现象→提新 heuristic。CapKV 把游戏规则换了——不是设计一个新 trick，而是 *给所有 trick 找一个公分母*。这种 "unification paper" 在领域成熟期会反复出现，比如 Adam → AdamW → SOAP 之于优化器，或 ELBO 框架之于各种生成模型。

*第三层：信息论被搬回了工程一线*。Information Bottleneck 在 2000 年提出后，长期被困在理论圈——因为算不了。CapKV 用的不是真 IB，是 *线性-高斯简化版* + *杠杆分数近似*。这是个值得注意的工程策略：*用一个明知道不精确的理论模型, 换取闭式可算的目标函数, 然后让实验来验证近似是否成立*。这比"理论必须精确"或"工程必须无理论"都更务实。

下一步往哪走？我下注两个方向：(1) 这套 IB 视角会迅速扩散到 *所有有"在固定预算下选 K 个"结构* 的 LLM 子问题——MoE routing、speculative decoding、retrieval chunking、tool selection。(2) 真正的下一个突破不是把杠杆分数算得更准，而是 *放弃 one-shot 假设*——容量本身是 decoding 推进时动态变化的，谁能把"流式容量"做出来，谁就把 KV 驱逐和 streaming LLM 真正合一。

* 启发

对做研究的人，这条线给三条具体启发：

*一、统一者比创新者更稀缺，但更值钱*。 一个领域里堆了二十多个 heuristic 之后，写第二十一个 heuristic 的边际收益已经接近零。但写一篇"这二十个其实是同一个东西"的论文，边际收益巨大——它把整个领域从"经验科学"推到"理论科学"。CapKV 就是这种工作。要敢于在某个领域已经"看似成熟"时去找统一框架，而不是去卷下一个 0.5% 的 SOTA。

*二、用"明知不精确"的模型换取"能算的目标"*。线性-高斯假设在 attention 里 *肯定不成立*——softmax 是非线性的，分布是重尾的。但作者选择接受这个近似，换来一个闭式互信息表达式。然后用实验证明：*近似虽然糙, 但够用*。这是工程化研究的核心套路——*别追求理论的完美, 追求理论的"够用且能算"*。

*三、读论文要倒着读*。如果只读 CapKV 一篇，你会觉得"哦，又一个 KV 压缩方法"。但倒读到 H2O、SnapKV、KeyDiff，你才看见这五年大家在反复试错地接近同一个真相。倒读法让你看见的不是"一个新方法"，而是 *一群人在黑屋子里摸大象*——摸了五年终于有人画出了大象的轮廓。下次你要进一个新领域，*先倒读三层*再读最新的，效率比正序高一个量级。

落点："能用"：
- 你做任何"在预算下选 K 个"的工程问题（特征选择、retrieval、prompt 压缩、batch 内样本选取），都可以先把它写成一个 IB / 信道容量目标，再考虑近似，比直接拍 heuristic 更稳。
- 你看一个领域有五个以上 heuristic 在互卷时，别急着加第六个，问自己：*它们的母公式是什么*？写出来，可能就是你的下一篇 paper。
- 你的下一次文献调研，试一次倒读法——从最新论文的 related work 反向追三层，再正向讲一遍。会比顺序读快很多，也理解得深得多。
