* The River of the Problem

LLMs are too large to run efficiently and must be compressed. The most direct
approach is quantization — cutting 16-bit weights down to 4 or 3 bits,
immediately shrinking memory footprint by four to eight times and speeding up
matrix multiplication. But the harder you cut, the dumber the model gets.
Since 1990, everyone working on this problem has asked the same question:
*when you shrink weights, which ones need careful handling, and which can be
chopped freely*?

The criterion for answering that question has kept shifting. Early on it meant
looking at a weight's Hessian — theoretically perfect but computationally
explosive. Then it became "look at how large the activation values are" —
crude but serviceable. Later it became "use calibration data to tell me which
channels matter" — faster, but calibration data and real user inputs diverge,
causing accuracy drops on unfamiliar tasks.

In 2026, TTQ proposes to move the calibration step entirely into inference
time — *for every incoming prompt, compute which channels matter on the spot,
and quantize on the spot*. The concept of calibration data disappears
altogether, because the input itself is the calibration data. Across 35 years
of evolution along this line, what has really been happening is a gradual
lightening and instantiation of a single thing: the criterion for weight
importance.

* Layer 1: OBD — Use Second Derivatives to Decide Which Weights Are Unimportant

#+begin_quote
LeCun, Denker, Solla. /Optimal Brain Damage/. NeurIPS 1989.
#+end_quote

In 1989, neural networks first became large enough that "prune some weights to
speed up inference" was a real problem.

The pruning method of the day was crude — simply set weights with small
absolute values to zero. LeCun's observation was that *small weights are not
necessarily unimportant*. A weight may be numerically tiny yet lie along a
steep direction of the loss surface; removing it causes the loss to spike.
Another weight may be large yet the loss is entirely insensitive to it, making
removal harmless.

His solution can be stated in one sentence: use the second derivative of the
loss with respect to each weight (the diagonal elements of the Hessian) as an
importance measure. The intuition is — *directions with large second
derivatives mean the loss is sensitive to perturbations there, so those weights
matter; directions with small second derivatives can be pruned freely*.

This is the founding moment of the "look at curvature, not magnitude" paradigm.
But OBD assumes the Hessian is diagonal — in other words, it pretends all
weights are mutually independent. This assumption is badly wrong, but in 1989
computing the full Hessian was out of the question, so it had to do.

* Layer 2: OBS — Weights Are Coupled

#+begin_quote
Hassibi, Stork. /Second order derivatives for network pruning: Optimal Brain Surgeon/. NeurIPS 1992.
#+end_quote

OBS targets exactly the hole OBD left: weights in a neural network are not
isolated — they are tightly coupled. Remove one, and the optimal values of all
the others shift. OBD pretends this coupling does not exist, so pruning past a
certain point causes collapse.

OBS's solution is to use the *full* inverse Hessian: not only compute "how much
does the loss rise if we remove this weight," but also compute "how should the
remaining weights compensate afterward." This is a more accurate second-order
expansion.

The price is O(d³) Hessian inversion. Networks of 1992 were tiny — a few
thousand parameters — and it barely ran. The framework was considered
"theoretically elegant but engineering-infeasible" and shelved for over twenty
years.

Then the LLM era arrived. Suddenly "theoretically elegant" was valuable again,
because a few points of accuracy loss can justify or block the deployment of
billions of parameters. OBS was dug up and reexamined.

* Layer 3: GPTQ — Rebuild OBS as Quantization, Make It Tractable

#+begin_quote
Frantar, Ashkboos, Hoefler, Alistarh. /GPTQ: Accurate post-training quantization for generative pre-trained transformers/. ICLR 2023.
#+end_quote

GPTQ directly borrows the OBS idea but makes two key adaptations that let it
run on 175B-parameter LLMs.

The first adaptation: replace "pruning" with "quantization." OBS sets a weight
to zero; GPTQ rounds a weight from fp16 to the nearest 4-bit grid point. The
mathematical structure is identical — both are a "perturbation," both require
compensating the other weights.

The second adaptation: instead of inverting the Hessian separately for each
weight, process columns greedily and use Cholesky decomposition to prepare all
the inverse elements needed in one shot. Complexity drops from O(d⁴) to O(d³);
with each layer processed independently, a 70B model can be quantized in a few
hours.

But GPTQ leaves a hole that AWQ later targets: *you still need calibration
data*. To compute the Hessian you must feed thousands of text samples through
the model in a forward pass and collect activation statistics. What calibration
data to use, how much, and how far it diverges from actual deployment scenarios
became one of the most debated questions in LLM quantization for years.

* Layer 4: AWQ — The Hessian Is Too Heavy; the Diagonal Approximation Is Enough

#+begin_quote
Lin, Tang, Tang, Yang, Xiao, Dang, Gan, Han. /AWQ: Activation-aware Weight Quantization for LLM Compression and Acceleration/. MLSys 2024.
#+end_quote

AWQ identifies two problems with GPTQ.

First, Cholesky is still too expensive. On edge devices you do not want to run
hours of Hessian computation every time you deploy.

Second, and more insightful: *in LLM linear layers, the coupling among weights
can overwhelmingly be described by a diagonal matrix*. That is — most channels
are independent of one another; only a few "salient channels" need protection.

AWQ's solution is clean: approximate the Hessian directly as
~diag(X X^T)^α~ — a diagonal matrix whose entries are the per-channel norms of
the input activations. Before quantization, multiply each column of the weight
matrix by this scale; after quantization, divide it back out. This pushes
"important channels" into finer regions of the quantization grid, naturally
yielding higher precision.

Complexity drops from GPTQ's O(d³) to roughly O(d²). Calibration data
requirements shrink substantially. AWQ became the de-facto industrial default
from 2023 to 2025 — the awq_gemm CUDA kernel in vLLM was written specifically
for it.

But AWQ introduces a new hole: *you still need calibration data to estimate
those diagonal norms*. Typically people run a pass over C4 or WikiText, compute
the scales, and bake them into the weights before deployment. If the user's
actual input distribution differs markedly from C4 — code, math, medicine,
multilingual text — AWQ's scales are biased and the model loses accuracy on
out-of-domain tasks.

Worse: *once quantization is done, the original fp16 weights are discarded*,
and there is no way to re-calibrate.

* Layer 5: TTQ — Move the Entire Calibration Step to Inference Time

#+begin_quote
Koike-Akino, Liu, Wang. /TTQ: Activation-Aware Test-Time Quantization to Accelerate LLM Inference On The Fly/. arXiv 2026.03.
#+end_quote

TTQ's entire motivation grows out of AWQ's hole: since calibration data and
real inputs do not match, why not *use the real input as the calibration data*?

The key technical observation: computing AWQ's diagonal norm
~D_i = (||X_i||^p + λ)^α~ is actually very cheap — it is just a p-norm over
each row of the input activations, an O(dT) operation. Relative to the original
linear projection's O(d'dT), the extra cost is only O(1/d' + 3/T). At typical
scales of d' in the thousands and T in the hundreds, *this overhead is
essentially zero*.

TTQ therefore transforms the entire pipeline into:

1. A prompt arrives; obtain the input activation X at each layer for this prompt.
2. Compute D on the fly; compute scale S and zero-point Z on the fly.
3. Quantize the fp16 weights to 4 bits on the fly (quantize W * D^(1/2), then
   divide by D^(-1/2) afterward).
4. Run the layer output using a 4-bit int_matmul kernel.
5. Repeat for the next layer.

Every prompt gets its own bespoke quantized version of the model.
*Calibration data = the current prompt. Out-of-domain problems cease to exist.*

The paper also stacks a low-rank compensation layer: ~W_hat = W_q + B A~,
where the low-rank factors B and A are static (initialized via top-r PCA),
but W_q is still recomputed for every prompt. This combination superficially
resembles QLoRA, but QLoRA freezes W_q and only updates B and A; TTQ does the
opposite — it updates W_q and freezes B and A.

Experimental results: on OPT-350M and Qwen3-1.7B at 3-bit quantization, TTQ
achieves lower perplexity than AWQ calibrated on C4, with *zero offline
calibration data whatsoever*.

The explicit cost: *every prompt requires re-quantizing all weights*. The paper
reports a 5× speedup on GPU because the memory-bandwidth savings from 4-bit
matrix multiplication outweigh the re-quantization overhead. Whether this holds
on edge devices is the hard engineering battle this direction must fight next.

* Frontier and Next Steps

TTQ is a 2026.03 paper and no public follow-up work has directly critiqued it
yet. But several natural directions of extension are visible:

- *The same author team* previously published micro-grained MoE
  (Koike-Akino 2025), which performs "test-time pruning." TTQ is test-time
  quantization. The next step will likely be test-time low-rank adaptation
  combined with test-time mixed precision, completing a full "test-time
  compression" toolbox.

- *Test-time scaling* (the line of work from Muennighoff 2025's s1 onward) has
  been hot for the past two years — spending more compute at inference time,
  via more steps or more samples, to buy performance. TTQ goes in the opposite
  direction — spending less memory at inference time to buy efficiency. Both
  are fundamentally about "moving training-phase decisions into the inference
  phase." This paradigm shift will continue.

- *Hardware co-design*: AWQ became mainstream partly because the awq_gemm CUDA
  kernel was there to support it. For TTQ to reach production, fused
  "quantize + matmul" operators are needed so that the re-quantization overhead
  truly vanishes into the data pipeline. That is the next engineering gate.

- *The end of calibration*: from OBD in 1990 to TTQ in 2026, thirty-five years
  of evolution is really the story of the importance criterion becoming
  progressively lighter and more immediate — from requiring a full Hessian, to
  requiring calibration data, to requiring only the current prompt. Pushing one
  step further: can we *require no statistics at all*? For instance, using the
  structural properties of the weights themselves (symmetry, low-rank spectrum)
  to determine quantization policy a priori? This is the contemporary echo of
  the question LeCun posed in 1989.

* What This River Is Really Changing

On the surface this looks like a story of "quantization algorithms getting
faster and faster." But what is actually happening underneath is a redefinition
of what "deployment" means in the LLM era —

OBD/OBS era: a trained model is fixed; pruning is a one-time surgery, done and
settled.

GPTQ/AWQ era: quantization is a one-time post-processing step, but it requires
"representative data" to perform. The entire industry began accepting "don't
worry about compression during training; compress right before deployment."

TTQ era: the model itself stays high-precision (fp16 weights are retained), but
*every inference re-compresses it from scratch*. Compression is no longer a
step before deployment; it is an embedded component of the inference process
itself.

This shift shares its underlying logic with the rise of test-time compute —
*the inference phase is no longer passively consuming a trained model, but
actively, adaptively, and per-prompt orchestrating a dynamic process*. The
algorithm no longer serves the model; the model has become an artifact that the
algorithm sculpts anew at inference time.
