* 问题之河

长上下文 LLM 推理时，几十万 token 的 KV cache 把显存吃干净。能不能边推理边*扔掉*一部分 KV，让显存撑下去——而又不让模型变笨？

这个问题难在哪？不是"扔多少"，而是*扔谁*。每个 token 在过去的"重要性"怎么估？最朴素的办法是看注意力权重高的就留着——但这忽略了一件关键的事：你扔掉一个 token 后，剩下 token 的注意力会*重新分配*。原来注意力低的可能突然变重要，原来"看起来不重要"的可能就是承担了关键的语义聚合。一代代人前赴后继，本质上都是在回答同一个问题：*要不要丢一个 token，到底应该看什么信号*？

* 演化叙事

下面这条线，每一篇都在回答上一篇没回答好的同一个问题：*怎么估计一个 KV pair 的重要性，使得删掉它对模型输出的伤害最小*。

** 2017 | Transformer: KV cache 是后来才发现的"债"

*** 它看到了什么问题

RNN 一步一步算，长序列上根本跑不起来。

*** 它的解法

砍掉循环，只留注意力。每个位置可以并行算 query/key/value，训练吞吐量直接上一个台阶。推理时为了不重复算前面 token 的 key/value，引入 KV cache——一旦算过，就缓存起来给后面的 query 用。

*** 为什么这招管用（以及哪里埋了新坑）

并行训练 + 缓存复用是 Transformer 之所以能 scale 的根。但 KV cache 是一笔*线性增长的债*：每多一个 token，缓存就多一份 key + value。当时谁也没在意——上下文也就几千 token。等到 GPT-4、Llama-3.1 把上下文推到 128k，债主上门了：一个 70B 模型在 128k 上下文下，单纯 KV cache 就能吃掉几百 GB 显存。问题从"训练吞吐"变成了"推理时显存撑不撑得住"。

** 2023 | StreamingLLM: 既然撑不住，就只留两头

*** 它看到了什么问题

最 naive 的办法是固定窗口——只保留最近 N 个 token 的 KV。听起来合理，实际跑出来 perplexity 直接崩。为什么？因为模型对*最开头几个 token*的注意力出奇地高，哪怕这几个 token 在语义上是 "the"、"<s>" 这种废词。把它们扔掉，softmax 的归一化失衡，整个分布就坏了。

*** 它的解法

承认这件事，把开头的几个 token 永远留住（命名为 "attention sink"），再加上一个滑动窗口保留最近的内容。两头都留，中间不要。

*** 为什么这招管用（以及哪里埋了新坑）

第一次让长上下文推理在显存有限的情况下跑得通。但代价很明显：*中段一律丢*。如果用户的问题答案就藏在中段（比如一份 80k 字报告里的某一段事实），模型就成了金鱼脑。规则越简单，遗漏的就越关键。

** 2023 | H2O: 中段也有"重要 token"，按累积注意力挑

*** 它看到了什么问题

StreamingLLM 的"两头规则"太粗暴。事实上，过去 token 里有一小撮被反复 attend——这些就是 "heavy hitters"。它们既不在开头，也不在最近，但模型每生成几步就要回头看一眼。一刀切丢中段，等于把这些 heavy hitters 一起埋了。

*** 它的解法

不再用固定模式，改成*动态*：每解码一步，把每个 token 至今为止被 attend 的累积权重累加起来；超过预算时，淘汰累积权重最低的那些。

*** 为什么这招管用（以及哪里埋了新坑）

第一次把"重要性"做成了数据驱动的信号。但"累积注意力"是一种*回看*——它说的是"过去这个 token 有多被看过"，没说"未来的 query 还会不会需要它"。在多轮对话或者长生成里，话题一转，过去的 heavy hitters 反而拖后腿。

** 2024 | SnapKV: 用 observation window 预测未来

*** 它看到了什么问题

H2O 的累积分数太"过去时态"。在 prefill 阶段一次性做剪枝时，更合理的代理信号是：*最近这一小段输入的 query，在 attend 哪些 token*？因为"最近"通常意味着"接下来生成时也会继续在这些位置附近发问"。

*** 它的解法

在 prompt 的最后保留一个 observation window（比如 32 个 token），把这些 query 对所有历史 token 的注意力做 pooling，然后取 top-K 作为要保留的 KV。等于让"最近的几个 query"先去"试一遍"，挑出来的就是"未来很可能还要用的"。

*** 为什么这招管用（以及哪里埋了新坑）

预测性质更强了。但根本上还是同一个套路：*用 attention weight 当作 importance 的代理*。两个被遮掉的事实没被处理：
1. *Value 向量根本没参与*。两个 token 的注意力一样高，但 value 模长差十倍，对输出的贡献完全不同——SnapKV 看不见。
2. *删除会引发重新分配*。删一个高注意力 token，剩下 token 的 softmax 会把那部分权重摊回去；如果摊得过来，损失就不大；摊不过来，输出才会坏。SnapKV 假设"每个 token 的影响相互独立"，但 softmax 不允许这样假设。

** 2024 | AdaKV: 每个 head 单独算预算，用输出 L1 误差的上界

*** 它看到了什么问题

之前所有方法都把"预算分配"和"挑哪个"混在一起做。AdaKV 把这两件事拆开：注意到不同注意力头的稀疏程度差异巨大，统一给每个 head 同样的预算是浪费。

*** 它的解法

为每个 head 推导一个 *output L1 loss 的上界*，按这个上界来分配预算——容量给"删了之后输出会更坏"的 head。挑哪个 token 留则继续用注意力权重排序。

*** 为什么这招管用（以及哪里埋了新坑）

第一次把"输出误差"明确写进框架。但它只用在*预算分配*上，不用在*选谁*上——选谁还是看 attention weight。而且 L1 是上界，不是真正的输出误差。这就留下了一个明摆着的下一步：能不能把"输出重建误差"直接做成 *eviction indicator 本身*？

** 2026 | ReST-KV: 把"删了之后输出会差多少"直接写成 indicator

*** 它看到了什么问题

之前所有方法的共同盲点：用 *attention weight* 当 *importance* 的代理，但真正的目标其实是 *输出*——你不在乎注意力分布像不像原来，你在乎下游的 MHA 输出还像不像原来。两件事不是一回事，因为：
- 注意力删一项后会 *重新归一化*，剩下的项分到的权重被等比例放大；
- Value 向量的方向决定了"放大之后"输出会被拉到哪儿；
- 注意力小但 value 模长大的 token，删掉之后输出可能崩得很厉害。

*** 它的解法

把 KV eviction 重新表述成一个 *优化问题*：在预算 B 之内，找一个 KV 子集，让选中子集算出来的 MHA 输出，和保留全部 KV 算出来的 MHA 输出，*L2 距离最小*。

直接求解太贵，所以走贪心：算每个 KV pair 单独被删掉后会引起多大的输出误差，按这个误差从大到小排序。经过推导，每个 token 的 importance 化简成两项的乘积（见论文 Eq. 7）：
1. =A[n] / (1 - A[n])= ——一个 *非线性放大器*。保留"高注意力的更重要"这一直觉，但用了一个单调上凸的曲线，能更好地区分"接近"的高注意力 token。
2. =||MHA(全) - v_n W_O||_2= —— *重分配敏感度*。如果删掉这个 token，剩下的 KV 能不能在加权平均里把它"代偿"回来。代偿不回来的，才是真正重要的。

光有更准的 indicator 还不够。作者继续往前一步：观察到这个 indicator *在时间上和空间上都在抖*——同一个 token 的重要性在相邻解码步之间会跳变；空间上相邻位置的重要性模式也会偏移。于是又加了两个平滑：
- *时间上*: 指数移动平均（EMA），让最近几步的 indicator 衰减加权，避免单步噪声 dominate。
- *空间上*: adaptive window-based smoothing，对附近位置做窗口平均，并自适应调整窗口偏移。

*** 为什么这招管用（以及哪里埋了新坑）

它第一次把"挑谁留"这件事从"基于代理信号的排序"升级到"基于真实目标的优化"。LongBench +2.58%、RULER +15.2%，128k 上下文解码延迟降 10.61 倍——证明这条路是对的。

留下的坑：
- 输出重建是 *逐层、逐 head* 算的，不考虑后续层的级联误差——一层的小错可能在更高层被放大。
- 贪心选择是一阶近似（local linearity assumption），删两个 token 的联合损失不一定等于分别删两个 token 的损失之和。
- EMA 和窗口平滑引入了新的超参数（衰减系数、窗口大小），通用性有待跨任务验证。

* 前沿延伸

ReST-KV 发表于 2026 年 5 月（ICLR 2026），距今不到一个月，目前还看不到正式的"批判它"的后续论文。沿着同一条问题线，最可能的下一步方向：

1. *跨层联合优化*: 既然单层的 reconstruction 有效，下一步自然是把 N 层的 KV 选择联合起来求解，把级联误差也吃进 indicator——但计算代价是 N 倍。
2. *Indicator + quantization 协同*: KV cache 量化（如 KIVI 系列）和 eviction 是互补的两条线，把"留哪个"和"留下来按几 bit 存"放在同一个优化框架里，可能进一步压低显存。
3. *训练感知的 eviction*: 目前所有方法都是 training-free。下一步可能让模型在训练时就知道"未来推理时部分 KV 会被丢"，从而学到对丢失更鲁棒的表示。

不确定项已标注——以上属推测，不是已有论文。

* 洞见

这条线表面上是"KV cache eviction 越做越好"，背后真正发生的是一次*认知坐标的位移*：

1. *从"靠经验规则"到"靠数据信号"再到"靠目标函数"*。StreamingLLM 用直觉（开头很重要+最近很重要）。H2O/SnapKV 用经验信号（注意力权重）。ReST-KV 终于把"我们到底想优化什么"写成了一个方程：保住 MHA 的输出。每往前一步，离"我们关心的东西"就近一步。
2. *代理变量是有寿命的*。注意力权重作为重要性代理用了三四年，最后被一个非常朴素的观察捅破：删了之后会重新分配，所以静态的权重根本不是删后输出的预测器。任何"用 A 作为 B 的代理"的研究都该被这个例子提醒：*代理用着用着就忘了它只是代理*。
3. *把"删除"这件事当成一个 perturbation 来建模*——ReST-KV 等价于在做一个 leave-one-out 的影响函数估计。这套思路（剪枝、量化、token drop 都是 perturbation）的统一视角，可能比 KV cache 这件具体的事走得更远。

下一步最可能往哪走？看上面的"前沿延伸"——*跨层联合优化*是最自然的延续，但*把"丢 KV"这件事融进训练目标*才是真正的范式跃迁。前者是把同一思路推到极致，后者是换战场。

* 启发

如果你在做"用代理信号代替真实目标"的研究——不只是 KV cache，比如知识蒸馏、特征选择、剪枝、检索——这条线给的方法论非常直接：

- *显式写出"我们最终关心什么"的损失函数*。不管它多难算，先写出来。然后问：我现在用的信号，离它有多远？
- *找代理失效的边界条件*。注意力权重作为重要性代理在什么时候失效？当一个 token 注意力低但 value 模长大时。这个反例就是下一篇论文的种子。
- *把"小扰动后的目标变化"作为 indicator*，往往比"当前状态"更准。这是 ReST-KV 借自经典的 Optimal Brain Damage / Molchanov 剪枝路线——并不新，但用对地方就能跳一代。

具体到工程上：这意味着如果你在工程里做 long-context LLM serving，现在可以考虑把 SnapKV/H2O 类的 eviction 换成 reconstruction-based 的；ReST-KV 的代码已经开源（=github.com/an-yongqi/rest-kv=），在低预算 (n=64) 场景下的提升尤其明显，是性价比最高的切换。
