* 问题之河

一段 36k token 的对话，在 LLM 推理时要占多少显存？*16GB 起步*——这还只是 KV cache，模型权重另算。然后你按下回车，模型开始一个 token 一个 token 地往外吐，每吐一个，就得把这 16GB 重新过一遍。生成速度被显存带宽锁死，长一点的 CoT 推理动辄等几十秒一个 token。

这条河要回答的问题就一个：*KV cache 太大了，怎么办？*

听起来很笨——不就是个缓存吗，删了就行。但这正是难的地方。LLM 的注意力是动态的：你以为没用的 token，下一句话可能就用上了。你以为重要的 token，再过 1000 步就过气了。"删什么、留什么、放哪里"，这三个问题被七八年的论文反复打磨，每一次都觉得"这次终于解决了"，然后下一篇就指着鼻子说"你这么干在 XX 场景下就崩"。

IceCache 是这条河流的当前最新一站。要看懂它为什么这么设计，得倒着挖到河源。

* 演化叙事

** 2023 | StreamingLLM: 注意力都在头和尾，把中间扔掉就行

*** 它看到了什么问题

最早暴力的做法是"窗口注意力"——只看最近 N 个 token，超出就丢。但这有一个反直觉的现象：模型一旦丢掉最开头那几个 token，整个生成就崩了，开始胡言乱语。

Xiao 他们盯着注意力图看，发现了 *attention sink*：开头那几个 token 拿走了大量注意力权重，不管你问什么。它们像锚一样，把后面的注意力分布稳住。删掉锚，船就漂走了。

*** 它的解法

简单到家：保留开头的 4 个 sink token + 最近的滑动窗口，中间全删。模型不会崩了，显存也省了。

*** 为什么这招管用（以及哪里埋了新坑）

它管用是因为对很多日常对话，"中间内容"确实不太重要——你聊到一半，前面具体说过什么话，模型确实不太需要逐字回忆。

但坑就在这里：*这是个静态、位置驱动的策略*。如果你做长文档问答，关键信息恰好在中间某段，删掉就再也调不回来了。它把"什么重要"等同于"位置在哪里"，这个等式在很多场景下不成立。

** 2024 | H2O: 按"实际被用的次数"删，比按位置删聪明

*** 它看到了什么问题

StreamingLLM 等于说"重要 = 头+尾"，但这显然太粗糙。一个 token 重要不重要，应该看它*实际上*被多少后续 token 注意力盯过——盯得多，说明它在生成过程中频繁被引用，那就是关键信息。

*** 它的解法

H2O 给每个 token 维护一个"被累积关注度"，每生成一步就更新。删除的时候，把累积分数最低的踢出去。这些幸存者叫 *Heavy Hitters*——它们是真正在干活的少数。

*** 为什么这招管用（以及哪里埋了新坑）

它解锁了一个观察：注意力是稀疏的，少数 token 真的扛起了大头。所以 5%-20% 的 token 留下来，质量损失就很小。

但新坑也来了：*这仍是一次性、不可逆的决策*。你在第 100 步觉得某个 token 没人理，把它扔了；可在第 1000 步，话题转了一圈又回到它身上——这时候你想找也找不回来了。长生成、多轮对话、CoT 推理，这种"过气又翻红"的情况非常普遍。

** 2023 | PagedAttention/vLLM: 借鉴操作系统，把 KV cache 当虚拟内存管

*** 它看到了什么问题

这是与 eviction 路线正交的一支。Kwon 他们没有问"删什么"，他们问的是"怎么放"。

当时所有推理框架把 KV cache 当作一整块连续显存来分配。问题是：每个请求长度不一样，分配的时候要预留最大长度，造成大量碎片和浪费。一个 80GB 的 A100 实际能用上的可能不到 30%。

*** 它的解法

照搬操作系统的虚拟内存：把 KV cache 切成固定大小的 *页*（比如 16 个 token 一页），用一张页表把"逻辑位置"映射到"物理页"。要扩容就分配新页，要回收就把页还回去，零碎片。

*** 为什么这招管用（以及哪里埋了新坑）

这是从"原始数组"进化到"虚拟内存"。一下子吞吐量翻了几倍，成了所有现代推理框架的标配。

但请注意：*它解决了管理问题，没解决增长问题*。Cache 还是会越涨越大，PagedAttention 只是涨得更整齐。要想真正省显存，还得叠加 eviction 或别的招数。

不过它埋下了一颗种子：*页是个好抽象*。后面的人会发现，页不仅适合做内存管理，还适合做"批量决策"——一次决定一整页要不要算/要不要载入。

** 2024 ICML | Quest: 给每页打个分，每次只算 top-k 页

*** 它看到了什么问题

Quest 把 H2O 和 PagedAttention 的两条线合到了一起。

H2O 那套"基于历史决定丢什么"的做法有两个缺陷：(1) 一次决定终身不变；(2) 决定基于*过去的 query*，不是*当前的 query*。Quest 看出来：如果我能为*这一次*的 query 临时算一下"哪些 token 重要"，就不用提前删，也不用押宝。

但你不可能为每个 token 都算一次相关性——那等于把 attention 算了两遍。这时 PagedAttention 救场了：*以页为单位算*，便宜得多。

*** 它的解法

每页存一个简单的"摘要"：这页里所有 key 向量在每个维度上的 min 和 max。给定一个新 query q，可以用 q 和这个 (min, max) 算一个 *上界*——表示这页里任何 token 与 q 的最大可能注意力分数。

然后按上界排序，只对 top-k 页（比如 128 页）真正算 attention，其他页跳过。

*** 为什么这招管用（以及哪里埋了新坑）

聪明在于：*用极便宜的元数据换攻击面*。每页的 (min, max) 几乎不占空间，但提供了一个不会漏掉重要页的安全上界。Self-attention 加速最高 7 倍，端到端 2 倍多。

但还是有坑：*显存没省下来*。所有 KV 都在 GPU 上，Quest 省的是算力，不是显存。你 36k context、200k context 该塞不下还是塞不下。

** 2024 NeurIPS | ArkVale: 全放 CPU，按 query 取回 GPU

*** 它看到了什么问题

Quest 解决了"该算谁"，但没解决"该放哪"。ArkVale 接着问：既然每次只用 top-k 页，那其他页能不能放 CPU 内存？需要的时候再拷回来。

CPU 内存几百 GB 都不嫌多，瓶颈反而变成 PCIe 带宽——所以"准确选页"变得更重要了。每错选一页都要付出 PCIe 传输代价。

*** 它的解法

继承 Quest 的页摘要 + top-k 选页思路，但加两个新东西：
1. *Offloading*：完整 KV 留 CPU，GPU 只留当前用到的若干页
2. *Recall*：被赶出去的页不是永久丢弃，而是从 CPU 召回。这是对 H2O "一次性决策"问题的修正

*** 为什么这招管用（以及哪里埋了新坑）

它把"够大的内存容量"和"够快的访问速度"分开放：CPU 给容量，GPU 给速度，PCIe 做调度。理论上你可以塞下任意长的 context。

但 ArkVale 留下了一个看起来不起眼、实际致命的坑：*页是按原始 token 顺序拼的*。第 0-15 个 token 一页，16-31 一页……，按时间顺序排进去。

为什么这是坑？想象你在一份 50 页的法律合同里搜"违约条款"。如果这份合同的"违约相关内容"散落在第 3, 17, 23, 41 段，你只能把这 4 段所在的*整页*都加载回来——可每一页里只有 1-2 个 token 真正相关，剩下的全是无关 noise，白白消耗 PCIe 带宽和 GPU 显存预算。

页选对了，token *命中率* 还是很低。

** 2026 ICLR | IceCache: 把页按"意思"重新拼，让相关 token 自动聚到一起

*** 它看到了什么问题

IceCache 是这一支最关键的认知反转。前面所有方法——StreamingLLM, H2O, Quest, ArkVale——都默认"页是按时间顺序拼的"。这个假设从来没被质疑过，因为 KV 就是顺序生成的，按生成顺序存"很自然"。

但"自然"不等于"高效"。Quest/ArkVale 的命中率瓶颈，本质就在这个隐含假设上。

IceCache 问的是：*页凭什么必须按时间顺序拼？我能不能按意思拼？* 把语义相近的 token 放到同一页里，那么一个 query 来的时候，相关 token 就*天然*聚集在少数几页上，每页几乎全是 hit，没有 noise。

*** 它的解法

构造一棵 *DCI-tree* 给所有 key embedding 做层次聚类（DCI 是 Li & Malik 2017 的一个高维 KNN 算法，作者之前在 sparse attention 里用过）：
- 叶子节点 = 一个内存页，存语义相近的 token
- 上层节点 = 子节点的"代表"
- 查询时从根往下走，到叶子就是 top-k 相关页

prefill 阶段就建好树，decoding 阶段每来一个新 token，把它*插入*正确的叶子（树是动态可更新的）。

*** 为什么这招管用（以及哪里埋了新坑）

整个流水线的*耦合点*变了：
- ArkVale: 选页 → 一页里只有几个 hit → 浪费带宽和预算
- IceCache: 选页 → 一页里全是 hit → 256 token 预算就能保住 99% 精度

数据说话：256 token 预算保 99% 原始精度，比其他 offloading 方法只用 25% 的预算就能达到甚至超过它们的延迟和精度。

新坑（也是河流接下来的方向）：*页这个粒度本身仍然偏粗*。即便语义聚类拼得再好，一页 16 个 token，里面也未必每个都是这次 query 真正需要的。下一代方法（FIER 等）开始往 *token 级检索* 推进，但 token 级又面临元数据爆炸的问题。河又往前流。

* 前沿延伸

IceCache 是 2026 年 4 月的工作，紧接着出现了两条延伸路线：

** SemantiCache (2026): 把"语义聚类"思路用到 KV 压缩

不做 offloading，做*压缩*。把 cache 按语义边界（标点、段落）切块，块内再做贪心种子聚类，把多个相似 token 合并成一个"语义核"——存得更少，但保住语义。和 IceCache 的差异：IceCache 是"全留 + 选着用"，SemantiCache 是"合并 + 精简存"。

** FIER (2025): 索性别按页选了，按 token 选

FIER 直接挑战页这个粒度："不管你怎么拼页，里面总有 noise。"它做 token 级的细粒度检索。代价是元数据更大、调度更复杂。这是 IceCache "页内可能仍有 noise" 这个坑的直接延续。

截至 2026 年 5 月，IceCache 仍是"语义聚类 + 分页 + offloading"这条具体路线上的 SOTA。

* 洞见

** 这条河真正在演化什么

表面上看是"KV cache 压缩越做越好"。但底下有个更深的变化轨迹：*管理 KV cache 的"原子单位"在不断重新定义*。

- 第一代 (StreamingLLM, H2O)：原子单位是 *token*。基于"这个 token 该不该留"做决策。
- 第二代 (PagedAttention)：原子单位是 *页*。"页"作为内存管理单位被引入。
- 第三代 (Quest, ArkVale)：原子单位是 *页*，但页同时承担了"选择粒度"的角色——一次决定整页用不用。
- 第四代 (IceCache)：页的*组成方式*被重新定义。从"时间相邻"变成"语义相邻"。
- 第五代 (FIER, 萌芽中)：发现页这个粒度本身有局限，向 token 级回摆，但带着前面学到的所有工程经验。

每一步都是在重新回答："*管理什么*和*以多大的粒度管理*"。

** 一个看似不起眼的隐含假设，可能是整条河的瓶颈

IceCache 之所以成立，关键不是 DCI-tree 多巧妙，而是它*意识到一个被默认了很多年的隐含假设*——"页按时间顺序拼"。

这个假设从 2023 年 PagedAttention 写进去，到 2024 年 Quest/ArkVale 完全继承，没人质疑。直到 IceCache 把它拿出来问"凭什么"。

整条河的隐藏教训：*当一个领域的多篇 SOTA 都共享某个不成文的约定时，那个约定往往就是下一个突破点*。

** 下一步往哪走

往两个方向走，而且这两个方向已经有人走了：

1. *更细的粒度* (FIER 这条线)：page-level → token-level → 甚至 sub-token level？瓶颈是元数据膨胀，需要更紧凑的 token 指纹。
2. *更智能的组织* (SemantiCache 这条线)：与其聚类后选择，不如*合并*相似 token，从根上减少需要管理的对象数量。

更远一点，可能会出现"语义聚类 + token 级检索 + 合并压缩"三合一的方法。这条河还远没流到大海。

* 启发

** 对找问题的启发

读这条河最大的体会是：*真正的突破往往在"已经被解决"的地方*。

ArkVale 在 NeurIPS 2024 拿到非常好的成绩，所有人都说"KV offloading 这条线差不多到顶了"。然后 IceCache 来了，发现瓶颈根本不在"选哪些页"，而在"页是怎么组成的"——一个谁都没注意的环节。

这意味着你*可以*：
- 找一个有共识的领域，问"大家共享了什么没被质疑过的假设"——那个假设很可能就是你的论文。
- 看 baseline 们的 ablation：如果某一组都"自然地"做了同一件事，但没人解释为什么，那里就有金子。

** 对做架构设计的启发

注意四代方法的*耦合点变化*：

每代方法都看似在"小修小补"，但实际上把*关键耦合点*换了一个位置：
- StreamingLLM/H2O: 耦合点在"删除策略"
- PagedAttention: 耦合点在"内存布局"
- Quest/ArkVale: 耦合点在"选页策略"
- IceCache: 耦合点在"组页策略"

如果你在做系统设计，可以*主动列出*"我的设计里哪些地方是和别人耦合的，哪些是我独立决定的"——后者就是你未来的护城河，也可能是别人的突破点。

** 对读论文的启发

读 IceCache 时，最容易犯的错是被 DCI-tree 这个算法名词唬住，去研究 Li & Malik 2017 那套 KNN 数学。但 *DCI-tree 在这篇里只是一个手段*——核心 insight 是"页按语义拼"。换成别的 ANN 算法（HNSW、IVF 都行）也成立。

读论文要分清*核心 insight* 和*实现手段*。前者是论文真正的贡献，后者是工程选择。混在一起读，你会觉得每篇论文都很神秘；分开读，你会发现 90% 的论文其实就改了一个 insight。
