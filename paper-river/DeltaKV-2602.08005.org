* 一句话

LLM 推理时那个越聊越大、最后挤爆显存的 KV 缓存，七年来人们一直在想怎么"瘦身"——先是删，再是挑着删，再是压成低位数，再是只算一部分，DeltaKV 走到了最后一步：不存原值，只存"和老朋友的差值"。

* 演化史：一个问题怎么长大的

** 故事的开头：那个没人在意的小副产品（2017，Transformer）

2017 年那篇 *Attention is All You Need* 火了之后，几乎所有人讨论的都是"注意力机制有多牛"。但论文里其实埋了一个不起眼的细节：自回归生成时，为了避免每生成一个新词就把前面所有词重算一遍 Q、K、V，工程师顺手做了一个优化——把已经算过的 Key 和 Value 缓存下来，下一步直接拿来用。

这就是 *KV cache*。

那会儿没人觉得这是个问题。模型上下文最多 512、2048 个 token，缓存一下几兆显存，不痛不痒。论文里甚至没专门讨论它，就是个"顺手的工程小聪明"。

但这里埋了一根雷管：*KV cache 的大小，随上下文长度线性增长*。

七年后这根雷管炸了。

** 雷管引爆：长上下文时代（2023）

到 2023 年，大家开始喂模型几万、几十万 token 的文档。一算账，发现 KV cache 比模型权重本身还大——一个 7B 模型在 128K 上下文下，KV 占的显存可能是模型权重的两三倍。

于是第一个反应非常朴素：*删*。

*** StreamingLLM（2023）：最粗暴的删法

它的观察是：注意力机制有个奇怪现象，开头那几个 token（哪怕是无意义的 BOS）总是被疯狂关注，叫"注意力沉淀（attention sink）"。同时，最近几个 token 也总有用。

所以方案是：*只留开头四个 + 最近的滑动窗口，中间全删*。

这管用——至少模型不会崩。但你只要稍微想想就知道问题：中间那一大段被删的内容，万一有重要信息呢？比如你贴了一个长合同让模型回答第三章某个条款，结果第三章正好在被删的范围里。完蛋。

它解决了显存爆炸，但是用"装作没看见"的方式解决的。

*** H2O（2023）：删得聪明一点

H2O 的反思是：删可以，但别瞎删，按"重要性"删。

它的观察更精细：每个 token 历史上被后续 token 关注的总和（叫 H2，Heavy Hitter），可以衡量它有多重要。注意力分数低的，说明大家都不爱搭理它，删了影响不大。

所以方案变成：*按累计注意力分数排序，分数低的优先删*。

更聪明，但又埋了新雷管：*一旦删了就找不回来*。当前看起来无关紧要，不代表未来用不到。一个最经典的失败场景叫 Retrieval KV：你前面定义了"X = 42"，模型当时没怎么关注（因为还没到用的时候），分数低，被 H2O 删了。后面问 X 等于几，模型彻底懵了。论文里测出来 SnapKV（H2O 同类）在这类任务上准确率掉到 0.4，几乎是瞎猜。

H2O 把删的策略从"按位置"变成"按重要性"，但没改变*删=不可逆*这件事。

** 思路一转：不删了，挑着用（2024，Quest）

Quest 是第一个跳出"删不删"框架的工作。它的反思非常关键：

*与其纠结删谁不删谁，不如全部留着，每次生成新 token 时动态挑一小部分参与计算*。

类比一下：H2O 是入学就把成绩差的学生开除，开除了就再也回不来；Quest 是所有人都留校，但每节课老师只点名叫几个有用的来回答问题——下节课需要谁，下节课再叫。

这是认知上的一次大跃迁：从"压缩=丢弃"，转到"压缩=选择性使用"。

但 Quest 留下一个非常现实的问题：*它没真的减少显存*。所有 KV 还是得存着，只是计算量降了。显存是长上下文最硬的瓶颈，Quest 没碰这块。

工程师把 Quest 这类方案变成"那就把 KV 卸载到 CPU 内存或者 SSD 吧"——结果遇到下一个瓶颈：PCIe 带宽。每次决定要用哪些 token，就要把它们从内存搬回 GPU，搬运成本压死延迟。

** 工程化：让"选择性使用"真的能跑（2025，OmniKV）

OmniKV 是这条线的工程总结者。它做的事情大致是：

- 设计一套"过滤层"架构，决定哪些 KV 该留 GPU、哪些卸载到 CPU
- 把整个流程包成一个能塞进 vLLM 这种生产推理引擎的模块

它解决了 Quest "理论上很美但工程上跑不动"的问题。在 LongBench 这类长上下文测评上，OmniKV 终于让"选择性使用"做到了既不掉精度又能跑得动。

但 OmniKV 自己最大的弱点写在论文里都是大白话：*GPU 显存占用还是 100%*。因为它选择"不删任何 KV"，全部存着，只是聪明地决定哪些放 GPU 哪些放 CPU。一个 128K 上下文跑下来，显存依然紧张。

这就是 DeltaKV 接手前的局面。

** 终点：既然你和过去那么像，干脆只存差值（2026，DeltaKV）

DeltaKV 的作者盯着 OmniKV 这个"显存还是降不下来"的尾巴看，做了两个观察：

*观察一*：把每个 token 的 K 和 V 向量两两算相似度，发现 *60% 以上的"高度相似 token 对"距离超过 16 个位置*。换句话说：一个 token 在历史上经常能找到一个跨度很远的"老朋友"，长得几乎一模一样。

*观察二*：所有 token 的 KV 向量里，存在一些共享的"主成分方向"——就像一群人的脸，鼻子嘴巴眼睛的大致位置都差不多，真正区分人的是细微差异。

这两个观察合起来推出一个很自然的想法：

*与其完整存储每个 token 的 KV（耗显存），不如只存它和某个"老朋友"的差（很小）*。

具体怎么做：

- 每隔固定步数 s 选一个 token，叫"参考 token"，KV 完整保留
- 剩下的"压缩 token"，对每一个做这件事：
  1. 用 L2 距离在历史里找它最像的 k 个参考 token
  2. 取这几个参考的平均向量
  3. 经过一个 MLP 把当前 KV 和这个参考均值都压到低维空间
  4. *只存两者的差值*——这个差值很小，编码起来很短
- 用的时候，把差值解出来，加回参考均值，就还原出原始 KV

类比：你写日记，过去七年每天都记。如果今天的日记和去年三月某天高度相似（都是"早上下雨，喝咖啡，读书"），你可以不写完整一篇，只写"参见 2025-03-14，多了一杯咖啡"。

效果：KV 缓存压到原来的 29%（接近三分之一），LongBench / SCBench / AIME 这些长上下文测评几乎不掉精度，配套写了 Sparse-vLLM 这套推理引擎来跑稀疏不规则的 KV 布局，最终长上下文吞吐量比原生 vLLM 快 2 倍。

它的精妙在哪：它把 OmniKV "全留着、聪明地用" 改成了 *"只留参考、其他用差值表示"*。既不像 H2O 那样不可逆地丢，也不像 Quest/OmniKV 那样不降显存——而是利用"KV 之间高度冗余"这个被忽视已久的统计事实，做了一次真正意义上的*信息压缩*而不是*位置压缩*。

* 洞见

** 这条线背后真正在发生什么变化

表面看，是 KV cache 压缩技术从 30% 进化到 70%、再到 90% 的"百分比游戏"。但真正发生的认知转变有两层：

*第一层：从"位置思维"到"信息思维"*。早期所有方法都在问"哪个位置的 token 该删？"——开头？结尾？注意力低的？这是一种"位置=信息"的暴力假设。DeltaKV 第一次把问题改成"信息熵在哪里？"——发现大量信息其实是冗余的，可以用更短的编码表示。这是从"位置压缩"到"信息压缩"的本质升级。

*第二层：从"删除范式"到"压缩范式"*。H2O 时代大家都在做减法：删谁。Quest 时代开始做选择：用谁。DeltaKV 是第一个真正在做"编码"：怎么用更少的比特表示同样的信息。这其实是把经典数据压缩（霍夫曼编码、差分编码、字典编码）的思想迁移到了 KV cache 这个语义对象上。

注意 DeltaKV 的名字里那个 "Delta"——它直接致敬的是差分编码（delta encoding），一个 60 年代就有的数据压缩老把戏。LLM 这边花了七年，才让人想起来"差分编码可以用在语义向量上"。

** 下一步最可能往哪走

往两个方向：

*方向一*：把"参考 token"也压了。DeltaKV 现在还要保留一部分完整的参考 token，那这些参考 token 之间也很相似，能不能递归地用差值表示？想象一个层次化的差值树：少数根节点存完整 KV，大量叶子节点存层层差值。这是经典的字典学习/向量量化思路。

*方向二*：把"训练时学一个压缩器"做得更智能。DeltaKV 现在是离线训一个 MLP 压缩器+解压器。下一步可能是 *运行时自适应*：根据当前任务、当前文档分布，动态调整压缩字典。类似神经网络里的 prompt tuning 但应用在 KV 编码上。

更野的方向是：等大家把 KV cache 压到 5% 以下时，会不会出现"语义级 KV cache"——不存向量，存语义索引，用的时候再生成？那时候 KV cache 这个概念本身就被颠覆了。

* 诚实标注

- DeltaKV → OmniKV：*确认*。论文里 +DeltaKV 总是叠加在 OmniKV 上做实验，明确说"complement sparse attention methods such as OmniKV"。
- OmniKV → Quest：*合理推测*。两者都属于"动态选择 KV"流派，OmniKV 是 Quest 思路的工程化延伸，但 OmniKV 论文里是否直接把 Quest 作为唯一前序，未在本次溯源中确认。
- Quest → H2O：*强相关*。Quest 的核心反思就是"H2O 这类静态删除会丢关键 token"，这是公开记录里的标准对比。
- H2O → StreamingLLM：*同期并行*。严格说两者是同年（2023）并行提出的，不是 H2O 改 StreamingLLM；这里按"由粗到精"的认知演化排列。
- 时间标签里的 2024、2025、2026 取自相关工作引用的发表年份，部分属于 arXiv 预印本时间，不一定等同正式会议年份。
- "差分编码"作为思想来源的归因是叙事性的，DeltaKV 论文本身并未明确引用经典数据压缩文献。
