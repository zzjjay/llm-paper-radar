* 问题之河

长上下文大模型推理时，最贵的不是算力，是显存里那份 KV cache。你让模型读 50 万 token 的文档，它就得把这 50 万个 token 的 Key 和 Value 全摁在 GPU 显存里，因为下一个字可能要回头看其中任何一个。序列每长一倍，这份 cache 就大一倍——一台 8 卡 H20 服务器，光 KV cache 就能吃掉几十 GB，比模型权重还沉。

整条研究线，七八年，都在跟这条 *线性增长* 死磕。但磕的方式一直在变，而且变的方向很有意思：

- 一开始大家想的是"每个 token 存得小一点"（压缩）；
- 然后是"算注意力时少看几个 token"（稀疏）；
- 再然后是"干脆只算重要的那几个"（可训练稀疏）；
- 到这篇，问题被重新问了一遍：*为什么解码当前这个字，要把整段历史都放在 GPU 里？*

这篇论文（FlashMemory-DeepSeek-V4，方法叫 Lookahead Sparse Attention, LSA）的尖锐之处在于一个被它放大的经验观察：*超过 64K 上下文的真实请求里，90% 以上只靠最后 8K token 就能正确回答*。也就是说，绝大多数显存压在了"此刻根本用不到的历史"上。那能不能把历史挪到 CPU，需要时再 *预测性地* 捞回来？

这就把一个"注意力机制问题"，悄悄变成了一个"检索系统问题"。

* 演化叙事

** 2024 | MLA: 不问"砍几个头"，问"为什么存全维 KV"

*** 它看到了什么问题

标准多头注意力（MHA）推理时，每个 token 都要把所有头的 Key、Value 缓存起来——下一个字要拿当前 query 去点乘所有历史 K。序列一长，这份 cache 线性爆炸。

前人（MQA、GQA）的思路是"让多个头共享 K/V"，少存几份。但共享是有损的：头本来各看各的位置，强行合并就掉点。这是一条"用质量换显存"的路。

*** 它的解法

DeepSeek 退一步问了个不同的问题：*我们干嘛要存完整的高维 K 和 V？*

MLA 把每个 token 的 K、V 联合投影到一个低维 latent 向量 c（维度 d_c 远小于 n_h·d_h），cache 里 *只存这个 latent*。用到时再用上投影矩阵把它解压回 K、V。配合"权重吸收"技巧（解压矩阵可以提前融进 Q 投影和输出投影），解压几乎不额外花钱。

*** 为什么这招管用（以及哪里埋了新坑）

低秩压缩抓住了一个事实：K/V 里真正有用的信息维度，远低于它们的名义维度。所以压成 latent 不太掉信息，反而因为表示更紧凑，实测质量还略超 MHA，KV cache 直接降 93%，吞吐涨到 5.76 倍。

但请注意 MLA 改的是 *每个 token 存多大*——它把斜率压低了，却没改变"每个 token 都要存"这件事。cache 仍然是 O(L)。读 50 万 token，你还是有 50 万份（小号）latent 压在 GPU 里。线性增长这堵墙，MLA 把它推远了，没推倒。

** 2023-2024 | 驱逐线（H2O / StreamingLLM）: 干脆扔，但扔不回来

*** 它看到了什么问题

既然每个 token 都存太贵，最直接的念头是：*那就别全存，扔掉一部分*。

*** 它的解法

两种哲学。H2O 看内容：维护每个 token 的累计注意力分数，只留被反复重击的 "heavy hitter"。StreamingLLM 看位置：发现开头几个 token 总被高度关注（"attention sink"，softmax 归一化逼出来的位置 artifact），于是留 sink + 最近滑窗，中间全扔。

*** 为什么这招管用（以及哪里埋了新坑）

它们便宜、训练无关、即插即用，在流式对话里能把显存摁成常数。

但埋了一个本篇会正面继承的大坑：*永久驱逐*。token 一旦被扔，信息就再也回不来。对"开头读了份合同、结尾要核对某条款"这种全局任务，被扫地出门的中段 token 正是答案所在。而且这是训练后外挂的启发式——模型从没在"稀疏"状态下被训练过，是被强行套上稀疏的。

这两个坑——"扔了回不来"和"模型没学过稀疏"——分别被后面两篇接手。

** 2025-02 | NSA: 让稀疏"原生可训练"，而不是事后外挂

*** 它看到了什么问题

驱逐线的稀疏是推理时硬套的，模型在训练时见的是稠密全注意力，部署时却被掐成稀疏——分布对不上，质量没保证。NSA 还点出第二件事：很多稀疏方法理论上省了 FLOPs，GPU 上却跑不快，因为访存不连续。

*** 它的解法

把稀疏做成模型 *与生俱来、端到端可训练* 的一部分。每个 query 走三条并行分支：

1. *粗粒度压缩*：把成组 token 压成摘要，看全局轮廓；
2. *细粒度选择*：算重要性分数，按 *块* 选出最相关的几块 token；
3. *滑窗*：保住本地上下文。

关键是"块选择"——不是逐 token 挑，而是挑空间上连续的块，因为现代 GPU 对连续访存的吞吐高得多（硬件对齐）。整个机制从预训练就在，模型一开始就学会了在稀疏下工作。

*** 为什么这招管用（以及哪里埋了新坑）

"原生可训练"补上了驱逐线"模型没学过稀疏"的坑：27B 模型上 NSA 的 loss 甚至略低于全注意力，64K 序列上前向、反向、解码全面加速。它证明了稀疏不必拿质量换。

新坑在"重"：NSA 是 *预训练时的架构*，稀疏与主干联合训练。你想给一个已经训好的现成大模型加稀疏，没法用 NSA——得从头再训一遍。成本极高。

** 2025-09 | DSA: 给现成模型续训一个"闪电索引器"

*** 它看到了什么问题

NSA 太重，要从头预训。DeepSeek-V3.2 想要的是：拿现成的 V3.1，*续训* 一下就能上稀疏。

*** 它的解法

DSA 拆成两件东西：

1. *Lightning Indexer（闪电索引器）*：一个超轻量的 FP8 打分器，快速给每个 query 算出所有历史 token 的相关分；
2. *Top-k token selector*：只留分数最高的 k 个（开源里 k=2048），其余在注意力里 mask 掉。

复杂度从 O(L²) 降到 O(Lk)。索引器随主干一起、用自蒸馏端到端训练，作为对 V3.1 的唯一架构改动续训上去。这条线第一次有了一个"专门负责挑 token 的轻量模块"。

*** 为什么这招管用（以及哪里埋了新坑）

把"挑哪些 token"外包给一个廉价索引器，既省了从头预训（比 NSA 轻），又保住了质量（128K 下几乎不掉点，长上下文 API 成本降一半）。

但留下了本篇要撬开的那个坑：*DSA 让"算谁"变稀疏了，KV 却还全躺在 GPU 里*。索引器只是决定注意力点乘时看哪几个 token，被 mask 掉的 token 的 K/V 依然占着显存。换句话说，DSA 省的是 *计算*，没省 *显存驻留*。解码第 50 万步、当前这个字明明只跟最近 8K 有关，前面 49 万 token 的 KV 还在 GPU 里白占着。

而且索引器是 *跟着千亿主干一起训* 的——想调它，就得把整个庞然大物搬上显存，研究迭代极慢。

** 2026-06 | LSA: 把索引器从"打分器"改造成"检索系统"

*** 它看到了什么问题

本篇先把 DSA 那个坑量化成一句扎眼的话：*90% 以上的长上下文请求，只靠最后 8K token 就能答对*。意思是绝大多数显存，压在了"此刻完全用不到的历史"上——解码当前 token 这一步，跟绝大部分历史是 *独立* 的。

它同时点出对立面：可那剩下 10% 真要全局综合的任务，纯滑窗（只留近窗）又会彻底崩。于是矛盾被逼到台前：*既要支持深度全局推理，又不想为每一步本地解码付全额显存税*。DSA 没解这个——它的 KV 还全驻留。

*** 它的解法

把 DSA 的闪电索引器，*原地改造成一个独立的检索系统*，叫 Memory Indexer。三处关键变形，把"模型内打分"连续地变成"检索召回"：

1. *KV 下沉到 CPU 冷池*。历史压缩 KV（沿用 V4 的 128:1 高压缩 HCA chunk）不再常驻 GPU，挪到 CPU。GPU 里只留滑窗 + 当前被召回的那一小撮。
2. *预测性前瞻召回（lookahead）*。不是每一步都挑，而是每隔 τ=64 步触发一次：索引器拿当前 query 的隐状态，去 *预测接下来这 64 步这一窗会用到哪些历史 chunk*，提前把它们从 CPU 捞进 GPU。"被动地看所有历史" → "主动预测未来需求"。
3. *Sigmoid + 阈值 取代 Top-k*。DSA 用 ReLU 打分 + 固定 top-k；LSA 把最后一层激活换成 Sigmoid，把分数对齐到 (0,1)，再用阈值（≥0.5）召回 *动态数量* 的 chunk——该多就多，该少就少，而不是雷打不动取 k 个。

最妙的是 *训练解耦*。既然历史的压缩索引键 K^IComp 是预先算好、训练时冻结的，那索引器就退化成一个标准 *双塔检索模型* 的 query 编码器——你只要训那几个低秩投影矩阵（W^DQ, W^IUQ, W^w），用预先抽好的隐状态和标签，按检索任务训（BCE / Focal Loss，3:1 负采样）。*千亿主干从头到尾不上 GPU*。结果：整个索引器在 *单张 H20、一个 GPU 小时* 内收敛；一周内能在 8 卡上跑完约 500 组配置搜索。

标签怎么来？也很讲究：在冻结主干上离线跑，对未来窗口里每个 token、每层，用"Softmax 归一化 → top-p(0.6) 阈值 → 跨层多数投票(≥3 层共识)"三步去噪，蒸出"金标准"该召回的 chunk 集。这是为了去掉 top-k 硬选带来的近万条噪声正样本。

最终生产配置：把独立索引器放在第 10、12、20 三层（Pareto 扫出来的甜点），推理时三层做 *并集（OR 路由）*——任一层判 ≥0.5 就召回，作为稳健的兜底召回边界。

*** 为什么这招管用（以及哪里埋了新坑）

核心转变：这条线从"在注意力机制内部做文章"，整个 *跳到了检索系统*。LSA 自称是个 "attention denoiser"——它不只省显存，还把几千个无关历史 chunk 挡在 GPU 外，等于顺手降噪，反而少了幻觉。实测：平均只用全量 13.5% 的 KV 显存（降 86.5%），下游精度还 *略升* 0.6 个百分点；500K 尺度下显存降 90%。"less is more"。

但论文自己（Limitations 一节，很诚实）摆出了几个新坑，每个都指向下一步：

1. *上下文无关任务，显存压不成常数*。本以为对"不依赖历史"的 query，Sigmoid 会自然召回近零、得到 O(1) 显存。实测 Sigmoid 门控在超长序列里会漏进一点背景假阳性，从 125K 到 500K，物理 chunk 留存反而膨胀约 2.5 倍。
2. *稠密全局记忆会崩（MRCR 失败案例）*。在 MRCR 这种"真的要靠大量历史"的基准上，精度从 76% 暴跌到 48%。Oracle 实验显示：哪怕喂给它 50% 的真·金标准 chunk，精度仍比全注意力低约 2%——它结构上没法处理"密集全局依赖"。
3. *冻结的键 + 浅交互*。为省算力，历史压缩键 K^IComp 全程冻结、没微调；索引器只靠 64 步一次的粗粒度点乘相似度，缺 ColBERT 式的 *late-interaction* token 级交叉匹配，解不开复杂稠密检索模式。
4. *解耦训练忽略了自回归漂移*。索引器和主干没联合优化，用的是静态伪标签，没建模在线解码时的分布漂移。
5. *长度泛化天花板*。本以为点对点 chunk 匹配能从短训长泛化到 1M+；实测只能安全外推到 *训练长度的 2 倍*，超过就退化成近随机选择（位置编码 OOD）。所以最终模型只敢声明训到 512K。

* 前沿延伸

截至 2026 年 6 月，这是 LSA / "把索引器当检索系统"这条线的最新公开进展，且项目本身已（按论文所述）因组织变动暂停。论文把后续路线图写得很明确，基本就是上面五个坑的逐一回填：

1. *Late-interaction 索引器*：用 ColBERT 风格的 token 级交叉匹配替换当前 64 步粗点乘，解 MRCR 这类稠密检索（针对坑 2、3）。
2. *端到端联合优化*：让索引器与主干联合训练，建模自回归漂移，摆脱静态伪标签（针对坑 4）。
3. *位置编码的长度外推*：解决 OOD 位置嵌入，突破 2× 训练长度的天花板（针对坑 5）。
4. *常数显存的门控*：让上下文无关 query 真正召回近零，拿到 O(1) 显存地板（针对坑 1）。

更大的悬念是：一旦"把 KV 当外部可检索存储"这个框架成立，它会不会和正交的压缩轴（MLA 低秩 latent）、量化轴（FP8 KV）叠加？论文里 HCA 128:1 压缩 + LSA 检索已经叠了一层，但三轴全叠的系统还没人做出来。

推理: 前四代的隐含约束都是"历史必须驻留 GPU"。
       LSA 撬开它——既然 90% 的步用不到历史，
       就把历史放 CPU，需要时再"检索"回来。
       注意力问题 → 检索问题。
#+end_example

* 洞见

读完这条线，表面是"KV 越省越多"，底下真正在发生的是 *问题被重新归类了三次*：

- *MLA 阶段*：这是个"表示"问题——KV 太大，那就压小。轴是 *每个 token 的字节数*。
- *NSA / DSA 阶段*：这是个"计算"问题——注意力是稀疏的，那就只算重要的。轴是 *参与点乘的 token 数*。
- *LSA 阶段*：这是个"系统/检索"问题——历史此刻用不到，那就别放 GPU，需要时再捞。轴是 *什么东西必须驻留在快存储里*。

每一次跳跃，都不是把上一代的曲线又压低一点，而是 *换了一个被优化的量*。MLA 在压字节时，默认"每个 token 都得在 GPU"；DSA 在减计算时，仍然默认"KV 得在 GPU"。LSA 的杠杆恰恰是把这个 *被所有人默认、从没人质疑的约束*——"历史必须驻留 GPU"——拎出来打破。它甚至没发明新的打分器，就是把 DSA 现成的闪电索引器，重新理解成一个检索系统的 query 编码器。

这背后的模式和 KV 驱逐那条线惊人地一致：*当一个领域的提升曲线开始变平，往往不是因为大家逼近了最优，而是因为大家共享了同一个隐含约束*。打破天花板的不是更精细的算法，是 *识别并解除那个约束*。MLA→DSA 解除了"全维存储""稠密计算"，LSA 解除了"GPU 驻留"。

下一个待解除的隐含约束是什么？我猜是 *"KV 是被动存储、只能被检索"*。目前历史 chunk 一旦压好就冻结（论文坑 3 正是冻结键），只能被查、不能被改写、不能被合并。如果哪天有人让冷池里的历史能 *在线被重写、被聚合成更高层的摘要*（像人脑把短期记忆固化成长期记忆），KV cache 就从"存储"变成"可演化的记忆"，可能又是一次归类跳跃。论文标题里那个 "Memory" 和暂停前喊的 "infinite long-context"，赌的大概就是这个方向。

* 启发

1. *找突破口，先列"大家在优化什么量"，再问"这个量背后默认了什么不动的约束"*。这条线五代人优化了三个不同的量（字节→token 数→驻留集），每次跳跃都来自质疑前一代没质疑的那个常量。MLA 没碰"驻留 GPU"，DSA 也没碰，LSA 碰了——这意味着：*前一代省下的维度，恰恰圈定了它没敢动的维度，那里就是下一个杠杆*。

2. *最值钱的创新，常常是"重新理解一个现成组件"，而不是发明新组件*。LSA 没造新打分器，它把 DSA 的闪电索引器 *重新归类* 成检索系统的 query 塔。一旦归类变了，整套成熟的检索工程（双塔、BCE、负采样、解耦训练）瞬间可用——这才是它能"单卡一小时训完"的真正原因。

3. *解耦能换来的，远不止省钱*。把索引器从主干剥离，表面是省显存（千亿主干不上 GPU），实际换来的是 *研究速度*——一周扫 500 组配置。当一个子模块能脱离庞然大物独立迭代，迭代速度的量级提升本身就是一种能力。找系统里"能被解耦出来单独训练的部分"，是一个被低估的杠杆。

4. *把局限写成路线图，比藏起来更有价值*。本篇最值得学的不是 LSA，是它 Limitations 一节的诚实：MRCR 崩到 48%、显存压不成常数、2× 长度天花板，每条都配了 oracle 实验定位根因，直接变成下一步的施工图。在结果被资源和项目暂停"腰斩"的情况下，把"哪里不行、为什么不行"讲清楚，反而让这份半成品对社区有了长期价值。

这意味着你可以：*拿任何一个"靠压缩/稀疏在省资源"的系统，去问它——被省下来的东西，此刻是不是还非得待在最贵的那层存储里？* 如果答案是"其实大部分用不到却还占着"，那把它挪走、按需检索回来，往往就是下一个数量级。
