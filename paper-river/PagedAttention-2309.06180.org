* 问题之河

一个大模型部署上线，怎么让一张 GPU 同时服务几十、几百个用户，而不是一个一个排队？

这事一直卡在同一个地方：模型每生成一个字，都要回头看一遍前面所有字的"印象"——这些印象叫 KV cache，它又大又活——一个请求来了不知道会写多长，写到一半也不知道还要写多久。你预留少了不够用，预留多了就大片浪费。GPU 内存就那么大，你浪费一半，吞吐量就掉一半。

七年时间，一代一代人都在啃这同一根骨头：怎么让一群长得不一样、活得不一样、说话长短不一样的请求，挤在一张 GPU 上和平共处。

* 演化叙事

** 2017 | Transformer: 为了快,埋下了一个慢的种子

*** 它看到了什么问题

那时候做翻译、做对话的主力是 RNN。RNN 的毛病很直观: 它必须像读小说一样一个字一个字地读, 第 100 个字的状态依赖第 99 个字的状态。GPU 这种喜欢一万个数同时算的家伙, 在 RNN 面前等于被绑住一只手。训练个模型动辄一两周。

*** 它的解法

Vaswani 团队的招很狠: 把 RNN 整个扔掉, 只留注意力机制。每个字直接看其他所有字, 不用排队。一句话 100 个字, GPU 一次性把 100 个字的关系算完——训练快了十几倍。

*** 为什么这招管用(以及哪里埋了新坑)

并行得起来, 是因为训练时全句已知。但是上线之后用户问你一个问题, 你得一个字一个字写答案——这部分还是只能串行。每写一个字, 都得回头算它跟前面所有字的注意力。于是大家想了个偷懒办法: 把之前算过的 Key 和 Value 向量存起来, 下次直接拿——这就是 KV cache。

KV cache 解决了重复计算, 但带来一个新东西: *每个请求都背着一坨随时间长大的状态* 。这坨状态后面会变成所有麻烦的源头。

** 2021前后 | FasterTransformer: 批量起来,但代价是填空

*** 它看到了什么问题

一个推理请求只占 GPU 算力的几个百分点, 太浪费。让 GPU 划算的方式是: 把好几个请求的同一个矩阵乘合并起来, 一次乘掉——这叫批处理。这是深度学习老掉牙的招。

*** 它的解法

NVIDIA 的 FasterTransformer 这类系统就是这么干的: 收到一批请求, 把它们的输入填到相同长度, 一起跑 Transformer。这是 "静态批处理(static batching)" ——批一旦组好, 跑到所有请求都生成结束才换下一批。

*** 为什么这招管用(以及哪里埋了新坑)

GPU 利用率立刻上去了。但问题接踵而至:

- 同一批里有人写 50 个字就结束, 有人要写 500 个。前者得在槽位里干等后者, 这 450 步全是浪费。
- 你不知道每个请求会写多长。保险起见, 给每个请求预留 "最大可能长度" 的连续显存(比如 2048 token)。结果一个只写 50 字的请求, 占着 2048 字的空间——内部碎片浪费暴击。
- 一批没结束, 新请求只能排队——队首延迟拉满。

这一套, vLLM 论文里直接管它叫 "前任系统(existing systems)" , 用红色柱子画在 Figure 2 里嘲讽: 实际用上的 KV 内存只有 20% 多。

** 2022 | Orca: 不要按请求调度,按 token 调度

*** 它看到了什么问题

Orca 团队盯上了静态批处理那个 "等队友写完才能下班" 的浪费。问题不在批处理本身, 在于 *批处理的颗粒度太大* ——一批一组队, 锁死到所有人都写完。

*** 它的解法

把调度颗粒度从 "整个请求" 砍到 "一次迭代(一个 token)" 。每生成一个 token, 调度器就重新看一眼: 谁结束了? 谁刚来? 立刻把结束的踢出去, 把新来的塞进来。这个机制后来被叫做 *continuous batching* (持续批处理), 是现代 LLM 服务系统的标配。

变形地看: FasterTransformer 是 "组团副本, 一起开始一起结束"; Orca 是 "组团副本, 随时有人进有人出"。

*** 为什么这招管用(以及哪里埋了新坑)

队首延迟瞬间塌缩——新请求最多等一个 token 时间就能进来。GPU 计算槽位永远是满的。Orca 一举把吞吐量推上一个台阶。

但 Orca 没碰 *内存* 这件事。每个请求进来时, 它还是按 "最大长度" 给你预留一整块连续 KV 内存。这就是 vLLM 接力的地方——计算调度灵活了, 但内存还在用 2017 年的笨办法分配。

把 Orca 想象成一家餐厅: 上菜变得很快, 翻台率高, 但每张桌子还是按 12 人桌预留, 哪怕来的是 2 个人。剩下的 10 个位置永远不许给别人坐——直到这 2 个人吃完才能收。

** 2023 | PagedAttention/vLLM: 把 KV cache 当成操作系统的内存来管

*** 它看到了什么问题

Kwon 团队的洞察一针见血: KV cache 的这些麻烦, 跟你电脑上跑程序时操作系统遇到的麻烦, *是同一类问题*——

- 程序占多少内存事先不知道(KV cache 长度不知道)
- 程序内存要能动态扩张(KV cache 边生成边长)
- 不同程序之间有共享需求(beam search、parallel sampling、共同前缀的 prompt)
- 物理内存有限, 必须避免浪费(GPU 显存是瓶颈)

操作系统怎么解决的? *虚拟内存 + 分页*——50 年前就解决了。把内存切成定长的小页(page), 程序看到的是连续的虚拟地址, 物理上爱在哪在哪, 缺页才分配, 多个进程可以共享同一页。

*** 它的解法

照搬到 GPU:

- 把每个请求的 KV cache 切成定长的小 block(比如 16 个 token 一块)
- 每个 block 在物理显存上爱在哪在哪, 不要求连续
- 逻辑层维护一张 "block 表" (相当于 OS 的页表), 把请求的逻辑序列映射到物理 block
- 缺块才分配(按需), 不再为 "可能用到的最大长度" 占着茅坑
- 多个序列可以共享同一个物理 block(比如 beam search 的兄弟们前面相同, 后面分叉), 分叉时再 copy-on-write

PagedAttention 这个名字就是: 注意力算子要去访问那些分散在各处的 block, 不能再假设 KV 是一整块连续矩阵。所以他们重写了 attention kernel, 让它支持 "分页访问" 。

*** 为什么这招管用(以及哪里埋了新坑)

直接结果: KV cache 内存利用率从 ~30% 干到 ~96%。同样的卡能装下 3-4 倍的请求, 吞吐量翻 2-4 倍。

更深的收益: 一旦把 KV cache 抽象成 "可寻址、可共享、可分页的内存", 后面所有玩法都打开了——beam search、prompt 共享、prefix caching, 全是这套抽象的应用。

留下的坑(后浪的起点):

- 共享只在 block 粒度, 而且需要请求显式声明关系(同一个 prompt 的多采样); *跨请求的、自动的 prefix 共享* 还没被原生支持
- vLLM 把 prefill(处理 prompt, 算力密集)和 decode(生成 token, 内存密集)塞在同一张 GPU 上轮转, 两种负载特性不同, 互相干扰

* 前沿延伸

vLLM 之后, 这条河没有停。它在三个方向继续往下挖:

** 2024 | SGLang / RadixAttention: 让共享变成默认动作

vLLM 的共享是 *被动* 的: 你告诉它 "这几个请求来自同一个 prompt", 它才让它们共享。但现实世界里大量请求其实有共同前缀(同一个系统提示、同一段 few-shot、同一段聊天历史), 调用方不一定有能力告诉系统。

SGLang 的招: 把所有活跃 KV block 组织成一棵 *radix tree* (基数树, 一种按前缀压缩的树)。每来一个新请求, 先在树里查它的 prompt 前缀, 命中的那一段直接复用对应的物理 block——共享变成系统自动识别的事, 不需要用户配合。

某些 workload 下吞吐量再翻 5-6 倍。这是把 vLLM "KV cache 是内存" 这个隐喻又推进了一步: 不仅是内存, 还是 *带索引的内存* , 像数据库一样可查。

** 2024 | DistServe: 别让两种工作打架

prefill(读完用户的 prompt, 算出第一个 token)是个算力密集型任务, 它决定 "首字延迟(TTFT)" , 用户感知最敏感。decode(逐字生成后续 token)是内存带宽密集型任务, 它决定 "总吞吐" , 决定每秒钱赚多少。

这俩塞在一张 GPU 上, 一个长 prompt 的 prefill 跑起来, decode 全卡住, 用户体验崩。DistServe(OSDI'24)的招: 直接把 prefill 和 decode 拆到不同的 GPU 上, 各自按各自的目标优化, KV cache 通过高速网络在两边搬。

这是认识到 LLM 服务不是单一负载, 而是 *两种性格完全不同的工作* 共用同一个数据结构。拆开之后两边都能舒展。

** 2024 | Mooncake: KV cache 是集群级资源

Moonshot(Kimi)团队的 Mooncake 把 vLLM 的隐喻拉到了顶: 如果 KV cache 真的是 "内存" , 那它就不该只属于一张 GPU。

他们把所有节点的 GPU 显存、CPU 内存、甚至 SSD 整合成一个 *分层的 KV cache 池* , 跨请求、跨节点共享。一个长聊天历史可能存在 SSD 上, 用到时再分级拉到 GPU。一个热门 prompt 的 KV 在多台机器间复制。

这是从 "OS 内存管理" 跳到 "分布式存储系统" 的层级。隐喻越走越深。

* 洞见

七年时间, 表面看是一连串工程优化: 批处理、调度、内存。但底下真正在发生的事是 *视角的迁移*——

*把生成式 LLM 从"模型问题"重新理解成"系统问题"*。

具体地, 这个迁移分了几步:

1. 2017 把它看成 *模型问题* : 怎么设计架构让它学得好(Transformer 替代 RNN)
2. 2021 把它看成 *算力问题* : 怎么把 GPU 喂饱(静态批处理)
3. 2022 把它看成 *调度问题* : 怎么让请求高效共享 GPU(Orca 的 iter-level)
4. 2023 把它看成 *内存问题* : 怎么管 KV cache 这块巨大的、动态的、有共享的存储(vLLM 引入 OS 抽象)
5. 2024+ 把它看成 *存储问题* : KV cache 是一个值得独立设计的、跨节点的、可查询的数据系统(SGLang/Mooncake)

每一步都是把熟悉的领域知识(OS、数据库、分布式存储)往这个新场景里搬。最有意思的是 vLLM 这一跳: 它没发明新东西, 它只是认出了 "KV cache 管理 == 虚拟内存管理" 这个 *同构* 。一旦认出来, 50 年的 OS 智慧全部立刻能用。

下一步最可能往哪走? 大概率是把 KV cache 进一步 *资产化* :

- 像 CDN 一样的 KV cache 分发网络(热门 prompt 的 KV 边缘缓存)
- KV cache 跨模型/跨厂商的标准格式(类似容器镜像)
- 把 attention state 当成可持久化、可版本化的数据对象——也许下一篇论文会叫它 "KV database"

* 启发

这条河给做研究的人三个明显的提示:

1. *找同构, 不要发明* 。vLLM 没发明分页, 没发明虚拟内存, 它只是看出了一个 *别人没看出的同构* 。最好的研究往往不是 "发明新东西" , 而是 "认出 X 和已知的 Y 是同一类问题" 。问题: 你手头那个棘手的工程问题, 在 OS、数据库、编译器、网络、文件系统里有没有同构的老问题?

2. *跟着瓶颈走, 不要跟着热闹走* 。从 2017 到 2023, 每一步突破都是因为前一步把上一个瓶颈打掉, 露出了下一个。Transformer 解决了训练慢, 露出推理慢; FasterTransformer 解决了算力浪费, 露出调度浪费; Orca 解决了调度浪费, 露出内存浪费。 *永远问: 这一代解法之后, 下一个最大浪费在哪里?* 那就是下一篇论文的题目。

3. *把熟悉领域的隐喻借出来当杠杆* 。"KV cache 像 OS 的页面" 这一句话, 价值抵得上半个 PhD。如果你在某个领域(OS、DB、编译、网络、操作系统调度)有深厚积累, 主动去看新兴的 LLM 系统问题——大概率你能贡献一个杠杆级的隐喻。

落点: 你手上的工程系统, 把每一个 "看起来很麻烦、动态、需要共享、又有限的资源" 找出来——它就是下一个 KV cache, 它就在等一个 vLLM。
