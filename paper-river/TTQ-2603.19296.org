* 问题之河

LLM 太大，跑不动，得压。最直接的压法叫量化——把 16 位权重砍成 4 位、3 位，
显存立刻缩四到八倍，矩阵乘也变快。但砍得越狠，模型越笨。从 1990 年起到今天，
这条线上的所有人都在问同一个问题：把权重砍小的时候，*哪些权重要小心，哪些可以
随便砍*？

判断标准一直在变。最早是看权重的 Hessian——理论上完美但计算量爆炸。然后变成
"看激活值有多大"——简单粗暴但够用。再后来变成"看校准数据告诉我哪些通道重要"——
快了，但校准数据和真实用户输入对不上，部署到陌生任务就掉点。

TTQ 在 2026 年提出的解决思路是把校准这一步整个搬到推理时——*每一个 prompt 进来，
现场算一遍哪些通道重要，现场量化*。校准数据这个概念彻底没了，因为输入本身就是
校准数据。这条线 35 年的演化，本质上是"权重重要性判据"这一件事的逐步轻量化与
逐步即时化。

* 第一层: OBD - 用二阶导数判断哪个权重不重要

#+begin_quote
LeCun, Denker, Solla. /Optimal Brain Damage/. NeurIPS 1989.
#+end_quote

1989 年，神经网络第一次开始大到"剪掉一些权重以加速推理"成为一个真问题。

当时大家剪枝的办法很糙——直接把绝对值小的权重置零。LeCun 看到的问题是：*权重小
不代表它不重要*。一个权重可能数值很小，但它处在损失函数的陡峭方向上，砍掉它损失
会暴涨；另一个权重数值很大，但损失函数对它根本不敏感，砍掉无所谓。

他的解法用一句话讲完：用损失函数对权重的二阶导数（Hessian 的对角元素）来量重要性。
直觉是——*二阶导大的方向，损失对扰动很敏感，这个权重就重要；二阶导小，砍它没事*。

这是"看曲率，不看数值"的范式起点。但 OBD 假设 Hessian 是对角的——也就是说，假装
所有权重之间互不影响。这个假设错得离谱，但 1989 年算不动完整 Hessian，只能将就。

* 第二层: OBS - 权重之间是有勾连的

#+begin_quote
Hassibi, Stork. /Second order derivatives for network pruning: Optimal Brain Surgeon/. NeurIPS 1992.
#+end_quote

OBS 看到 OBD 留下的那个坑：神经网络里的权重不是孤立的，它们高度勾连。砍掉一个，
其他权重的最优值都得跟着调整。OBD 假装这种勾连不存在，所以剪到一定程度就崩。

OBS 的解法是用*完整*的 Hessian 逆矩阵：不仅算出"砍这个权重损失会涨多少"，还能算出
"砍完之后其他权重该怎么补偿"。这是一个更精确的二阶展开。

代价是 O(d³) 的 Hessian 求逆。1992 年的网络小得可怜，几千个参数勉强能跑。这套数学
当时被认为"理论漂亮但工程不可行"，搁置了二十多年。

直到 LLM 时代——突然之间，"理论漂亮"重新值钱了，因为差几个点的精度可以换几十亿
参数的部署。OBS 被翻出来重新审视。

* 第三层: GPTQ - 把 OBS 重做成量化, 算得动了

#+begin_quote
Frantar, Ashkboos, Hoefler, Alistarh. /GPTQ: Accurate post-training quantization for generative pre-trained transformers/. ICLR 2023.
#+end_quote

GPTQ 直接挪用了 OBS 的思想，但做了两个关键改造，让它在 175B 参数的 LLM 上能跑。

第一个改造：把"剪枝"换成"量化"。OBS 是把一个权重置零，GPTQ 是把一个权重从 fp16
舍入到最接近的 4 比特格子点上。在数学结构上是一样的——都是一次"扰动"，都需要补偿
其他权重。

第二个改造：不再对每个权重单独求 Hessian 逆，而是按列贪心处理，用 Cholesky 分解
一次性把所有列要用的逆元素备好。复杂度从 O(d⁴) 降到 O(d³)，再加上层与层独立处理，
70B 模型几小时能量化完。

但 GPTQ 留了一个 AWQ 后来要砍的坑：*你仍然需要校准数据*。要算 Hessian，得先把
几千条文本喂进模型走一遍前向，收集激活统计。校准数据选什么、选多少、和真实使用
场景有多远——这些都成了 LLM 量化界吵了几年的话题。

* 第四层: AWQ - Hessian 太重, 对角近似就够了

#+begin_quote
Lin, Tang, Tang, Yang, Xiao, Dang, Gan, Han. /AWQ: Activation-aware Weight Quantization for LLM Compression and Acceleration/. MLSys 2024.
#+end_quote

AWQ 看到了 GPTQ 的两个问题。

第一，Cholesky 还是太重。在端侧设备上，你不想每次部署都跑几个小时的 Hessian 计算。

第二，更深的洞察：*权重之间的勾连，在 LLM 的线性层里，绝大部分能用一个对角矩阵描述*。
也就是说——大部分通道彼此独立，只有少数"显著通道"需要保护。

AWQ 的解法干净利落：把 Hessian 直接近似成 ~diag(X X^T)^α~——一个对角矩阵，对角线
就是输入激活每一个通道的范数。然后量化前把权重矩阵的每一列乘以这个 scale，量化完
再除回去。这就把"重要的通道"用更大的 scale 推到量化格子的细处，自然得到更高精度。

复杂度从 GPTQ 的 O(d³) 砍到 O(d²) 量级。校准数据需求也大幅减少。AWQ 成了 2023-2025
事实上的工业默认——vLLM 里的 awq_gemm CUDA kernel 就是为它写的。

但 AWQ 留下一个新坑：*你还是需要校准数据来估那个对角范数*。一般大家用 C4 或者
WikiText 跑一遍，把 scale 算出来烤进权重，部署。如果用户的实际输入分布和 C4 差很远
（代码、数学、医学、多语言……），AWQ 的 scale 就估偏了，模型在域外任务上掉点。

而且——*一旦量化完成，原始 fp16 权重就丢了*，没法重新校准。

* 第五层: TTQ - 校准这件事整个搬到推理时

#+begin_quote
Koike-Akino, Liu, Wang. /TTQ: Activation-Aware Test-Time Quantization to Accelerate LLM Inference On The Fly/. arXiv 2026.03.
#+end_quote

TTQ 的整个 motivation 是从 AWQ 的那个坑长出来的：既然校准数据和真实输入对不上，
那为什么不直接*用真实输入当校准数据*？

技术上的关键观察：AWQ 那个对角范数 ~D_i = (||X_i||^p + λ)^α~ 算起来其实非常便宜——
就是对输入激活每一行做个 p-范数，O(dT) 操作。相比原本线性投影的 O(d'dT)，额外开销
只占 O(1/d' + 3/T)。在 d' 几千、T 几百的尺度上，*这个开销几乎是零*。

于是 TTQ 把整个流程变成：

1. prompt 来了，拿到这个 prompt 在每一层的输入激活 X
2. 现场算 D，现场算 scale S 和 zero-point Z
3. 现场把 fp16 权重量化成 4 比特（量化的是 W * D^(1/2), 用完除回 D^(-1/2)）
4. 用 4 比特 int_matmul kernel 算这一层的输出
5. 下一层重复

每个 prompt 都有自己专属的量化版本，*校准数据 = 当前 prompt*，*域外问题不存在*。

文章顺手还叠了一层低秩补偿：~W_hat = W_q + B A~，其中低秩因子 B、A 静态（top-r
PCA 初始化），但 W_q 仍然每个 prompt 重算。这个组合和 QLoRA 表面相似，但 QLoRA
的 W_q 是冻死的，只动 B、A；TTQ 反过来——动 W_q，冻 B、A。

实验结果：在 OPT-350M / Qwen3-1.7B 上 3 比特量化，TTQ 比 AWQ（用 C4 校准）的
perplexity 更低，而且*完全零离线校准数据*。

但代价是显式的：*每个 prompt 都要重新量化所有权重*。文中说 GPU 上能做到 5x 加速，
是因为 4 比特矩阵乘的内存带宽节省盖过了重新量化的开销。这件事在端侧设备上能不能
站得住，是这个方向接下来要打的硬仗。

* 前沿与下一步

TTQ 是 2026.03 的论文，目前还没有公开的后续工作直接批判它。但能看出几条它自然
延伸的方向：

- *同一作者团队*前面有 micro-grained MoE（Koike-Akino 2025），做的是"测试时剪枝"。
  TTQ 是测试时量化。下一步大概率是测试时低秩 + 测试时混合精度，凑齐"测试时压缩"
  这一套完整的工具箱。

- *测试时扩展* (test-time scaling, Muennighoff 2025 的 s1 那一脉) 这两年很火——
  推理时多算几步、多采样几条来换性能。TTQ 是反方向——推理时少占点资源换性价比。
  两者本质上都是"把训练阶段的事情挪到推理阶段"。这个范式转变还会持续。

- *硬件配合*：AWQ 之所以能成主流，是因为有 awq_gemm CUDA kernel 配套。TTQ 要落地，
  需要"量化 + 矩阵乘"融合算子，让重新量化的开销真正消失在数据流里。这是工程上的
  下一关。

- *校准的尽头*：从 1990 OBD 到 2026 TTQ，三十多年的演化其实是"重要性判据"的逐步
  轻量化与即时化——从需要完整 Hessian，到需要校准数据，到只需要当前 prompt。
  再往前推一步：能不能*完全无需任何统计*？比如根据权重本身的结构（对称性、low-rank
  谱），先验地决定量化策略？这是 LeCun 1989 年提的老问题的当代回声。

* 这条河真正在变什么

表面上看，这是"量化算法越做越快"的故事。但底下真正在发生的，是 LLM 时代对
"部署"这个概念的重新定义——

OBD/OBS 时代：训练好的模型是死的，剪枝是一次性的手术，做完就定了。
GPTQ/AWQ 时代：量化是一次性的"事后处理"，但需要"代表性数据"来做。整个工业界
开始接受"训练时不管，部署前再压"。
TTQ 时代：模型本身保持高精度（fp16 权重还在），但*每次推理都现场压一遍*。
压缩不再是部署前的一个步骤，而是推理过程的内嵌环节。

这个转变和"测试时计算"的兴起是同一个底层规律——*推理阶段不再是被动消费训练好的
模型，而是主动地、自适应地、按 prompt 调度的一个动态过程*。算法不再服务于模型，
模型变成了算法在推理时塑造的产物。
