* 问题之河

让一个大模型「想清楚」一道数学题，它要在草稿纸上写几万字。每多写一个字，GPU 显存就被多占一格——而且永远不释放。这就是这条河要回答的问题：*怎样让模型在长时间「思考」的时候，不被自己的草稿撑爆显存？*

这问题难，是因为 Transformer 的「记忆方式」太诚实——它把每个生成过的 token 的 key/value 都老老实实存着 (KV cache)，下一个 token 来了，要和这堆历史里的所有 key 算注意力。生成 16k token 的回答，Llama-3.1-8B 光 KV cache 就要 16GB+ 显存。这不是工程优化能省下来的，是架构的底层税。

一代一代人前赴后继：有人砍历史（StreamingLLM 只留头尾），有人按重要性挑（H2O 算注意力分数），有人按段落选（Quest 分页检索），有人把历史压缩成摘要（LightThinker 用 gist token 替换）。每一步都在「丢什么」和「留什么」之间拉扯，丢多了答错，留多了爆显存。ZoomR 这一步说：*别在丢/留里二选一，搞个分层——粗粒度索引 + 细粒度按需放大。*

* 演化叙事

** 2017 | Transformer + KV cache: 用空间换时间，埋下未来的债

*** 它看到了什么问题

RNN 的并行性太差——要算第 t 步必须先算完第 t-1 步，训练慢得让人发指。

*** 它的解法

自注意力。每个 token 同时看到所有其他 token，矩阵乘一把梭。但推理时如果每生成一个 token 都重算全序列的注意力，O(n^2) 算力直接爆炸。于是引入 KV cache——把过去所有 token 的 key/value 存下来，下一步直接复用。

*** 为什么这招管用（以及哪里埋了新坑）

KV cache 把每步注意力从「重算 n 个 token」降到「拿当前 q 和 n 个 k 算一遍」，省时间。但*它把时间问题转化成了空间问题*——cache 随生成长度线性增长，模型越「能写」，显存越紧张。

接下来几年大家发现：推理模型动辄写两万 token 的草稿，KV cache 单独就吃 10+ GB。于是「怎么压 KV cache」成了独立赛道。

** 2023 | H2O: 注意力分高的留下，分低的扔掉

*** 它看到了什么问题

KV cache 不能一直无脑攒。但简单地丢「最旧的」也不对——很多老 token 还是会被反复关注。需要一个「重要性」标准。

*** 它的解法

观察：注意力分数高度集中——少数 "heavy hitter" token 吸走了绝大部分注意力权重。那就维护一个累计注意力分，每步淘汰得分最低的那些 token。

*** 为什么这招管用（以及哪里埋了新坑）

H2O 第一次把「KV 淘汰」做成数据驱动的策略，性能比固定窗口好不少。但留下两个新坑：第一，*每生成一个 token 都要重算并维护所有 KV 的累计分*，本身有计算开销；第二，*淘汰是 token 粒度的*——一段连贯的推理可能被打成碎片，「这个常数等于 0.5」前半句留下来、后半句被扔了，模型再也拼不回来。

** 2024 | StreamingLLM: 开头几个 token 是「锚」，留住它们就不会崩

*** 它看到了什么问题

H2O 那条路要算分、要排序，开销不小。更便宜的做法是「窗口」——只看最近 N 个 token。但实验发现：单纯滑动窗口会让模型在长序列下「精神崩溃」，输出全乱码。

*** 它的解法

Xiao 团队发现一个反直觉现象：*模型对最开头那几个 token 投了过量的注意力*，称为 "attention sink"。这几个 token 像锚一样稳住注意力分布——一扔，softmax 就失衡，整个生成崩掉。

解法朴素到极致：永远保留头部几个 sink token + 一个最近的滑动窗口。不算分、不排序，O(1) 维护，能让 LLM 无限流式生成。

*** 为什么这招管用（以及哪里埋了新坑）

这是「分布稳定性」的洞察——LLM 不是被砍历史砍死的，是被砍 sink 砍死的。

但留下了 ZoomR 直接点名的坑：*lost-in-the-middle*。中段被永久丢弃。对聊天可以，对长推理致命——你不能在解题第 38 步还要回看第 7 步推出的引理。

** 2024 | Quest: 不是 token 级丢, 是页级选

*** 它看到了什么问题

StreamingLLM 砍中段，长上下文场景（长文档 QA）经常错。问题是：*历史里哪段重要，取决于当前在问什么*——静态规则注定不行。

*** 它的解法

把 KV cache 切成「页」(page)，每页几十到上百个 token。每生成一步，用当前 query 跟每页的代表 key 算一个粗略匹配分，只把 top-K 页搬到 GPU 上做注意力，其余留在 CPU。

这是「query-aware 检索」+「CPU offload」的组合：精度由 query 决定，显存由 K 决定。

*** 为什么这招管用（以及哪里埋了新坑）

页级粒度比 token 级保了语义完整，每步只动 top-K 页大幅省显存。但 Quest 的预设场景是 *长输入 prefill*——给一篇长论文做 QA，KV cache 是 prompt 编码完一次性生成的、固定的。

到「长输出 decoding」时这个假设崩了：cache 是边生成边长出来的，每步都在变。Quest 这套机制没有为「边写边检索」设计。这就是 ZoomR 要补的洞。

** 2025 | LightThinker: 写完一段就压成几个字，原文丢掉

*** 它看到了什么问题

无论 H2O 还是 Quest，都是在「选哪些 KV 留下来」的框架里打转。但选来选去，没动过模型生成的*总量*——草稿还是几万 token。

*** 它的解法

换个思路：训模型自己写摘要。给定一段推理，让模型生成几个 "gist" token 来代表这段，然后*把原始的几百个 token 直接从 KV cache 里删掉*。每段思考从几百压到几个，几十倍压缩比。

*** 为什么这招管用（以及哪里埋了新坑）

把「选择」问题变成「重写」问题——cache 里存的是模型自己提炼过的浓缩版。对压缩率友好。

但 ZoomR 直接点名的坑：*这是不可逆的有损压缩*。如果第 50 步推理需要回看第 5 步那个不起眼的细节（某个常数、某个临时引入的变量），LightThinker 那里已经只剩摘要了——细节再也找不回来。摘要里说「我证了引理 A」，但「引理 A 的具体表达式是什么」已经被压没了。

这是 ZoomR 在引言里画的那张图想说的事：*不同推理步骤需要不同粒度的历史*。

** 2026 | ZoomR: 既要目录，也要原文，按需放大

*** 它看到了什么问题

它把前面四代的坑摆到一起看：
- H2O 在 token 粒度上打碎语义；
- StreamingLLM 丢中段；
- Quest 不为 decoding 设计；
- LightThinker 把细节焊死了。

更根本的：*没人同时持有两种粒度*。要么只有 token，要么只有摘要。

*** 它的解法

让模型在生成过程中自己写段落级摘要（用 LoRA 微调 + Bespoke-17K 数据集，配 ~begin_of_summary~/~end_of_summary~ 分隔符）。然后:

1. *全量 KV cache 卸到 CPU*（包括摘要和原文）；
2. *GPU 上只留摘要的 key*——作为「目录」；
3. 每生成一步, 当前 query 和摘要 key 算匹配分，挑出 top-K 个最相关的段；
4. *把这几段的原文 KV 临时搬回 GPU*，做注意力。

像用 Ctrl+F 跳到书的某一章，再读那一章的原文。多头之间还做一次「投票」（per-head 评分 + global consensus），避免单个头判断失误。

*** 为什么这招管用（以及哪里埋了新坑）

抓住了一个朴素事实：*推理过程中, 大多数步骤只需要摘要级理解, 少数关键节点才需要原文*。所以平均下来, GPU 上只需要装一点点；偶尔需要细节, 临时从 CPU 拉。

报的数字: AIME2025 / MATH500 上接近 full cache 的精度, 显存省 4 倍+。

埋下的新坑（论文 limitation 里自己承认）:
- 摘要质量靠 Llama-70B 离线生成, 模型小了会塌；
- 段落边界是固定粒度, 不一定最优；
- 没在代码生成、创意写作上验证, 那里的「段落」结构不像数学题那么清晰。

* 前沿延伸

ZoomR 是 2026 年 4 月的新论文, 暂时没有公开的直接后续。但能看出几条明显的延伸方向：

1. *学到的段落边界* —— 不固定按 paragraph 切, 让模型自己决定「这里该收一个摘要了」。和 SepLLM (Chen et al., 2025) 把段落塞进 separator token 的思路接近。
2. *自适应粒度* —— 当前 K (选几段)、压缩比都是固定超参; 下一步会按 query 难度动态调。
3. *多级层次* —— 现在只有「摘要 / 原文」两层。可以做「全书 → 章节 → 段落 → 原文」的递归。
4. *跨域推广* —— 代码、长对话、agentic workflow。这些场景的「段落」语义需要重新定义。
5. *与量化 / 低秩 / 投机解码组合* —— ZoomR 自己也强调和 KV 量化 (KIVI, KVQuant) 正交, 叠加预计还能省一档显存。

* 洞见

把这条线拉直了看, 表面是「KV cache 怎么压」, 底下是另一个更大的转变：

*模型推理正在从「单层平面记忆」走向「分层有结构的记忆」。* 早期所有 token 都是一等公民——同样存、同样查、同样淘汰。一代代往后, 慢慢承认了一件事：*历史不是均质的*。有的位置是锚 (sink), 有的是骨干 (heavy hitter), 有的是细节 (可被摘要替代), 有的是中段 (需要按需检索)。模型记忆的结构性, 是被「显存不够用」逼出来的——但它的方向, 越来越像人类记忆。

第二个观察：*每一代都在「丢什么 / 留什么」的二元题里挣扎, 直到 ZoomR 把它换成「分几层」的三元题。* H2O 选 token, StreamingLLM 选位置, Quest 选页, LightThinker 选改写——都是 2D 选择。ZoomR 加了「粒度」这个新轴, 让选择不再非黑即白。这个升维, 大概率会扩散到 KV cache 以外的领域（比如长上下文 retrieval、agent memory）。

第三个观察：*"CPU 当冷存储, GPU 只放热数据"* 这个朴素的存储分层观念, 从 Quest 开始已经稳了。下一代该卷的不是「要不要 offload」, 而是「热/冷的判定标准, 是不是该 query-aware 且 token-aware」。

下一步最可能的方向: 把「段落边界」「压缩粒度」「top-K 大小」都做成可学习的, 让模型自己决定什么时候记多细。

* 启发

读到这里, 落点在「能用」:

1. *做长输出推理优化时, 别只盯 prefill。* 整条 KV 压缩文献的盲区是「decoding 阶段 cache 边生成边膨胀」, 这里仍有大块空白可挖（ZoomR 也只是开了个头）。
2. *看到「二元选择」的工作, 试着加一根轴让它变多元。* 从 H2O 的「留/丢」, 到 ZoomR 的「粗看 + 细看」, 多一根维度往往就是新论文。
3. *评估别只看精度, 显存曲线和长输出场景下的衰减更能暴露真实差距。* 很多 KV 压缩方法在短上下文上数字漂亮, 一到 16k 输出就崩——ZoomR 选的 AIME2025 (数学竞赛, 长推理) 是个好基准。
4. *人类记忆的隐喻不是装饰, 是设计指南。* ZoomR 直接引了 1932 年 Bartlett 的工作 (schematic memory)。下一波 efficient inference 可能来自更深地读认知科学——working memory、episodic memory、retrieval cue 的机制都有可借鉴的形状。
5. *如果你在做模型部署 (尤其消费级 GPU)*, ZoomR 这类「摘要 + 按需 zoom」的方案值得关注——4x 显存节省意味着同样显卡能跑长两倍的推理上下文, 或同样上下文翻倍 batch size。
