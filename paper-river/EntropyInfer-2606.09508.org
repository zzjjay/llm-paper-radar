* 问题之河

让大模型读一份 50 万字的文档，它就得把这 50 万个 token 的 Key、Value 全压进 GPU 显存——因为生成下一个字时，理论上可能要回头看其中任何一个。序列每长一倍，这份 KV cache 就大一倍，很快比模型权重还沉。整条研究线都在跟一个朴素的念头死磕：*这 50 万份历史，其实大部分用不着，能不能少存、少算？*

难点不在"砍",在"砍得准"。你怎么知道哪个 token 现在没用、之后也没用？过去七八年，一代代方法都在回答这个问题，而它们共享了一个越来越可疑的隐含假设：*重要性可以被某种统一的规则提前定下来*——要么按位置（留开头留近窗），要么按一个全局分数（留被反复关注的），要么给所有注意力头发同样大的预算。

这篇论文（EntropyInfer）的尖锐之处，是把这个隐含假设直接摆上台面用实验打掉。它发现：注意力头压根不是一类东西。有的头（*Rigid Heads*，刚性头）注意力极度集中，熵几乎为零，无论输入是什么都死盯着固定几个位置；有的头（*Dynamic Heads*，动态头）注意力随上下文剧烈起伏，承载着真正的语义结构。更要命的是，*同一个头在不同输入下可能是刚性的、也可能是动态的*——这意味着"哪些头重要"根本无法离线预先确定，必须 *在线、逐输入* 地判断。

于是问题从"留哪些 token"被重新问成了：*为什么要用一把尺子去量所有头、所有上下文？*

* 演化叙事

** 2023 | H2O: 注意力是稀疏的，那就只留"重击者"

*** 它看到了什么问题

长文本生成时，KV cache 随序列线性膨胀，很快吃光显存。H2O 的出发点是一个实测观察：很多预训练大模型的注意力 *超过 95% 是稀疏的*——也就是说每一步真正起作用的，可能只有 ~5% 的历史 token。那能不能只留这 5%，把 cache 砍到 1/20？

*** 它的解法

H2O 提出 "Heavy Hitter（重击者）" 概念：少数 token 吸走了绝大部分注意力。它给每个 token 维护一个 *累计注意力分数*（所有历史 query 对它的注意力之和），cache 固定大小，一半留最近 token、一半留累计分最高的重击者，每步贪心驱逐当前分最低的。

打个比方：一个固定容量的 VIP 俱乐部，永远留住刚进门的（近窗），加上历史上最受瞩目的几位（重击者），谁现在人气垫底就请谁出去。

*** 为什么这招管用（以及哪里埋了新坑）

它便宜、训练无关、即插即用，把显存压成常数。但 H2O 自己也承认一个硬伤：*驱逐是永久且贪心的*——"被驱逐的 KV 在未来步里再也见不到，丢掉关键 KV 会导致严重的功能崩溃"。一个此刻看着没用、之后才变关键的 token，一旦被扔就回不来。而且按 *累计* 分数打分，天然偏袒"活得久"的老 token（积累的步数多），把"被频繁关注"误当成"对未来重要"。

这个"扔了回不来"的坑，是整条线接下来反复要修的东西。

** 2023 | StreamingLLM: 别按内容扔，先搞懂为什么不能只留近窗

*** 它看到了什么问题

H2O 要维护分数、逐步驱逐，还是有点重。最朴素的念头是：干脆只留最近 N 个 token（滑窗），不就常数显存了？StreamingLLM 发现这条路 *会崩*——文本一旦超过窗口大小、开头 token 被挤出，困惑度立刻爆炸。

*** 它的解法

它揪出了原因：开头几个 token 是 "attention sink（注意力池）"。softmax 要求注意力权重加起来等于 1，当某个 query 在哪儿都找不到强匹配时，多出来的注意力必须 *倒到某处*——而开头 token 因为对每个后续位置都可见，被训练成了这个"零钱罐"。把罐子（开头 token）扔了，预算的数学就崩了。

解法极简：永久留住 ~4 个开头 sink token + 一个滑动近窗，不用微调，就能稳定流式处理 400 万 token。

*** 为什么这招管用（以及哪里埋了新坑）

它点破了"重要性不只看近不近，还看是不是 sink"这件反直觉的事。但它的局限和 H2O 殊途同归：StreamingLLM 自己写明"不扩展上下文窗口、不增强长期记忆"——任何掉出 sink+近窗的中段信息 *一去不回*，检索准确率"随距离衰减，超过 cache 容量就归零"。

到这里两条朴素路线（按内容扔、按位置扔）都撞上同一堵墙：*靠某种固定规则做的永久取舍，总会在某些任务上扔错*。下一篇换了个角度——能不能让取舍发生得更聪明、更靠后一点？

** 2024 | SnapKV: 模型读完 prompt 就已经知道要看哪了

*** 它看到了什么问题

SnapKV 盯上的是 H2O 这类方法的 *时机*：它们在解码循环里逐 token 地累计、驱逐，一路维护到底，太啰嗦。SnapKV 的判断是——其实没必要：模型对 prompt 的注意力分配，*在生成开始前就基本定型了*。如果能从 prompt 本身把这个模式读出来，那就 *一次性* 在 prefill 结束时把 cache 压好，不用全程维护。论文标题就是这个主张："LLM Knows What You are Looking for Before Generation"。

*** 它的解法

SnapKV 盯住一个 *观察窗口*——prompt 末尾的一小段（LongBench 默认 32 个 token）。直觉是：prompt 最后这几个 token 是 query 最新的"声音"，它们关注哪里，能很好地预测接下来生成的 token 会关注哪里。

1. *投票*：拿观察窗口里的 query 对所有前文 key 的注意力，沿 query 维求和，得到每个前文位置的重要性分。
2. *选择*：每个头各留 top-k 个前文位置，再拼上观察窗口。
3. *池化*：选之前先做一维最大池化（核大小 7），保住重要 token 的 *邻居*。否则会出现"只留了电话号码的国家码、剩下全靠瞎编"那种碎片化问题。

类比：会议快结束时，你记下最后一位发言人反复指着哪几份文件，就只留那几份（连同它们周围几页），赌后续讨论会引用同一批。

*** 为什么这招管用（以及哪里埋了新坑）

把"逐步维护"换成"一次性快照"，又快又省，质量还稳。但它埋下两个本篇会正面继承的坑：

1. *prefill 盲点*：选择完全由 *prefill 阶段* 测到的注意力驱动。SnapKV 默认这个模式生成后还稳，可一旦生成开始、相关上下文发生迁移（比如多步推理浮现出新依赖），prefill 时读到的快照就过时了——而压缩 *一次定终身，再不回头*。本篇把这个叫"基础性盲点"：prefill 信号下看着重要的 token，不一定就是真正驱动解码的 token。
2. *所有头同一份预算*：每个头独立挑 *哪些* 位置，但 *留几个*（预算）是全局统一的固定值。注意力集中的头和分散的头被迫留同样多——前者浪费、后者饿肚子。这正是 Ada-KV 要补的口子。

** 2024 | Ada-KV: 选法不变，但预算该按头重新分

*** 它看到了什么问题

Ada-KV 认同 SnapKV 的 *选择规则*，但盯上它的 *预算切分*：给每个头发一样多的槽位，"忽略了每个头独特的注意力模式"，在理论上不是误差最小的分法。

*** 它的解法

Ada-KV 是个挂在现有驱逐法之上的 *预算再分配层*（于是有了 "Ada-SnapKV"）。它不再让每个头在固定预算下各取 top-k，而是：

1. 把 *所有头* 的候选注意力权重倒进一个池子；
2. 全局取 top-B（B = 总预算）；
3. 每个头的预算 = 它有几个权重挤进了全局这一刀。

集中的头自然分得少，分散的头分得多。理论上（Theorem 3.1/3.3）它把"驱逐前后注意力输出的 L1 距离"上界压到最小——等价于在固定总预算下 *最大化所有头保留的注意力质量*，并证明这种全局自适应分配比均匀分配更优。还留了个安全阀（α=0.2）混入均匀分配，防止某些头被饿到零。

类比：SnapKV 给每个部门发同样的预算，不管需求；Ada-KV 把所有申请倒成一堆，全局先批最高优先级的，每个部门最终拿多少由它有几个申请过线决定——还设了地板，没人被彻底饿死。

*** 为什么这招管用（以及哪里埋了新坑）

预算分得更准了，同样的总开销下保住更多有用注意力。但坑也很清楚：Ada-KV 优化的是"怎么分预算"，*分的依据还是从 SnapKV 继承来的那个 prefill 盲点信号*。它从同一张可能过时的快照里做了更聪明的分配——如果快照本身就错判了生成需要什么，那再优的分配也是在错信号上重新洗牌。而且预算只在 *头* 之间分，是一次性算定的，没有随生成推进、也没有在"段"这个粒度上按需调整。

** 2024 | CritiPrefill: 把稀疏从解码端搬到 prefill 端

*** 它看到了什么问题

前面几篇省的都是 *解码* 显存。但 CritiPrefill 指出一个被忽视的瓶颈：长上下文里，*prefill 阶段本身* 就是延迟大头——注意力 O(L²)，1M token 光 prefill 就要算半小时。光省解码救不了这个。

*** 它的解法

关键观察是 query 侧的局部性：相邻的 query token 往往关注差不多的历史 KV。逐 token 算 criticality 对算力密集的 prefill 太贵，于是它走 *段* 粒度：

1. 把 query 切成固定大小的段（512）、KV 切成块（32）；
2. 段级算法估每个 KV 块对每个 query 段的 "criticality"；
3. 加一步层间融合细化分数，每段在预算（1024）内选 top 块，剪掉不重要的（段,块）对。

在 128K 上下文、单张 A100 上对 Llama3-8B 提速 2.7 倍，几乎不掉点。

*** 为什么这招管用（以及哪里埋了新坑）

它第一次把"稀疏"这把刀干净地用在 prefill 阶段，证明 prefill 也能大幅加速而不伤质量。但留下本篇要撬的口子：预算是 *单个全局值*、按段施加，*一个段里所有头共享同一个算力信封*。一个做稠密检索的头和一个做局部流式的头，几何完全不同，却被塞进同一个 k。换句话说，CritiPrefill 把稀疏从"逐 token"提到了"逐段"，但还没提到"逐头"。

** 2026 | EntropyInfer: 头不是一类东西，别用一把尺子量

*** 它看到了什么问题

本篇把前面所有方法的共同病根总结成一句话：*它们都假设某种"齐性"——跨位置（StreamingLLM）、跨头（SnapKV 同预算、CritiPrefill 同 k）、或跨上下文（离线预定头的角色）——而这种齐性，实测上根本不成立。*

它用注意力熵把"头不齐"量化出来：有的头是 *刚性头*，熵"无论输入都低于 10⁻⁵"，近乎确定性地死盯固定位置；有的头是 *动态头*，熵"随 query 位置剧烈起伏，承载真正依赖上下文的语义结构"。而且 *同一个头在一个输入上刚性、在另一个输入上动态*——所以"哪些头重要"无法离线预定（这恰好否定了 DuoAttention 用学到的分类器、RazorAttention 用静态分析预先给头分角色那条路）。结论：分类 *必须在线发生*。

它还顺手把 SnapKV/Ada-KV 的 prefill 盲点也收进同一框架：这些方法默认"prefill 注意力可靠预测解码重要性""头的行为可在见到输入前就刻画"——两个假设都被熵分析证伪。

*** 它的解法

EntropyInfer 是个训练无关、即插即用的"注意力替换件"，两端都改：

1. *Prefill 端——按熵逐头逐段分配算力*。在线测每个头的行注意力熵：刚性头给一个固定的最小预算（反正它就盯那几个位置，多给也浪费）；动态头给的预算 *随相邻段之间的熵波动而缩放*——"把算力集中在注意力最不确定的地方"。这等于把 CritiPrefill 的"逐段同 k"和 SnapKV 的"逐头同预算"两个齐性假设一起打破，做到 *逐头 × 逐段* 自适应。

2. *解码端——延迟压缩（latent KV compression）修盲点*。不在 prefill 一结束就锁定要留哪些 token，而是 *等少量输出 token 先生成出来*，把这些 *输出* token 纳入观察窗口，用它们重新给 cache 条目排序。这直接针对 SnapKV 那个"prefill 快照预测不了解码真实需求"的盲点——既然生成开始后模式会变，那就让真实生成出来的几个 token 来投票，而不是只信 prompt 末尾的代理信号。

把这一招连续变形地理解：SnapKV 用 prompt 末尾 32 个 token 当"未来的代理"；EntropyInfer 说代理不够准，那就 *等真未来露一小截脸再做决定*——观察窗口从"prompt 的尾巴"挪成了"prompt 尾巴 + 已生成的几个真 token"。

*** 为什么这招管用（以及哪里埋了新坑）

核心转变：这条线从"找一把更好的统一尺子"（更聪明的全局分数、更优的均匀替代分配），跳到了 *承认根本没有统一尺子*——重要性是逐头、逐段、逐上下文变化的，度量它的工具是 *熵*，而且只能在线测。论文称 EntropyInfer 在 Llama、Qwen、openPangu 上稳定超过 SnapKV、AdaKV、CritiPrefill 这些基线。

它留下的开放问题（也是这条线天然的下一站）：在线测熵、逐段缩放预算本身有运行时开销，自适应省下的算力能不能盖过判定的代价，在更长序列、更多头数下是否仍划算，论文之外还需更多验证。而"延迟压缩"要等输出 token，意味着压缩时机被推后——前几个解码步仍背着较大的 cache，这对极致显存场景是个待权衡的折中。

* 前沿延伸

截至 2026 年 6 月，EntropyInfer 是"用熵在线区分头、逐头逐段自适应"这条具体路线的最新公开进展。围绕它要解的同一个问题——*prefill 到 decoding 的注意力错配* 与 *逐头自适应*——同期有几条相邻的线在并行推进（这些是相关工作，并非确证引用本篇的后续）：

- *DuoAttention*（2024-10, ICLR 2025）：把头二分为"检索头"（留全量 KV）和"流式头"（只留 sink+近窗），用合成数据上的轻量优化识别。和本篇对立的一点：它的头划分是 *预先学好的静态分类*，而本篇坚持头的类型随输入在线变。
- *RazorAttention*（2024-07, ICLR 2025）：训练无关地按"检索/非检索"给头分角色，非检索头丢远端 token、把丢掉的信息折进一个补偿 token。同样是 *推理前的静态分析*——正是本篇用熵证伪的那类假设。
- *LoopServe*（2025-07）：面向多轮对话的双阶段方案，在线做 prefill 稀疏化 + 解码时渐进 KV 压缩，直接攻 prefill→decode 的注意力错配——和本篇的"延迟压缩"动机同源。
- *LouisKV*（2025-10）：把 KV 当成可 *检索* 的对象，利用时间局部性只在语义边界触发检索，并把输入/输出的 KV 管理解耦。它指向一个更激进的方向：与其在注意力机制内部反复调预算，不如把历史当外部存储按需召回。

更大的悬念：本篇把"重要性必须在线、逐头度量"立住之后，下一步多半是把这个在线判定的 *代价* 再压下去（让测熵、分预算本身接近免费），或者干脆顺着 LouisKV 那条线，把"逐头自适应稀疏"和"KV 外部检索"两轴叠起来。

* 洞见

读完这条线，表面是"KV 越省越准"，底下真正在发生的是 *"重要性"这个概念被不断追问，从"可预定"一路退到"必须在线、逐头测量"*：

- *H2O / StreamingLLM 阶段*：重要性 = 一个 *全局规则*（累计分 / 位置）。一把尺子量所有 token。
- *SnapKV / Ada-KV 阶段*：重要性 = prefill 末尾观察窗口投出来的 *快照*。尺子细化到"用 prompt 的尾巴预测未来"，并开始承认不同头该分不同预算——但量的工具还是同一张快照。
- *CritiPrefill 阶段*：重要性下放到 *段* 粒度，却仍在一段里对所有头用同一个 k。
- *EntropyInfer 阶段*：重要性 = *逐头、逐段、逐上下文* 变化的量，没有任何离线规则能预定，度量它的工具是 *熵*，且只能在线测。

每一次跳跃，都不是把上一代的曲线又压低一点，而是 *缩小了"可以提前假设"的范围*。H2O 假设全局分够用；SnapKV 假设 prefill 快照够用；Ada-KV 松开了"所有头同预算"，CritiPrefill 松开了"所有位置同规则"，但都还攥着"重要性可被某个离线/prefill 规则刻画"。EntropyInfer 的杠杆，恰恰是把这个 *被所有人默认、没人质疑的约束* 拎出来打破：用熵证明头的行为既不齐、也不可预定，于是把判定整个挪到在线。

这和它继承的"prefill 盲点"批判是同一个母题的两面：SnapKV 假设"prefill 能预测解码"，本篇说不能——那就 *等真解码露一小截脸再决定*。两处改动（在线测熵、延迟压缩）共享一个认知转变：*从"提前猜"转向"边跑边量"*。

下一个待解除的隐含约束，我猜是 *"在线度量本身必须便宜到可以忽略"*。EntropyInfer 把判定挪到在线，换来准，但也背上了在线测熵、动态分预算的运行时成本。如果哪天有人能让"逐头逐段的重要性度量"本身近乎免费（比如让模型在预训练时就自带一个轻量的熵预测器），或者干脆像 LouisKV 那样把历史外置成可检索存储、只在语义边界才触发一次度量，这条线可能又是一次归类跳跃：从"在线度量重要性"跳到"重要性度量被摊薄进架构本身"。

* 启发

1. *当一个领域的提升曲线变平，往往不是逼近了最优，而是大家共享了同一个没人质疑的假设*。这条线五代人都在做"更好的统一重要性尺子"，直到 EntropyInfer 用熵指出：尺子再细也错，因为根本没有统一尺子——头不是一类东西，且随输入变。找突破口的姿势因此很具体：*列出前人都在"假设什么可以提前定下来"，那个被默认为常量的东西，就是下一个杠杆*。

2. *"证伪一个隐含假设"常常比"优化一个指标"更值钱*。Ada-KV 在 SnapKV 的框架内把预算分得更优（优化指标），EntropyInfer 则直接证明这个框架的地基（prefill 信号可靠、头可离线刻画）是错的（证伪假设）。前者把曲线压低一点，后者换掉了被优化的对象。先问"这个领域有没有一个被反复继承、却从没被实测检验过的前提"，比闷头调参回报高一个量级。

3. *一个好的"诊断量"能撬开整条路线*。EntropyInfer 真正的支点不是某个新算法，是把"注意力熵"当成区分刚性/动态头的诊断指标——量一立住，"必须在线分类""逐头逐段分配"这些设计就都顺理成章地长出来了。找研究问题时，*能不能找到一个简单、可测、能把'大家以为齐的东西'照出不齐的量*，本身就是一种稀缺能力。

4. *修复"前人信号的盲点"，有时只需把观察的时机往后挪一点*。SnapKV 用 prompt 尾巴当未来的代理，本篇只是把观察窗口延后到"已生成的几个真 token"——一个时机上的小变形，就修掉了整条线扛了两年的 prefill 盲点。改动不必复杂，关键是认出"我们到底在拿什么当未来的代理，这个代理为什么会偏"。

这意味着你可以：*拿任何一个"靠某个提前算好的分数/规则在做取舍"的系统，去问它两件事——这个分数是在"真正需要决策的那个时刻"之前算的吗？它是不是默认了被打分的对象都是同一类？* 如果答案是"提前算的"且"假设了齐性"，那把度量挪到在线、按真实子类分别度量，往往就是下一个台阶。
