* 一句话

把几道题塞进一个 prompt，模型自己就会把每道题的思考压短——ConPress 把这条"挤一挤就瘦了"的副作用，反过来当成不用老师、不用 RL 的自监督瘦身信号。

* 问题怎么长出来的

** 起点：Chain-of-Thought —— "让它把过程写出来"

2022 年之前，大语言模型回答数学题、推理题就是一个字：蒙。给它一道应用题，它直接吐答案，错得离谱。

Wei 他们那年提了个简单到不像论文的东西：在 prompt 里塞几个"先想一步、再想一步"的示范，让模型也学着把推理过程写出来。这就是 Chain-of-Thought（CoT，思维链）。

结果出奇地好。同一个模型，加上"让我们一步步想"，数学题准确率能从十几个百分点跳到五六十。

这立住了一条 *不证之物*：写过程 > 直接答。后面所有故事都在这条路上跑。

** 第二跳：长推理模型 —— "既然写过程好，就让它狠狠写"

2024 到 2025 这一年，OpenAI 推 o1，DeepSeek 推 R1，Qwen 推 Qwen3 的思考模式——这一波被叫做 *Large Reasoning Models*（LRM，大推理模型）。

它们的共同套路是：用强化学习训模型在回答之前先做一长串"内心戏"——自我怀疑、自我验证、试错回退、换思路重来。准确率又上了一个台阶，AIME、GPQA 这种竞赛题都开始能做了。

但代价露出来了：

- 一道 AIME 题，模型动不动写一万个 token 的思考
- API 一秒一千 token，等一道题答案出来要十几秒
- 真上线，单题成本几毛钱起步

写过程好这件事，被推到了极致——也推到了荒谬。

** 第三跳：Overthinking 被点破 —— "其实大部分都是废话"

2025 年初，Chen 他们一篇叫 *"Do NOT Think That Much for 2+3=?"* 的论文一锤定音：长推理模型在简单题上的思考过程里，大部分内容是冗余的——重复检查、绕弯路、自我安抚式的"让我再确认一下"。

把这些废话砍掉，准确率不变，甚至略涨。

Sui 他们紧跟着做了一篇 *Stop Overthinking* 综述，把这个现象正式命名： *overthinking*（过度思考）。

到这一步，问题清晰了：

#+begin_src
原始问题: 长推理模型答得准,但太啰嗦,太贵,太慢
真问题:   它的啰嗦里大部分是冗余,本来就该砍
新问题:   怎么砍? 谁来告诉它"这段是废话"?
#+end_src

** 第四跳：RL 派的解法 —— "用奖励函数告诉它"

最直接的思路：搬出训长推理时用的那套 RL，奖励"短而对"的轨迹，惩罚"长而对"和"短而错"。

代表作：

- *O1-Pruner*（Luo+）：用 PPO 类方法，把长度作为奖励信号的一部分
- *L1*（Aggarwal+）：让模型在指定长度预算内完成推理
- *ThinkPrune*（Hou+）：边训边剪，逐步压短

这条路走得通，但门槛高：

- 要设计奖励函数（准确率 × 长度 怎么权衡？）
- 要做 rollout（每步采样很多轨迹）
- 要算力（RL 训练成本远高于 SFT）

对小团队，这套基础设施搭不起来。

** 第五跳：SFT/DPO 派 —— "退一步用监督学习"

既然 RL 太重，就退一档：用 SFT（监督微调）或 DPO（直接偏好优化）。代表作：Arora+、Cheng+ *Incentivizing Concise Reasoning*、Yi+ *ShorterBetter*、Zhang+ *AdaptThink*。

思路统一：搞一批"短而对"的轨迹当训练目标，或者搞"短的 vs 长的"配对当偏好数据，让模型学着像短的那样写。

简单、便宜、能复现。

但有个绕不开的问题： *短而对的轨迹从哪来*？

主流做法是：
1. 找一个更强的"老师模型"生成简短答案（蒸馏）
2. 人工写示范（贵）
3. 截断长推理（容易截坏）

每一条都有依赖：要么靠外部老师，要么靠人，要么靠启发式。

** 第六跳：ConPress 的转身 —— "原来它自己就会变短"

这就到本篇了。

Deng 他们做了一个看起来无关、其实关键的观察：

#+begin_quote
把几道独立的题塞进一个 prompt，让模型一次答完——它会自动给每道题写更短的思考。
#+end_quote

他们管这个叫 *Self-Compression*（自压缩），背后的机制叫 *multi-question contextual pressure*（多题情境压力）。直觉上很顺：当上下文里已经堆了一堆题，注意力被分摊了，模型自己就不愿意再为某一道题展开冗长的内心戏——它会更直奔答案。

这跨过去了前一派的死结：

#+begin_src
SFT/DPO 派:  "短而对的轨迹得从老师/人/启发式拿"
ConPress :   "模型在多题场景下自己就会生成短而对的轨迹"
#+end_src

剩下的就顺理成章：
1. 构造一批多题 prompt，让模型自己生成
2. 解析出每道题的思考片段
3. 过滤掉答错的、长度异常的
4. 拿剩下的"短而对"轨迹去 SFT 同一个模型

整个流程 *不用老师，不用人工，不用 RL*。8k 条样本就够。

结果：MATH500 上 token 砍掉 59%，AIME25 上砍掉 33%，准确率基本保持。

** 这条线的真正变化是什么

表面上是：长 → 短，CoT 越来越省。

深层的变化其实是 *"压短信号"从哪里来* 这件事的迁移：

#+begin_src
  外部老师告诉   ->  奖励函数告诉   ->  偏好数据告诉   ->  *它自己告诉自己*
  (蒸馏)            (RL)              (SFT/DPO)          (ConPress)
#+end_src

每一步都在降低"知道什么是短而对"的成本。到 ConPress，这个成本几乎归零——模型 *本身* 就是简短答案的提供者，只要你换个 prompt 模式让它表现出这一面。

这是个很优雅的迁移：从依赖外部判断，到挖掘内部已有的能力。

* 内核：一句话带走

ConPress 抓住了一个反直觉的副作用—— *给模型加压（多题挤在一起）反而让它瘦下来* ——并把这个副作用变成了不需要任何外部监督的自瘦身回路。这是"自监督"思想在推理压缩领域的一次干净落地：你想要的能力其实已经在模型里，只是要找到对的 prompt 把它逼出来，再拿出来训自己。

* 可能的下一步

- *更猛的 contextual pressure*：从多题扩到多文档、多角色、多模态，看压短能压到什么程度
- *推理时直接 prompt 化*：如果多题就能压短，是不是可以训练阶段都省了，纯靠 inference-time prompt 工程
- *压短的边界*：哪些题型/难度上 self-compression 会反噬准确率？什么时候模型会"压过头"

第一个方向最有可能——已经有 LongGenBench、batch prompting 这条线在铺路。ConPress 等于在那条线上又开了一个新出口：原来批量推理不只是省 API 调用，还能反过来变成训练信号。
