* One-liner

Teach a reasoning model to answer correctly even when its thinking is cut short mid-way — not by making it think less, but by making it capable of "handing in the exam" at any point where it is interrupted.

* How the Problem Evolved — One Continuous Thread

** Stop 1: 2022, CoT Teaches Models to "Think While Answering"

The story starts with a simple observation: if you ask a model to write out its reasoning before giving an answer — "let's think step by step" — its accuracy on math problems jumps noticeably. That is Wei et al.'s chain-of-thought prompting.

The benefit is immediate: large models shift from "guessing an answer" to "reasoning toward an answer."

But it brings what looks like a harmless side effect: the model must produce a long "thinking" segment before it will commit to an answer. A primary-school arithmetic problem that could be answered in a few dozen tokens now routinely consumes hundreds or thousands.

At the time nobody saw this as a problem — getting the right answer was what mattered, and a few extra tokens seemed trivial.

** Stop 2: 2025.01, DeepSeek-R1 Bakes "Long Thinking" into the Model's Core

Three years later, the situation has intensified. DeepSeek-R1 uses reinforcement learning to train models to *spontaneously* produce extremely long reasoning chains — a single problem can trigger thousands of tokens of internal deliberation, repeated self-doubt, self-correction, and "wait, let me reconsider."

The payoff: competition-level mathematics problems are solved.

The cost: inference time and token expenditure explode. Worse, R1 launches a distillation pipeline — using R1 as a teacher to train smaller models to reason the same way. The students inherit not only the reasoning ability but also the *teacher's verbosity* — even trivial problems get long-winded treatments.

This is when the community names the disease: reasoning models suffer from *overthinking*.

** Stop 3: 2025.01, s1's Blunt Experiment — What Happens If We Just Cut the Thinking Off?

s1 takes a direct approach: since the model thinks too much, just intercept it at inference time. The authors call this *budget forcing*.

Two specific moves:
- *Think shorter*: once a token budget is reached, forcibly insert an end-of-thinking marker and make the model deliver its answer immediately.
- *Think longer*: whenever the model tries to stop, covertly replace the stop token with "Wait," forcing it to continue.

The second trick actually works — models often correct errors after being made to pause. But *the first trick is costly*: cutting off a problem that genuinely requires more thought causes a cliff-edge drop in accuracy.

Here the problem is stated clearly for the first time: *a model trained to think at length simply cannot be interrupted mid-chain at inference time*. It has no ability to "hand in the exam halfway through."

** Stop 4: 2025.02, TokenSkip Thinks More Cleverly — Skip the Tokens That Don't Matter

s1's "cut at a fixed position" is too crude. TokenSkip takes a different angle: if much of a CoT is filler, why not *learn which tokens can safely be skipped*?

The method:
1. Use an importance scorer (drawing on LLMLingua-2) to assign a score to every token in the reasoning chain.
2. Drop the low-scoring tokens according to a compression ratio γ, producing a "condensed" chain.
3. Fine-tune the model on these condensed chains so it learns to generate them directly.

Results are solid: dropping 40 % of tokens costs only 0.4 % in accuracy.

But a new problem is now buried in the approach: *how do you know which tokens are truly "unimportant"?* The importance scorer is an external judge, and nobody can verify whether its judgments are correct. Concurrent work — LightThinker, CoT-Valve, O1-Pruner — all travel the same "select-and-compress" road, each answering the same underlying question: which words stay, which words go?

** Stop 5: 2026.02, TRSD Flips the Table — Stop Selecting; Selection Was Never the Right Question

The genuine insight in TRSD is that it sees through the hidden assumption every previous method was resting on.

It asks: why should we assume "some tokens matter more than others"? If a model has truly learned to reason, then *at any point where it is interrupted it should be able to give the best possible answer from whatever thinking it has already done*.

In other words: the bug is not in the chain — it is in the model. The model was never trained to "hand in the exam halfway through."

TRSD's approach is disarmingly simple:
1. Take a frozen teacher model; let it think through a problem *completely* and record its final answer distribution.
2. Train a student model (same architecture, same initialization) to match that "fully-thought-out" answer distribution *while seeing only a prefix of the thinking chain*.
3. Crucially, the truncation position is *random* — no selection, no scoring, no weighting.

What does this training objective say? *No matter where you are interrupted, produce the same answer you would have reached by thinking it all the way through.*

Results:
- At any token budget, truncation no longer causes accuracy to collapse.
- *No length penalty is applied anywhere*, yet the trained model spontaneously generates shorter chains — because it has learned to "arrive at clarity sooner."

** What Is Really Happening Across This Whole Arc

From CoT to TRSD, the surface story is a technical progression about "how to shorten reasoning chains." The deeper story is a shift in how the problem is framed:

| Stage        | Hidden Assumption                                         |
|--------------+-----------------------------------------------------------|
| CoT          | Writing more tokens is harmless                           |
| DeepSeek-R1  | The longer the chain at training time, the stronger the   |
|              | capability                                                |
| s1           | Length can be controlled by external intervention at      |
|              | inference time                                            |
| TokenSkip    | Information density varies across the chain; compress it  |
| TRSD         | Varying information density is a false premise —          |
|              | the real problem is that models cannot "wrap up mid-way"  |

Each step peels away one hidden assumption from the step before. TRSD peels away the seemingly innocent idea that "tokens have importance scores" — arguing that rather than expending effort judging which tokens matter, it is better to train the model directly to conclude gracefully from any position.

* Where This Could Go Next

If the TRSD line of reasoning holds, research premised on "which tokens matter" will gradually lose its footing. Promising next steps:

- *Recast "truncation robustness" as a pretraining objective* rather than a post-training patch.
- *Combine with speculative decoding*: speculative decoding already "guesses ahead"; if the model is inherently robust to interruption, the error tolerance of speculation grows considerably.
- *Adaptive budgeting*: let the model decide "I have thought this through — I will answer now." TRSD already gestures at this implicitly, but it has not been modeled explicitly.
