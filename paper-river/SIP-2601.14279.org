* 问题之河

大模型推理为什么越聊越慢、越聊越占显存？因为它要把对话里每一个 token 的"记忆"（KV 向量）都背在身上。一百轮对话，可能就是几十 GB。

所以一个朴素的想法冒出来：*能不能丢掉一些没用的 token，只留有用的？*

听起来简单。但"哪些 token 是没用的"——这个看似工程问题的问题，已经被研究了五年，方法一代比一代花哨：从"留前几个+留最后几个"，到"看历史注意力分数"，再到"训练一个小模型来预测重要性"。每一代都觉得自己抓到了前人的命门。

直到这篇论文站出来说：你们这一整条路可能根本走不通。

* 演化叙事

** 2017 | Transformer: 一切麻烦的起点

*** 它看到了什么问题

RNN 太慢。要算第 100 个词，必须先把前 99 个词一个一个跑完。机器闲着算不动。

*** 它的解法

扔掉时序依赖，让每个 token 同时看到之前所有 token——*自注意力*。这样训练时整批数据可以并行算。

*** 为什么这招管用（以及哪里埋了新坑）

并行了，训练快了，模型也变强了。但推理时麻烦来了：生成第 N 个 token，要让它"看"前 N-1 个 token 的 Key/Value。如果每生成一个 token 都重算，O(N^2) 直接爆炸。

解决办法是 *KV cache*：把每个 token 的 K/V 算一次存住，后续生成直接复用。看起来很美好——直到上下文从 2K 涨到 32K 再涨到 1M。这堆 cache 就成了显存的怪兽。

*这个怪兽，就是后面所有论文的起点。*

** 2023 | StreamingLLM: 砍掉中间, 只要头和尾

*** 它看到了什么问题

长对话 (比如客服、多轮 agent) 一开就是几千 token，KV cache 显存吃不消。最简单的想法：那就丢一些 token 嘛。但具体丢哪些？

*** 它的解法

作者观察到一个奇怪的现象：模型对前几个 token (尤其是第 1 个) 的注意力分数极高——哪怕这些 token 是 "the" 或 "<s>" 之类的语义垃圾。他们管这叫 *attention sink* (注意力汇点)。

于是策略很简单：*留前 4 个 token + 留最近的 N 个 token*，中间全砍。

*** 为什么这招管用（以及哪里埋了新坑）

attention sink 是模型 softmax 归一化产生的副作用——总得有个地方"卸"掉无处可去的注意力。砍掉这几个 sink, 模型崩。留住它们，模型基本不掉点。

但代价显而易见：*中段信息全丢了*。如果用户在第 500 个 token 提了个关键名字，到第 5000 token 你想引用它，对不起，扔了。

下一篇论文要解决的就是这件事。

** 2023 | H2O: 看谁历史上"被关注"得多, 谁就留

*** 它看到了什么问题

StreamingLLM 太一刀切。中段的 token 不是都没用，有些是被反复回看的关键信息（比如人名、术语、数字）。怎么把这些"重要的中段 token"挑出来？

*** 它的解法

作者提出 *Heavy Hitter Oracle*：对每个 token，累加它从出生到现在所有后续 query 给它的注意力分数。分数高的就是"重击者"（heavy hitter），留下；分数低的，砍。

类比：开会时，谁的发言被反复引用，谁就是关键人物。

*** 为什么这招管用（以及哪里埋了新坑）

直觉上，过去被频繁注意的 token，未来大概率还会被注意。这个假设在很多任务上确实成立。

但有两个新坑：
1. *累积分数有偏*：早出生的 token 攒分时间长，天然占便宜
2. *过去不等于未来*：如果用户突然换话题，问一个之前没人在意的细节呢

** 2024 | SnapKV: 别看一辈子, 只看最后一眼

*** 它看到了什么问题

H2O 的累积分数包含太多"过时信息"。你在长 prompt 开头讨论 A，结尾问 B，关于 A 的 token 历史分数虚高，但其实快被丢弃了。

*** 它的解法

只看 *prefill 最后一个观察窗口* 的注意力——也就是 prompt 处理完那一刻，模型正在"瞄"哪些 token，留这些就行。

类比：不看一个员工五年的考核记录，只看他最近一周在干什么。

*** 为什么这招管用（以及哪里埋了新坑）

prefill 末尾的注意力是离生成最近的信号，比累积分数更"新鲜"。在很多 benchmark 上确实赢过 H2O。

新坑：*所有这些方法都在赌"未来 query 长得像现在 query"*。可是 LLM 之后要回答什么问题，本质上是个未知函数。用现在的 attention 预测未来的需求——是不是从根上就有问题？

这个怀疑就是学习派登场的契机。

** 2024 | 学习派 (DMC / TRIM-KV / ChunkKV): 既然人想不清, 让模型自己学

*** 它看到了什么问题

启发式 (位置、累积分数、prefill 注意力) 都是人在拍脑袋。机器学习领域有句老话：*能学的别手工*。既然"哪些 token 重要"是个预测问题，那就训一个小模型去学。

*** 它的解法

各家细节不同，但骨架一致：
- 输入：每个 token 的 K/V 向量
- 输出：一个重要性分数
- 训练：用某种代理目标 (重建 loss / 蒸馏 / 监督) 让模型学会打分

比如 DMC 学一个 "merge or keep" 的决策，TRIM-KV 学保留门控，ChunkKV 假设注意力服从高斯分布做期望计算。

*** 为什么这招管用（以及哪里埋了新坑）

paper 们都报告 SOTA。看起来学到了启发式没看到的模式。

但 TRIM-KV 自己 2025 年的后续更新里悄悄承认：训练出来的门控 *收敛到了一个近似位置启发式的解*。

也就是说——模型学了半天，学出来的东西约等于"留前 4 个 + 留最近 N 个"。

这个尴尬的发现，需要一篇论文系统性地把它讲清楚。

** 2026.01 | SIP (本文): 把学习派做到顶, 然后宣布它不行

*** 它看到了什么问题

之前的学习派论文有个共同毛病——*评估不严谨*。
- 不跑多个随机种子
- 不和"随机选 token"这个最弱的 baseline 对照
- 在少数 benchmark 上调好就发

如果学到的真的是位置的代理，那为什么没人做个干净的对照实验把这件事钉死？

*** 它的解法

作者 Brady Steele 自己设计了一个 *尽可能强的学习派*——叫 SIP (Speculative Importance Prediction)：
- 1.7M 参数 (不算小)
- 多步前瞻：不止预测下一步重要性，预测未来多步
- 交叉注意力机制
- 各种 trick 拉满

然后用一套 *诚实到苛刻* 的评估：
- 5 个随机种子 (看方差)
- 4 个保留预算 (看不同压缩率)
- 3 个任务 (看泛化)
- baseline 包括：纯随机选、StreamingLLM (前 4 + 后 N)、H2O、SnapKV、还有 prefill attention 本身

*结果*：SIP 打不过随机。或者勉强打平 H2O，但显著不如最强的位置启发式。

*** 为什么这招管用（以及哪里埋了新坑）

这是这篇论文最漂亮的部分——它不止说"学不动"，还给了一个机制解释：

> 在 non-query-aware 设定下 (只用 KV 自己, 不知道未来 query)，重要性预测有一个 *循环依赖*：
>
> 哪些 token 重要 取决于 未来生成什么
> 未来生成什么 取决于 你保留了哪些 token
>
> 你压根没法跳出这个循环。

所以 KV 向量里本来就没有足够的信号，让你能预测一个还没出现的 query 会需要什么。学得再花哨也没用。

留下的新问题：*那要么放弃压缩，要么把 query 拉进来。*

* 前沿延伸

这篇论文 2026 年 1 月刚出，但回声已经响了。

** 同向证据: Guo et al. 2025 "Hold onto that thought"

几乎同期，另一组研究者从不同角度得到了相似的负面结论。他们在更大的模型上验证：学习派的"学到的"几乎可以被简单的"留长尾位置"复现。两篇互为佐证。

** 异向突破: TokenButler (Wang et al. 2025)

如果 non-query-aware 的学习派注定失败，那 *query-aware* 呢？

TokenButler 把每一步生成时的 query 当输入，*动态* 给历史 KV 打分。这避开了 SIP 提出的循环依赖：query 已经出现，不用预测了，直接算就行。

结果：query-aware 的学习派开始稳定打赢启发式。

这反过来印证了 SIP 的诊断——*循环依赖才是真正的瓶颈*。打破它的方法不是更聪明的模型，是更晚的决策时机。

* 洞见

把整条线压平了看，真正发生的不是"压缩技术的迭代"，而是 *信号边界的发现过程*：

1. *第一阶段 (2023)*——人们以为 KV cache 里到处都是冗余，随便砍
2. *第二阶段 (2023-2024)*——发现砍不能乱砍，得有依据。先找位置依据，再找注意力依据
3. *第三阶段 (2024)*——以为依据可以学得更好，进入学习派狂欢
4. *第四阶段 (2026)*——SIP 用严格实验钉死：KV 向量里的信号已经被简单启发式榨干了，更复杂的模型只是在重新发现"位置"

这是机器学习领域反复出现的剧本：*一个看似可学的任务，其实信息论上就没法学*。线性回归打不过决策树不是模型问题，是数据没那么多模式可学。这里类似——KV 自己没说出未来 query 会问啥，再大的 scorer 也变不出来。

*下一步往哪走？* 三个方向已经清楚：
- *Query-aware*：决策推迟到 query 出现时 (TokenButler 路线)
- *换层面*：不在"token 选择"上较劲，转向量化 (KIVI)、跨层共享 (MiniCache)、prompt 压缩
- *接受不可压*：承认部分场景就是要付显存，把工程焦点转回 batching / paged attention

* 启发

这条河给做研究的人三个落点：

1. *负面结果有价值，但必须做得干净*
   SIP 的贡献不是发明了新方法，而是用 5 seeds x 4 budgets x 3 tasks 的严格评估，把"学习派比启发式好"这个共识戳破了。落点在能用：*如果你怀疑某个领域在自欺欺人，先把对照实验做到无可挑剔，再开口*

2. *警惕"代理目标"陷阱*
   学习派训出来的 scorer 收敛成了位置启发式——因为训练目标 (重建 loss / 蒸馏) 和真正想要的"未来 query 的需求"之间隔了一层。机器只能优化你给的目标。落点：*下次想训一个模型解决问题前，先问自己——我的训练信号和真正的目标之间有多少层代理？每一层都可能让模型滑向一个 trivially 的解*

3. *从循环依赖里跳出来的唯一办法是改变时序*
   SIP 的诊断很优雅：哪些 token 重要取决于未来 query，未来 query 取决于保留了哪些 token——闭环。打破闭环的不是更大的模型，是 *把决策推迟到信息可得之后* (query-aware)。落点：*遇到任何"鸡生蛋蛋生鸡"的问题, 先别想着用算力硬怼, 想想能不能把决策时机往后挪*
