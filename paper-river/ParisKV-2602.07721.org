* 问题之河

一个大模型读完一百万个 token 的上下文以后，每生成一个新 token，理论上要把这一百万个 token 全部"回看"一遍——这件事既慢又费内存。能不能只回看其中"真正重要"的那一小撮？

这个问题难就难在：你不知道哪些 token 重要，要找出来又要够快、够准、还要在生成过程中不会"看走眼"。过去七八年里，人们一代一代换思路——从"砍掉不重要的"到"留着但只取重要的"，再到"把不重要的搬去 CPU"，每一代都解决了上一代的问题，又留下了新的坑。ParisKV 是这条线 2026 年初的最新一站。

* 演化叙事

** 2017 | Transformer: 一切的起点

*** 它看到了什么问题

RNN 必须一个 token 接一个 token 算，无法并行，训练长序列贼慢。

*** 它的解法

扔掉循环结构，纯靠 attention：每个位置同时"看"前面所有位置，并行算完。

*** 为什么这招管用（以及哪里埋了新坑）

并行起来了、训练快了，模型也能学到更长的依赖。但留下一颗定时炸弹：attention 的算量和显存都是序列长度 N 的平方。当后来人们想做"100 万 token 上下文"时，这颗炸弹就引爆了——光是把每个 token 的 Key/Value 缓存（KV cache）存下来，显存就先满了。

后面所有的故事，都是在解这颗炸弹。

** 2023 | H2O: 砍掉不重要的

*** 它看到了什么问题

长生成时 KV cache 像滚雪球：每生成一个新 token，就要给它存一份 K 和 V，永久占着显存。生成几万 token 之后，显存直接爆。

*** 它的解法

观察到：注意力分数大部分集中在少数"重锤"token 上（Heavy Hitters）。那干脆——给 cache 设一个固定预算，超出就把"不重要"的 token *永久扔掉*。

*** 为什么这招管用（以及哪里埋了新坑）

显存有了上限，推理也快了。但"永久扔掉"是赌博：你怎么知道现在不重要的 token，五千步之后不会突然变重要？真扔错了，模型没法再回头看，回答就崩了。这种"早期被丢的 token 后期变关键"的失败模式，正是下一代要解决的。

** 2024 | Quest: 别扔，只是先放着

*** 它看到了什么问题

H2O 那一类"扔 token"方案有一个根本毛病：扔了就回不来。但 KV cache 大归大，其实你不需要每一步都"看"全部——只是每一步看的"那一小撮"会变。所以问题不是"该扔哪些"，而是"该取哪些"。

*** 它的解法

KV cache 全部留着，但按 page 分块，给每个 page 算一个粗略的"代表向量"。每次 query 来时，先用这些代表向量算一遍，挑出 top-k 个最相关的 page，只 attend 这些 page 里的 token。

*** 为什么这招管用（以及哪里埋了新坑）

不会扔错了，因为本来就没扔。但全留着的 KV cache 仍然要全部驻留在 GPU 显存——100 万 token 还是装不下。问题被推到了下一关：怎么让"全留着"在物理上可行。

** 2024 | RetrievalAttention / MagicPIG: 搬去 CPU，用近似搜索

*** 它看到了什么问题

Quest 思路对了，但显存是硬约束。GPU 显存有限，长上下文的 KV 必须找地方放——CPU 内存有几百 GB，但 GPU 读 CPU 慢。同时，Quest 那种"算所有 page 的代表向量再取 top-k"在百万 token 量级也开始变慢，因为 top-k 本身就是 O(N)。

*** 它的解法

两件事：(1) 把大部分 KV 卸载到 CPU 内存；(2) 在 CPU 上建一个近似最近邻（ANN）索引，比如用 LSH（局部敏感哈希）。每步 query 不再"扫所有 page"，而是丢进哈希函数，只去几个"碰撞桶"里取候选。MagicPIG 用 LSH 采样估算 attention；RetrievalAttention 用 ANN 索引。

*** 为什么这招管用（以及哪里埋了新坑）

显存问题解决了——百万 token 装得下。但搜索从 GPU 跑到了 CPU：CPU 调度本身就是开销，CPU 到 GPU 的数据拷贝又是开销。LSH 的召回率还不太稳，有时候真正最相关的 token 哈希散到了别的桶里，就被漏了。

** 2024 | PQCache: 用产品量化压缩，再聚类

*** 它看到了什么问题

LSH 的随机性让召回率不可控；MagicPIG 那种采样估计算出来的 attention 又有偏差。需要一种更"工程化"的索引：召回稳定，预算可控。

*** 它的解法

借鉴近似最近邻搜索社区的产品量化（PQ）：把每个 key 向量切成几段，每段单独聚类成 256 类，存成几个字节的"码字"。查询时也切段，查表算近似距离。CPU 端跑 k-means 学到的聚类中心当索引，固定取出 20% 的 token 给 GPU 算 attention。

*** 为什么这招管用（以及哪里埋了新坑）

召回比 LSH 稳，预算也确定了。但埋下了 ParisKV 要解的关键坑——*decoding drift*：聚类中心是在 prefill 阶段（喂入提示词那一刻）用当时的 keys 学出来的。随着模型一步步生成新 token，新的 keys 不停加入，原来的聚类中心越来越"不像"真实的 key 分布。Figure 1(b) 直接画出来：prefill 学到的蓝色质心，和后续 decode 产生的红色 key 越拉越远。结果就是长生成时召回率塌方。再加上索引和搜索还在 CPU 端，CPU 仍然是瓶颈。

** 2026 | ParisKV: 解析式聚类 + GPU 原生 + UVA 按需抓取

*** 它看到了什么问题

PQCache 和 MagicPIG 都有同一个病根：聚类/哈希是"学出来的、依赖数据的"——一旦数据分布漂移，它就退化。在百万 token、长生成的场景下漂移不可避免，召回率必然崩。同时整条流水线被 CPU 端搜索和 CPU→GPU 拷贝卡住，延迟下不去。

*** 它的解法

三个动作连起来：

1. *消除漂移*：先把每个 key 做 ℓ2 归一化，再过一个共享的随机正交旋转（SRHT）——把所有 key/query 都映射到单位球面上的"均匀分布"。然后用一组 *解析式*、不依赖任何数据的聚类中心：所有符号模式 {±1/√m}^m，2^m 个方向均匀铺满球面。因为是预定义的，新生成的 key 永远靠近某个中心——根本不会漂移。

2. *GPU 原生粗筛*：把上一步的多个子空间各自当作一票，看每个 key 的"分配中心"是否撞上 query 的 top 中心。撞上就加分，高分 key 进候选——本质是 GPU 上的多哈希碰撞投票。

3. *4-bit 量化重排 + UVA 按需抓*：候选里再用 4-bit 量化码（1-bit 符号 + 3-bit 量级 + 一个标量校正项）算近似内积排序。最后选中的 top-k，靠 CUDA 的 Unified Virtual Addressing 直接从 CPU 内存里"逐 key 抓取" full-precision KV，不走 CPU 调度、不走显式 memcpy。

*** 为什么这招管用（以及哪里埋了新坑）

把"学出来的索引"换成"几何上预定义的索引"，一刀切掉了漂移的根。把搜索全搬回 GPU，再用 UVA 跳过 CPU 调度，延迟拉下来了——百万 token 解码相对 MagicPIG 快 17×，相对 PQCache 快 44×。在 AIME25 等长生成任务上，pass@8 比两者高 40-77 分。

新坑大概在这些方向：(a) 子空间维度 m 的选择是 2^m 个中心的权衡，子空间太小覆盖差、太大爆内存；(b) UVA 抓取在多用户/多请求并发下，CPU 内存带宽可能成为新瓶颈；(c) 4-bit 量化重排虽然加了校正项，但极端分布下仍可能低估真实内积。

* 前沿延伸

截至 2026-05，ParisKV 才发布三个月，尚无明确公开的"后续批判者"。但从这条线本身的节奏看，可以预测两个方向：

- *训练阶段就考虑 retrieval*：MoBA、NSA 这一线（"训练时 retrieval"）正在和 ParisKV 这种"推理时 retrieval"汇流。下一步可能是把 ParisKV 的解析式聚类直接烧进模型训练目标里——让 query/key 天生就适合这种几何索引。
- *跨请求共享索引*：单请求场景下 ParisKV 已经很快，但服务化时如果上千个请求共享同一个超长上下文（如 RAG 知识库），如何让索引和 UVA 抓取在多请求间复用，是下一个工程瓶颈。

* 洞见

这条线表面上是"KV cache 怎么压"，背后真正在发生的是 *两个隐喻的死循环和破局*：

1. *"学习"vs"几何"的钟摆*。十年前深度学习的胜利让大家本能地"什么都学一下"——索引也要学（PQ 学聚类、LSH 学投影）。但学出来的东西必然依赖训练分布，一旦分布漂移就退化。ParisKV 是这条线里第一个明确放弃"学索引"的——它说：球面上均匀铺点就完了，不要学。这是一个反潮流的判断，但它管用，因为它把"漂移"这个问题从根上铲掉了，而不是去打补丁（增量更新聚类、定期重训之类）。

2. *瓶颈的"打地鼠"*。这条线非常清晰地呈现了系统优化的规律：你解决了显存瓶颈（搬到 CPU），瓶颈就变成 CPU 调度；你解决了 CPU 调度（搬回 GPU + UVA），瓶颈就变成内存带宽。每代论文真正的贡献不是"快了多少倍"，而是"把瓶颈推到了下一个地方"。ParisKV 把瓶颈推回了 CPU 内存带宽和 PCIe，这就是下一代要打的。

预测：下一代要解的，要么是 ParisKV 留下的"多请求并发下 CPU 内存带宽瓶颈"，要么是更激进的——干脆让训练目标天生适配这种几何索引，让 retrieval 和模型本身合二为一。

* 启发

对"怎么做研究"的启发：

- *系统性能优化的论文，价值不在"快了 X 倍"，而在"找到了下一个瓶颈在哪"*。读这类论文，要追问的是"它把瓶颈推到哪了"，而不是"它快了多少"。
- *当一个领域反复用"学习"打补丁却越打越多时，反方向"扔掉学习"的解法值得严肃考虑*。ParisKV 的 SRHT + 符号格点中心，本质上是 30 年前 ANN 社区就有的几何工具，在深度学习狂热里被冷落了——直到漂移把所有"学出来的索引"都打趴下，它才重新进场。
- *读最新论文时，先倒着挖三层*。不挖你看到的是"又一个 KV cache 优化"，挖完你看到的是"长上下文 LLM 推理这十年的核心矛盾"——前者你两周后就忘了，后者你两年后还能用。

落点在能用：这意味着，你以后看到任何"为加速 X 而学习 Y"的论文，可以本能地问一句"Y 漂移的时候怎么办"——如果作者答不上来，这篇论文大概会被下一代用"几何方法"取代。
