* 问题之河

大模型推理时，历史 token 的 Key-Value 缓存（KV Cache）该留哪些、扔哪些？

这个问题的核心矛盾是：*注意力权重说一个 token 重要，它就真的重要吗？*

在短文本时代这不算大事。但当 o1-style 推理链动辄几千 token，KV Cache 占满显存，
模型只能靠「注意力得分高的 token」来决定取舍——而这个标准，天然是近视的：
注意力权重主要反映当前 token 跟哪些历史 token 关系近，却看不见那些远在几百 token
之外、但在关键时刻才猛然被需要的「信息密集点」。

于是一代代研究者反复问同一个问题：*什么信号能真正衡量一个 token 对未来的贡献？*

* 演化叙事

** 2020 | BigBird / Longformer: 训练期稀疏注意力——问题的远祖

*** 它看到了什么问题

原始 Transformer 的全注意力是 O(n^2)：每个 token 都要跟所有历史 token 算相似度。
长文本直接算不起。

*** 它的解法

BigBird 和 Longformer 在*训练阶段*就把注意力图稀疏化：保留局部窗口（看附近的）
加上几个全局锚点（看全局的），其余连接全砍。

把注意力矩阵想象成一张密织网——这两篇论文把它改成了一张渔网：
大多数线是连的，但有规律地留了许多孔。

*** 为什么这招管用（以及哪里埋了新坑）

训练期稀疏化效果不错，但有个根本问题：
模型推理时，KV Cache 还是把所有历史 token 全存着，等到内存撑不住才慌。
这条线关心的是「计算量」，没有解决「存储量」。
这个坑，留给了 2023 年的一批研究者。

** 2023 | Scissorhands / H2O: 推理期动态丢 token——第一把剪刀

*** 它看到了什么问题

大模型生成长文本时，KV Cache 持续增长，内存和带宽是瓶颈。
没有人在推理期动态地丢掉「不重要」的历史 token。

*** 它的解法

Scissorhands（Rice，2023-05）和 H2O（UT Austin / Stanford，2023-06）几乎同时
各自独立到达同一个答案——*持续重要性假设*：

一个 token 在过去的生成步骤里受到高注意力，它在未来的步骤里也会被高频关注。
所以只需要维护一个固定大小的缓存，每步扔掉累积注意力得分最低的 token。

H2O 还给这套逻辑加了一点数学支撑，把 token 选择问题建模为动态子模优化，
并证明了贪心算法有常数比保证。

想象成一个办公桌：桌面空间有限（显存），你只保留最近最常用的文件。
不同的是，「最常用」这里是按注意力权重统计的。

*** 为什么这招管用（以及哪里埋了新坑）

确实管用：H2O 在 OPT-6.7B 上做到了 29x 吞吐提升。
但两个坑同时浮出：

坑一：*累积注意力得分有位置偏差*。序列越早的 token 被关注的机会越多，
分数天然就高，不反映真实重要性。

坑二：*只压缩了生成阶段的 KV，没压缩 prompt 阶段的*。
而对真实业务来说，一个 128K 的 prompt 才是内存大头。
这个坑，直接导出了 SnapKV。

** 2023 | StreamingLLM: 发现注意力汇点——一个小洞见解决大崩溃

*** 它看到了什么问题

如果用滑动窗口注意力（只看最近 N 个 token）来省内存，
当文本超过窗口大小，之前的 token 全部移出窗口，模型 PPL 直接爆炸崩溃。

*** 它的解法

MIT Han Lab 仔细分析注意力图发现：*最开始的几个 token 永远得到远超正常的注意力*，
不管这几个 token 内容是什么。他们称之为「注意力汇点（Attention Sink）」。

原因是 softmax 必须给所有位置分配概率质量，当模型不确定该看哪里时，
它会把多余的注意力「倾倒」到序列头部。这是 softmax 的一种「卸载阀」行为。

解法很简单：滑动窗口保留最近 N 个 token 的同时，*永远保留最前面 4 个汇点 token*。
实验跑到 4M token 长度，困惑度平稳如初。

*** 为什么这招管用（以及哪里埋了新坑）

管用，但只是位置启发式——跟 token 内容无关，跟 token 信息量无关。
它解决的是「模型不崩」的工程问题，没有回答「哪些 token 真的重要」的科学问题。

这个问题，信息论方向的研究者在同一时期已经开始从另一个角度切入了。

** 2023 | Selective Context + LLMLingua: 信息论的平行支线——难预测的才有用

*** 它看到了什么问题

上面所有方法都用「注意力得分」衡量 token 重要性。
但注意力得分是模型对「现在需要看哪里」的主观判断，
不是 token 本身携带多少信息的客观度量。

有没有一种不依赖注意力的重要性度量？

*** 它的解法

Edinburgh 的 Yucheng Li（也就是后来 InfoKV 团队的合作者所在机构的生态圈）
2023 年底发表了 Selective Context：
用语言模型在当前上下文下给每个 token 打一个*自信息*分数：

   I(x) = -log p(x | 上下文)

意思是：这个 token 在当前语境下越难猜，它携带的信息就越多，越应该被保留。

把这个想象成猜词游戏。如果句子写着「我去了___买了苹果」，
「超市」很容易猜，信息量低；「岛屿」几乎猜不到，信息量高。
Selective Context 就是优先留下「难猜的」，扔掉「一眼就能猜到的」。

Microsoft 的 LLMLingua 同期做了类似的事，
用一个小模型的困惑度来做大模型 prompt 的粗到细压缩，实现 20x 压缩率。

*** 为什么这招管用（以及哪里埋了新坑）

这条线做的是*压缩输入 prompt 本身*，不是 KV Cache。
它证明了「自信息/熵 = token 重要性的好代理」，但还没有人把这个信号
搬进推理时的 KV Cache 淘汰决策里。

** 2024 | SH2: 同一作者组的前序实验——高熵 token 是事实的栖息地

*** 它看到了什么问题

大模型产生幻觉时，问题通常发生在哪些 token 上？

*** 它的解法

Jushi Kai 和 Zhouhan Lin（就是 InfoKV 的第一作者和通讯作者）在 SH2 里发现：
*模型预测不确定性最高的 token（熵最大的 token）往往是专有名词和事实性词汇。*

他们的方法是：先找出高熵 token，把它们重新附加到上下文里，然后做对比解码来抑制幻觉。

这是一个关键实验：它不直接处理 KV Cache，
但给了团队一个坚实的经验支撑——*熵高的 token 信息密度高*，
这个直觉可以从幻觉减少迁移到 KV Cache 的 token 选择。

*** 为什么这招管用（以及哪里埋了新坑）

EMNLP 2024 发表，效果验证充分。但它的「重新附加 token」方法不能直接用于推理期 KV 淘汰——
那需要一个和注意力分数可以融合的连续评分函数，而不是二值的「高熵/低熵」标签。

** 2024 | SnapKV: 把压缩移到 prefill——解决了对的问题

*** 它看到了什么问题

H2O 和 Scissorhands 只压缩*生成阶段*新增的 KV。
但对一个有 128K prompt 的任务，prefill 阶段产生的 KV 才是内存怪兽。

*** 它的解法

SnapKV 的关键洞见是：*在开始生成的第一个 token 之前，我们已经能知道哪些 prompt token 重要了。*

方法分三步：
1. 把 prompt 分成「前缀」和末尾的「观察窗口」（后 16-64 个 token，通常是指令/问题）
2. 用观察窗口里的 query 向量，给所有前缀 token 投票打分
3. 每个注意力头独立选出得分最高的 top-k prefix token，其余的直接丢掉

一次性决定，之后整个解码过程 KV Cache 大小固定，不再增长。

把它想象成考前复习：在真正开始答题之前，先扫一眼题目，
划出哪些知识点相关，然后只翻那些页的笔记。

*** 为什么这招管用（以及哪里埋了新坑）

在 A100 上，16K 输入下内存减少 8.2x，速度提升 3.6x，LongBench 上精度几乎不降。

遗留的坑：*所有层用同等大小的 KV 预算*。
但实验表明，Transformer 浅层的注意力图密集（很多 token 都有贡献），
深层的注意力图稀疏（只有少数 token 重要）。
统一预算要么浪费了深层的空间，要么压死了浅层的信息。

** 2024 | PyramidKV: 让预算跟着注意力走——金字塔形分配

*** 它看到了什么问题

SnapKV 每层固定同等预算，和实际注意力稀疏度不匹配。

*** 它的解法

仔细测量了 32 层 LLaMA 的注意力分布，
发现了「金字塔形信息漏斗（Pyramidal Information Funneling）」：

- 第 1-8 层（浅层）：注意力分散在几乎所有 token 上，需要大缓存
- 第 9-24 层（中层）：注意力开始汇聚，中等缓存
- 第 25-32 层（深层）：注意力高度集中在极少数 token，小缓存足够

于是 PyramidKV 就按照这个形状分配 budget：底层宽、顶层窄。
同样的总预算，效果比 SnapKV 好——在极低保留率（0.7%）下，
TREC 任务精度比竞品高 20.5 个百分点。

*** 为什么这招管用（以及哪里埋了新坑）

金字塔 budget 解决了层间分配问题。
但 *token 选择标准本身还是注意力得分*，没有变。

核心遗漏：注意力得分衡量的是「和当前上下文的相关性」，
是近视的——它捕捉不到「这个 token 对未来远距离上下文的贡献」。
在短文本里这不是大问题，但在长达几千 token 的推理链里，
一个关键的中间推导步骤可能在几百 token 之后才被用到，
注意力得分会把它误判为低重要性然后丢掉。

这个坑，就是 InfoKV 的切入口。

** 2026 | InfoKV: 注意力 + 熵——两把尺子量不同的东西

*** 它看到了什么问题

前人所有方法（SnapKV、PyramidKV、D2O、FastKV...）选 token 只靠一把尺子：注意力得分。

InfoKV 的研究者做了一个关键实验：定义了「前向影响力（Forward Influence）」——
把某个 token 从上下文里抠掉，看对后续预测分布影响多大（用 KL 散度量）。

实验结论令人意外：
- *注意力高分 token* 的前向影响力主要集中在*近邻 context*（几十个 token 以内）
- *高熵 token*（模型预测不确定的 token）的前向影响力却延伸到*远距离 context*（几百 token 之外）

推理链天然就是长程依赖的场景——一个中间步骤的结论，可能在 500 token 后才被引用。
靠注意力得分选 token，恰好会系统性地漏掉这类「高熵的远程关键 token」。

*** 它的解法

InfoKV 同时用两个信号给每个 token 打分：

*信号一：层内注意力得分 A_i^(l)*
同 SnapKV/PyramidKV 的做法，用观察窗口的 query 给 prefix token 打分。

*信号二：熵得分 E_i^(l)*
对每个 token 位置，用当前层的 hidden state 推算一个预测分布（对词表 top-k）
算 Shannon 熵 H(x_i)，再按层间表示漂移（cosine 距离）加权，
越接近最终输出层的层，权重越大。

融合公式：
  S_i^(l) = alpha * A_i^(l) + (1 - alpha) * Softmax(E^(l))_i
  默认 alpha = 0.9

两个分数归一化后加权求和，选得分最高的 top-k token 保留。

把它想象成用两种方式筛选会议记录：
注意力得分找「刚才刚讨论过的要点」（近程相关），
熵得分找「说错了或者争议很大的句子」（高不确定性=信息密集）。
两种方式捕捉的信息是互补的。

*** 为什么这招管用（以及哪里埋了新坑）

在 Llama-3.1 / Llama-3.2 / DeepSeek-R1 上，
在 AIME 2024 / LiveCodeBench / IFEval / LongReason 等推理 benchmark 上，
InfoKV 一致优于纯注意力的压缩基线。

遗留的坑：
- alpha 超参数需要调，论文默认 0.9（注意力权重 90%），但对不同任务可能不同
- 熵计算需要额外的前向推算，有一定计算开销
- 还是 token 级别淘汰，KV 向量本身（channel 维度）的冗余没有处理
- 和 FlashAttention / Paged Attention 等生产级内核的集成尚未明确

* 前沿延伸

截至 2026-06-26，InfoKV 于 2026-06-25 提交，是该研究线的最新进展。

同期有 *CapKV*（arXiv 2604.25975，2026-04）从信息瓶颈（Information Bottleneck）理论
出发，用互信息最大化导出 KV 淘汰目标，与 InfoKV 形成理论上的呼应：
两篇文章独立地验证了「注意力得分是更深层信息论目标的近似，而不是目标本身」。

下一步最可能出现的方向：
- 把 alpha 参数变成可学习的、输入自适应的混合权重
- 将信息论 token 选择和 channel 维度量化结合（InfoKV + KIVI / KVSharer）
- 面向长推理链的*动态* budget 分配（推理过程中实时调整，而不是 prefill 一次性决定）

* 洞见

读完这条线，真正在发生的变化不是「压缩方法越来越好」，
而是*研究者对「重要性」这个概念的理解越来越深*。

第一阶段（2023）：重要性 = 注意力权重高。
这是最自然的定义，也是最短视的定义——注意力本质上在问「现在需要看哪里」，
而不是「这个 token 对之后一千步还有没有用」。

第二阶段（2024）：意识到层间重要性分布不同，开始精细分配。
但还是在注意力框架内做优化，没有跳出去。

第三阶段（2026，InfoKV 的位置）：*引入了第二把尺子——信息论*。
重要性的定义扩展了：一个 token 可以不被注意力「选中」，
但因为它预测困难（高熵）而携带大量信息，对远处的 token 有强烈的因果影响。

这实际上是两类不同的重要性被分开：
- *局部相关性*：这个 token 跟当前任务有多相关？（注意力捕捉）
- *全局信息密度*：这个 token 携带了多少难以被上下文预测的内容？（熵捕捉）

在短文本里这两类几乎一致，所以之前的方法够用。
在长推理链里它们开始分裂，InfoKV 是第一篇明确区分并利用这种分裂的工作。

*下一步预测*：长推理链的 KV 压缩将朝「自适应双信号」演化——
不同的推理阶段（问题理解 / 中间推导 / 最终归纳）对两类重要性的权重不同，
静态的 alpha=0.9 只是起点，下一篇论文很可能让 alpha 动态化。

* 启发

这条线的研究方法有一个反复出现的模式：*先做测量，再做设计*。

SnapKV 的洞见来自测量「注意力模式在生成过程中是否稳定」——发现了稳定性，
才有了「一次性 prefill 压缩」的可行性。

PyramidKV 的洞见来自测量「注意力稀疏度随层深度怎么变」——发现了金字塔形，
才有了按层分配 budget 的理由。

InfoKV 的洞见来自测量「KL 散度衡量的前向影响力如何分布」——发现了注意力/熵的
远近分工，才有了双信号融合的必要性。

*这意味着你可以*：在现有方法上找到一个具体的「测量实验」——
不是设计新方法，而是先精确测量现有方法*哪里在失效*。
失效的地方，就是下一篇论文的开篇。
找不到值得做的研究？先找到一把不够准的尺子，然后造一把更准的。
