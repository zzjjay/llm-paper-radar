* 问题之河

让一个大模型读懂整个代码仓库——几万行代码塞进上下文——它的显存会爆。怎么把 KV cache 压小，但又不把模型读懂代码所必须的那几个关键 token 扔掉？

这条线难在一个分裂：从 2023 年开始，所有人都用 *注意力分数* 来判断哪个 token 重要、哪个可以扔。在自然语言里这招挺好使。可代码不一样——代码的语义靠的是 *结构*（谁调用谁、哪个分支管哪段逻辑、哪个变量定义在哪），不是表面字词出现频率。注意力高的 token 未必是结构上关键的 token，结构上关键的 token 反而经常注意力低。

于是整条河，是一群人在 *"如何判断 token 重不重要"* 上越走越细——从"留最近的就行"，到"留注意力高的"，到"留整块的"，到 CodeComp 这一篇终于跳出来说："注意力本身就是错的代理，得换信号源。"

* 演化叙事

** 2023.09 | StreamingLLM: 别想全留下, 留头和尾就够了

*** 它看到了什么问题

2023 年大家在用 Transformer 跑流式对话, 一个棘手现象冒出来: 上下文超过预训练长度后, 模型胡言乱语。原因是 KV cache 不能无限攒, 攒到一定长度滑动窗口一砍, 模型就崩。

*** 它的解法

Xiao 等人做了个反直觉的发现: Transformer 会把"注意力配额"大量倾倒到序列最开头那几个 token 上, 哪怕它们语义无关 -- 像一个排水沟, 把多余的注意力都倒进去, 这叫 *attention sink*。

所以解法极其简单粗暴: 保留最开头几个 sink token + 最近的滑动窗口, 中间全扔。模型不崩了, 也能流式跑下去。

*** 为什么这招管用 (以及哪里埋了新坑)

它管用是因为模型本来就在"利用"位置, 而不是真的需要中间所有内容。

但坑很明显: 中间那些 *语义上重要但位置不靠头也不靠尾* 的 token 全没了。比如对话开头第 50 句话里提到的关键事实——既不是 sink 又不在最近窗口, 直接消失。

H2O 看到了这个坑。

** 2023.10 | H2O: 不看位置, 看谁被反复盯着

*** 它看到了什么问题

StreamingLLM 用位置决定去留, 太粗暴。一个 token 重不重要, 应该看它在历史 decoding 里 *实际被多少次重点关注* —— 这才是它在模型脑子里的"地位"。

*** 它的解法

Zhang 等人提出 "Heavy Hitter Oracle" (H2O): 把每个 token 在所有过去 decoding step 上收到的注意力分数累加起来, 累积高的就是 "heavy hitter", 留; 累积低的, 扔。

变形地看: StreamingLLM 是 "按位置打分", H2O 是 "按累积注意力打分"。前者是先验, 后者是观测。

*** 为什么这招管用 (以及哪里埋了新坑)

它把"重要性"从位置启发式升级成了行为统计, 在自然语言上效果明显。

但 heavy hitter 是在 decoding 跑起来之后才能识别出来的。如果你的瓶颈在 prompt 阶段 (prompt 本身就几万 token 一上来直接爆显存), H2O 没法在你"开口提问"之前就把无关 token 砍掉。

SnapKV 看到了这个坑。

** 2024.04 | SnapKV: 问题问出来之前, 模型其实已经知道要找什么

*** 它看到了什么问题

H2O 要先 decoding 才能识别 heavy hitter, 但很多场景下你早就知道"用户要问什么"——问题就写在 prompt 末尾。能不能 *用问题本身做线索*, 反向去筛 prompt 前面那一大坨上下文里哪些 KV 值得留?

*** 它的解法

Li 等人发现: prompt 最末尾那几个 token (通常是用户问题的结尾) 的注意力分布, 已经"预演"了模型解码时会去看哪些位置。把这个分布拿出来当评分器, 给前面所有 prompt token 打分, 高分留低分扔。

变形地看: H2O 是 "累积过去的注意力", SnapKV 是 "借用问题末尾的注意力当未来的代理"。前者向后看, 后者向前预测。

*** 为什么这招管用 (以及哪里埋了新坑)

把 query-aware 引入压缩, 一下让 prompt 阶段就能压。这是个工程上的大跃迁。

但 SnapKV 还是 *逐 token* 评分。一句话里的 token 评分有高有低, 留下来的可能是"the"、"is", 真正承载意思的名词反而因为注意力低被扔了 —— 语义单元被打散。

ChunkKV 看到了这个坑。

** 2025.02 | ChunkKV: 别拆字, 留整块

*** 它看到了什么问题

SnapKV 把上下文当成松散 token 集合, 但语言是有"块"的——一个短语、一个子句、一个论点单元, 拆开任何一个都损失意义。逐 token 选保留, 等于把书撕成单字再选哪些字留下。

*** 它的解法

Liu 等人提出 ChunkKV: 把上下文切成固定大小的 chunk, 评分以 chunk 为单位, 决定哪整块留、哪整块扔。chunk 内部所有 token 命运绑定。

变形地看: SnapKV 是 "把 KV 当沙子, 一粒粒筛", ChunkKV 是 "把 KV 当石头, 一块块搬"。

*** 为什么这招管用 (以及哪里埋了新坑)

chunk 化保住了局部语义连续性, 也方便并行。一个意群完整留下比 "重要 token 散落各处" 对下游 generation 友好得多。

但 chunk 怎么评分? 依然是注意力。chunk 级评分只是把单点信号聚合成块信号, *信号源没换*。

ParallelComp 把 chunk 化推到工程极致, 但也没换信号源——CodeComp 才换。

** 2025.03 | ParallelComp: 把 chunk 压缩并行起来, 喂超长上下文

*** 它看到了什么问题

ChunkKV 走对了 chunk 路线, 但当上下文真的飙到几十万 token, 单进程串行处理 chunk 评分都吃不消。怎么让 chunk 压缩本身可并行, 吃下"代码仓库级"或"长文档级"输入?

*** 它的解法

Xiong 等人 (注意: 这位 Xiong 也是 CodeComp 的二作) 提出 ParallelComp: chunk 之间并行做注意力评分和压缩, 让流水线吞吐量上去, agentic / 长 context pipeline 真的能跑起来。

*** 为什么这招管用 (以及哪里埋了新坑)

工程上, 它把"压缩"从串行瓶颈变成并行算子, 让 ChunkKV 这类方法在生产规模下可用。

但 —— 这是 CodeComp 全部张力的来源 —— 信号源仍然是注意力。在自然语言里这没问题。可一旦输入是 *代码*, 一个根本性错误就暴露了:

代码的语义不在"哪个 token 被反复看", 而在"谁调用谁、谁定义谁、哪个 if 控着哪段执行"。callsite、branch condition、assignment 这些结构关键 token, 在某次 decoding 时注意力分数可能很低, 但删了它们模型就理解不了程序。

CodeComp 看到了这个坑——而且这次, 坑不在 *实现细节*, 在 *信号源本身*。

** 2026.04 | CodeComp: 换信号源, 让程序结构来发言

*** 它看到了什么问题

作者做了个直白测量: 拿 attention-based 评分排出来的 top-20% chunk, 和 CPG (Code Property Graph) 结构重要度排出来的 top-20% chunk, 算交集。Jaccard 只有 *0.0944* —— 几乎不相交。

这意味着五年来基于注意力的所有压缩方法, 在代码场景下都在系统性地砍错地方。Figure 2(b) 显示, callsite、return、assignment 这些类的结构 token, 被注意力 baseline 误删的比例触目惊心。

*** 它的解法

跳出注意力, 用 *静态程序分析* 当独立信号源:

1. 用 Joern 从代码抽 CPG (Code Property Graph, 统一了 AST/CFG/PDG 的图表示)
2. *Span-level structural protection*: 把 callsite、branch、assignment 这些结构关键 span 标记出来, 压缩时强制保留, 注意力低也不准扔
3. *Structure-aware budget allocation*: 不再给每个 chunk 平均分配 KV 预算, 按 chunk 里结构关键 token 的密度分配 —— 结构稠密的 chunk 多给点
4. 在剩余 budget 里再用传统 attention-based 方法压

变形地看: 前五代是 "在注意力这一个信号源里不断换打分公式", CodeComp 是 *并联* 了第二个信号源, 用结构信号给注意力打补丁。

训练-free, 接到 SGLang 里就能用, 不动模型权重。

*** 为什么这招管用

LCA 代码生成 benchmark 上, API precision 从 SnapKV 的 0.0196、ParallelComp 的 0.1318 跳到 CodeComp 的 0.1929, edit distance 也明显降。callsite token 保留率从 ParallelComp 的 0.48、SnapKV 的 0.52, 跳到 1.00。

它管用的根本原因是: *注意力是模型学到的查询驱动代理, 而 CPG 是程序运行时真实存在的结构*。两个信号互相校正, 比单一信号靠谱。

* 前沿延伸

CodeComp 提交于 2026-04-11, 几周前的新论文。截至目前 (2026-05-28), 还没看到公开发表的直接后续。

但可以预判会被推往三个方向:

- *信号源扩展*: 不止 CPG, 还可以引入 type information、dataflow trace、test coverage 信号
- *动态化*: 静态分析是 prompt 阶段一次性的, 能否在 decoding 过程中动态更新结构图 (例如当模型在生成 patch 时, 新生成代码也进结构分析)
- *跨领域复用*: "找一个独立信号源给注意力打补丁" 这个范式, 可以推到数学 (用形式化证明结构)、SQL (用 schema 结构)、化学 (用分子图) 等任何有强结构先验的领域

* 洞见

这条河表面是"KV cache 压缩"的技术迭代, 真正在发生的事情是 *信号源的演化*。

前五代研究者都默认了一个前提: token 重要性 == 注意力函数 f(attention)。他们改的全是 f 的形式——从"位置 f", 到"累积 f", 到"query-aware f", 到"chunk-aggregate f"。每一代都在把 f 调得更精, 但谁也没问"这个等号本身对吗?"

CodeComp 的真正贡献不是用了 CPG, 是它 *把这个等号打开了*。一旦你承认"注意力不是 token 重要性的唯一代理", 整个游戏规则就变了——你可以接入任何独立信号源作为校正信号。

这是研究里典型的 *框架重置* 时刻: 前几代在水平方向上优化已知公式, 这一代在垂直方向上 *换了被优化对象本身*。一旦换框架, 一大片新论文位置就开放了——每一种结构先验、每一种符号信号、每一种工具产物, 都可能成为下一篇 CodeComp。

下一步最可能往哪走? 不在 KV cache 内部, 在 *"信号源生态"*。谁能给注意力找到第二个、第三个独立信号源, 谁就接管下一节河道。

* 启发

研究方法层面:

- 看到一个领域里所有论文都在改 f 的形式而不质疑 f 的输入, 那里就有 *框架重置* 的机会。问一句 "如果信号源本身错了呢?"
- 跨学科信号引入往往比同领域公式优化威力大: 程序分析 (软件工程) 嫁接到 KV cache (系统/ML), 直接打开整片新地。"两个不相干的领域之间, 隔着一个新论文。"
- 训练-free 的方案比训练-based 在工业界传播更快。能不动权重就能用, 等于绕开了所有 finetune 成本和兼容性争论

落点在能用的事:

- 你在做任何"基于注意力打分"的工程系统 (rerank、检索、压缩、剪枝), *先量一下注意力分数和你真正关心的 ground-truth 重要性的相关性*。如果 Jaccard 低于 0.2, 你可能在系统性地砍错地方——这正是 CodeComp 一开始做的测量。
- 想给已有系统提一档, 不一定要训新模型——先找 *第二个独立信号源*。CodeComp 全文核心就这一句话。
