* 问题之河

让大模型读完一本书、一个代码库、一份病历，然后回答任意问题。这件事今天卡在一个很物理的地方：模型每生成一个字，理论上都可能要回头看前面任何一个字，所以它必须把读过的每个 token 的 Key、Value 都存进 GPU 显存。这份 KV cache 随上下文长度 *线性* 膨胀，到了几十万 token，它比模型权重本身还重，而且每生成一步都要把整份 cache 从显存里搬一遍——长上下文推理因此不是 *算得慢*，是 *搬得慢*，是被显存带宽卡死的。

整条研究线都在跟同一个朴素的念头死磕：*这几十万份历史，大部分时候用不上，能不能少存、少搬？* 难的不是"压"，是"压完还得好用"。过去几年分裂出了两条互不服气的路线：

- 一条是 *事后做减法*：先把完整的 cache 老老实实建出来，再按某种规则把不重要的踢掉（H2O、SnapKV、KVzip……）。它们快、不用重训，但天花板很低——压过 2 倍质量就掉，而且很多方法的 cache 是"为某个具体问题量身定的"，换个问题就废。
- 另一条是 *事前换表示*：训练一个小模型，把长文本压成一小撮"软 token"（连续向量），让大模型直接读这撮浓缩向量，而不是原文（Gist、AutoCompressor、ICAE、REFRAG……）。它原则上能压得狠得多，但一直有个尴尬：要么把大模型本来的通用能力压坏了，要么只在某个窄领域管用，*在信息密集的长上下文 benchmark 上从来打不过第一条路线*。

这篇论文（LCLM, Latent Context Language Model）的主张听起来几乎没有新 gadget，却很刺：第二条路线之所以一直打不过，*不是方法不对，是没人把它认真做大过*。把 0.6B 编码器和 4B 解码器端到端联合训练 350B+ token，软 token 路线第一次在 RULER、LongBench 这类硬 benchmark 上越过了 KV 压缩的精度-效率前沿，而且天然兼容 vLLM / SGLang。

于是问题被重新问了一遍：*我们到底是在"删除"历史，还是在"重写"历史？删一直打不过重写，会不会只是因为重写从来没吃饱过训练预算？*

* 演化叙事

先讲清根问题，再分两条线往下走，最后在 LCLM 处合流。

** 根问题 | 为什么长上下文推理是"搬得慢"而不是"算得慢"

要看懂后面所有方法在跟什么较劲，先得明白 KV cache 到底卡在哪。Transformer 解码时，为了不把前文重算一遍，它把每个历史 token 的 Key/Value 缓存下来。这份 cache 大小 = 序列长度 × batch × 层数 × 头数，随上下文 *线性* 长。KVQuant（Hooper 2024）点破了关键一句：到了百万 token，cache 才是显存的第一大消耗，超过权重本身；而且每个解码步都要把整份 cache 从 HBM 流一遍——*这是带宽瓶颈，跟 batch 多大、序列多长无关*。这就是"搬得慢"的来历。

另一边，Lost-in-the-Middle（Liu 2024）补了第二刀：就算你显存装得下，模型对长上下文的利用也是 U 形的——开头结尾记得清，中间段基本"看不见"。两条加起来给出整条河的前提：cache 又大、又是带宽受限、还不是均匀有用的。*那就压它。* 问题只剩下——怎么压才不掉精度。两种思路就此分家。

** 2023-06 | H2O: 把"删 cache"这件事第一次做成可行的

*它看到了什么问题*：最朴素的做法是 cache 全留，于是线性爆炸、OOM。H2O 的观察是：注意力分布极度稀疏，累计下来少数 token（"heavy hitter"，重击者）吸走了绝大部分注意力，剩下的留着纯属浪费显存。

*它的解法*：给每个 token 算一个累计注意力分数，只保留分数高的 heavy hitter，再配一个最近窗口（保证局部连贯），其余在解码过程中贪心驱逐。约 20% 的 cache 就能逼近全 cache 的质量。

*为什么管用，又埋了什么坑*：稀疏性是真的，所以删得动。但贪心+永久驱逐意味着——此刻看着没用、之后才关键的 token 一旦扔了就回不来；而且"累计分数"天然偏袒活得久的老 token。更深的坑是：H2O 的重要性判据是 *静态启发式*，跟当前要回答什么问题无关。下一篇就盯着这一点。

** 2024-04 | SnapKV: 让"删"这件事看着具体问题来删

*它看到了什么问题*：H2O 这类固定重要性策略有个硬伤——"哪些 token 重要"会随指令变。同一段上下文，问"主角叫什么"和问"结局如何"，该留的 token 完全不同。用一把离线定死的尺子量所有问题，必然有问题被照顾不到。

*它的解法*：拿 prompt 结尾的一小撮 query token（默认 16 个）当"观察窗口"，让它们的注意力给前文投票，每个头各选各的重要位置；再沿序列做一次池化（max-pool），让选中的 token 带上邻居，保住上下文连续性。这是 prefill 时一次性压完，不用重训。

*为什么管用，又埋了什么坑*：用真实 query 来挑，自然比盲选准。但代价是——压出来的 cache 是 *为这个 query 量身定的*。换一个问题、进入多轮对话，这份 cache 就不对味了（KVzip 后来报告：这类方法在多 query 下，即使留 90% 预算也会崩）。"query 绑定"成了新的靶子。

** 2025-05 | KVzip: 压一次, 任何问题都能用

*它看到了什么问题*：现实里同一份上下文（一个代码库、一份文档）要服务成千上万个不同问题。SnapKV 那种 query 相关的 cache 没法复用——你不可能为每个问题各压一遍。需要的是 *一份 query 无关的、压一次到处用* 的 cache。

*它的解法*：换掉打分信号。不再问"这个 token 对当前 query 重要吗"，改问"这个 token 对 *重建原文* 重要吗"——用自监督的方式（提示模型"复述前面的上下文"），看哪些 KV 删了会让模型重建不出原文。这个信号跟具体 query 无关，所以压一次能服务任意后续问题，报告 3-4x 压缩、约 2x 解码加速、质量几乎不掉。

*为什么管用，又埋了什么坑*：把"重要性"从 query 解耦，复用问题解决了。但两个坑暴露出来：一是它只压 *预算维*（每个位置留多少 KV），*压不了长度维*——序列还是那么长，FlashAttention 的 query×key 计算量没省下来；二是那趟重建打分本身要跑额外 forward，prefill 时间多花 2-3 倍。删得更聪明了，但"先建全 cache 再删"这个范式本身的成本还在。

** 2025-06 | Cartridges: 不删了, 干脆把 cache 训出来

*它看到了什么问题*：所有"事后删"的方法都撞到同一堵墙——在硬的长上下文任务上，压过 2 倍左右质量就快速崩。如果一份大语料（128K 上下文，70B 模型要 ~84GB cache）要被反复查询成千上万次，与其每次现建再删，不如一次性把成本摊到离线。

*它的解法*：冻结模型，*训练* 一个小小的 KV cache（一个"Cartridge"/弹夹）来代表这份语料。训法叫 self-study：先让模型围绕语料自问自答生成合成对话，再用上下文蒸馏目标去训这个 cache（直接拿语料做 next-token 训练只会死记硬背，效果差）。报告 ~38x 省显存、~26x 提吞吐，成本摊到所有命中该语料的查询上。

*为什么管用，又埋了什么坑*：把"压缩"从一次前向变成一次优化，压缩比的天花板一下抬高了。但坑很直白——每个语料都要 *梯度训练*，是 GPU-小时量级的离线开销（后续 Attention Matching 直接吐槽这"贵得离谱"）。路线 A 走到这里，已经从"实时删"滑向"离线训"，离路线 B 只剩一层窗户纸：*既然都要训了，为什么不直接训一个通用的压缩器，而不是每个语料训一个 cache？*

—— 路线 A 讲完。它的轨迹是：从删 → 删得准 → 删得通用 → 干脆训。终点撞上了一个天花板和一个"要不要训"的纠结。现在掉头看路线 B，它从一开始就在训。

** 2021 | Prefix-Tuning: 软 token 的源头

*它看到了什么问题*：给每个任务都全量微调一遍大模型，等于每个任务存一整个模型副本，模型越大越扛不住。而离散的手工 prompt 又难优化、受限于词表。

*它的解法*：冻结整个大模型，只在输入前面接一小撮 *可训练的连续向量*（virtual tokens / 前缀），让后续 token 去注意它们。每个任务只训、只存这一小撮前缀。这就是"软 token"的源头——用连续向量而非离散文字去携带信息。

*为什么管用，又埋了什么坑*：连续向量比离散词表达力强、能直接梯度优化。但此时前缀是 *每个任务学死的一组固定向量*，跟具体输入内容无关。它还不是"压缩器"——它压的是任务，不是上下文。

** 2023-04 | Gist Tokens: 把前缀从"每任务学死"改成"看着输入现压"

*它看到了什么问题*：Prefix-Tuning 的前缀对每个任务是死的。但真实场景里浪费在哪？是同一段长 prompt（指令）每次调用都要重新编码一遍。能不能把这段 prompt *现场* 压成几个 token，存下来复用？

*它的解法*：Gisting——只改注意力 mask，让 gist token 后面的 token 不能再看原始 prompt，逼着 gist token 把 prompt 的信息全吸进去。这几个 gist token 缓存起来反复用，最高 26x prompt 压缩、~40% FLOPs 下降，几乎不增加训练成本。

*为什么管用，又埋了什么坑*：从"每任务一组固定前缀"变成了"看着具体输入预测出来的软 token"——这是质变，压缩器第一次会"读内容"。但 Gist 压的还是 *短指令*，长文档塞不进它的窗口。

** 2023-05 | AutoCompressor: 把压缩从短指令推到长文档

*它看到了什么问题*：Gist 只能压短 prompt。可真正吃显存的是长文档。怎么把"现压软 token"这招用到几千、几万 token 的长上下文上？

*它的解法*：分段 + 递归。把长文档切成段，每段压出 summary 向量，再把前面段的 summary 累积着喂给后面段——像滚雪球一样把任意长的输入滚成一小撮 summary 向量（如 6144 token → ~150 个）。无监督训练。

*为什么管用，又埋了什么坑*：递归让它突破了单窗口长度限制，第一次能压真·长上下文。但递归=串行，summary 不缓存时延迟很高；而且 summary 向量只能放在前缀位置，用法被钉死在"前缀"上（REFRAG 后来正是批这一点）。还有，它的训练目标主要是 next-segment 语言建模，没有"忠实重建原文"的约束。

** 2023-07 | ICAE: 给压缩加一条"必须能还原"的硬约束

*它看到了什么问题*：AutoCompressor 靠语言建模目标学 summary，但没人保证这撮向量真的 *忠实* 装下了原文——会不会压着压着把细节丢了、甚至编造？

*它的解法*：自编码器框架。用 LoRA 适配大模型当编码器，把上下文 + 一组可学 memory token 压成 memory slots；再用 *冻结的目标大模型* 当解码器去读这些 slots。训练目标里明确加一条 *自编码重建*——slots 必须能让解码器把原文重建出来。论文发现这条重建预训练"至关重要"，能显著降幻觉，~1% 额外参数做到 4x 压缩。

*为什么管用，又埋了什么坑*：编码器/解码器分工 + 重建约束，让软 token 第一次有了"忠实度"的保证，也定型了后来所有编码器-解码器压缩器的骨架。但它不递归（长度受限），而且仍是小规模、特定训练。

** 2024-02 / 2025-09 | CEPE 与 REFRAG: 把编码器做轻, 但格式还是老格式

*它们看到了什么问题*：ICAE 这类要么不递归（压不了超长），要么训练规模小、用法窄。能不能用一个 *轻量编码器* 分块并行处理上下文，配冻结解码器，既快又能扩到长上下文？

*它们的解法*：CEPE——小编码器把上下文分块并行编码，冻结解码器通过新增的 *交叉注意力* 层去读（FiD/并行窗口的血统）。REFRAG 更进一步——编码器把每个定长块（如 16 token）压成 *一个* embedding，用投影层映射进解码器的 token 嵌入空间，*内联* 喂进去（不改解码器结构），再用 RL 策略挑出少数关键块展开回原文。REFRAG 报告最高 30x 的 TTFT 加速。

*为什么管用，又埋了什么坑*：REFRAG 直接点名批 CEPE——CEPE 只放前缀、破坏因果结构（不适合多轮/摘要），而且 *根本不压 token*，延迟没真降下来；REFRAG 则真压、能放任意位置、可预计算复用块嵌入。但 REFRAG 自己也留了坑，而这恰恰是 LCLM 的起点：它只训了 ~20B token，还在用"前半段压缩、后半段可训"的旧格式，*通用能力到底保住没有，没有在信息密集的长 benchmark 上验过*。

** 2026-06 | LCLM: 把软 token 路线端到端喂饱, 第一次正面打赢删 cache

*它看到了什么问题*：把前面两条线并排放——路线 A（删 cache）走到 Cartridges 已经在"离线训"了，却受困于 2x 压缩天花板和每语料重训；路线 B（软 token）从 Gist 到 REFRAG 一脉相承，架子早就搭好了，却 *从来没人把它认真做大*：训练 token 少（REFRAG ~20B）、用窄格式、编码器解码器没真正端到端联合优化。LCLM 的判断是——软 token 一直打不过删 cache，*缺的不是新机制，是训练预算和端到端优化*。

*它的解法*：把 ICAE 定下的编码器-解码器骨架 *做大、做透*。0.6B 编码器 + 4B 解码器，都从预训练模型初始化，然后 *联合端到端继续预训练 350B+ token*，压缩比 1:4 / 1:8 / 1:16。文本用 `<|memory_start|>`/`<|memory_end|>` 包起来，编码器把每个窗口池化成隐向量，适配器投影进解码器嵌入空间，解码器把它们当普通输入读——*保留标准 KV cache 结构*，所以 vLLM / SGLang 直接能跑。配套还做了一轮架构搜索，把前人拍脑袋定的设计逐个验过：
  - *MLP-only 适配器* 比注意力适配器预训练 loss 更低还更省算力——直接 *推翻 GMSA（Tang 2025）* 的结论。
  - 编码器窗口扫 {16, 256, 1024}，16→256 收益大，最终默认 W=1024。
  - 因果 mask 稳定优于非因果；池化看压缩比（16x 用 mean pooling）。
  - 格式上抛弃"前半压/后半训"，改成 *交错的压缩/非压缩格式*，让解码器在很多位置都能条件于隐上下文，而不只是前缀——直接回应 AutoCompressor/REFRAG 的前缀局限。

*为什么管用，又埋了什么坑*：baseline 全是路线 A 的方法（Expected Attention、Attention Matching、KVzip、FastKVzip、SnapKV，同一个 Qwen3-4B 解码器），在 RULER / LongBench / LongHealth 上，LCLM 第一次把软 token 推到 *新的精度-效率前沿*：删 cache 的方法因为"先建全 cache 再删"，压缩时间几乎不随压缩比变（图上是竖线），还在 512K-1M token 处 OOM；LCLM 峰值显存低得多，还能继续往上扩。代价也很实在——它需要 350B+ token 的继续预训练，是一次性的重投入；编码器对超长输入的处理、以及压缩比再往上推的极限，都是开放问题。

* 前沿延伸

LCLM 是 2026-06-08 提交的，目前 *没有确认引用它的后续工作*。但同期有几篇平行方向的论文，能看出这个问题正被从不同角度同时推进（以下均为已核实的真实 arXiv 论文，但与 LCLM 是 *同期平行*，非互相引用）：

- *Meta-Soft*（2026-05, arXiv:2605.22337）：从一个可学的正交"元 token 库"里动态合成输入自适应的软 token，用注意力流把被驱逐 token 的信息重分配回去，而不是硬删。仍属"在删 cache 框架内引入软 token"，没走到端到端换表示。
- *KV Packet*（2026-04, arXiv:2604.13226）：在冻结的预计算文档 KV cache 外面包一层可训软 token 适配器（Header/Trailer），用自蒸馏让多个 cache 能直接拼接复用而不重算。目标是 cache 复用而非极限压缩，是 LCLM 最近的"软 token 适配器"表亲。
- *REFRAG*（2025-09, arXiv:2509.01092, Meta）：LCLM 直接对位的前驱。LCLM 的"做大 + 端到端"几乎就是对 REFRAG"只训 20B、用旧格式"的正面回答。

截至目前，这条"把软 token 压缩端到端做大、正面挑战删 cache"的线，LCLM 是最新、也是第一个跨过前沿的进展。

* 洞见

这条河表面是"KV cache 越压越狠"，底下真正在发生的是 *三次认知转弯*：

1. *从"删除"到"重写"*。路线 A 的隐含信念是：历史是既成事实，我只能挑哪些留、哪些扔（离散的保留/丢弃）。路线 B 的信念是：历史可以被 *重新编码* 成一种更紧凑的表示（连续的软 token）。删除有信息论硬上限——一旦在离散 token 空间丢了，精确的字面和结构就找不回来了；重写没有这个上限，因为它在连续空间里重新分配信息。LCLM 的胜利本质上是 *重写* 对 *删除* 的胜利。

2. *从"设计判据"到"喂训练量"*。早期每一篇都在发明更聪明的重要性判据（累计注意力→query 投票→重建打分）。但 Cartridges 和 LCLM 都在说同一件事：与其手工设计该留什么，不如让模型 *自己学* 该压什么。判据之争最终让位于训练预算之争。LCLM 最反直觉的一点是——它几乎没提新机制，它的论点是"前人的机制没错，只是没喂饱"。这在一个痴迷于发明新结构的领域里，是一种克制的、却往往更对的判断。

3. *从"研究 demo"到"能进生产引擎"*。KVzip、Cartridges 强在压缩比，弱在落地（非均匀驱逐、特种 kernel 没人实现、每语料重训）。LCLM 反复强调"保留标准 KV 结构、vLLM/SGLang 直接能跑"——这说明这条线已经从"刷 benchmark"进入"必须能被现有推理栈吃下"的阶段。可部署性本身成了一等约束。

下一步最可能往哪走：(a) 把编码器也做大、或让压缩比自适应内容（信息密的少压、信息疏的狠压）；(b) 软 token 与 RAG/agent 记忆融合——压缩后的隐上下文天然就是一种可存取的"记忆条目"；(c) 端到端这条路一旦验证有效，会反向逼问预训练本身——为什么不在预训练阶段就让模型学会读隐上下文，而要事后继续训。

* 启发

这条线对"怎么做研究"最大的启发是：*当一个方向"原理上更优但实践上一直打不过"时，先别急着换方向，问一句"它被认真做大过吗"*。软 token 压缩从 2021 年就有了，五年里大家默认它"质量不行"，于是不断在删 cache 那条线上做增量。LCLM 没发明新东西，它只是把一个老想法第一次喂到了 350B token 的规模、并端到端打通——结果前沿就被推过去了。很多"此路不通"的结论，其实是"此路没走够远"。

落点在"能用"：这意味着你在评估一个失败的旧方法时，可以先把它的"失败原因"拆成 *机制错* 和 *规模/训练不够* 两类——前者该放弃，后者可能只是时机未到。具体到工程上，如果你要给长上下文系统选压缩方案：要可部署、要兼容现有推理引擎、要一次压到处用，LCLM 这类"换表示 + 保留标准 KV 结构"的路线，比"事后非均匀删 cache"更值得押注；而如果你手上只有一份反复查询的静态语料、又训得起，Cartridges 式的离线训练仍是压缩比上限最高的选择。
