* 问题之河

长上下文大模型怎么又快又准地"翻字典"？

模型读完一本 100 万字的小说，每生成一个新词，理论上都要回头扫一遍所有读过的内容——
存这些"读过的内容"叫 KV cache，它撑爆显存；扫这些内容叫 attention，它慢得让人发指。

七年来一群人前赴后继，就是想破这个死结：
- 全存全扫：准但慢、爆显存；
- 扔一部分：省了显存，但扔错了就答不出问题；
- 只扫一部分：省了时间，但扫的粒度太粗或太细都不行。

每一代解法都解决了上一代的一个痛，又留下一个新痛。CTkvr 是这条河目前的下游。

* 演化叙事

** 2017 | Transformer: 把注意力公式钉在 LLM 的地基上

*** 它看到了什么问题

RNN 必须一个 token 接一个 token 地算，长序列训练慢、并行不起来。

*** 它的解法

扔掉循环，让每个 token 同时看其他所有 token（self-attention）。算 Q.K^T 拿到一张
n x n 的注意力图，加权 V 输出。

*** 为什么这招管用（以及哪里埋了新坑）

并行起来了，质量也上去了。但 O(n^2) 是写在公式里的：n 一旦到 100K，KV cache 几十 GB，
每生成一个 token 都要重新读一遍全部 K 和 V。这就是后面所有论文都在挖的那个坑。

** 2023 | StreamingLLM: 砍掉中间，只留头和尾

*** 它看到了什么问题

显存不够，最朴素的想法是"滑窗"——只保留最近 N 个 token。可一旦把最前面的 token 扔了，
模型就崩了，输出全是乱码。

*** 它的解法

发现前几个 token 是"注意力黑洞"（attention sink），模型默认把多余的注意力倒在那里。
保留开头 4 个 + 最近 N 个，剩下全扔。

*** 为什么这招管用（以及哪里埋了新坑）

显存稳定下来，模型不崩了。但你问"第 500 段讲了啥"——它真的不知道，因为第 500 段已经物理删除了。
"长上下文"这事根本没解决，只是不崩而已。

** 2023-2024 | H2O / SnapKV: 用注意力分数挑"重要 token"

*** 它看到了什么问题

StreamingLLM 太机械——按位置扔，不看内容。明明中间有句关键话，结果被滑窗滑掉了。

*** 它的解法

观察哪些 token 在历史 attention 里累计分数高，那些是"Heavy Hitter"。留它们，扔分数低的。

*** 为什么这招管用（以及哪里埋了新坑）

比纯滑窗聪明：信息不再按位置丢，按"被注意过的频率"丢。问题是——*现在不重要不等于以后不重要*。
比如一段陈述前期没人引用，但到了多轮对话第 5 轮突然成关键证据。一旦驱逐就回不来。
论文里这条线（eviction）在长输出和多轮对话上崩得最厉害。

** 2024 上半 | InfLLM: 别扔，全存到 CPU，按需拉回来

*** 它看到了什么问题

驱逐方法的根本错误是"一旦丢了就回不来"。能不能 *全部留着* ，只是不全都放在 GPU？

*** 它的解法

把 KV 全部 offload 到 CPU 内存里，切成 8-16 token 一块。每块算一个代表向量（mean-pool）。
生成时新 query 跟所有块代表算相似度，挑 top-k 块拉回 GPU 跑 attention。

*** 为什么这招管用（以及哪里埋了新坑）

这是范式转折：从"裁剪 KV"变成"检索 KV"。再也不用永久丢东西。

但块是粗粒度的——一块 16 个 token，里面只要有 1 个相关的就整块拉回来，剩下 15 个是噪声。
长上下文里这种稀释会让 attention 算得糊里糊涂。论文实验里 ShadowKV、Quest 在 128K
上下文下都被这个问题咬。

** 2024 中 | Quest: 给块代表加个紧界，少漏关键块

*** 它看到了什么问题

InfLLM 用 mean-pool 当块代表，对 query 的相关性估计不准——一块里有个高相关 token，被其他低相关
token 平均掉了，结果整块没选上。

*** 它的解法

每块不存一个均值向量，存 *逐维 min 和 max*。给定 query，用 min/max 算这块跟 query 的相似度
*上界* ，按上界排块。这样不会漏掉"块平均不高但里面有亮点"的块。

*** 为什么这招管用（以及哪里埋了新坑）

块选得更准了。但块还是块——选中的块里还是 8-16 个 token，里面仍然有不相关的。
精度天花板被"块粒度"卡住，论文报告 Quest 在 128K Ruler 上比 FullKV 掉 5-7 分。

** 2024 中 | RetrievalAttention: 索性下到 token 级，建 ANN

*** 它看到了什么问题

块粒度本质上是个妥协——为了快放弃了准。能不能直接在 token 级别检索？

*** 它的解法

每个 token 的 Key 都当成数据库里一条向量，整体建一个 Approximate Nearest Neighbor 索引（HNSW 之类）。
query 来了直接查 top-k token，绕开块。

*** 为什么这招管用（以及哪里埋了新坑）

精度直接顶到接近 full KV——这条路在精度上是对的。但 ANN 索引这东西，建一次贵得离谱：
HNSW 在 128K 上下文要 800+ ms，KMeans 要 6000+ ms。论文统计 prefill 拖慢 3x、decode 拖慢 5x。
精度赢了，吞吐崩了。

** 2024 下 | MagicPIG: 换个索引——LSH 哈希，丢到 CPU 上

*** 它看到了什么问题

ANN 索引太重，建得慢、查也慢。换个轻的哈希索引（LSH）应该会快。

*** 它的解法

用 9 个随机哈希函数 + 120 张哈希表，定位"哈希桶"里跟 query 同桶的 token。把哈希表全放 CPU，
GPU 只算 attention。

*** 为什么这招管用（以及哪里埋了新坑）

查询快了，CPU/GPU 也分工了。但哈希表本身占内存巨大：120 张表 x 长上下文 x 头数，
batch 一大 CPU 内存就吃不下。论文里 MagicPIG 在 batch=8 之后就吞吐崩盘。
并且 LSH 是概率索引，碰撞率高时精度仍然不稳，128K Ruler 上掉 7 分。

** 2025-12 | CTkvr: 别给 token 建复杂索引——用 query 自己当索引

*** 它看到了什么问题

回头看这条线：一边是 block-level（快但糙），一边是 token-level + ANN/LSH（准但索引太重）。
*整个领域都在 Key 这一侧做文章——给 Key 建索引、按 Key 聚类* 。但 Key 是离线的、海量的，
怎么折腾都贵。能不能换边想问题？

CTkvr 做了一个实验：把同一个长文本中位置不同的 query 向量拿出来算相似度。结果发现——
*经过 RoPE 之后，位置相邻的 query (距离 1K 以内) cos 相似度持续超过 0.8* 。再做第二个实验：
*query 相似度越高，它们各自取出来的 top-k Key 重叠率就越高（正相关）* 。

也就是说：你不必给每个 query 都重新扫一遍 Key——附近的 query 早就替你扫过了，
它们找出的 top-k 就是你大概率也要的那批。

*** 它的解法

两阶段：
1. *Query-Centroid IVF (粗筛)* ：取最后 C 个 query 向量当"质心"，预存每个质心对应的
   top-rho 个 Key 的下标。新 query 来了，先找最近的 C' 个质心，把它们的 Key 列表合并去重，
   得到一个"召回集" L^Recall（约 10K）。
2. *Token-Level Rerank (精排)* ：在召回集上算真正的 Q.K^T，取 top-rho' 个 Key（约 1K）做 attention。

工程上：召回那一步留在 GPU，精排那一步丢到 CPU 上（KV cache 本就在 CPU），用自定义 CUDA kernel
让两端并发跑。再加一个 *Dynamic Centroid Update* ——多轮对话场景下，FIFO 队列滚动更新最老的质心。

*** 为什么这招管用（以及哪里埋了新坑）

这是真正的换边思考——*把索引从 Key 侧搬到 Query 侧* 。Key 是海量、静态的，给它建索引一定贵；
Query 是少量、动态的、而且天然有"位置相邻则相似"的结构（拜 RoPE 所赐）。用 query 自己做索引，
索引构建快 1000x（HNSW 6 秒，CTkvr 0.4 毫秒），精度损失 < 1%，decode 吞吐 3-4x。

留下的坑：质心策略依赖 RoPE 这个特定位置编码——换成 ALiBi、NoPE 之类还成不成立没验证；
多轮对话超过 8 轮时质心漂移，DCU 也只能部分补救；CPU 算 attention 这条路需要 CPU FLOPs 够高，
小机器吃不消。

* 前沿延伸

CTkvr 是 2025 年 12 月 17 日刚挂的预印本，截至目前是这条线的最新进展。

可以预见的下一步：
- *Query 侧索引这条思路会被推广* ：既然 RoPE 让 query 在位置上聚集，能不能更细——分头、分层
  各自学一组质心？
- *跨请求复用质心* ：相似的 prompt 模板（agent 工作流、RAG）质心高度重合，缓存起来跨请求复用。
- *与 KV 量化正交组合* ：CTkvr 解决"取哪些"，量化解决"每个有多大"，两者乘起来还有空间。

* 洞见

*真正的进步往往不是"把现有那条路再优化一倍"，而是"换一边想问题"。*

这条河前 7 年所有论文（包括 InfLLM、Quest、RetrievalAttention、MagicPIG）都在 Key 侧做文章——
给 Key 切块、给 Key 建 ANN、给 Key 算哈希。Key 是海量的、静态的，再怎么折腾索引构建都贵。
CTkvr 的转身在于发现：*Query 才是少量的、动态的、且 RoPE 给了它天然的局部相似结构* 。把索引
搬到 query 侧，瞬间从"给一本字典建索引"变成"借隔壁刚查过的人的笔记"。

更深一层：这是 *"从优化数据结构到利用问题本身的结构"* 的转变。前几年大家在做 ANN/LSH，
是在跟数据库领域借现成的索引技术。CTkvr 用的"位置相邻 query 相似"这件事，是 LLM 内部
（RoPE + 自回归生成）独有的性质——只有把领域结构吃透才能想到。

*下一步最可能往哪走？* 沿着同一个方向继续——"还有什么 LLM 内部的结构没被利用？"
比如：不同 head 的 query 分布差异大，能不能 per-head 用不同的质心策略？
比如：agent 工作流里 prompt 模板高度重复，能不能跨请求做 query 索引复用？
比如：reasoning 模型生成长 chain-of-thought 时，query 的语义聚集模式比普通生成更强，
专门为 reasoning 设计的 KV 检索可能再压一个数量级。

* 启发

*做研究找突破口时，先问一句：当前所有人都在哪一侧做？另一侧动过没？*

KV cache 这条河整整 7 年都在 Key 侧切割、索引、压缩。CTkvr 之所以能拿到 1000x 索引构建
加速，不是因为他们的算法更精妙，是因为他们换了一边。当某个领域所有论文的改进幅度都在
10-30%、所有人都用一样的术语吵架时，往往不是这条路没榨干，而是大家忘了还有另一条路。

落点在"能用"：这意味着你在做任何"找 X 的 top-k"问题时，可以先停一下——
- 现在所有方法都给 X 建索引吗？查询那一侧 Y 有什么内部结构没被利用？
- Y 的查询是否有局部性、周期性、模板复用？
- 如果有，能不能"用 Y 自己做索引"，把 X 那一侧的负担甩掉？

具体到 LLM 推理工程：如果你在做 RAG、agent 系统、长文档问答，CTkvr 这条路意味着你可以
*把推理服务的索引粒度从"按知识库 chunk"换成"按 query 模板"* ——
用户的 query 在语义空间里其实是稀疏聚集的，缓存"query 模板 -> 召回集"的映射，可能
比缓存"chunk embedding"省更多算力。
