* 问题之河

LLM 推理时，每生成一个新词都要回看前面所有词的"记忆"——这份记忆就叫 KV cache。
上下文越长，缓存越大，显存越炸，速度越慢。问题是：能不能把这份记忆压一压，
让模型一边省内存一边不变笨？

这事难在哪儿？记忆里每一个 token 都"有可能"被未来某次注意力用到。你扔掉任何
一个，都可能扔掉一句话的关键。所以这条研究线从 2023 年起就在一个核心矛盾里打转：
压得越狠，性能掉得越多。十几篇论文一直在围着同一个权衡撕扯——到底什么是"真正
重要"的 token，以及怎么在不还原原文的情况下把次要的东西安全地丢掉或合并掉。

* 演化叙事

** 2017 | Transformer + KV cache: 问题的源头

*** 它看到了什么问题

注意力机制要求每个新 token 都和所有历史 token 算相似度。如果每次都从头算
一遍历史 token 的 Key 和 Value，那是 O(n^2) 的灾难。

*** 它的解法

把已经算出来的 K 和 V 缓存下来，下一步直接复用。这就是 KV cache 的由来。

*** 为什么这招管用（以及哪里埋了新坑）

它把生成阶段每一步从 O(n) 算 Q·K 摊到了 O(n) 读缓存，节省了重复计算。
但缓存本身要常驻显存，且随上下文长度线性增长——8K 上下文还好，32K、128K
就开始挤爆 GPU。整条研究线的起点就是：这份越长越大的缓存，能不能瘦身？

** 2023 | H2O: "把不重要的 token 直接踢出去"

*** 它看到了什么问题

实际跑下来发现，绝大多数 token 在后续注意力分布里的权重接近零——它们躺在
缓存里就是占地方。既然如此，为什么不挑出真正"重要"的少数留下，其他全扔？

*** 它的解法

定义一个简单指标：累积注意力分数。每个 token 在历史所有 query 下被点名
的总分。分数高的就是 "heavy hitter"（重击者），保留；分数低的，驱逐。
缓存预算固定，谁分高谁留下，谁分低谁出局。

*** 为什么这招管用（以及哪里埋了新坑）

直觉很对——稀疏注意力本来就是 transformer 的常态。但 H2O 的硬伤是
*不可逆*：一旦一个 token 被踢，它的信息就永久消失。如果后面的 query
忽然需要它呢？模型就只能瞎编。这成了之后所有 eviction 派必须面对的疤。

** 2024 | StreamingLLM: "前几个 token 是神圣的"

*** 它看到了什么问题

H2O 跑长流时会神秘崩溃。研究者一查发现：模型对最前面几个 token 有
异常高的注意力——哪怕这些 token 内容毫无意义。原因是 softmax 必须把
分配权重某处，没有"重要"的地方它就硬塞给开头。这叫 *attention sink*
（注意力沉降点）。

*** 它的解法

简单粗暴：永远保留前几个 sink token，再加一个滑动窗口（最近 N 个）。
中间老旧的部分全扔。

*** 为什么这招管用（以及哪里埋了新坑）

模型不崩了，能流式跑无限长。但代价是中间整片上下文消失——你给它喂一本
50 页的文档问第 20 页的事，它已经看不见了。这暴露了"按位置丢"的根本局限：
重要性和位置不是一回事。

** 2024 | SnapKV: "丢什么取决于要问什么"

*** 它看到了什么问题

H2O 和 StreamingLLM 都是 *任务无关* 的——它们不知道用户接下来要问什么。
但实际上，同一段历史在不同问题下，重要 token 是不一样的。

*** 它的解法

引入一个 "observation window"——把 prompt 末尾的若干 token 当作"问题的
代理"，让它去打分历史 token，按这个打分来选保留谁。这样保留下来的就是
"对当前问题有用的"那批。

*** 为什么这招管用（以及哪里埋了新坑）

性能立刻上来一截，因为有了任务信号。但它的范式没变：仍然是 eviction，
仍然以单个 token 为单位。被踢的 token 还是死的。CaM 一派就在这里登场了。

** 2024 | CaM: "别踢，合并"

*** 它看到了什么问题

eviction 永远存在"踢错"的可能。既然每个 token 都还带一点信息，能不能
*不丢*，而是把它的信息揉进保留的 token 里？

*** 它的解法

对被选中要丢的 token，按其注意力相似度，把它的 K/V 加权合并到保留的 token
上。原本"删除"变成了"折叠"。

*** 为什么这招管用（以及哪里埋了新坑）

信息不再断崖式丢失，长上下文任务上明显比 H2O 系稳。但 CaM 仍然以单个
token 为粒度判断合并对象——一个名词短语里的修饰词可能被并到完全无关的
另一个 token 上，把语法关系搅烂。下一篇 SepLLM 就盯上了"边界"这个东西。

** 2025 | SepLLM: "标点是天然的摘要槽"

*** 它看到了什么问题

观察注意力热力图会发现一个怪现象：英文的逗号、句号、换行符这些"看上去
没意义"的 token，注意力分数异常高。它们像是模型自己学会的"段落摘要点"。

*** 它的解法

利用这一点：把每段非分隔符 token 压进它后面紧邻的分隔符里——让分隔符
本身承担"这段话的浓缩版"的角色。推理时只留分隔符的 K/V，其他全部扔。

*** 为什么这招管用（以及哪里埋了新坑）

第一次把"语义结构"引进了 KV 压缩。分隔符当锚点，比按位置丢、按 token 丢
都更尊重语言的天然单位。但它依然太粗：一整段塞进一个 token，段内的细
结构（多个并列子句、多个实体）就被一锅端了。这正是 SemantiCache 切入的
缝隙。

** 2026.03 | SemantiCache: "按语义层级压，不按 token 压"

*** 它看到了什么问题

把之前所有方法摆在一起看，它们犯了同一个错：要么以单个 token 为单位
（H2O / SnapKV / CaM），要么以"任意一段"为单位（一些用固定长度块的方案
如 MoBA、Native Sparse Attention），要么"一段塞一个"过于粗（SepLLM）。
这都是 *semantic fragmentation*——把一句"修饰语 + 名词"拆开、把一句
条件从句和它的主句切开。

*** 它的解法

把人类记忆长文的方式搬过来，做成三段流水线：

1. *语义切块*（Semantic Chunking）。沿用 SepLLM 的洞察，把分隔符当成天然
   语义边界，按它们切成块。但分隔符自己的 K/V 完整保留，只压块内的内容。
2. *贪心聚类*（GSC, Greedy Seed-Based Clustering）。在每个块内，挑第一个
   未分配 token 作"种子"，把后续余弦相似度高于阈值 τ 的 token 都吸进同一
   簇。这是单遍 O(l^2/2) 的轻量聚类，不需要 k-means 那样反复迭代。
3. *簇合并 + 比例注意力*（Proportional Attention）。每个簇用 mean-pool
   做出一个 "semantic core"。但合并完一个簇缩成一个 token，注意力会被
   稀释——加权计算时给 logits 加一个 log(|S|)，等价于把注意力分数乘以
   簇大小。换句话说，"这个 core 代表 5 个 token"，那它在 softmax 后就拿
   到 5 倍权重。

*** 为什么这招管用（以及哪里埋了新坑）

三件事咬合得很整齐：
- *切* 不破坏语义边界（继承 SepLLM 但放过块内细节）
- *合* 不丢冗余（继承 CaM 但语义粒度更细）
- *配重* 不让合并稀释影响力（这是新的——以前没人显式处理"一个 core 应该
  代表几个 token 的话语权"）

实测上 decoding 加速到 2.61×，且 LongBench / NIAH 性能逼近全 cache。

留下的新坑：(a) 阈值 τ 是固定的，不同层、不同 head 的最佳值可能不一样；
(b) GSC 的"第一个未分配 token 当种子"是次序敏感的——换个顺序聚类结果
就变了；(c) 依赖分隔符，对没有显式标点的语言或代码就尴尬。

* 前沿延伸

SemantiCache 之后（2026 Q1-Q2 同期），同一条线已经在分叉：

- *AdmTree* (Li et al., 2026)：把 SemantiCache 的"块内一层聚类"扩成
  *自适应语义树*——多层级聚合，每层粒度自动选。回答了"τ 该多大"这个
  遗留问题：用层次结构代替单一阈值。

- *COMI: Coarse-to-fine Context Compression* (Tang et al., 2026)：从
  marginal information gain 的角度选择保留 token，比"相似度阈值"更有
  信息论依据。

- *"Read as Human"* (Tang et al., 2026)：进一步把人类阅读策略
  形式化——close reading（精读关键段）+ skimming（粗读其余段）并行做。
  这是 SemantiCache "模仿人类记忆"动机的更系统化版本。

整条线的方向已经从"怎么丢得少"转到"怎么按人类认知的层级去压"。

* 洞见

把这条线压成一句话：*KV cache 压缩的进化史，就是把"压缩单元"从"任意 token"
一步步往"自然语言单元"对齐的过程。*

更深一层看到的是研究者们对"重要性"定义的迭代：
- 第一代用 *注意力分数* 当重要性 → 但模型自己产生的注意力分布是偏的
  （sink 现象）。
- 第二代加入 *任务/query 信号* → 但仍是 token 级的二分法。
- 第三代换思路 *不删而合* → 信息守恒，但单位仍错。
- 第四代承认 *语言本身有结构* → 让压缩单位对齐到这个结构（分隔符块、
  语义簇）。
- 然后必然出现的新问题：合并完信息被稀释怎么办？答：在 softmax 之前
  加 log(|S|) 配重——这是一个非常优雅的小操作，把"代表多少 token"显式
  写进 attention 公式。

下一步会往哪走？可猜两个方向：
1. *阈值 τ 学起来* —— 让模型自己学每层、每个 head 的最佳聚类粒度，
   取代手调。AdmTree 已经开了头。
2. *压缩-检索一体化* —— SemantiCache 仍是"一次性压"，未来很可能跟
   long-context retrieval 融合：被压成 core 的内容如果某次 query 需要
   它的细节，可以反查原始 K/V。这相当于给压缩加上一层 cache 的 cache。

* 启发

对"怎么找问题"的启发：

1. *盯着权衡曲线上同一个点撕扯的方向不对*。前三代方法都在"踢多少 / 怎么
   选"上做文章，性能-内存曲线被推平了。真正的突破是 SemantiCache 这种
   *换坐标轴* 的——别问"丢哪个 token 好"，问"压缩的最小单位应该是什么"。

2. *别人嫌脏不看的现象往往是金矿*。SepLLM 之所以成立，是因为有人愿意
   认真看"为什么标点 token 注意力这么高"——很多人会归因为"模型有 bug"
   略过去。一个被忽略的稳定现象，常常正好就是下一个解法的支点。

3. *对齐到领域的天然结构往往是终极方向*。这条线的终点（也是 SemantiCache
   的核心论点）就是：自然语言不是 token 序列，是层级结构；压缩这件事
   不该跟语言对着干。任何"把领域天然结构搬进算法"的工作，方向上都不会
   错——只是早晚和实现的事。

落到能用上：你下次做"删/压/选"类工作时，先停一下问——*我现在用的"操作
单位"，是问题天然的单位，还是工程上的方便单位？* 如果是后者，那里就有
一个 SemantiCache 在等。
