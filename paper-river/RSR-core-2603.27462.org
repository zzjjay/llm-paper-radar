* 问题之河

LLM 推理的瓶颈是矩阵乘向量。权重矩阵几亿到几百亿个数，每出一个 token 都要把它们读一遍、乘一遍。怎么让这件事既不掉精度、又跑得飞快？

这个问题难在两个方向往一块挤：
- *硬件方向*：把权重从 16 位压到 1 位，省内存、省带宽。但越压精度越掉，怎么压到极限还能用？
- *算法方向*：标准矩阵乘是 O(n^2)，做了几十年都没人能突破。但如果权重只有 {-1, 0, 1} 这三种值呢——它还是 O(n^2) 吗？真的没有捷径吗？

两条路各走各的，谁也帮不了谁。直到有人发现：硬件把权重压到了三值之后，算法的旧地基塌了一块——可以挖出 log 因子。再然后又有人发现：算法挖出来的好处，落不进硬件 kernel，只是纸面胜利。这条河就是讲这两条路怎么从分头跑到撞在一起。

* 演化叙事

** 2023.10 | BitNet: 从头训一个 1-bit Transformer

*** 它看到了什么问题

LLM 推理慢、贵、烫。FP16 的权重一个数占 2 个字节，70B 模型光权重就 140GB。INT8 训后量化能省一半，但再往下压、压到 4 位甚至 1 位，精度断崖式下跌——因为模型从来不是为低精度训练的，硬塞给它一个 {-1, +1} 的世界，它当然蒙。

*** 它的解法

不做训后量化了——从头就用 1 位训。把 PyTorch 里的 ~nn.Linear~ 全换成 ~BitLinear~：前向传播时权重被压成 ±1，反向传播时还是高精度梯度去更新一个"潜在的"实数权重。这样模型从第一步就在学怎么在 1 位的世界里生活。

*** 为什么这招管用（以及哪里埋了新坑）

模型自己适应了低精度，比事后硬压保留更多能力，且展现出和 FP16 类似的 scaling law。但纯 1-bit 表达力还是有上限：每个权重只能"开/关"，连"不参与"都说不出来。在大模型上还是有可观的能力差距，催生了下一步。

** 2024.02 | BitNet b1.58: 加一个 0, 引爆 ternary 浪潮

*** 它看到了什么问题

1-bit BitNet 在小模型上能凑合，scale 到几十 B 就吃力。问题不在训练框架，问题在词汇量——一个 0/1 的开关，表达不了"这条边不重要"这种很常见的情况。

*** 它的解法

只多加一个值: 0。权重集合从 {-1, +1} 变成 {-1, 0, +1}。三个值用 log2(3) ≈ 1.58 位编码，所以叫 b1.58。

*** 为什么这招管用（以及哪里埋了新坑）

那个 0 等于给模型一个"剪枝"按钮：不重要的连接可以归零。3B 起就开始追平同尺寸的 FP16 LLaMA，70B 上比 FP16 快 4.1 倍、矩阵乘能耗低 71 倍。

但 b1.58 只是一个 *模型架构*——它没说怎么把这些三值矩阵真的跑快。GPU 上的 cuBLAS/cuDNN 全是为 FP16/INT8 写的，没人为三值优化。结果是: 模型权重小了，推理却没快多少。两条修路的工程接力出现了：bitnet.cpp 走硬件向，RSR 走算法向。

** 2024.10 | bitnet.cpp: 硬件向的 CPU kernel

*** 它看到了什么问题

b1.58 模型放出来了，但用 HuggingFace + PyTorch 跑，等于把 bfloat16 的乘法做在三值矩阵上——三值的好处没占到一点。

*** 它的解法

为 x86 和 ARM 手写 SIMD kernel，把"三值乘 float 激活"这件事压进几条指令里。同尺寸模型，CPU 上 2.4x-6x 速度。

*** 为什么这招管用（以及哪里埋了新坑）

利用了三值乘法不需要真正的乘法器——查表、加减就够了。但它本质上还是 O(n^2)：每个矩阵元素都要被访问一次。没有触及"是不是可以更少地访问"这个问题。

** 2024.11 | RSR: 算法向, 第一次破开 O(n^2)

*** 它看到了什么问题

之前所有人——FP16、INT8、BitNet、bitnet.cpp——都默认了一件事: 矩阵乘向量必须是 O(n^2)。这是几十年的常识。但当权重只有三个值的时候，常识有个漏洞: *很多列是一样的*。

具体说: 把矩阵切成 k 行一个的横条，在每个条里看每列的 k 位 pattern。如果 k=8、ternary，pattern 一共只有 3^8 = 6561 种。矩阵宽 n=10000 列的话，按鸽笼原理，平均每种 pattern 出现 1.5 次——重复严重。

旧的做法：每列单独和输入向量 v 算点积，重复算 n 次。RSR 的洞察：每种 *pattern 只算一次*，然后散布到所有用了这个 pattern 的列上。

*** 它的解法

两阶段:
- *离线预处理*：把矩阵切横条，对每条做 counting sort 把相同 pattern 的列聚到一起，存好列置换和分组边界。
- *在线推理*：先按预处理记录的分组，把输入 v 中对应位置的元素聚合（gather + sum），再和每个 *唯一* pattern 算一次乘加，最后把结果散回去（scatter）。

复杂度从 O(n^2) 降到 O(n^2 / log n)，纸面上拿到一个 log 因子。

*** 为什么这招管用（以及哪里埋了新坑）

利用了"权重训完就不变"——一次预处理可以反复用。利用了"低位宽自带重复"——位宽越窄重复越多。

但论文里的实现是 NumPy + C++ 实验室代码：拿来跑 benchmark 可以，塞进真实的 LLM 推理 pipeline 就不行——Python 解释器、通用排序、可变长度的 scatter list、非合并的内存访问……这些工程性的损耗，把算法挣回来的 log 因子全吃光。理论 29x，实际进 HuggingFace 几乎没有提升。这就是 RSR-core 的起点。

** 2026.03 | RSR-core (本文): 让 log 因子真的落地

*** 它看到了什么问题

RSR 算法本身没问题，问题是 *Python 写的 RSR* 太慢——慢到吃掉自己挣的所有便宜。具体卡点:

- 解释器调度，每次 matmul 都要绕一圈 Python
- 通用比较排序 O(n log n)，但 pattern 空间是有限的离散值，根本不需要比较排序
- scatter 用 Python list of lists，内存四处乱跳
- 三值需要正负两个 scatter 方向，原始实现用两个 list，分支密集
- 多个共享同一输入的 linear (Q/K/V) 各自调一次 kernel，浪费

*** 它的解法

把整条 RSR 重写成原生 kernel:

1. *Counting sort 替换通用排序*：pattern 空间只有 2^k 或 4^k 个桶，O(n + buckets) 一遍扫过去。
2. *Metadata 紧凑化*：列置换索引压成 16 位、scatter 用 bitmask 而不是变长 index list、CUDA 上一个组的元数据塞进一个 64-bit word，kernel 一次 load 就开干。
3. *CPU 上 gather 和 aggregate 融合*：一次扫过输入向量直接累加，不落地中间数组。还用 scalar unrolled load + software prefetch，比硬件 gather 指令在 RSR 这种随机访问 pattern 下更快。
4. *CUDA 上 row block → thread block, warp 并行处理 group*：每 warp 一个 shared memory 局部 buffer 避免写冲突；预处理时把零贡献的 group 直接扔掉。
5. *活化量化和矩阵乘融合, Q/K/V 一次批掉*：少一层 Python dispatch，给多核更大的并行池。

*** 为什么这招管用（以及哪里埋了新坑）

它做的不是"算法创新"，是把已有的算法 *从论文翻译到硅片*——每一处优化都对应了一个被前一版实现浪费掉的硬件特性。最终: CPU 上对 ternary LLM 推理快 62x（vs HuggingFace bf16），CUDA token 生成快 1.9x，精度完全无损。

新的局限: CUDA 上提速只有 1.9x，因为 GPU 的随机访问代价高，scatter 模式不够友好。算法的下一步可能要专门为 GPU 的 memory hierarchy 重新设计 pattern 分组——这是个开放问题。

* 前沿延伸

截至目前 (2026-05)，RSR-core 是这条线上"算法 × 硬件"结合最新的工程化里程碑。下一步可见的方向：
- GPU 友好的 RSR 变体：CUDA 上的 1.9x 远没榨干算法的潜力，瓶颈是随机访问。需要重新设计 pattern layout 以贴合 GPU 内存合并访问。
- 跨 layer 的 pattern 共享：当前每个 linear 层独立预处理。如果不同层共享一个 pattern 字典，预处理产物可以进一步压缩。
- 训练时知 RSR：在训练 loss 里加一项鼓励列出现重复 pattern，可能进一步放大 log 因子。这条 BitNet 和 RSR 还没正式合流过。

* 洞见

这条河表面上是"LLM 推理优化"，深一层是 *硬件压缩和算法压缩两条独立的路, 在某个点上互相打开了对方的门*。

- 当权重是 FP16 时，没人想"算法上能不能更快"，因为矩阵看起来"满"且"连续"——结构性冗余被表示精度掩盖了。
- 当权重被压到 ternary，结构性冗余暴露出来：列只有 3^k 种可能 pattern，n 大时必然大量重复。这时 O(n^2) 这堵几十年的墙第一次被压缩开了一道缝。
- 但算法上的胜利在工程里会被解释器/通用库/内存模式默默吃掉。"算法 → kernel" 之间的距离，往往比算法本身的发明更长，也更不被学界重视。

更深的一条线: *压缩表示的代价是产生新的冗余*。FP16 没冗余但贵，INT4 有点冗余，ternary 有大量冗余。冗余从"坏事"变成"加速可利用的资源"——这是 RSR 的核心认知反转。

下一步的押注: 下一个突破可能不是"再把 bit 砍小"，而是"重新设计训练目标让冗余出现得更结构化、更容易被 GPU 吃掉"。bit 数已经接近底线 (1.58 已经很接近 1)，剩下的空间在 *冗余的形状*。

* 启发

1. *看一篇论文先看它批判谁*：RSR-core 一句"existing implementations operate at the application level"，就把上游的 RSR 论文位置点清了。读论文先读 introduction 里的 "however"。
2. *算法和工程的鸿沟比想象大*：理论 29x 进生产变 0x，是常态不是例外。一个有 log 因子的算法不一定比一个常数因子小的工程实现快——这意味着，如果你想做有影响的工作，做"把别人的算法 kernel 化"是被严重低估的高杠杆机会。
3. *硬件趋势会改变算法的可行性*：ternary 让 O(n^2) 那堵墙第一次有缝。下一次硬件趋势变化（比如 in-memory compute、新的内存层级）会再开新缝。盯着硬件路线图选算法问题。
4. *冗余是被表示方式定义的*：换一种数的表示方式，"无规律的矩阵"会变成"高度重复的矩阵"。当你卡在某个问题上时，问一句"是不是我用的表示方式自己藏住了结构？"
