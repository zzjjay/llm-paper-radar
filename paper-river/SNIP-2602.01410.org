* 这条河在讲什么

LLM 训练越来越贵。从 2018 年到 2026 年，整整八年，整个领域都在做同一件事：*把训练用的数字位数压低*。从 32 位压到 16 位，再压到 8 位，现在试图压到 4 位。

每往下压一档，矛盾就更尖锐：*要省钱，但模型不能崩*。这条线上每一篇关键论文，都是在回答同一个问题的不同变体——"这一档精度下，怎么不让模型死？"

倒读这条线，你会发现一个很美的规律：解法在不断从"全局一刀切"走向"局部精打细算"。SNIP 是这条演化线目前的终点，它把"哪一层该用什么精度"做成了一个可以求解的数学题。

* 第一段：从 32 位到 16 位（2018）

故事得从 Micikevicius 2018 那篇《Mixed Precision Training》开始讲。

在那之前，神经网络训练默认用 32 位浮点数（FP32）。一个数字占 4 字节，一个 GPT-3 规模的模型，光是权重和激活就能把显存撑爆。所以大家想：*能不能把数字位数减半，用 16 位（FP16）？*

直接换不行。FP16 表示范围窄得多，梯度一旦变小就会下溢成 0，模型直接学不动。

Micikevicius 给出三个补丁：

1. *FP32 主权重副本*：参数本身保留一份 32 位的"原版"，用 16 位算梯度，但更新时回写到 32 位
2. *损失缩放*：训练前把 loss 乘以一个大常数（比如 1024），让梯度都被放大到 FP16 能表示的范围，更新前再缩回去
3. *算子级别的精度选择*：矩阵乘用 FP16，但累加用 FP32

这套打法奠定了一个隐含范式——*"混合精度"不是均匀压低，而是关键的地方留高，不关键的地方压低*。这是后面所有故事的种子。

但 Micikevicius 留下了两个口子。第一，FP32 主权重副本本身还在占显存，"省了 2 倍"打了折扣。第二，他没说更低的位数（8 位、4 位）该怎么办——他只解决了 16 位这一档。

* 第二段：从 16 位到 8 位（2022-2023）

四年后，硬件追上来了。NVIDIA H100 原生支持 FP8。新的问题来了：*16 位能不能继续往下压成 8 位？*

直接换还是不行。FP8 比 FP16 表示范围更窄，原来的"FP32 主权重 + 损失缩放"补丁不够用了——梯度在 FP8 下更容易爆掉。

这一段有两篇关键论文。Micikevicius 2022 那篇《FP8 Formats for Deep Learning》先定义了 *两种 FP8 变体*：E4M3（更精确，适合前向）和 E5M2（更大范围，适合反向梯度）。这其实是在说："8 位太窄了，我们要分两种用法。"

紧接着 Peng 2023 的 FP8-LM 把这个想法落地到 LLM 训练。他们做了一件很狠的事——*把 FP8 从前向矩阵乘扩展到梯度、优化器状态、跨卡通信*。这是个分级框架：

- O1：只把梯度 all-reduce 通信压到 FP8
- O2：连优化器状态都压到 FP8（Adam 的 m 和 v）
- O3：连分布式权重广播也压到 FP8

GPT-175B 实测，显存减 39%，速度快 75%。这等于宣布："FP8 不再只是个矩阵乘的精度，它是整条训练管线的精度。"

DeepSeek-V3（2024）顺着这条路把 FP8 训练真正用在了产业级万亿参数模型上，证明这套打法在最大规模也站得住。

但 FP8-LM 系列留下了一个明显的洞——*它默认所有 GEMM 都用 FP8*。这是个"全压"策略。每一层、每一个矩阵乘，都按同一档精度处理。

* 第三段：尝试压到 4 位（2025）

硬件路线图继续往下走。Blackwell 架构开始支持 FP4。问题再次重演：*8 位能不能压成 4 位？*

这次的难度跃升了一个量级。FP4 只有 16 个可表示值（2^4=16），离群值（outlier）一来，整个量化网格就被撑歪。原来 FP8 时代的"加点损失缩放、注意一下范围"已经远远不够。

Wang 2025 那篇《Optimizing LLM Training Using FP4 Quantization》是第一篇严肃的 FP4 训练论文。它的核心两招：

1. *可微量化估计器*：量化函数本身不可导，所以反向传播时梯度估计是错的。他们设计了一个平滑的近似，让权重更新方向不被量化噪声带偏
2. *离群值钳制和补偿*：检测激活里的极端值，单独处理，避免它们撑爆整个量化网格

这套打法在 13B 模型上把性能损失压到了"几乎可忽略"的水平。但代价也很明显——*梯度计算更复杂，离群值还得用稀疏 GEMM 单独算*。

更关键的是，Wang 2025 仍然延续着"全压"思路：所有层都用 FP4，靠精巧的数值技巧硬扛误差。

到这里，一个核心矛盾彻底暴露了：

#+begin_src
  现实：不同层对精度损失的敏感度差异巨大
  - Attention 投影层：稍微掉点精度无所谓
  - LayerNorm 附近的层：稍微一动就崩
  - 第一层和最后一层：通常比中间层敏感
  - 训练前期 vs 后期：敏感分布会变

  做法：所有层一刀切，要么全 FP4，要么全 FP8

  这是浪费。可以更省。
#+end_src

* 第四段：每层独立挑精度（2026，SNIP 本篇）

SNIP 看到这个矛盾，问了一个新问题：*能不能给每一层独立挑精度？敏感的层留 FP8，不敏感的层压到 FP4？*

这个想法不是 SNIP 原创——*推理领域早就有人在搞*。HAQ（2019）用强化学习挑 bit 数，OBC（2022）用 Hessian 算敏感度，BBS（2024）用 KL 散度。但 SNIP 对这些方法的批评很硬核：

#+begin_quote
"它们都只算了局部的量化误差，没考虑训练动力学。"
#+end_quote

这是什么意思？推理时，你只关心"这一层的输出和原版差多少"。训练时不行——*量化误差会通过梯度传到优化器，再被 AdamW 的动量放大，最终改变权重的更新轨迹*。一个局部误差很小的方案，可能在训练 1000 步后让整个模型偏离原轨道。

所以 SNIP 设计了两个新指标：

** 指标一：损失发散度（前向通路）

如果把第 i 层量化了，*整个网络的训练 loss 会涨多少？* 这个量按定义需要重新跑一次前向，太贵了。SNIP 用 Theorem 4.1 做近似——把量化看成给输入加了一个小扰动 δ，于是 loss 的变化近似等于 *梯度 · 扰动幅度*。梯度反正都要算（反向传播本来就要算），所以这个估计基本免费。

** 指标二：权重发散度（反向通路）

光看前向 loss 不够，因为优化器会"记忆"。SNIP 还问：*如果把第 i 层量化了，跑一步优化器之后，权重和原版差多少？* 这个量涉及二阶偏导（loss 对权重和输入的混合偏导），训练时根本没算。怎么办？

SNIP 用了一招很聪明的:*往输入里注入一个小高斯噪声，多跑一次前向反向，看梯度变化*。这相当于用噪声去"探针"那些没有显式计算的二阶项。代价是每隔一段时间多跑两次前反向，但比每层单独评估便宜多了。

** 把两个指标接到整数规划

有了 *(损失发散度 + 权重发散度) = 这一层选这个精度的代价*，剩下的就是个数学题了：

#+begin_src
  最小化   sum( q_{i,j} * x_{i,j} )       // 总质量损失最小
  约束 1   sum( e_{i,j} * x_{i,j} ) >= E_t // 总加速比达标
  约束 2   sum_j x_{i,j} = 1               // 每层只能选一个精度
  约束 3   每个 pipeline stage 内单独满足约束 1  // 不让流水线打不平
#+end_src

这是个经典的 *背包问题变种*，用 scipy 的 milp 求解器（HiGHS 后端）30 秒搞定。输出就是一份"哪层用 FP8、哪层用 FP4"的全局最优清单。

实测在 1B / 3B / 7B / 70B Llama 上，最多省 80% 的 FLOPs，质量基本不掉。

* 这条线背后真正在发生什么

表面上是位数在降——32, 16, 8, 4。但底下的认知转变是另一回事。

*第一层转变：从"精度是数值问题"到"精度是分配问题"*。早期大家想"怎么让 FP16 不崩"，是个数值技巧问题。SNIP 这个时代变成"在固定的精度预算下，怎么分配最优"——这是个 *资源调度问题*。

*第二层转变：从"训练时的精度策略不依赖模型"到"精度策略和模型本身耦合"*。FP8-LM 的策略对所有 LLM 一样，SNIP 的策略每个模型、每个训练阶段都不同。混合精度从"框架特性"变成了"模型相关的超参数"。

*第三层转变：从"硬件能跑什么我就用什么"到"硬件 / 算法 / 编译器联合优化"*。SNIP 明确把 pipeline parallelism 约束写进了 ILP——你不能只在某些 stage 集中用 FP4，否则流水线会失衡。精度选择不再是孤立决定，它和分布式策略耦合在一起。

* 下一步可能往哪走

按这个演化趋势推：

1. *动态精度*：SNIP 目前是周期性重算（gather statistics 后求解一次），但训练动力学会变。下一步是 *训练每一步都重新分配精度*，可能需要更便宜的敏感度估计
2. *硬件协同设计*：既然每层精度都能算出来，那能不能让硬件支持的精度组合反过来指导芯片设计？"哪些组合最常被 ILP 选中"是芯片设计者想知道的数据
3. *跨任务迁移*：同一个模型架构在不同任务（pretrain / SFT / RLHF）下敏感度分布可能不同。能不能学一个 *"敏感度的元模型"* 而不是每次都重新统计？
4. *再往下到 1-2 位*：FP4 之后，1.58 bit（BitNet）已经在做了。但那是另一条线——*改架构去适应低精度*，而不是 *在固定架构上分配精度*。两条线最终可能合流。
