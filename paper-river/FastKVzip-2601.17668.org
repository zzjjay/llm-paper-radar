* 问题之河

一个大模型在你输入十万字之后，为什么会突然变慢、爆显存？因为它要记住每一个字对应的"中间态"——这叫 KV cache。上下文越长，这个缓存越大，最后撑爆显卡。

那能不能扔掉一部分？当然能。问题是：*扔哪些*。扔错了，模型立刻变蠢；想扔得准，又得花一堆算力去判断哪些重要——结果"省下来的钱不够交学费"。这条河，整整三年，每一代人都在和这个两难死磕：*要么快但蠢，要么准但慢*。

* 演化叙事

** 2017 | Transformer + KV cache: 给"快"种下了"贵"的种子

*** 它看到了什么问题

在 Transformer 之前，RNN 一个 token 一个 token 地算，慢得令人发指。注意力机制把这条序列依赖砍掉了——每个位置可以并行地"看"所有其他位置。

但推理时还是要一个 token 一个 token 地出。如果每出一个新 token 都要把前面所有 token 的 K（key）、V（value）重算一遍，那等于白搭。

*** 它的解法

把每个 token 算出来的 K、V 存起来，下一个 token 直接复用——这就是 *KV cache*。生成第 N+1 个 token 时，前 N 个 token 的 K、V 不用重算。

*** 为什么这招管用（以及哪里埋了新坑）

时间换空间：算力省下来了，但是要存东西。每个 token 在每层、每个 head 都要存一对 K 和 V。模型越大、上下文越长，这个缓存就越大。Llama-2 70B 处理 128K 上下文时，KV cache 要占 60GB 以上——比模型权重本身还重。

埋下的坑：*KV cache 的体积与上下文长度成正比，没有任何上限*。等到 "100 万 token 上下文" 成了标配，这就是必爆的雷。

** 2023 | H2O: 第一次拿起剪刀

*** 它看到了什么问题

整条 KV cache 都装着，但注意力矩阵其实非常稀疏——多数位置的注意力权重接近 0，等于"没人看"。既然没人看，为什么还要占显存？

*** 它的解法

H2O（Heavy-Hitter Oracle）观察过去一段时间每个 KV 被看了多少次（累计注意力分数），把"重器"（heavy hitters）留下，剩下的扔掉。本质是 *用过去的注意力分布预测未来还会不会被用到*。

*** 为什么这招管用（以及哪里埋了新坑）

实证上的稀疏性是真的——多数情形下 20% 的 KV 承担了 80% 的注意力。所以扔掉 70% 也能基本不掉点。

埋下的坑：这是 *query-dependent* 的——它假设"过去常被关注 → 未来还会被关注"。但用户可能下一秒就抛出一个全新的问题，这个问题需要的 KV 在过去恰恰没被怎么看。结果就是：*多轮对话或多个查询的场景下，扔过的 KV 正好是新问题需要的*。

** 2024 | SnapKV: 把剪刀移到 prefill

*** 它看到了什么问题

H2O 主要在 decoding 阶段慢慢扔。但很多场景（长文档问答、代码理解）是 *prefill 重*——把十万 token 的文档读进去这一步就已经爆显存了，等不到 decoding 才动手。

*** 它的解法

在 prefill 即将结束的最后一小段窗口里，看每个 KV 被这段窗口里的 query 关注了多少次，做一次大扫除。压缩动作发生得更早，省得更多。

*** 为什么这招管用（以及哪里埋了新坑）

prefill 末尾的 query 通常带着"问题信号"（用户的提问），用它们来挑 KV 比用文档自身的 query 更靠谱。

但本质和 H2O 同病：*还是 query-dependent*。如果用户问了第二个完全不同的问题，前一次扫荡留下的 KV 不一定对得上。

** 2024 | StreamingLLM: 攻击一个隐藏假设

*** 它看到了什么问题

H2O、SnapKV 都按"注意力分数高 → 留"这条规则扔。但人们发现一个怪现象：*把最早的几个 token 扔掉，模型立刻崩溃*——哪怕这些 token 看起来根本不相关（比如 "<s>" 或者 "The"）。

*** 它的解法

提出 *attention sinks* 概念：模型把开头几个 token 当成"蓄水池"，softmax 必须有地方"泄洪"，不然分布就乱了。所以无论怎么压缩，*开头几个 token 永远不能动*。

*** 为什么这招管用（以及哪里埋了新坑）

这是一个对"什么样的 KV 重要"的全新理解——不全是语义上的重要，有些是 *机械上的重要*（撑住数值稳定性）。Fast KVzip 后面的 "sink-attention gating" 设计就直接借用了这个洞察。

留下的坑：sink 之外的 KV 怎么挑，还是没解决，依然依赖"过去注意力"这种事后信号。

** 2025 | KVzip: 范式翻转——不看 query，看"原文"

*** 它看到了什么问题

前面所有方法都有同一个软肋：*靠过去预测未来*。多查询、多轮对话场景下这就会漏。能不能换一种思路：判断一个 KV 重要不重要，*跟具体 query 无关*，只看它在"承载原文信息"中有多关键？

*** 它的解法

让模型做一件事：基于当前 KV cache *重建出原文*。重建过程中谁被频繁用到，谁就重要。这是 *query-agnostic* 的——不管未来用户问什么，这个重要性分数都成立。

*** 为什么这招管用（以及哪里埋了新坑）

重建是"无监督"的内省任务，类似 BERT 的 mask-language-modeling 的精神：靠任务无关的目标拿到一个能广泛迁移的特征。结果：在多查询基准上，KVzip 显著优于 H2O / SnapKV。

埋下的坑：*重建过程本身要再跑一次模型前向*——prefill 时间直接翻倍。准是准了，但 14B 模型处理 170K token 要 150 秒（无压缩 90 秒），上线场景没人敢用。

** 2026 | Fast KVzip: 不重建，直接"预言"

*** 它看到了什么问题

KVzip 的核心算力开销是"用整个 LLM 跑一遍重建"。但仔细想：模型其实在 *prefill 的那一遍前向* 里就已经把每个 token 的语义都编码进 hidden states 了。能不能不再跑第二遍，而是 *直接从已有的 hidden states 里读出"这个 KV 以后会有多重要"*？

*** 它的解法

在每一层注意力旁边挂一个超轻的 *gating module*——输入是该层的 hidden state，输出是每个 KV head 的重要性分数。

训练这个 gate 的目标信号怎么来？*就用 KVzip 那套重建分数*——但重建只在 *训练阶段* 跑一次，*推理阶段* 完全用不到。

架构借鉴 StreamingLLM 的 attention sinks：让 gate 看几个 sink token 的注意力模式来打分，效果最好。

训练只更新 gate 的参数（LLM 冻住），14B 模型不到 1 个 H100 GPU 小时就训完。

*** 为什么这招管用（以及哪里埋了新坑）

这是一次精彩的"目标分离"：
- *准*（generalization）来自 query-agnostic 的重建目标——继承 KVzip
- *快* 来自把昂贵的判断过程蒸馏成一个轻量 gate——绕开 KVzip 的瓶颈

结果：RULER-4K 上 25% 预算几乎无损，170K token prefill 比 KVzip 快 ~2x，比无压缩还省内存。

新的坑：gate 还是一个外挂模块，没有融进预训练。如果未来的基础模型本身就把"KV 重要性预测"作为预训练目标的一部分，可能根本不需要这个外挂。

* 前沿延伸

Fast KVzip 自己就处在"前沿"——但它不是孤军：

- *KVzap* (Jegou & Jeblick, 2026)：和本文 *并发*，几乎同一时间提出非常接近的 gating 优化目标，架构和训练细节略有差异。两篇独立工作收敛到同一方向，强烈暗示这是问题演化的"必然下一步"。
- *TrimKV* (Bui et al., 2025)：用时间衰减重要性的 retention gate，主要针对数学推理；在 prefill 重场景下掉点明显。
- *DMS* (Łańcucki et al., 2025)：用 logit distillation 训 eviction policy。
- *Locret* (Huang et al., 2024)：用 gate 预测 KV 重要性，但只在 prefill 阶段生效，decoding 不管。

可以看到，*门控（gating）* 已经成为 2025-2026 KV 压缩的共识词。Fast KVzip 的核心贡献是把"gate 该看什么、训什么目标、放哪一层"三件事都系统化了，不是开辟了一个新方向，而是把一个已经在涌现的方向 *规范化*。

* 洞见

这条线的真正主线，不是"怎么压缩 KV cache"，而是 *"怎么从一个昂贵的 oracle 里蒸馏出便宜的近似"*。

- H2O 用 *过去的注意力分数* 当 oracle —— 便宜，但 oracle 本身不准（query-dependent）
- KVzip 用 *完整的重建任务* 当 oracle —— 准了，但 oracle 太贵（要跑两遍前向）
- Fast KVzip 把 KVzip 的 oracle *蒸馏到一个小 gate 里* —— 准的来源不变，但访问 oracle 的成本降到 1%

这是一个被反复验证的模式：*第一代方法用启发式，第二代方法用昂贵但正确的目标做 oracle，第三代方法把 oracle 蒸馏成轻量模块*。同样的剧本在 RLHF (PPO -> DPO)、speculative decoding、模型蒸馏里都演过一遍。

下一步最可能往哪走？我猜两个方向：

1. *把 gate 融进预训练* —— 不再外挂，而是模型一开始就被训练成"会自我评估 KV 重要性"的形态。MoE / MoD / MoBA 的趋势暗示这条路是走得通的。
2. *Gate 和量化/低秩压缩组合* —— 当前 gate 解决的是"扔哪些"，量化解决的是"剩下的每一项怎么存得更小"。两条线正交，组合空间巨大。

* 启发

这条演化线对"怎么做研究"的启示：

- *先把问题分解成"准"和"快"两个独立维度*。多数突破不是同时提升两者，而是发现可以 *分别优化* 它们。KVzip 先把"准"打到天花板（不计成本），Fast KVzip 再把"快"打回来——这种分阶段攻克比一次性求最优要务实。
- *昂贵的 oracle 是宝藏*。如果你已经有一个慢但正确的方法，别急着扔掉，*它是天然的训练信号源*。蒸馏一个轻量近似往往比从头设计启发式更有效。
- *并发工作是信号*。Fast KVzip 和 KVzap 几乎同时出现且方案相似——当你在一个方向上看到多个独立工作收敛，说明这是"问题演化的必然下一步"，值得 all in。

落点：这意味着你可以——下次遇到一个"准但慢"的方法，不要轻易放弃，先问 *"这里面有没有一个 oracle 可以蒸馏成 gate？"* 这个问题问出来，半个解法就有了。
