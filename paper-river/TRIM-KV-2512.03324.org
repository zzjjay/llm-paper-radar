* 问题之河

LLM 一边聊一边记。聊得越久，要记的东西越多——记到显存撑不住为止。

那能不能边聊边忘？人脑天天这么干。问题是，*哪些该忘、哪些得留*——这件事没人能给出一个干净的答案。十年了，一代一代方法都卡在同一个地方：判断一个 token "重不重要"，靠的是它*现在被注意到多少*。可"现在没人看"和"以后没人看"是两件事。于是模型常常把后面才用得上的关键事实，在它还没被用到之前就扔了。

TRIM-KV 想换个思路：能不能让模型在生成 token 的那一刻，*就直接说出来*这个 token 该不该长留？把"重不重要"从一个事后猜测的东西，变成 token 自己带的一个属性。

* 演化叙事

** 2017 | Transformer: 把 "记住一切" 变成默认动作

*** 它看到了什么问题

RNN 一个 token 一个 token 顺着算，慢，且远距离依赖会糊掉。

*** 它的解法

注意力机制。每个新 token 直接和所有历史 token 做点积，谁相关就看谁。为了不每步重算历史的 K 和 V，*把它们缓存下来*——这就是 KV cache。

*** 为什么这招管用（以及哪里埋了新坑）

并行训练、长距离依赖都解决了。但代价是：cache 大小随生成长度*线性增长*，注意力计算*二次增长*。十万 token 的对话，cache 能塞满整张显卡。后面所有 KV 压缩工作，都是在还这笔账。

** 2023 | StreamingLLM: 留头留尾, 中间全扔

*** 它看到了什么问题

一个真实场景：你和模型聊了三天。cache 撑不住了。最朴素的做法——把最早的 token 扔掉——结果模型直接崩，开始胡言乱语。

*** 它的解法

观察到一个怪现象：模型对*最开头几个 token*的注意力高得离谱，哪怕这些 token 内容没意义（比如 BOS）。这几个被称为 "attention sinks"，相当于注意力的泄洪口。

方案：*永远留住开头那几个 sink + 最近的一个滑窗*，中间全扔。

*** 为什么这招管用（以及哪里埋了新坑）

简单、稳定、几乎不用算。但它把"中间"等同于"没用"——只要你的对话里有"三天前提到的密码是 7421"，滑窗一过就再也找不回来。这是一个*结构性偏见*：按位置砍，不按内容砍。

** 2023 | H2O: 不按位置砍, 按 "被关注度" 砍

*** 它看到了什么问题

StreamingLLM 太粗暴。中间的 token 不是平均没用，是*分布不均*——少数关键 token 反复被注意，大多数是噪声。能不能保留关键的、扔掉噪声的？

*** 它的解法

定义一个分数：每个 token 的*累计注意力得分*（被后续 token 看了多少次的总和）。得分高的叫 "heavy hitters"——重要 token。当 cache 满时，扔分数最低的那些。

*** 为什么这招管用（以及哪里埋了新坑）

效果显著优于 StreamingLLM。但这套逻辑有个隐藏假设——*过去被关注的，未来还会被关注*。在数学推理、代码生成、长程规划这些场景下，这个假设经常崩：模型早期推导出来的中间结论，可能要到第 8000 个 token 才用到。在它"还没被用到"的几千步里，它的累计注意力分一直很低，早就被 H2O 扔了。

*更深的坑*：注意力分是*事后线索*。等你能看到一个 token "重要"，它已经在 cache 里晃了几百步。这本身就是浪费。

** 2024 | SnapKV: 索性在 prefill 阶段一次压缩到底

*** 它看到了什么问题

H2O 在解码阶段每步都更新分数、做淘汰，成本高、还得不到答案。而且很多场景里，prompt 本身就特别长（几万 token），光是 prefill 阶段 cache 就已经吓人。

*** 它的解法

prompt 一进来，用最后一小段（"观察窗"）的注意力去给所有前面的 token 打分，*一次性*选出 top-k 留下，剩下的当场扔。解码阶段就轻装上阵。

*** 为什么这招管用（以及哪里埋了新坑）

prefill 大场景下又快又准。但它本质还是 H2O 那条路——*用注意力当重要性的代理*。而且它管不了解码阶段：一旦开始生成，cache 又会重新涨起来。问题被搬到了后面，没消失。

** 2025 | SeerAttention-R: 别再靠启发式了, 让模型自己学

*** 它看到了什么问题

不管 H2O 还是 SnapKV，都是研究员拍脑袋设计的"重要性判据"，调参全靠经验。直觉是：模型自己肯定比我们更懂哪些 token 关键，*让它自己学怎么挑*。

*** 它的解法

训练一个 gate 网络，给 query 学一个 attention pattern，按 block 去取相关的 KV。全量 KV cache 仍然保留——只不过卸到 CPU 主机内存，需要时按需 fetch 回 GPU。

*** 为什么这招管用（以及哪里埋了新坑）

第一次让"挑哪些 token"成为一个*可学*的能力，效果比启发式稳。但代价很重：*全量 cache 还在那*，只是换了地方放。每次 fetch 都有 GPU-CPU 编排开销，长对话下来这个开销线性堆。它没真正"扔"任何东西，只是延迟付账。

** 2025 | TRIM-KV: token 生下来就告诉你它能活多久

*** 它看到了什么问题

把前面五代串起来看，作者发现一条暗线：所有方法都在*事后判断*——要么靠注意力分（过去线索），要么靠 query fetch（当前需求）。但注意力是噪声大的代理，query fetch 要付全量保存的钱。

更根本的问题：一个 token 重不重要，*难道不是它被生成的那一刻就大致定了吗*？常识告诉你，"the" 几乎肯定不重要，一个变量定义几乎肯定重要——这种判断不需要等几千步才下。

*** 它的解法

每一层每个 head 加一个轻量的 gate（线性层 + sigmoid），token 一生成就吐出一个 β ∈ [0,1]——它的*内在保质期*。

这个 β 控制*指数衰减*：token 在第 i 步生成，到第 t 步它的有效权重是 β^(t-i)。
- β 接近 1：慢衰减，长期记忆
- β 接近 0：快衰减，几步就被遗忘

注意力 logit 直接加上 (t-i)·log β，等价于*把遗忘塞进注意力公式本身*。cache 满了就扔当前 β^(t-i) 最小的那个——真扔，不卸载。

训练只学这些 gate，backbone 冻住。损失 = 蒸馏 + 容量惩罚（不让 Σβ 超过预算 M）。

*** 为什么这招管用（以及哪里埋了新坑）

它把"忘"从一个外挂的淘汰策略，变成模型*生成过程的一部分*。作者还指出一件有意思的事：学出来的 β 模式*自发*重现了 sink token、sliding window、gist 压缩这些前人手工设计的启发式——也就是说，过去十年里人们靠经验摸索出来的"哪些该留"，本来就是模型自己能学出来的，只是没人给它学的机会。

更反直觉的结果：在某些任务上，TRIM-KV *打过了全 cache 的原模型*。作者的解读是——选择性遗忘等价于一种*正则化*，把无关 token 的噪声压住了。

留下的新问题：β 是 token 生成时定的，*之后能不能修正*？如果某个 token 后来被发现意外有用，目前没机制把它的 β 拉回来。这条线下一步大概率往"可重估的保质期"方向走。

* 前沿延伸

TRIM-KV 是 2025 年 12 月的论文，截至 2026 年 5 月，公开文献里直接对它的批判性后续还不多。可见的几条延伸方向：

- *可重估的 retention*：在 TRIM-KV 框架上加一个机制，允许后续步骤抬高某个 token 的 β——把"一次性命名保质期"变成"动态续命"。
- *与量化的复用*：β 不仅决定扔/留，还能决定一个 token 用 fp16 还是 int4 存。重要的精存，次要的粗存。这跟 KVQuant / KIVI 这条量化线天然能接。
- *推理时的预算自适应*：当前 M 是固定的，能不能让模型自己根据问题难度决定要多少 cache。这跟 test-time compute scaling 那条线对得上。

* 洞见

把这条线压缩成一句话：*"什么时候决定"比"用什么决定"更要紧。*

前五代都在升级"判据本身"——位置 → 注意力分 → 学出来的 attention pattern。但*判断发生的时刻*一直没动过：永远是 token 进入 cache *之后*，靠外部观察来反推它的价值。TRIM-KV 把这个时刻往前挪到了 token *生成的那一刻*，由 token 自己带着一个保质期标签出生。

这是一个不太显眼但很关键的换轴。它对应的更深的认知转变是：从"重要性是观察出来的"到"重要性是模型本来就知道的"。如果模型在生成 "the" 的瞬间，内部隐状态里就已经知道这是一个 stop word——那为什么要绕一大圈，再花算力去事后验证？

类似的换轴在别的地方也在发生——比如 speculative decoding 把"该输出什么"从"逐 token 验证"往"块状预测"前移，比如 test-time compute scaling 把"该想多少"从"固定预算"往"模型自评"前移。一个统一的趋势是：*越来越多的判断从外挂搬进模型本体*。

下一步可能往哪走？我赌两个：
1. *可逆的保质期*——β 不再是 token 生成时定死，后续步骤能对它重估。这会把 TRIM-KV 的"提前定性"和 SeerAttention-R 的"按需 fetch"两条线缝起来。
2. *联合量化*——β 同时决定扔/留 *和* 精度。这能把 KV 压缩、KV 量化两条独立工作线统一进同一个框架。

* 启发

对做研究的启发，最直接的一条：*当一个领域所有人都在改进"用什么判据"的时候，问一句"判断发生的时刻能不能挪动"。*

这条经验普遍适用。看那些跨代际的工作——Transformer 把"循环计算"挪到了"并行计算"；speculative decoding 把"逐 token 决策"挪到了"块状预测"；TRIM-KV 把"事后筛选"挪到了"生时定性"。它们的共同点不是发明了更聪明的判据，是*换了判断发生的时间点*。

落点在能用：你下次读一个领域，先别急着评估各种方法的精度差异。先画一根时间轴，把每个方法的*判断时刻*标出来。如果大家都挤在同一个时刻，那个"还没人占的时刻"往往就是下一篇好论文的位置。
