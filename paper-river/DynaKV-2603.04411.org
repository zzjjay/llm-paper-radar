* 倒读：从一个问题开始

DynaKV 这篇论文在说一件事：*同一个序列里，不是每个 token 都值得用同样多的字节来存它的 KV cache*。BOS、句号、连接词——这些 token 在注意力里几乎不起作用，却和那些承载语义的关键 token 占着一样大的内存。给它们等量的压缩预算，是浪费。

听起来很显然。但要把这件显然的事做对，整个领域走了大概七年。倒过来看，这条线的每一步都在解一个非常具体的"前人留下的痛"：

- KV cache 太大装不下 → 砍掉一半 KV 头
- 砍多了质量塌陷 → 改成分组共享
- 共享到极限了还不够小 → 不共享，改投影到低维
- 投影需要从头预训练 → 改成事后 SVD
- 事后 SVD 一刀切丢信息 → 改成可伸缩的嵌套维度
- 嵌套也还是对每个 token 一视同仁 → 按 token 重要性动态分配

每一步都是被前一步的失败逼出来的。下面是这条线。

* 正向叙事：问题怎么长大的

** 第一刀：MQA (2019, Shazeer)

最早意识到 KV cache 是问题的人是 Shazeer。他在 Google 看到 Transformer 解码时显存占用主要不是模型权重，是缓存——每生成一个 token，前面所有 token 的 Key 和 Value 都得留着，等下一步的 attention 用。一个 32 层、32 头的模型，存几千个 token 的 cache，能吃掉好几 GB。

他的解法极端粗暴：*所有 query head 共用同一组 Key 和 Value*。原来 32 套 K/V，现在 1 套。内存瞬间砍 32 倍。

代价是质量崩。每个 head 本来应该看不同的东西——有的看语法、有的看长距离指代、有的看局部模式。强迫它们用同一个 K/V，相当于把 32 个独立专家拍成一个。短任务还能凑合，长任务里立刻看出问题。

但他指出了那个 *后面所有人都在解的问题*：KV cache 是新的瓶颈。

** 第二刀：GQA (2023, Ainslie et al., Google)

四年过去，社区接受了 KV cache 是瓶颈，但谁也不想接受 MQA 的质量损失。GQA 做的事很简单：*别全共享，也别全独立，分组共享*。32 个 query head 分 8 组，每组 4 个共用一套 K/V。压缩 4 倍，质量基本不掉。

LLaMA-2 70B、Mistral、几乎所有 2023 年后的大模型都用了 GQA。它成了事实标准。

但 GQA 的天花板很低：分组数必须是整数。你只能 1、2、4、8、16 组——不能"3.7 组"。如果你想压到 10 倍、20 倍，没办法。粒度卡死了。

** 第三刀：MLA (2024, DeepSeek-V2)

DeepSeek 的人换了思路：*不在 head 维度上做文章，改在向量维度上做*。

原来每个 K 是个 128 维向量。MLA 把它先压到比如 32 维（"latent"），用的时候再投影回 128 维。32 维就是真实存储的东西，128 维只是计算时临时算出来的。

这一招漂亮在压缩比 *连续可调*——你想压到 4 倍就用 32 维，想压到 8 倍就用 16 维。GQA 的离散诅咒被打破了。DeepSeek-V2、V3、R1 都靠这个把 KV cache 压到极致。

代价是：MLA 是一种 *新架构*。你必须从头预训练。已经训好的 LLaMA、Qwen、Mistral——它们的 K/V 投影矩阵不是这个结构。想用 MLA？要么重训（几十亿 token 起步），要么用 MHA2MLA 这种转换方法（成本依然像"半个预训练"）。

对学术界和小团队，这就是一道高墙。

** 第四刀：Palu (2024)

Palu 把问题反过来问：*我能不能不重训，直接对已训好的模型做低秩压缩？*

办法很经典——SVD。把 K/V 的投影矩阵 W 分解成两个小矩阵相乘，W ≈ U·V，中间那个秩 r 就是压缩程度。运行时不存原始的 K/V，存它们在低秩空间里的表示，要用时再乘回去。

这条路完全 training-free。任何 LLaMA、Qwen 都能套用。

但 Palu 的死穴是 *高压缩比下崩盘*。论文里的数据：Llama-3-8B 在 C4 上，保留 20% rank 的时候 perplexity 飙到 113.90——基本是胡言乱语。原因不难想：SVD 是从矩阵奇异值里砍掉最小的那些。当你砍掉 80%，再小的奇异值汇总起来也很要紧，重建误差爆炸。

** 第五刀：MatryoshkaKV (2025)

MatryoshkaKV 注意到一个工程现实：*同一个部署里，不同请求需要的压缩比不同*。短对话可以大方一点，长文档必须狠压。但 Palu 每次想换压缩比都要重新做一次 SVD，不现实。

它的解法借鉴俄罗斯套娃——*把 K/V 表示成嵌套结构，重要信息塞在前几个维度*。要 50% 压缩就取前一半维度，要 75% 压缩就取前四分之一。一份 cache，多种切法。

这是一次工程上的胜利。但它在 *一个根本假设上和 Palu、MLA 一脉相承*：序列里每个 token 用同样的 rank。不管这个 token 是 "the" 还是核心实体名词，待遇一致。

** 第六刀：DynaKV (本文, 2026)

DynaKV 的洞察很简单：*这个一致是错的*。

注意力图早就告诉我们了——大量 token 在后续推理里几乎不被看到。BOS 是个 attention sink，特殊但不需要高保真度。停用词、标点的 KV 信息含量很低。真正承载语义的可能就 20-30% 的 token。

如果你的压缩预算是平均 6%，那把所有 token 都压到 6% 是傻事。让琐碎 token 压到 1%，把省下的预算给关键 token 提到 15%——总平均还是 6%，质量却完全不同。

技术上，DynaKV 借用了句法分析里的 *可微门控* 思想（PRPN、ON-LSTM 那条线）。给每个 token 一个连续的"重要性"打分，按打分动态分配该 token 要保留多少 rank。整个过程可微，可以端到端微调。

结果数据：和 SnapKV（一个独立的"丢掉不重要 token"的方法）组合，能在只留 *6% KV cache* 的情况下保住 LongBench 上 94% 的性能。这是之前任何低秩方法都做不到的。

DynaKV 也证实了 StreamingLLM 当年的观察——它学出来的分配里，BOS 始终拿着大约 0.75 的高保留率。

往下看这张表，你能感受到这条线不是单维度的演化，而是 *三个维度并行推进*：

1. *压缩对象*：从粗粒度的 head，到细粒度的向量维度——做得越来越细
2. *压缩获取方式*：从预训练时定死，到训练后零成本——成本越来越低
3. *压缩粒度*：从全局一刀切，到 token 粒度自适应——预算分配越来越聪明

DynaKV 是第三个维度上的第一次突破。前两个维度已经走了多年，第三个一直没人动——大家默认了"序列里每个 token 平等对待"这条假设。DynaKV 把这条假设拆了。

* 内核与下一步

读完这条线，你看到的其实不是 KV 压缩技术的进步，是一个 *认知转变*：

  早期：*KV cache 是个均质的内存对象*，问题是"如何整体缩小它"
  现在：*KV cache 是个异质的信息载体*，问题是"如何按信息密度分配预算"

这和 LLM 推理领域更广的趋势一致——MoE 是按 token 分配计算，Speculative Decoding 是按 token 分配验证开销，CoM 是按 token 分配计算量。token 不再是被均匀处理的单位，而是 *被差异化对待的个体*。DynaKV 把这个范式带到了内存领域。

下一步最可能往哪走？三个方向已经露头：

1. *联合优化*：DynaKV 解决了"维度"维度的自适应。SnapKV 解决了"序列"维度的自适应（丢哪些 token）。下一步会有人把这两个 join，让维度和序列上的分配一起学
2. *跨层自适应*：现在 DynaKV 是层内 token 级。前期已有 CommonKV 这种跨层共享 basis 的工作。token 级 × 层级的联合分配是个空白
3. *推理时再分配*：DynaKV 的分配在微调阶段学好。但同一个 token 在不同上下文里重要性不同——动态在推理时根据当前 attention 模式重新分配 rank，是更激进的一步

最后一个 takeaway：*"一刀切"几乎总是一个领域的临时解，不是最终解*。哪里有"对所有 X 用同样的 Y"，哪里就有 DynaKV 式的工作等着出现。
