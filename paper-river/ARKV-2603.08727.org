* 问题之河

一句话：一个 LLM 在跟你聊到第十万个字的时候，它脑子里"刚才说过什么"那块小本本（KV cache），会比模型本身还大。怎么让它在装不下的时候，*忘掉不重要的、压缩半重要的、留住关键的*——而不是粗暴地一刀切？

这个问题为什么难，卡了三四年都没解干净？

- *未来不可知*：要决定"这个 token 还重不重要"，你得知道后面会问什么。可后面还没生成，根本不知道。
- *一刀切代价巨大*：全压成 2-bit，数学题立马算错；全保留，64GB 显卡装不下 70B 模型的长对话。
- *每层都不一样*：Transformer 浅层在做语法，深层在做推理，对"丢一个 token"的容忍度天差地别——但绝大多数方法把所有层一视同仁。
- *在线 vs 离线的两难*：离线分析每层敏感度很准，但部署时模型一变就废；在线决策灵活，但每步都算一遍开销大到吃掉收益。

这条河上每一篇论文，都是在这四个夹角里试着挪一小步。

* 演化叙事

下面这一串论文不是孤立成果——每一篇都是在解前一篇留下的具体痛点。沿着这条线走一遍，你会看到一个朴素的直觉（"重要的 token 留住，不重要的丢"）是怎么被一步步逼到"重要性是分层的、是动态的、是和量化耦合的"这种精细认知的。

** 2023 | H2O：发现 KV cache 里大多数 token 是"陪跑的"

*** 它看到了什么问题

在 H2O 之前，KV cache 怎么管基本没人当回事——反正每个 token 都要留着，因为"万一后面注意力要查到呢？"。结果 LLaMA-70B 跑长上下文，KV cache 一个人吃掉几十 GB 显存，比模型权重还重。

*** 它的解法

H2O 做了一个非常朴素但震撼人的观察：把每个 token 历史上被注意力看了多少次累加起来，会发现这个分布极度长尾——*20% 的 token 吸走了 80% 以上的注意力*。剩下那 80% 的 token，留着也没人理，可以扔。这 20% 就叫 heavy hitter（重击者）。

具体操作：每生成一步，更新所有历史 token 的累积注意力分数，留前 K 名，剩下的从 cache 里删除。

*** 为什么这招管用（以及哪里埋了新坑）

管用是因为：注意力机制本身就是稀疏的——大多数 query 只盯着少数几个 key 看。所以丢掉那些没人看的 token，模型几乎察觉不到。

但是它埋下了三个坑，全是后面这条河要还的债：

1. 边生成边算累积分数，*开销不便宜*（SnapKV 接手）
2. *每层都用同一个 K*，没考虑深层和浅层敏感度差几个数量级（PyramidKV 接手）
3. 一旦把 token 删了就 *永远找不回来*，未来如果突然需要它，模型就瞎了（Q-Hitter 和 ARKV 用"量化"作为中间档来缓解）

** 2024 春 | SnapKV：能不能不在生成时算，提前算？

*** 它看到了什么问题

H2O 在 decoding 的每一步都要更新一次累积注意力分数。当 cache 已经很大时，这一步本身就成了瓶颈。SnapKV 注意到一个有趣现象：*你 prompt 里最后那几十个 token 怎么"看"前面的 token，就基本预示了整个 decoding 阶段会怎么看*。

*** 它的解法

那干脆，prompt 进来后做一次性分析——拿末尾一个小窗口（"observation window"）当探针，看它对前文的注意力分布，挑出 top-K，剩下的全删。decoding 阶段就不再动了。

*** 为什么这招管用（以及哪里埋了新坑）

管用是因为 LLM 的注意力模式在同一段对话里相当稳定，prompt 末尾的"视野"确实能预测后续。

新坑：*预测的代价是僵化*。一旦 decoding 中话题转了，新生成的 token 可能需要去查一个 prompt 阶段被判"不重要"的旧 token，但那 token 已经没了。SnapKV 的核心局限是 "一次决定，终生执行"。

ARKV 后来直接吸收了这个 observation window 的思路（论文里 W=32），但把"一次决定"换成了"持续决定"。

** 2024 春 | PyramidKV：每层的胃口不一样，凭什么用同一份预算？

*** 它看到了什么问题

H2O 和 SnapKV 都把所有 transformer 层看成"同一种东西"——预算 K 拍下去，每层都留 K 个。PyramidKV 实测发现，*深层的注意力高度集中，浅层却分散*。换句话说，深层只留几个关键 token 就够了，浅层得留多一些。一刀切要么浪费要么伤精度。

*** 它的解法

按层分配不同预算，整体呈金字塔形状——底层（浅层）预算大，顶层（深层）预算小。

*** 为什么这招管用（以及哪里埋了新坑）

管用是因为 transformer 不同层确实在做不同抽象级别的事，"注意力集中度"是个真实信号。

新坑：金字塔形状是 *人为先验*——基于"深层比浅层集中"的总体观察，但具体到某个模型、某个任务，曲线长什么样？硬塞个金字塔进去未必最优。ARKV 后来就用每层实测的统计量（熵/方差/峰度）替代了这个人为形状。

** 2023-2024 | KIVI / KVQuant：另一条路——干脆不丢，全压

*** 它们看到了什么问题

H2O 系的"丢 token"虽然省内存，但有个根本风险：*被丢的 token 再也回不来*。如果生成到一半模型突然想引用一个被判死刑的 token，那段上下文就废了。KIVI 和 KVQuant 反过来想：所有 token 都留着，但用更少的比特存。

*** 它们的解法

- KIVI：2-bit 量化，对 K 和 V 分别设计不同方案，无需重训
- KVQuant：3-bit + 学习到的 scale + 分组量化，号称能撑 10M 上下文

*** 为什么这招管用（以及哪里埋了新坑）

管用是因为信息保留——每个 token 还在，只是模糊了一点。短任务上几乎无损。

但 ARKV 的实验把这条路打得很狠：在 512 token 预算下做 GSM8K 数学题，*纯量化的 baseline 准确率 0.010*——基本就是瞎猜。原因是数学推理对数值精度极度敏感，每步的小误差累乘起来就是灾难。另外，所有 token 都留着意味着 *注意力计算量还是线性增长*，量化只省了存储没省算力。

这条线和 eviction 那条线必须合并——这就是 Q-Hitter 出场的时候。

** 2024 | Q-Hitter：第一次把"丢"和"压"合体——但是僵的

*** 它看到了什么问题

最朴素的合体方案是"先用 H2O 选出重要 token，再把它们量化"。Q-Hitter 实测发现这是个陷阱——*量化会污染注意力分数本身*，导致 heavy hitter 选错；同时，*被 H2O 选出来的重要 token，不一定耐量化*（它们可能恰好是 outlier-heavy 的）。

简而言之：把两个独立优化串起来，1+1 < 1。

*** 它的解法

设计一个联合分数 S = 注意力重要性 × 量化友好度。同时高的才留下，留下后量化到 4-bit。重要但不耐量化的、耐量化但不重要的，都不要。

*** 为什么这招管用（以及哪里埋了新坑）

管用是因为它真正承认了"重要性"和"耐量化"是两个独立维度，必须联合优化。

新坑（也是 ARKV 直接攻击的点）：

1. *量化友好度是离线算的*——上线前给每层算一个静态值，跑起来不变。这意味着换个任务、换段输入，这张表可能就过时了
2. 仍然是 *二元决策*：要么留(并量化)要么丢，*没有"重要的留原始精度、次重要的留量化、不重要的丢"这种三档结构*
3. 维护"敏感度表"本身耦合了模型结构，复杂度上去了

** 2026 Feb | ARKV：三状态 + 在线统计——这条线目前最精细的形态

*** 它看到了什么问题

ARKV 在 introduction 里点名了两个核心 challenge：

1. *Token 的重要性是 layer-sensitive 的*——同一个 token 在浅层是噪声，在深层可能是关键。所有前人方法都假设跨层重要性一致
2. *Token 的量化容忍度也是异质的*——把不该量化的 token 量化掉，比量化错一个不重要 token 损失大十倍

PyramidKV 用先验金字塔解了一半（层异质），Q-Hitter 用离线表解了一半（量化异质），但都不够动态、不够细粒度。

*** 它的解法

把 Q-Hitter 的二元决策升级成 *三状态*：每个 token 要么 Original（保留全精度 bf16），要么 Quantization（压成 fp8），要么 Eviction（丢）。

逐层预算怎么算？这是 ARKV 最巧的地方。它在 prefill 阶段对每层注意力分布算三个统计量：

- *熵 H*：注意力分散不分散？（分散说明每个 token 都重要，要多留）
- *方差 V*：tokens 之间差异大不大？
- *峰度 K*：分布尾巴是不是又长又胖？（outlier 多）

合成一个 OQ 分数 q_ℓ = H^(1/τ₁) · V^(1/τ₂) · K^(1/τ₃)，归一化后得到每层的 OQ 比例 ρ_ℓ。ρ_ℓ 高 → 这层重要 → 多分预算给"原始精度"；ρ_ℓ 低 → 多分预算给"量化"，少量原始。

decoding 阶段，用一个轻量的 heavy-hitter 分数 S_k = μ_k + γ·σ_k² 给 token 排名，按层预算切成三档。

*** 为什么这招管用（以及哪里埋了新坑）

为什么管用：

- *三档而非二档*——给了模型一个"我不确定要不要丢"的中间选项，配合 fp8 用很少代价保留次要信息
- *熵/方差/峰度* 这三个矩本质上是在描述"这层注意力的浓缩程度"，比金字塔先验更贴合实际数据
- *14.4% 量化率*——这个数字很说明问题：ARKV 主要靠驱逐省内存（~78%），量化只占 ~14%，剩下是原始精度。量化是"次重要 token 的保险"，而不是省内存的主力

实测：4× 内存压缩、~97% 的 LongBench 精度、86% 的 throughput；GSM8K 上准确率 0.789（base 0.791），而 pure quant 是 0.010。

新坑（这一篇还没解的）：

1. 熵/方差/峰度三个温度参数 τ₁=7.774, τ₂=5.407, τ₃=5.528 是 Optuna 调出来的——*换模型可能要重调*
2. 只支持 *fp8 一种量化精度*，没有 2/4/8-bit 的灵活档位
3. 决策粒度是 *token-level*，没有进一步到 channel-level 或 head-level

这三个坑，正是 2026 年其他几篇接力论文各自在啃的方向。

* 前沿延伸

ARKV 是 2026 年 2 月放出来的。在它前后几个月，至少有四篇论文从不同角度继续推同一个问题：

** 2026 Feb | KVTuner (arXiv 2502.04420)

把"层敏感度估计"做成 *离线 Pareto 搜索*——牺牲在线适应性，换更精确的层间精度配比。和 ARKV 形成对照：ARKV 是"轻量在线统计"，KVTuner 是"重量离线最优"。两条路各有市场。

** 2026 Feb | "Don't Waste Bits!" (arXiv 2604.04722)

把 ARKV 的"两档精度（bf16/fp8）"扩展到 *{2-bit, 4-bit, 8-bit, FP16} 四档*，用一个学到的小控制器（基于 token frequency / quality score / attention variance / entropy）选 bit 数。本质是受 Huffman 编码启发——重要 token 给高精度，不重要给低精度，比例由数据决定。

直接继承了 ARKV 的 attention 统计量思路（entropy + variance），但把"三状态"细化成了"连续比特分配"。

** 2026 Dec | MixKVQ (arXiv 2512.19206)

把决策粒度从 *token-level 下沉到 channel-level*。观察是 key cache 里有 outlier channel——不是整个 token 难量化，而是 token 里某些维度难量化。MixKVQ 还把 query 也拉进来：哪些 channel 对当前 query 重要，给它们高精度。

这是把 ARKV 的"token 重要性"思路延伸到了一个新维度——通道。

** 2026 | PM-KVQ (OpenReview)

针对 *Long-CoT 推理模型*（DeepSeek-R1 一类）的特化版本：block-wise（不是 layer-wise）精度分配，加上 positional interpolation 校准。在 7B-70B 长 CoT LLM 上比 SOTA 高 8%。

可以理解为"PyramidKV + ARKV"在推理模型场景的特化变种。

* 洞见

读完这条河，三个真正在发生的转变浮出水面——它们比"又一篇 KV cache 论文"重要得多：

** 一、决策粒度的下沉

#+begin_example
  全局 K -->  层级 K -->  token 级 -->  channel 级
  (H2O)    (PyramidKV)    (ARKV)      (MixKVQ)
#+end_example

每往下沉一层，建模的异质性就多一个维度，节省也跟着上一台阶。但相应地，*在线决策的开销也在涨*。ARKV 用三个矩（熵/方差/峰度）算 OQ 比例，其实是在"足够精细"和"足够便宜"之间找了个甜点。

** 二、"丢" 和 "压" 的边界在消失

H2O 时代，eviction 和 quantization 是两条独立支线，做各自的事。Q-Hitter 第一次承认它们要联合优化。到 ARKV，"丢/压/留"成了一个 *连续光谱上的三个采样点*。再到"Don't Waste Bits!"，干脆变成连续 bit 分配。

*下一步的极限形态：每个 token、每个 channel、每个时间步，独立决定一个 0-16 之间的精度数*。这个数会被一个轻量控制器实时算出来。

** 三、注意力统计量正在成为"模型自我感知"的接口

ARKV 真正深刻的地方在于：它没有训练任何东西、没有改任何架构，*仅靠模型自己产生的注意力矩阵的三个统计量*，就推断出了每层的敏感度。这是一种"模型自带的健康监测仪"。

这个思路远不止 KV cache 能用——*任何需要"哪些参数/激活/层更重要"的决策*（剪枝、推理路由、专家选择），都可以从注意力的高阶矩里读出信号，而不依赖额外的校准数据或训练阶段。

下一步最可能的方向：

1. 把 ARKV 的"统计量 → 精度分配"做成 *可学习模块*（已有：Don't Waste Bits!）
2. 把"在线决策"和"硬件 kernel"打通——目前 ARKV 的 throughput 是 86%，意味着决策开销还是吃了 14%
3. 应用到 MoE 模型——expert 选择本质上也是 "哪些参数当前重要" 的问题，和 KV cache 同构

* 启发

这条线对"怎么找问题"有几个具体启发：

1. *复合问题不要二阶段串联做*。Q-Hitter 之前所有人都在"先 eviction 再 quantization"，因为这俩问题各自有成熟解法。但分开优化的结果是 1+1<1。真正的进步来自承认它们 *相互污染*，必须联合建模。
2. *中间状态往往是金矿*。ARKV 最大的设计创新——从"留/丢"二元到"留/压/丢"三元——表面看很朴素，但二元决策被研究三年也没人加这一档。每当你看到一个领域陷入二选一争论时，问一句"有没有 1.5 选项？"。
3. *先验形状（金字塔/线性/指数）总会被数据驱动替代*。PyramidKV 假设深层注意力更集中所以预算少——大致正确。ARKV 直接测出每层真实统计量——更准确。任何"按 X 形状分配"的方案，都是等一个"按数据测量分配"的方案来替代。
4. *轻量统计量胜过重量学习模块*。ARKV 没有训练任何 retaining head（对比 Locret），也没有维护学习到的 scale（对比 KVQuant）——它只是算了三个标量统计量。在线决策场景下，"算得快"比"算得准"重要，因为决策开销直接吃掉收益。
5. *实验最有说服力的不是平均数，是断崖式对比*。ARKV 在 GSM8K 512 budget 上的 0.789 vs 纯 quant 的 0.010——这一个数字比所有 LongBench 上的 ~97% 都更有冲击力。找到"前人方法会断崖式失败"的场景，是论文立论最有效的方式。

这意味着你可以——把上面五条任意一条，套到你正在做的领域里：你的问题是不是在被"二阶段串联"地解决？你的方案是不是默认了某种先验形状？你的实验有没有展示一个"竞争对手会断崖失败"的场景？任一答案为否，都是一个潜在的下一篇论文。
