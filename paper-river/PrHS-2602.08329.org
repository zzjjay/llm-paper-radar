* 一句话
长上下文推理里"挑哪些 KV 来算注意力"这件事，被 PrHS 第一次从"先算分数再挑"翻成"不算分数就能挑，并且能给出可证明的精度下界"。

* 演化叙事：问题怎么长出来的

** 第 0 篇：原罪是怎么留下的（Vaswani 2017）

Transformer 的注意力机制说穿了就一句话：每生成一个新 token，都要去看前面所有 token 一眼，看看谁跟我相关。"看一眼"=算一次点积。前面有 N 个 token，就要算 N 次。

序列长度从 2K 涨到 128K 之后，这件事开始崩。不是模型不行，是这个 O(N) 的"每个 token 都看一眼"扛不住——KV cache（存下来的"前面所有 token 的 key 和 value"）能撑爆显存，注意力本身的算力也线性涨。

整条线的源头问题就在这里：*我必须看所有人吗？*

** 第 1 篇：先验稀疏（Sparse Transformer / Longformer, 2019-2020）

第一批人的回答很朴素：不必。我先验地告诉你哪些位置之间需要注意——比如"只看最近 512 个"+"每隔 64 个看一个"。这是 *固定稀疏模式*。

问题立刻浮现：哪个 token 重要是 *跟内容有关* 的，不是跟位置有关的。一篇文章里关键人物可能出现在第 3 段，固定窗口会把它整个漏掉。固定模式对"按规律分布"的数据还行，对真实长文档差得远。

下一步就被这个问题逼出来：*稀疏模式应该跟着输入走，不能写死。*

** 第 2 篇：动态稀疏的第一招——按历史分数挑（H2O, 2023）

H2O 给出了一个看起来很自然的答案：既然每次注意力都会算出分数，那分数高的 token 就是重要的（"heavy hitter"），把它们留下，其他的丢掉。

这一招把稀疏从"位置驱动"变成"分数驱动"。第一次让稀疏选择跟着语义走。

但它埋了一颗雷：*分数是历史的*。某个 token 在前几步分数不高，不代表它后面不会变重要——你已经把它从 cache 里赶出去了，再也回不来。这种"我过去看不见你，未来也不要见你"的逻辑在多跳推理里特别脆。

** 第 3 篇：抓住一个反常现象（StreamingLLM, 2023）

StreamingLLM 不是来批 H2O 的，但它发现了一个奇怪的事实：开头几个 token（"attention sink"）几乎在所有 head 里都拿走一大块注意力——哪怕它们语义上没什么用。

这件事的副作用是：H2O 那种"按分数挑"的方法会把开头几个 sink token 永远留着，真正占用 KV 预算。它把问题暴露了——*你以为分数告诉你谁重要，其实分数告诉你的是模型自己的小习惯*。

StreamingLLM 的解法是 sink + 滑窗的极简组合，效率漂亮，但召回率天花板很低（长程依赖直接没了）。

这里出现了关键认识：*posterior 分数本身就是不可信的信号*。

** 第 4 篇：把粒度从 token 变成 page（Quest, 2024）

Quest 想要既保留长程依赖（不像 StreamingLLM），又比 H2O 更省（不要每步重算所有分数）。它的招数：把 KV 切成 page，每个 page 用一个上界估计代表"这页里最大可能的分数"，按 page 整块挑。

这是一个折中：分数还是要算，但是算的是"近似上界"而不是真分数，按页粗粒度挑。

它的进步是 query-aware（每次查询都重新评估），它的代价是引入了一个新的近似——*上界估计本身是有偏的*。Quest 算得快，但你不能告诉用户"我保证丢掉的部分小于 ε"。

** 第 5 篇：复用与共享，但 posterior 仍是 posterior（HShare, 2025）

HShare 在 Quest 这条路上走得更远：它发现不同 head、不同 layer 选出来的"重要 KV"高度重叠，所以可以 *跨头跨层共享* 一次选择，把选择本身的开销摊薄。

这是工程上漂亮的优化，但概念上没改变什么——它仍然是 *posterior selector*：先要看到（或近似看到）注意力分数，再决定挑谁。

这条 H2O → Quest → HShare 的演化线，越走越精细，但都困在一个共同的天花板下：

#+begin_src
你想知道哪些 KV 重要 -> 你要先打分 -> 打分本身就要看 KV
                                          ^
                                          |
                                  自指的死循环
#+end_src

谁能跳出这个循环，谁就能进入下一个时代。

** 第 6 篇：PrHS 的翻转——根本就不打分（2026）

PrHS 的核心动作非常违反直觉：*在算任何分数之前就把 KV 选好*。它叫 pre-hoc sparsity（事前稀疏）——区别于上面所有人的 post-hoc（事后/近似事后）。

凭什么不打分就能挑？PrHS 用 marginal-to-mutual-information 的关系做了一个推导：你丢掉的那部分 KV，其"信息损失上界"只跟一个量有关——*dropped mass δ*。也就是说，只要你能保证"我丢掉的部分总质量不超过 δ"，模型行为偏差就被一个跟 δ 相关的上界框住。

这就把整件事从"启发式挑选"变成了"带可证明保证的预算分配"：

| 维度    | 选择器       | 干的事                       |
|---------+--------------+------------------------------|
| 时间轴  | TimeSelector | 按时间分布预先丢稀少时段     |
| 深度轴  | DepthSelector| 按层深决定该层能保留多少     |
| 层轴    | LayerSelector| 跨层做协调，避免重复浪费预算 |

三个选择器正交，可以叠加。结果：
- GSM8K / CoQA 检索开销减 *90%+*
- 检索稀疏度比 HShare 高 *3×*，精度相当或更好
- LongBench 平均掉点 *<1%*
- 注意力算子 *9.9× 加速*，吞吐 2.8×

* 真正在发生的认知转变

表面看是"KV 选得越来越准、越来越省"。

底层在发生两件事：

1. *从启发式到可证保证*。整个 KV-selection 子领域之前像调参炼丹——什么策略好用是跑出来的，没人能事先告诉你"我会牺牲多少精度"。PrHS 引入 δ 之后，第一次把这个领域拽到"算法 + 数学保证"的轨道上。

2. *从 posterior 到 prior*。所有先前方法都假设"重要性需要被观测才能知道"。PrHS 证明：在一定条件下，重要性的 *上界* 不需要观测——你可以用分布的结构性质（稀疏程度、分布形状）来事先预算，而不必去看每个具体 token 的分数。这是一个挺哲学的转向。

* 下一步会往哪走

- *δ 的自适应*：现在 δ 是用户给的，下一步必然出现"模型自己学怎么挑 δ"——把"愿意牺牲多少精度"也变成一个可学习量。
- *训练阶段的 pre-hoc*：目前只用在推理，但同样的 marginal-MI 分析没理由不能搬到训练时的 attention dropout / sparse training 里。
- *跟硬件的合谋*：pre-hoc 选择意味着选择决策可以提前到 KV 进 GPU 之前——这给 *KV 放在 CPU/SSD、按需 prefetch* 这条路开了一道全新的门，可能比所有 GPU 内部的稀疏招数都更省。

* takeaway

整条 KV 稀疏化的进化史，本质上是一个"自指困境"的破解史：你想选 KV，就要先评分；你要评分，就要先看 KV。每一代人在这个循环里把"评分"做得更便宜（H2O→Quest→HShare），直到 PrHS 干脆退出循环——不评分，用分布性质直接给出"丢这些不会差太多"的可证上界。

这个动作的真正份量不是 9.9× 加速，是把一个炼丹领域第一次接到了"算法 + 证明"的轨道上。
