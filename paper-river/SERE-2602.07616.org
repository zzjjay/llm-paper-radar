* 问题之河

*一个号称「只激活一小部分参数」的大模型，为什么一旦真的拿去服务用户，反而变得又慢又费显存？*

MoE（Mixture-of-Experts）的卖点很简单：一个模型有几百个「专家」（小 FFN），每个 token 只走其中两三个。听起来天然省算力——参数量翻十倍，算力翻一倍。

但这只是理论。一旦你把模型部署到 vLLM、给几十个用户同时回复，事情就翻车了。一个 batch 里有 32 个用户的 token，每个 token 各自挑两三个专家——挑来挑去，最后这个 batch 几乎把所有专家都「点亮」了。本来想躲开的「全员上工」，被 batch 硬生生召回了。

而且 decoding 阶段是 memory-bound：瓶颈不是算力，是「把专家权重从 HBM 搬到 SRAM」的带宽。专家激活越多，搬运越多，越慢。

这条线，从 2020 年的 GShard 到 2026 年的 SERE，一代代人都在围着同一个问题打转：*怎么让 MoE 真的稀疏起来？*

每一代的答案都不一样。早期的人在改路由本身；后来的人在改专家结构；再后来的人在压缩专家权重；到了 SERE，他们换了一个角度——*别动专家，也别动路由，去动 batch 内部的冗余*。

* 正向叙事

** 2020 · GShard：把「全员上工」改成「点名上工」

故事的起点是一个工程问题。

2020 年之前，要让 transformer 变强，办法只有一个：堆参数，堆算力。每个 token 进 FFN 都要把整个矩阵走一遍——参数翻一倍，算力翻一倍，电费翻一倍。

Google 的 GShard 团队想了个绝招：与其让每个 token 都走完整的 FFN，不如把 FFN 切成 N 份（叫「专家」），让一个小路由器看一眼 token，挑出 K 个最相关的专家（一般 K=2），只走这两个。

参数还是 N 倍那么多（都存在显存里待命），但每个 token 只用 K/N 的算力。这就是 *稀疏激活* 的起点。

留下了什么问题？两个。

第一，*K 是写死的*——不管 token 是「the」还是「薛定谔方程」，都走 2 个专家。聪明 token 浪费算力，笨 token 不够用。

第二，*专家之间知识高度重叠*。N 个专家随机初始化、独立训练，没人告诉它们「你专攻语法、你专攻代码」。结果就是每个专家都什么都会一点，专业化稀薄。

后面的故事，基本就是在轮流修这两个问题。

** 2024.01 · DeepSeekMoE：让专家真的「专」起来

DeepSeek 团队第一个出手，修的是「专家不专」。

他们的观察很直接：如果每个专家都会一点点，那 top-2 路由其实在挑「两个差不多的」，相当于没挑。要让 top-2 真的有意义，专家之间必须有明显分工。

他们的做法分两步：

*第一步：把专家切碎。* 原来 N=8 个胖专家，每个 hidden_dim 是 4096。现在切成 mN=64 个瘦专家，每个 hidden_dim 是 512，但每次激活更多个（mK=8）。参数总量、算力总量都不变，但「挑选空间」大了 8 倍——更精细的组合让每个专家可以专注更窄的领域。

*第二步：抽出共享专家。* 既然「通用知识」每个专家都会一点是浪费，干脆抽出 1-2 个专家，让它们 *永远* 被激活，承担所有 token 都要的通用能力。剩下的 routed 专家就可以彻底专精了。

效果立竿见影：DeepSeekMoE 16B 用 40% 的算力达到 DeepSeek-7B（dense）的水平。

但 DeepSeekMoE 仍然没动「K 是固定的」这一刀。每个 token 激活的专家数还是写死的。

** 2024.03 · Top-P / Dynamic MoE：让 token 自己决定要几个专家

北大 Huang 等人盯上了第二个问题。

他们的逻辑特别朴素：在 NLP 里，有些 token 就是简单（语法填充词），有些 token 就是难（推理步骤的关键词）。给每个 token 派同样多的专家，要么浪费要么不够。

他们的解法叫 *top-P 路由*：路由器先按 softmax 排序专家分数，然后从最高的开始累加，*直到累积概率超过阈值 P* 才停。

- 如果最高分专家就有 0.9 概率：好，只用它一个。
- 如果分数稀稀拉拉、谁都不到 0.3：那就一直加，加到累积超过 P 为止。

结果：复杂推理任务（BBH）平均激活更多专家，简单任务激活更少。平均激活参数低于 90% top-2 的水平，效果反而更好。

留下的问题：这套路完全是 *单 token 视角* 的。每个 token 自己决定要几个专家——好极了。但 batch 维度呢？32 个 token 各自挑各自的，最后挑出的专家集合是 *并集*，几乎覆盖了所有专家。从 batch 级看，激活率根本没下来。

** 2024.08 · AdapMoE：边缘设备的另一种焦虑

北大另一组（Zhong et al.）走的是另一条路——他们关心的是边缘部署。

在数据中心，所有专家都常驻 HBM，加载几乎免费。但在 4090、A6000 这种消费/工作站级 GPU 上，根本装不下所有专家。专家平时存在 CPU 内存里，要用的时候 *按需加载到 GPU*——这一加载就是几十毫秒，硬生生卡住推理。

AdapMoE 的洞察：在边缘场景，决定速度的不是激活几个专家，而是 *能不能提前把要用的专家搬过来*，以及 *缓存里命中率高不高*。

他们做了三件事：
1. *敏感度分析* 决定每层激活几个专家——浅层多用、深层少用（因为浅层错一点会传递放大）。
2. *Speculative prefetch*：用本层的中间结果去预测下一层会路由到哪些专家，提前搬。
3. *动态规划分配每层缓存大小*：哪层专家变化快就分大缓存。

但 AdapMoE 还是没看到 batch 内部的冗余。它解决的是「加载延迟」，不是「激活率本身」。

** 2025.03 · ResMoE：换个思路，直接动权重

到这里，路由侧的招数差不多被用完了：固定 K、动态 K、按敏感度调 K……每个 token 视角的优化空间见底了。

UIUC 的 Ai 等人换了个完全不同的方向：*别动路由，动专家权重本身*。

他们的观察：训练出来的 N 个专家，权重之间高度相似。如果你画一下 N 个 expert 的权重分布，会发现大家围着一个「中心」打转，每个专家只是中心 + 一个小残差。

那能不能用一个「公共专家」（barycenter）+ N 个小残差，把整个 MoE 层压缩？

他们用 Wasserstein barycenter 做这件事，得到一个公共 FFN + N 个低秩残差。参数能减少 75%，效果只掉一点点。

但这是 *静态* 的——一次压缩，写死在权重里，不可逆。而且推理时该激活的专家还是要激活，省的只是存储和加载带宽，不省运行时算力。

更深的问题是：这种「事前压缩」的视角，*把所有 batch、所有输入一视同仁*。它假设专家间的冗余是固定的。但实际推理时，冗余是 *动态的*——同一个 batch 里哪些专家彼此相似、哪些不相似，会随输入变化。

** 2026.02 · SERE：所有人都漏掉了 batch 这个维度

终于轮到 SERE 出场。

Wu 等人的视角转换很有意思——他们不是在前人基础上做"+1"，而是指出 *前面所有人都漏了同一个观察点*：

- GShard、DeepSeekMoE：在专家结构上做文章
- Top-P、AdapMoE：在每 token 的激活数上做文章
- ResMoE：在静态权重上做文章

*没有一个人在「batch 内部、相似专家被重复调用」这件事上做文章。*

SERE 的观察非常具体：当 vLLM 把 32 个用户的 32 个 token 打成一个 batch、走到某个 MoE 层时，路由结果是 32 × 2 = 64 次专家调用。这 64 次调用看似稀疏，但 *因为很多 token 路由到了彼此非常相似的专家*，整个 batch 几乎把所有专家都激活了一遍。

如果我们能识别：
- 这个 batch 里哪些专家彼此相似
- 把「次要相似专家」的 token 改派到「主要相似专家」

那么一次专家搬运就能服务更多 token，激活的专家总数下降，memory-bound 的瓶颈被缓解。

这就是 *Similarity-based Expert Re-routing*：

1. *离线*：用 CKA 相似度（Kornblith 2019）算出专家两两之间的相似矩阵。
2. *在线*：每个 batch 进来后，看哪些专家被调用了，把「相似度高的次要专家」的 token 重新派给「主要专家」。
3. *合并执行*：用一个自定义 CUDA kernel 实现这个 re-routing，接入 vLLM 只需一行代码。

效果：在复杂推理 benchmark 上达到 *2.0× 加速*，精度损失极小。

为什么不掉点？因为相似度高的专家，本来对同一个 token 的输出就接近——派给主专家 vs 派给次专家，差异在噪声范围内。

* 这条河真正在发生什么

表面看，这是「MoE 推理加速」的迭代——一代比一代快。

但底层在发生的认知转变是这样的：

*第一阶段（GShard → DeepSeekMoE）：架构视角*。问的是「专家应该长什么样？」答案是结构上的——切细、抽公共。

*第二阶段（Top-P → AdapMoE）：单 token 视角*。问的是「这一个 token 真的需要那么多专家吗？」答案是激活数上的自适应。

*第三阶段（ResMoE）：权重视角*。问的是「专家权重之间能压缩吗？」答案是静态的、一次性的。

*第四阶段（SERE）：batch 视角*。问的是「同一个 batch 里，这些独立的路由决策叠加起来，到底有没有冗余？」

这是从「单点优化」到「群体调度」的转变。前三阶段都假设「每个 token 是独立的」，所以优化也是 token 级或权重级的。SERE 跳出来说：*token 不是独立的——它们要一起被服务*。在 serving 场景下，batch 不是一个无奈的工程现实，而是一个 *可以利用的统计结构*。

这个视角转换很像 transformer 历史上 KV-cache → PagedAttention 那一次。PagedAttention 之前，每个请求的 KV 是独立的；之后，KV 是可以共享、可以分页、可以跨请求复用的。SERE 在做的是同一件事，只不过对象换成了 expert activations。

* 下一步最可能往哪走

1. *KV-cache 与 expert 联合调度*。既然 batch 视角已经被打开了，下一步是把 attention 侧（KV）和 FFN 侧（expert）的 batch 级冗余一起调度。
2. *在线学习的相似度矩阵*。SERE 目前用离线 CKA 算相似度——但相似度其实是输入分布的函数，应该在线更新。
3. *跨 batch 的 expert 复用*。如果两个相邻 batch 调用的专家集合高度重合，第二个 batch 是不是可以跳过部分加载？
4. *和 quantization / pruning 的复合*。SERE 不动权重，正好和 ResMoE 那种静态压缩正交，可以叠加。
