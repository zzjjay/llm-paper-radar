* 问题之河

长上下文 LLM 推理为什么这么贵？因为每多一个 token，模型都要把所有历史 token 的 key/value 缓存在显存里——这块缓存（KV cache）随序列长度线性膨胀，几万 token 就能把 GPU 显存吃完。

九年来，整个领域一直在问同一个问题：*能不能扔掉一些 KV，让模型仍然记得它需要记得的东西？* 一代一代人换不同姿势回答：先训练时扔（Longformer），再推理时按位置扔（StreamingLLM），再按注意力权重扔（H2O），再按"窗口预测"扔（SnapKV）。

但有一个隐含假设从来没人质疑过——*所有 transformer 层都一样重要，扔的时候每层扔一样多*。DepthKV 把这个假设戳破了：层与层之间敏感度差了一个数量级，凭什么平均分配？

* 演化叙事

** 2017 | Transformer: 把所有过去都背在身上

*** 它看到了什么问题
RNN 处理长序列必须一拍一拍来，第 1000 个 token 必须等前 999 个算完。GPU 并行能力浪费掉。

*** 它的解法
"attention is all you need"——每个 token 直接看其他所有 token，全部并行算完。再也不用按顺序排队。

*** 为什么这招管用（以及哪里埋了新坑）
注意力是矩阵乘法，GPU 跑得飞快。但这一刀切下去，留下了之后所有 KV 工作的根：
1. *注意力是 O(N^2)*——每个 token 看所有 token，长序列就炸
2. *KV cache 必须存*——为了避免每生成一个新 token 都把全部历史重算一遍，必须把每层每个 token 的 key、value 缓存下来；显存于是和序列长度成正比涨

后面九年的论文，本质都在还这两笔债。

** 2020 | Longformer / BigBird: 训练时就把注意力剪稀疏

*** 它看到了什么问题
Transformer 想处理整本书，O(N^2) 直接劝退——一个 16K token 的文档，注意力矩阵就是 16K x 16K，单是这一个矩阵就吃光显存。

*** 它的解法
不是每个 token 都要看所有 token——大多数远距离 token 之间根本不相关。强制让注意力只看：(1) 自己周围一个滑动窗口，(2) 少数被指定的"全局" token。这样复杂度从 O(N^2) 降到 O(N)。

*** 为什么这招管用（以及哪里埋了新坑）
训练时就这么设计，模型学会在稀疏注意力里凑合工作。但坑很致命：*你必须从头训练*。后来 GPT-3.5、LLaMA、Claude 这些已经按 dense attention 训好的模型，没法把它们的注意力换成 Longformer 那种——一换性能就崩。

于是问题转向：*能不能不动模型权重，光在推理时省 KV？*

** 2023 | StreamingLLM: 原来开头那几个 token 是黄金

*** 它看到了什么问题
拿一个预训练好的 LLaMA 跑长对话，超出训练长度后输出就乱套。研究者本来打算用"滑动窗口"（只保留最近 N 个 token 的 KV）来限制内存——结果性能崩得一塌糊涂。

*** 它的解法
偶然观察到一个怪现象：*前几个 token（包括 BOS）的注意力分数异常高，模型疯狂往那里看*。把它们扔了，attention 分布会塌掉。于是解法很简单——保留开头的几个"attention sink" + 最近的滑动窗口。其他全扔。

*** 为什么这招管用（以及哪里埋了新坑）
attention sink 像一个"垃圾桶"——softmax 必须把权重总和加到 1，如果中间没什么可看的，模型就把多余的概率倒进开头那几个 token。维护这个垃圾桶，注意力分布就稳定。

但 StreamingLLM 是个*位置规则*——它只关心 token 在序列里的位置。一个长文档中间藏着一句关键信息（比如"答案是 42"），StreamingLLM 不管它有没有用，照扔不误。下一篇论文要解决的就是：*按语义重要性，而不是按位置来扔。*

** 2023 | H2O: 谁拉到的注意力多，谁就是 Heavy Hitter

*** 它看到了什么问题
StreamingLLM 太机械——只看位置，不看内容。文档里真正关键的那个 token 可能在第 5000 个位置，StreamingLLM 照样把它扔了。

*** 它的解法
"重要的 token = 历史上一直被高度关注的 token"。具体做法：维护一个累计计数，每生成一个新 token，把每个历史 token 在这一步收到的 attention 加上去；保留累计分数最高的那些，外加最近的几个。这就是 *Heavy-Hitter Oracle (H2O)*。

*** 为什么这招管用（以及哪里埋了新坑）
直觉：模型一直在看的 token，多半是真有用的（比如人名、关键数字、问题中的实体）。实证上 H2O 在长上下文任务上确实显著超过 StreamingLLM。

但它埋了两个坑——一个是这条线后续都在补的，一个是 DepthKV 真正要捅的：

1. *基于过去预测未来*：早期看起来不重要的 token，可能在 5000 步之后突然变关键（"那个我刚才提到的化学反应名字"）。H2O 一旦扔了就追不回来。
2. *每层用同样的逻辑、同样的预算*：H2O 默认 32 层 transformer 每层都按同样比例剪——这正是 DepthKV 要打破的假设。

** 2024 | SnapKV: 用"观察窗口"提前预判哪些 token 会被需要

*** 它看到了什么问题
H2O 的累计注意力是"全历史"加和——但生成阶段最相关的，其实是离当前最近的那些 query 看了什么。全局累计反而会被早期噪声污染。

*** 它的解法
取 prompt 末尾一小段（比如最后 32 个 token）作为"观察窗口"。看这些末尾 query 都在重点关注前面的哪些 token，把这些 token 选出来当做"会被频繁访问的"，留下；其他扔。

*** 为什么这招管用（以及哪里埋了新坑）
直觉：用户问问题的最后几个词，已经把"我接下来要去找什么"暴露了。让这几个词去给前面的 token 投票，比无差别地累计更准。SnapKV 在 LongBench 上几乎是 SOTA。

但还是没动那个底层假设——*每层多少预算*。SnapKV 把"选哪些 token"这一步做到了极致，但还是 32 层都用同一个预算。这就把舞台留给了 DepthKV。

** 2026 | DepthKV: 终于有人敢问"为什么每层一样多？"

*** 它看到了什么问题
所有前人——H2O、SnapKV、StreamingLLM、FastGen——都在认真研究"每层内部怎么挑 token"，但没人质疑"层间预算怎么分配"。它们隐含地假设：32 层 transformer，每层都一样重要，所以每层都剪同样比例。

DepthKV 做了一个简单但没人做过的实验：*一次只对一层做激进剪枝，看模型性能掉多少*。结果非常戏剧化：

- 剪掉某些中间层的 KV，性能塌方
- 剪掉首尾层的 KV，几乎没影响
- 用 *permutation test* 验证："各层一样重要"这个原假设被一致拒绝

更进一步，作者发现敏感层有个共同特征——剪它们时，模型输出会变短、信息量降低。他们把这种层叫 *content amplification layers*（内容放大层），猜想这些层负责把上下文里的信息"放大"成可生成的内容。

*** 它的解法
固定全局 KV 预算 B_total，按层敏感度重新分配每层的 B^(l)。三种策略：
- *MLP (Middle-Layer Protection)*：直接给中间层更多预算
- *MGA (Metric-Guided Allocation)*：用 InfoNCE 等表征度量给层打分，分高的给更多
- *MLMA*：两者混合

在 LongBench、QA、数学推理上，同样的全局预算下 DepthKV 都比 SnapKV/H2O 这类均匀剪枝强。

*** 为什么这招管用（以及哪里埋了新坑）
这是"该花的钱花到刀刃上"的标准故事——预算总量没变，只是分配方式变了。能赢说明：之前的均匀剪枝是把预算浪费在了对性能没影响的首尾层，而真正需要 KV 的中间层却饿着。

新坑：
1. 敏感层是离线测出来的，每个 (模型, 任务) 组合可能不一样——*跨任务泛化*没保证
2. 整篇文章假设"层重要性"是静态属性——但敏感度可能随输入分布变化
3. 没有和 PyramidKV/AdaKV 这些已经做过层间分配的工作对比（见下节）

* 前沿延伸

DepthKV 出现得太新（2026 年 4 月，距今刚一个月），还没有论文专门批判它。但同时期有几条平行线值得知道：

** 平行线 1: 层间非均匀分配（DepthKV 没引用）

- *PyramidKV / PyramidInfer (2024)*：观察到 attention pattern 自下而上逐渐变锐，提出金字塔型预算分配（底层多、顶层少）
- *AdaKV (2024)*：在 head 级别做自适应分配
- *CAKE (2025)*：层间分配的另一个变体

DepthKV 自我定位为"第一个挑战层间均匀假设的"，但严格说不准确——这条线已经被人走过一年多了。DepthKV 的真正独特之处在于：*用 permutation test 做了严格的统计验证*，并且把"内容放大层"这个解释挂出来。

** 平行线 2: query-aware 动态选择

- *Quest (2024)*：把 KV 切成 page，每步生成时按当前 query 重新选最相关的 page
- *RetrievalAttention (2024)*：把 KV 做成向量索引，每步检索
- *MorphKV (2025)*：迭代更新固定大小的 KV

这条线攻击的是 H2O 留下的另一个坑——"基于过去预测未来"。它们承认 token 重要性会随 query 漂移，所以每步重选。代价是额外计算。

** 平行线 3: 学习式选择

- *DuoAttention (2024)*：学一个 per-head gate，分"检索头"和"流式头"
- *SeerAttention (2024)*：学一个 block-level 稀疏 attention 模块

启发式 vs 学习式的老对垒，搬到 KV pruning 场景。

* 洞见

读完这条线，最大的转变不是技术上的，是*提问方式*上的：

1. *从"剪什么"到"剪多少分给谁"*。前五代论文（StreamingLLM、H2O、SnapKV、FastGen、DuoAttention）都在精细化"选 token"。DepthKV 第一次（虽然不是真的第一次，但是把它讲得最清楚的一次）跳出来问"预算分配本身合不合理"。这是一种典型的*降一个抽象层次*的提问——别人都在抠"哪些 token 重要"，它问"哪些层重要"，把问题从 token 空间推到 layer 空间。

2. *被忽视的"对照实验"价值*。这篇文章的核心贡献其实不是 DepthKV 算法，而是那个 *permutation test* 和 *逐层 ablation*。算法是平庸的（按敏感度分配预算谁都想得到），但实验设计第一次给出了统计意义上"层不平等"的证据。一个领域被卡住时，往往不是缺新方法，是缺一个能戳破隐含假设的实验。

3. *"content amplification layers"的提法值得追下去*。如果敏感层真的在做"把上下文放大成可生成内容"的事，那它就连接到了机制可解释性领域。这条桥后面会不会有人接着走？

下一步最可能往哪走：

- *动态层间分配*——DepthKV 是离线测出敏感度，但敏感度大概率随输入变化。下一步是 query-aware 的层间分配（结合 Quest 那条线）
- *微调时显式优化"层等价性"*——既然层不平等是模型自己长出来的，能不能在预训练或微调阶段就让它变平等？这样推理时均匀剪就够了
- *层间分配 × token 选择*的联合优化——目前 DepthKV 是把 SnapKV 当 token 选择器、自己只管分配预算。两步合一步联合优化也许有更多甜头

* 启发

对"怎么找问题"：

1. *找一个所有人都默认但没人验证的假设*。"每层一样重要"在 KV pruning 领域被默认了 3 年，谁都没去做层间 ablation。一旦做了，发现假设不成立——文章自然有人发。
2. *跨子领域借工具*。DepthKV 用了 *permutation test*（统计学）+ *InfoNCE / 表征相似度*（表征学习）+ *layer ablation*（机制可解释性）。这些工具单独都不新，组合起来产生新角度。
3. *新方法弱、新观察强也能发*。这篇文章的算法部分（按敏感度分配预算）三句话能说完，但前 4 节几乎全是 pre-study 和 ablation。审稿人吃这一套——*"我重新看了大家都看过的东西，得出了不一样的结论"* 比 *"我提了一个新算法"* 往往更有冲击力。

这意味着你可以：

- 在你熟悉的任何领域，列出"3 年没人质疑过的隐含假设"，逐一做对照实验——大概率能挖出至少一个能发的洞
- 做 KV cache 工程时，*不要默认每层均匀剪*——拿 DepthKV 的方法（或更简单的：先剪首尾层，留中间层多一点）测一下，可能白捡 1-2 个点
- 在做模型压缩 / 量化 / 剪枝 这类"预算分配"问题时，记得问一句"这个预算单位（per-layer / per-head / per-token）选对了吗？"——更换预算分配的粒度，常常是免费的性能来源
