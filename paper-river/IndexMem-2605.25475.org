* 问题之河

大模型要读 100 万 token 的代码库或者视频流，可是只要一开始解码，每一步都得回头看一遍前面所有 token 的 K 和 V——这堆缓存（KV cache）随长度线性膨胀，一个 70B 模型读 100 万 token 就要 320GB 显存。算力反而不是瓶颈，*显存和带宽才是*。

于是九年来一代一代人都在琢磨同一件事：*前面那么多 token，到底哪些必须留、哪些可以扔？* 扔早了模型答不出来，留多了显存爆炸。听起来像个工程问题，其实是个认识论问题——你怎么提前判断一个 token "以后还有没有用"？

每一代论文都在回答这个问题，每一代都被下一代嫌弃"你判断得不准"。

* 演化叙事

** 2017 | Vaswani et al.: 把 RNN 扔了, 但留下了一个会爆显存的怪物

*** 它看到了什么问题

RNN 太慢, 一个 token 接一个 token 地算, 没法并行. 模型读一篇长文章, 训练的时候 GPU 大部分时间在等.

*** 它的解法

注意力机制. 每个位置直接和所有其他位置算相关性, 一步到位, 完美并行.

*** 为什么这招管用（以及哪里埋了新坑）

并行训练之后, 这个架构吞下了整个 NLP 领域. 但有个细节当时没人在意: *推理时每生成一个新 token, 都要重新看一遍前面所有 K 和 V*. 工程师为了不重算, 把所有 K, V 缓存下来——这就是 KV cache. 训练阶段它无所谓; 推理阶段, 它随上下文长度线性增长. 当上下文从 2K 涨到 1M, 这只怪物就开始把 GPU 显存吃光.

接下来九年的故事, 都是在跟这只怪物搏斗.

** 2023 | StreamingLLM: 第一个发现 "其实大部分都能扔"

*** 它看到了什么问题

Vaswani 留下的 KV cache 在长上下文场景下完全撑不住. 但作者注意到一个反直觉现象: 如果你把窗口外的 token 简单丢掉, 模型崩了; 但如果你保留 *开头那几个 token* + *最近的窗口*, 模型居然能稳定生成几百万 token. 开头那几个 token 像 "锚点", 接住了所有 softmax 的概率质量.

*** 它的解法

滑动窗口 + attention sink. 永远保留头部 4 个 token, 加上最近的 N 个 token, 中间的全扔.

*** 为什么这招管用（以及哪里埋了新坑）

它证明了一件事: *KV cache 大量冗余*. 但代价是中间所有 token 都丢了——这意味着 "请总结第 5 章的内容" 这种长程依赖任务直接报废. 它打开了门, 但门里全是位置启发式, 不看内容.

下一代必须问: 凭什么按位置扔? 应该按 *重要性* 扔.

** 2023 | H2O: 让历史的注意力分数自己投票

*** 它看到了什么问题

StreamingLLM 太机械, 中间的 token 里也有关键的人名、数字, 全扔太亏. 应该让模型自己说"这个重要".

*** 它的解法

Heavy-Hitter Oracle. 每生成一步, 累加每个历史 token 收到的注意力分数. 保留分数最高的 K 个 (heavy hitters) + 最近窗口. 相当于让历史的注意力分布自己投票决定谁活下来.

*** 为什么这招管用（以及哪里埋了新坑）

第一次把 "重要性" 引入 KV 选择, 比纯位置启发式好多了. 但有两个软肋:

1. *过去重要 ≠ 未来重要*. 一个 token 前 100 步被高频引用, 不等于第 101 步还有用.
2. *因果颠倒*. 你得先把注意力算完, 才能知道谁是 heavy hitter, 那时候 KV 已经在显存里了——省下来的只是未来步骤的事情.

H2O 把视角从 "位置" 推到了 "历史使用频率". 下一代要继续问: 能不能 *提前* 知道, 而不是事后追认?

** 2024 | SnapKV: 偷看一眼 prompt 末尾, 就知道 query 想问什么

*** 它看到了什么问题

H2O 是 decode 阶段慢慢累加, 但其实 *prompt 已经把意图写出来了*——比如 "...请总结上面文档的核心结论". 末尾这句话告诉你: 现在 query 关心的是 "结论" 类的东西, 而不是文档里的细节. 没用它太可惜.

*** 它的解法

拿 prompt 末尾一小段 (比如最后 32 个 token) 作为 "观察窗", 用它对前面所有 KV 的注意力来打分, 选出 top-K 留下. 在 prefill 一结束就压缩, decode 阶段只需要看压缩后的小 cache.

*** 为什么这招管用（以及哪里埋了新坑）

第一次把 *query-aware* 引入选择, 跨上了一大步. 但留下两个坑:

1. *局部偏置*. 观察窗只能看 prompt 末尾, 如果 query 关心的是文档开头的某个细节, 选出来的全是窗口附近的 token, 真正需要的早就被淘汰了.
2. *依然依赖真实注意力分数*. 计算开销没省掉, 只是省了 decode 阶段的内存.

SnapKV 之后, 这条线分裂成两派——一派要 *绕开注意力分数*, 一派要 *学一个更好的判断器*.

** 2025 | KeyDiff / Expected Attention: 用代理指标绕开注意力计算

*** 它看到了什么问题

SnapKV 还得真去算一次注意力. 在超长上下文下, 这一次注意力本身就是负担. 能不能不算注意力, 也猜出谁重要?

*** 它的解法

两条不同的代理路线:

- *KeyDiff* (Park 2025): 一个 token 的 key 如果和其他 key 都很 *不像*, 说明它编码了独特信息, 应该留下. 用 key 之间的 cosine 距离当 "信息量" 代理, 完全不碰注意力分数.
- *Expected Attention* (Devoto 2025): 从语料库里平均出一个 "典型 query 分布", 用它对历史 keys 打分. 等于预设了一个 "想象中的未来 query".

*** 为什么这招管用（以及哪里埋了新坑）

省了计算, 也省了对当前 query 的依赖. 但全是 *先验假设*:

- KeyDiff 假设 "key 独特性 = 重要性", 但对某些任务 (比如检索任务) 重要的恰恰是 key 之间相似的那些.
- Expected Attention 假设 "平均 query 能代表具体 query", 但长尾任务和典型语料偏差很大.

这一代是 "聪明的猜". 但 *猜得再聪明也是猜*. 下一代的问题呼之欲出: 既然我们手上有训练数据, 为什么还在用人工设计的代理? 让模型自己学一个 importance 函数, 不就完了?

** 2026 | IndexMem: 干脆把判断器学出来, 顺便给被扔的 token 留条后路

*** 它看到了什么问题

两个坑同时出现:

1. *猜的时代结束了*. SnapKV / KeyDiff / Expected Attention 都是 "训练-free + 手工启发式". 它们假设的 query 分布、key 几何、注意力模式, 在不同任务上根本不一致. 既然我们能训, 为什么不直接训一个 indexer?
2. *扔掉 = 永久遗忘*. 整条线从 H2O 到 Expected Attention, 全都是 "选哪些留, 其他丢". 但有些 token 当时看不重要, 50 步之后 query 一变就需要了——这时已经被永久删除. 这叫 *irreversible forgetting*.

*** 它的解法

两个互补的模块:

1. *Learnable Indexer*. 一个非常轻量的小网络, 输入是 K (和位置信息), 输出每个 token 的重要性分数. 端到端训一次, 推理时直接打分, 不需要任何启发式. 训练目标是 "和真实注意力分数对齐 + 下游任务表现".

2. *Latent Memory*. 一个固定大小的潜状态 M (比如几百维), 在推理过程中在线更新——每次 KV 被淘汰, 它的信息就被压缩进 M. 注意力输出 = 主流稀疏注意力 + 从 M 读出的残差. 这样即使某个 token 的 K, V 没了, 它的"印记"还在 M 里, 可以以残差方式被读回.

类比: 主 KV cache 是 *快记忆* (高分辨率, 容量有限); Latent Memory 是 *慢记忆* (低分辨率, 容量恒定但累加无穷历史). 这个区分直接借自 Titans / 快慢权重那一脉.

*** 为什么这招管用（以及哪里埋了新坑）

效果上, RULER 4K/16K 在 Qwen/Mistral/Llama 上提升最高 25 个点, NIAH 检索更稳, LongBench 整体更好.

但更重要的是 *视角转变*: 整条线从 "怎么选" 进化到了 "怎么学" + "怎么留下扔掉的痕迹". 这其实是把 KV 压缩问题从 *硬选择* 变成 *软压缩*——你不需要真的判断对每一个 token, 因为判断错的还能从 memory 里救回来.

新留下的坑:
- Indexer 需要训练, 部署到新模型时要重新跑一次 (虽然作者强调 lightweight).
- 跨层是否可以共享 indexer? 作者 appendix 已经在试 (Running Mean / IndexCache 方向).
- Latent Memory 的容量上限和 Titans 一样, 没有给出明确的容量-精度 trade-off 理论分析.

* 前沿延伸

** 2026 | IndexCache (Bai et al.): 训完一个 indexer, 能不能跨层复用?

IndexMem 的 appendix 里已经在做这件事了——观察发现各层的 importance 分数高度相关, 那么算一次, 多层复用. 这条线刚刚开始. 它把 "学 indexer" 从 per-layer 推向 per-model.

** 2026 | Ada-KV (Feng et al.): 不同 head 应该有不同的预算

之前的方法把所有 head 的 cache budget 平均分. Ada-KV 论证: 有些 head 是 "检索 head", 应该多分; 有些 head 是 "局部聚合 head", 少分即可. 它和 IndexMem 正交——一个解决 "选谁", 一个解决 "每个 head 该选多少".

截至 2026.05, 这条线最新的进展是 *学 + 自适应预算 + 跨层复用* 三者结合, 但还没有一篇论文把它们完全打通.

* 洞见

整条线表面上是 "KV 剪枝越剪越准", 但底层其实是 *两个范式转变同时在发生*:

1. *从硬选择到软压缩*. 前五代都在做"留谁/扔谁"的二元决策, 失败一次就是永久信息损失. IndexMem 第一次承认: 任何选择器都会犯错, 那就让被扔的也别真死——压成低分辨率的状态留着. 这是从 *分类问题* 滑向 *压缩问题*. 这背后是承认 "完美选择不可能", 转而追求 "容错".

2. *从启发式到可学习*. 前五代花了九年在比谁的启发式更聪明 (位置 -> 频率 -> query -> 代理). 这是典型的 "在第一性原理之前先穷尽所有手工方案". 一旦发现 *训练成本可接受*, 学一个 indexer 就成了必然. 这和 retrieval 领域从 BM25 走到 dense retriever 的轨迹完全一样.

下一步往哪走? 三个方向已经发了芽:
- *跨层共享* (IndexCache): 一次训练多层复用.
- *自适应预算* (Ada-KV): head 之间差异化分配.
- *Memory 容量理论* (TBD): Latent Memory 究竟能压多少历史? Titans 也没回答, 这是个开放问题.

更激进的猜想是: 这条线最终会和 *训练时* 的稀疏注意力 (NSA, MoBA) 合流——不是 "训完模型再剪", 而是 "训练时就让模型学会自己剪". 推理阶段的 indexer 会被吸收回训练阶段, 整个 attention 模块原生就是稀疏 + memory 的.

* 启发

对做研究的启发:

1. *启发式是过渡, 学习是终局*. 任何看起来需要 "判断" 的问题, 如果有训练数据, 启发式方案早晚会被学到的方案替代. 但启发式不白做——它告诉你 *正确答案大概长什么样*, 学的方案才知道往哪儿优化. SnapKV / H2O 的 query-aware 视角, 直接成了 IndexMem 训练 indexer 的监督信号.

2. *看到一个领域里所有人都在 "比谁的代理指标更聪明", 就该提学习方案了*. 这是范式跃迁的窗口期.

3. *硬决策遇到不确定性, 转向软决策*. 当你的算法每一次错误都造成不可逆损失时, 加一个 "残差通道" 通常比 "把判断器做得更准" 容易得多. IndexMem 加 Latent Memory 这一招, 在分类、检索、推荐里都有同构应用.

落点: 你可以拿同样的视角 (硬决策 → 软压缩 + 残差) 去重审任何带 "丢弃" 操作的系统——量化里的 outlier 处理、MoE 里的 router、推荐里的 candidate pruning, 都是同一个故事.
