* Target Paper

*More Than a Quick Glance: Overcoming the Greedy Bias in KV-Cache Compression*
LASER-KV (Layer Accumulated Selection with Exact-LSH Recall)
Aryan Sood, Tanvi Sharma, Vansh Agrawal — IIT Roorkee, 2026.

One sentence: Prior KV-cache compression methods all relied on "keep the tokens
with high attention scores," but this paper argues that the premise itself is
wrong — high attention score ≠ token importance. An independent similarity
signal must be added to handle long contexts correctly.

* Evolutionary Narrative: How One Problem Forced LASER-KV into Existence

** Layer 0: Where Is the Root Cause?

When a Transformer generates tokens it must cache the Key and Value tensors for
every token it has seen, so that the next token can attend to them all.
The longer the context, the more memory the KV cache consumes — *linear growth*.

Running a 128k-token context means the KV cache alone fills an H100, let alone
a consumer GPU.  Starting in 2023 an entire research thread chased one question:

#+begin_quote
If we cannot fit all KVs in memory, which ones should we discard?
#+end_quote

This sounds innocuous, but it hides an untested assumption — *"which ones can be
discarded" is something we can determine*.  Every subsequent paper has been
paying the price for that assumption.

** Layer 1: The Two Most Naïve Intuitions — StreamingLLM and H2O (2023)

The first wave of researchers proposed two intuitions.

*StreamingLLM* argued: the first few tokens are "attention sinks" (the resting
place of a language model's attention) and must be kept; the most recent tokens
are what is currently being discussed and must be kept; *everything in between
can be dropped*.  This is a sliding window.  Simple and brutal — but the moment
a question concerns something in the middle of the context, the model fails
immediately.

*H2O* was more thoughtful: instead of arbitrarily dropping the middle, *look at
attention scores* — tokens that are attended to frequently (called "heavy
hitters") are kept; low-scoring tokens are dropped.  This was the first time
"token utility" was quantified as "attention score."

H2O established the law that all subsequent work implicitly obeyed:
*attention score = importance*.  Nobody challenged this law head-on before
LASER-KV.

** Layer 2: SnapKV — Moving the Selection Earlier (2024)

H2O drops tokens on the fly during generation.  Every new token triggers a
recomputation of who stays and who goes.  Two problems: the overhead is high,
and tokens dropped early can never come back.

SnapKV noticed this and proposed: *select which tokens to keep once, during the
prompt phase*.  The trick is to use the last few tokens of the prompt as an
"observation window" — wherever they attend is where later generation is most
likely to need — keep those positions and discard the rest.

SnapKV's contribution was turning KV compression from "online, greedy" into
"one-shot, predictive," but *it still used attention scores to select tokens*.
The law was intact.

** Layer 3: Quest and PyramidKV — How to Allocate the Budget (2024–2025)

SnapKV applied a uniform policy: every layer, every position, the same number of
retained tokens.  That is obviously too coarse.

*Quest* argued: which tokens to select should not depend only on historical
average attention scores, but on what the *current query* cares about.
Query-aware token selection, decided dynamically at each decoding step.

*PyramidKV* noticed waste along a different axis: different layers have different
attention patterns.  Lower layers (close to input) attend broadly and need more
tokens; higher layers (close to output) attend sharply and need fewer.
So *give lower layers a larger budget and higher layers a smaller budget* —
a pyramid shape.

By this point the line had exhausted all variations on *how to distribute the
budget*: to whom (H2O → SnapKV), conditioned on the query (Quest), conditioned
on the layer (PyramidKV).  But *the selection criterion was always the attention
score*.

** Layer 4: FINCH — Trying to Break the Budget Ceiling (2024)

Every prior method assumed a fixed budget.  FINCH pushed back: can we handle
longer contexts within a fixed budget?

Its approach was *recursive compression*: chop the very long context into small
chunks, apply Top-K compression independently to each chunk, pass the result to
the next chunk, compress again … layer by layer, theoretically supporting
arbitrary length.

It sounds elegant, but LASER-KV builds directly on the failure mode of FINCH —
*each compression round uses the current query's attention scores as the
criterion*.  A token that nobody attends to right now may still be needed later.
Compress once, lose some; compress again, lose more.  After several rounds,
genuine long-range dependencies are gone.

This is why, in LASER-KV's 128k experiments, FINCH *scores zero* (Table 2,
QA1–QA6 all 0%).

** Layer 5: MagicPIG — The First Attempt to Escape Attention Scores (2024)

MagicPIG was the first to vaguely sense that "attention scores might not be
enough."  It argued: instead of exactly computing all query × key attention
scores (expensive), use *Locality-Sensitive Hashing (LSH)* as a sampler —
similar query/key pairs fall into the same bucket with high probability, and
collision probability serves as an approximate score.

This was originally a speed trick, but it quietly opened a crack: *a token's
importance can be judged by how structurally similar it is to the query, not
only by its attention score*.

But MagicPIG was still *replacing* the attention score, not coexisting with it.
The result: it could capture structurally similar tokens but missed the true
attention heavy hitters — in LASER-KV's ablation table, MagicPIG alone scores
only 2% on QA2 at 16k.

At this point the problem surfaced clearly for the first time:

- Attention-score-only (H2O / SnapKV / Quest / PyramidKV / FINCH): misses
  structurally critical tokens.
- Similarity-hash-only (MagicPIG): misses the true heavy hitters.

*The two signals are complementary.  Nobody had combined them.*

** Layer 6: LASER-KV — Binding the Two Signals Together (2026, This Paper)

LASER-KV finally stitches the two lines together, and also heals the wound
inflicted by FINCH's recursive compression.  Three moves:

1. *Replace recursion with accumulation* — the budget no longer "overwrites prior
   selections" with each new chunk; instead it is *append-only*.  Tokens
   selected from a chunk are added to the KV cache and never evicted.  Early
   critical tokens cannot be flushed out just because nobody attends to them
   later.  The paper calls this "accumulative, append-only memory."

2. *Exact + LSH hybrid selection* — the true core.  Given a block's budget:
   - A portion of the budget (75% in the paper) goes to *Exact attention*:
     sum attention scores across all layers/heads and keep the top scorers —
     this is the H2O-style heavy-hitter component.
   - The remaining budget (25%) goes to *LSH hashing*: from the pool of tokens
     not selected by Exact attention, sample by query–key hash collision
     probability — this is the MagicPIG-style structural-similarity component.
   - The two groups are merged to form the final cache.  Exact captures "what
     everyone attends to"; LSH captures "what looks like the query structurally
     but whose score is suppressed."

3. *Divisor n controls the local window* — an engineering detail that matters.
   Given a block's total budget B, a divisor n splits it into three parts:
   - =2B/n= is reserved for "anchors" + "local sliding window" (syntax /
     local-coherence safety net).
   - =B - 2B/n= is reserved for the long-term memory pool (selected by Exact+LSH
     above).

   *Smaller n → larger local window → more stable generation; larger n → more
   budget for long-range memory.*  This knob is specifically designed to
   *separate the effect of "compression itself" from the "sliding-window
   side-effect"* — which is the scientific contribution of LASER-KV's ablation:
   the paper claims that previous benchmark numbers conflated the two.

Experimentally: at 64k context, LASER-KV scores 25% on QA1 vs. SnapKV 7% and
FINCH 22%; at 128k (the critical setting), LASER-KV QA1 = 38%, PyramidKV = 25%,
FINCH = 0%, SnapKV = 22%.  The gap LASER-KV opens is precisely the portion of
tokens captured by the 25% LSH budget — tokens that attention-only methods miss.

* Core Insight: What Is Really Happening Beneath This Thread

On the surface this is "how to compress KV caches more cleverly."  Underneath it
is a debate over *whether token importance can be measured by a single signal*.

- *2023–2024*: Everyone defaulted to attention score being sufficient; the debate
  was only about "how to compute it, when to compute it, and how to allocate more
  of it."
- *2024 MagicPIG*: For the first time someone used a different signal (LSH
  similarity) to select tokens, but in a *replacement* fashion — one side won,
  the other lost.
- *2026 LASER-KV*: For the first time a paper explicitly states "one signal is
  not enough; two signals must be combined," and provides an ablation to prove it.

Going one level deeper: this thread is moving from *single-view greedy
selection* toward *multi-view complementary selection*.  The "heavy hitter"
metaphor from H2O is vivid, but it assumes importance is one-dimensional and
totally-ordered.  LASER-KV is effectively arguing: *importance is
multi-dimensional, and any single ranking will have blind spots*.

This is the same logic as ensemble learning / multi-view learning, applied this
time to KV-cache management.

* Where Does This Line Go Next?

Extrapolating along this thread:

1. *Learn a utility function*: Right now Exact and LSH are two fixed heuristics,
   with the 0.75/0.25 split tuned by hand.  The natural next step is to train a
   small network to dynamically determine each token's "importance" in different
   contexts.  The trade-off is that this would sacrifice the training-free
   advantage.

2. *Abandon pruning, switch to compressed storage*: All pruning methods assume
   "what is dropped is gone forever."  But if low-priority tokens are stored in
   4-bit quantization while high-priority ones remain in FP16, more information
   can be retained within the same memory budget.  The MagicPIG team has
   reportedly moved in this direction.

3. *Push this line toward agent settings*: All current benchmarks are retrieval
   tasks like BAbILong.  Once deployed in multi-turn tool-calling agent
   scenarios, the judgment of "which tokens matter" becomes far more dynamic,
   and the accumulative / recursive strategies of every current method will need
   to be redesigned from scratch.
