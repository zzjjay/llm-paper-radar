* 问题之河

大模型要处理 128K 个 token 的超长上下文时，GPU 内存装不下——怎么让它记住全文，
又不爆显存？

这个问题难在：你得既快又准。快是指推理时不能为了省内存把速度拖垮；准是指被
压缩的内容，一旦后面用到，还能找得回来。每一代方案都在"丢什么、怎么丢、丢了
还能不能找回来"这三件事上折腾——每往前一步，新的代价就冒出来。这就是为什么
从 2023 年到 2026 年，这个问题吸引了一代代研究者前赴后继。

* 演化叙事

** 2023 | StreamingLLM: 发现"注意力汇点"，让 LLM 能无限生成

*** 它看到了什么问题

在 LLM 推理中，每生成一个 token，就要把这个 token 的 key/value 向量存进缓存。
序列越长，缓存越大，最终撑爆 GPU 内存。一个自然的想法：只保留最近 N 个 token
的缓存，像一扇滑动的窗口。Xiao 等人发现，一旦这扇窗口滑过训练时的最大序列
长度，模型的困惑度（perplexity）就突然崩塌——因为位置编码跳了，模型从未见过
这种状态，完全不知道该怎么算注意力。

*** 它的解法

他们仔细扫描了各层各头的注意力权重分布，发现了一件怪事：序列最开头的几个
token（哪怕是句号、逗号、BOS 符号这类没语义的词）在几乎所有层、所有头上都
吸引了极高的注意力分数。这不是因为这些词有多重要，而是 softmax 要强制让分数
加起来等于 1——当没有一个 token 特别突出时，注意力就"倾倒"到这些全局可见的
早期 token 上，像水流向洼地。Xiao 等人把这些 token 叫做 *注意力汇点*
（attention sinks）。

解法直接：把前 4 个汇点 token 的 KV 永久钉住，加一个滑动窗口保留最近的
token。softmax 有地方倒注意力，位置编码也不会跳变，模型可以持续生成，速度
比重新计算快 22 倍。

*** 为什么这招管用（以及哪里埋了新坑）

它解决了 softmax 的数值稳定性问题，让无限长生成成为可能。但代价是：中间所有
内容——文档的主体、对话的核心上文——一律抛弃，不管多重要。这是纯粹的位置
驱逐，完全不看内容。能不能根据 token 自身的信息价值来决定留谁，是下一篇论文
要回答的问题。

** 2023 | H2O: 按内容选留谁，重击手优先

*** 它看到了什么问题

StreamingLLM 丢中间 token 不问缘由。但注意力分数本身就暗示了哪些 token 重要：
被高频关注的 token，后续解码时大概率还会被关注。Zhang 等人统计了真实模型的
注意力矩阵，发现了类似幂律的分布——大约 20% 的 token 累积了 80% 以上的注意
力分数。他们把这些 token 称为 *重击手*（Heavy Hitters，H2）。

*** 它的解法

把 KV 驱逐问题形式化为动态子模优化问题：在每个解码步，保留两类 token 的 KV
—— 最近的窗口（保局部连贯）和累积注意力最高的 H2 token（保全局相关性）。
每步解码后重新评估 H2 集合，动态增删。理论上能证明这是对注意力质量的最优近
似保留。实验上，相比 StreamingLLM，在长文档任务上准确率提升显著，KV 缓存缩
减 5-10 倍。

*** 为什么这招管用（以及哪里埋了新坑）

内容自适应选留确实比固定位置强。但有一个巨大的盲点：H2O 只压缩解码阶段的
KV，完全不处理 *prompt*（输入提示）的 KV 缓存。而实际应用中，用户的 prompt
往往有数万 token，远比生成长度长——压缩的真正大头根本没动。

** 2024 | SnapKV: 在生成之前，先把 prompt 压缩好

*** 它看到了什么问题

Li 等人指出：真实使用场景里，prompt 动辄几千上万 token，生成的回答却只有几
百 token——压缩的重点根本不在解码上，而在 prefill 上。H2O、StreamingLLM 都
没碰这一块。

*** 它的解法

SnapKV 在生成 *开始之前* 就把 prompt 的 KV 压缩好。怎么知道 prompt 里哪些
token 重要？用 prompt 最后一小段（*observation window*，通常是最后 64 个 token）
的查询向量，对整个 prompt 前缀做注意力打分，加总投票——这些查询代表了"模型
即将生成时最关注什么"。为了避免选出零散的单个 token（语义不完整），SnapKV
在投票分数上做一维最大池化，把相邻高分 token 聚拢在一起选。每个注意力头独立
操作，保证多样性。

如果你把 H2O 的"每步解码后重新计算重击手"去掉，换成"在 prefill 时一次性
基于 observation window 投票决定谁留"，你就得到了 SnapKV 的核心思路。

*** 为什么这招管用（以及哪里埋了新坑）

它把压缩时机提前到了正确的地方，且每头独立选择很灵活。但它仍然把每一层的
KV 预算设成一样大。实验数据表明，LLM 不同层的注意力模式差异巨大——低层看
得很分散（需要大预算），高层已高度聚焦（小预算就够）。统一预算是一种结构性
浪费。

** 2024 | PyramidKV: 低层宽、高层窄，按层动态分配预算

*** 它看到了什么问题

Cai 等人系统性地分析了多个模型各层的注意力熵，发现 LLM 信息流有一个清晰的
"漏斗"形状（Pyramidal Information Funneling）：底层注意力分散到很多 token，
中层逐渐集中，顶层几乎只盯着少数几个关键 token。SnapKV 全层统一预算，是把
这个漏斗形结构强行拍平。

*** 它的解法

让 KV 预算沿层深度递减，形成金字塔形状：底层大预算，顶层小预算。token 选择
机制沿用 SnapKV 的 observation window 投票，只是把每层的 top-K 改成随层深度
变化的函数，总预算保持不变。

结果：在 LLaMA-3-8B 上，只保留 12% 的 KV 缓存时接近满 KV 性能；激进压缩到
0.7% 时，仍比其他方法高 20 分。

*** 为什么这招管用（以及哪里埋了新坑）

预算分配更聪明了，但底层逻辑没变：最终还是在 token 级别做选留，把某些单个
token 丢掉。问题在于：语义是跨 token 编码的。一个连续的语义片段里，扔掉其中
一个 token，整个片段就变得支离破碎——"电话号码 +86-138"，你留了"138"、
丢了"+86"和后面的号码，留下的缓存毫无意义。能不能以语义单元而不是单个
token 作为最小驱逐粒度？

** 2025 | ChunkKV + SentenceKV: 以语义单元为原子，不打散连贯性

*** 它看到了什么问题

Liu 等人（ChunkKV）和 Zhu 等人（SentenceKV）几乎同时指向同一个根本问题：
token 级别驱逐会破坏语义连贯性。语言的意义跨越多个 token 编码，孤立地保留
高分 token 而丢弃其上下文，相当于保留了词根却扔掉了整个句子。

*** 它的解法

*ChunkKV* 把 token 按固定大小（默认 10 个 token）分成 chunk，以 chunk 为单位
打分，整体保留或整体驱逐。一个完整的短语要么全在要么全不在，杜绝语义碎片。
额外发现：相邻层选留的 chunk 索引高度重合，因此做"层间索引复用"，减少重复
计算，实验上吞吐量提升 26.5%。

*SentenceKV* 把粒度提升到句子。每个句子的 KV 在 prefill 后压缩成一个语义向量
存在 GPU 上（很小）；完整 KV 卸载到 CPU（大但用时才取）。解码时，当前查询
向量和所有句子语义向量做相似度排序，只把最相关的几个句子 KV 从 CPU 取回 GPU
参与注意力计算。

这两者的关系：如果你把 ChunkKV 的"固定大小 chunk"换成"按标点分的句子"，
再加上"GPU 存摘要、CPU 存全量、按需取回"，你就得到了 SentenceKV。

*** 为什么这招管用（以及哪里埋了新坑）

语义单元保留确实比 token 级更好，且 SentenceKV 引入 CPU 卸载让 GPU 内存大幅
降低。但两者有同一个根本局限：*压缩在 prefill 时一次性完成，分辨率从那刻起
固定*。某个语义 span 在 prefill 时被评为"不重要"，哪怕解码时发现它其实关键，
也再找不回 token 级别的细节。此外 chunk/句子的粒度是静态的——无法根据当前
解码步的需要动态决定"这段我要放大，那段压缩就够"。

** 2026 | SeKV: 分辨率自适应，按需放大语义 span（目标论文）

*** 它看到了什么问题

Abaskohi 等人把前人问题归结为一句话："一旦压缩完成，token 级别的细节就永久
消失，没有任何机制能在解码时按需恢复。"无论是 ChunkKV 的 chunk 还是 SentenceKV
的句子，压缩都是单向的——压了就没了。

*** 它的解法

SeKV 做了三件事的组合：

1. *熵引导的语义 span 分组*：用信息熵检测语义边界，把 context 切成语义完整
   的 span——不是固定长度的 chunk，也不是按标点分的句子，而是按内容信息密度
   自适应切割，高熵区域（信息密集，边界模糊）切细，低熵区域（重复或稀疏）
   切粗。

2. *GPU-CPU 两级存储*：每个 span 在 GPU 上只存一个轻量摘要向量（用于快速路
   由判断，极省 GPU 内存）；完整 KV 在 CPU 上以低秩 SVD 基底的形式压缩存储
   （比直接存完整 KV 省得多，且可以无损/近似无损重建）。

3. *可训练的 zoom-in 机制*：解码时，如果某个 span 的摘要向量和当前查询的相
   似度超过阈值，就从 CPU 取出它的 SVD 基底，在线重建出 token 级完整 KV，用
   于高分辨率注意力。这个判断机制本身可训练，但只增加不到 0.05% 的参数，
   基础模型完全冻结。

如果你把 SentenceKV 的"一次性选留句子"替换成"GPU 只存摘要，CPU 存 SVD 压
缩的可重建完整 KV"，再加上"解码时自适应决定哪些 span 要放大到 token 级"，
你就得到了 SeKV。

*** 为什么这招管用（以及哪里埋了新坑）

关键创新是把"压缩"和"可恢复性"解耦——GPU 侧永远只存摘要，但"真实 KV"
以 SVD 压缩形式留在 CPU 上随时可以重建。相比直接丢弃（ChunkKV/SentenceKV），
恢复能力是真实存在的。实验结果：128K context 下 GPU 内存节省 53.3%，比最强
语义压缩 baseline 平均高 5.9 个百分点，跨 4 个 benchmark（LongBench、RULER、
Needle-in-a-Haystack 等）。

新坑：CPU-GPU 数据传输的延迟在高并发或大批次场景下是真实代价；SVD 重建本身
有计算开销；zoom-in 触发阈值若调不好，要么过度取回（省不了内存），要么漏取
关键 span（准确率掉下去）。

* 前沿延伸

SeKV 于 2026 年 6 月 30 日提交，目前尚无引用它的后续论文（提交不足一周）。
但同时期有三篇高度相关的并行工作，指向相同问题的不同攻法：

*CompressKV（arXiv 2606.24467，2026 年 6 月，与 SeKV 相差一周）*：发现 GQA
架构中有一小部分"语义检索头"（Semantic Retrieval Heads）主导语义定位，针对
这些头做异构驱逐策略，其余头用标准方案。方向和 SeKV 相同——语义结构 > 单个
token——但 CompressKV 全在 GPU 内完成，不做 SVD 重建。SeKV 的 zoom-in 能恢
复 token 级细节，CompressKV 不能。两篇论文互不引用，是真正的并行收敛。

*xKV（arXiv 2503.18893，2026 年 3 月）*：跨层联合做 SVD 分解，把相邻层的 KV
投影到共享低秩子空间，解码时做选择性重建。SeKV 的 SVD 是在语义 span 维度展
开，xKV 是在层维度展开——两种正交的压缩轴，理论上可以叠加组合。

*ScoutAttention（arXiv 2603.27138，DAC 2026）*：CPU 超前一层做注意力预计算，
与 GPU 流水线重叠，把 CPU-GPU 传输延迟藏进计算时间里，实现 2.1 倍加速。SeKV
的短板恰好是 CPU-GPU 传输开销，ScoutAttention 的思路是天然的工程补丁。

截至 2026 年 7 月，SeKV + ScoutAttention 的组合尚未有人实现，但是一个明显的
下一步方向。

* 洞见

这条演化线表面上是"KV 缓存压缩技术的进步"，本质上是一次次对
*什么是最小有意义的记忆单位* 的重新定义：

token -> chunk -> 句子 -> 语义 span（自适应）

每次升级都是把记忆的"分辨率可调范围"向上推了一档。粒度越粗省内存但信息损
失越大，粒度越细信息完整但根本没省多少——这是一个内在的张力。前五篇论文的
做法都是"在某个固定粒度上做取舍"，而 SeKV 第一次把这个粒度本身变成了解码
时动态可调的参数。

真正的认知转变发生在 SeKV 这里：从 *压缩 = 丢失* 变成 *压缩 = 延迟重建*。
这不是工程技巧，而是一种范式迁移——把数据库领域"存储层级 + 按需加载"的思
想引入注意力机制。以前研究者问"丢哪些 token 损失最小"，SeKV 改问"怎么设
计压缩格式使得任何时候都能按需还原"。

*下一步最可能的方向*：
- 把 zoom-in 触发从阈值判断升级为连续注意力路由（类似 MoE 的 gating），端到
  端学习"什么时候值得花 CPU 带宽重建"
- 把 SeKV（span 维度 SVD）和 xKV（层维度 SVD）叠加，同时在两个正交维度压缩
- 用 ScoutAttention 的流水线思路消除 SeKV 的 CPU 传输延迟瓶颈
- 把 zoom-in 机制扩展到 MoE 架构，不同 expert 对不同 span 的放大需求本身就
  可以提供自然的路由信号

* 启发

这条线对"怎么找研究问题"有一个清晰的示范：每一篇论文的贡献都不是从零开始
的创意，而是把前人方案的 *一个具体矛盾* 推进到了逻辑上的下一步。
StreamingLLM 解决了无限生成，但漏了内容感知；H2O 加了内容感知，但漏了 prompt；
SnapKV 补了 prompt，但漏了层差异；PyramidKV 修了层差异，但打散了语义单元；
ChunkKV/SentenceKV 保住了语义单元，但锁死了分辨率……

这意味着你可以：把任何你感兴趣的方法当成一个"有矛盾的系统"来读，不问它
做了什么，只问它的逻辑自洽地走到终点会在哪里撞墙。那堵墙，就是下一篇论文
的入口。
