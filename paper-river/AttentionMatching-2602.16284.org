* 问题之河

一个语言模型读完一份 60k 字的病历，准备回答医生的提问。这份病历在显存里被压成了一张巨大的「记忆表」——KV cache——每多一个 token 就多一行。现在医生想问十个问题，每问一次模型都要把这张表完整扫一遍：搬数据，算注意力。表越大，每次回答越慢，并发越少。

最直观的省法是「让模型自己写个摘要」——把六万字压成两千字，再让模型基于摘要回答。代价是：摘要丢的信息没法找回来，关键诊断可能就在没保留的那一行。

另一条路更激进：不在文字层面压缩，直接在 KV 这张「记忆表」本身上动手。能不能把六万行压成六百行，但让模型几乎察觉不到？过去两年这条路已经被走出了三派——扔掉不重要的行（H2O、SnapKV、KVzip）、合并相近的行（KVMerger）、以及最豪华的「把一张小表当参数训出来」（Cartridges）。

每一派都看到了前一派的伤疤。本篇 Attention Matching 站在所有人肩上，问了一个所有人都没问透的问题：*如果我们直接要求「压完后注意力的行为不变」，能不能避开训练，靠解一个小的最小二乘问题就把表压扁？*

* 演化叙事

** 2023 | StreamingLLM: 第一次承认「不能全留」

*** 它看到了什么问题

2023 年大家想让 LLM 跑无限长对话, 但 KV cache 涨一段就 OOM。最暴力的办法是滑动窗——只留最近 N 个 token, 老的扔掉。一扔, 模型立刻胡言乱语。

*** 它的解法

Xiao 那一组发现一个意外现象: 开头几个 token (即使语义无意义, 比如 "The") 在所有层里都收到极高的注意力——它们是 *注意力 sink*。只要保住这几个 sink + 最近的滑动窗, 模型就稳定。

*** 留下的坑

中间内容彻底丢失。这对纯生成 (chat) 够用, 但要做长文档问答, 关键事实就在被丢的中段。下一篇就是来救这点的。

** 2023 | H2O: 不是最近的, 是"打分高"的

*** 它看到了什么问题

StreamingLLM 默认"只有最近重要"。但 H2O 团队观察到: 在生成时, 注意力分数集中在少数 token——他们叫 *Heavy Hitters*。这些 token 可能在文档任何位置, 不一定靠近末尾。

*** 它的解法

prefill 完后, 累积每个 KV 收到的注意力分数, 保留 top-k。其余扔掉。结果在生成任务上比纯滑窗强不少。

*** 留下的坑

两个: (1) 这个排名是 query 相关的——后续 query 用同一份缩水 cache 就不再合适; (2) 「扔掉」这个动作本身有副作用——剩下的 KV 在 softmax 分母里要"代表"原来全部 token 的 mass, 否则 softmax 的尺度被低估。本篇 Attention Matching 后来正是从这里下刀。

** 2024 | SnapKV: 拿 prompt 尾巴当侦察兵

*** 它看到了什么问题

H2O 要看到 query 才能打分。但实际系统里, 我们想在 prefill 阶段就把 KV 压完——等用户问完再压, 延迟太大。

*** 它的解法

巧思: 用 prompt 末端的一小段 token (比如最后 64 个) 作为"代理 query", 用它们的注意力分数给前文 KV 打分。因为 prompt 通常已经暗示了任务方向, 这个代理打分跟真实 query 高度相关。

*** 留下的坑

仍然没解决"多 query"问题。同一份文档, 用户接连问"病人的诊断是什么"和"用药剂量是多少", 两个问题的相关 KV 完全不同。SnapKV 压一次就锁死了。

** 2024 | DuoAttention: 不同的头干不同的活

*** 它看到了什么问题

前面几个方法都在 token 维度上挑——决定哪些 token 留下。但 Xiao 这次的观察是: 注意力 *头* 才是关键。有的头是"检索头", 会跨长距离找信息; 有的头是"流式头", 只看附近几个 token。前者要全 cache, 后者只要滑窗。

*** 它的解法

学一个二值 mask: 每个头要么标"检索"(全保留) 要么标"流式"(只留 sink + 滑窗)。整体压缩率取决于流式头的比例。

*** 留下的坑

二值太粗——一个头要么全保要么砍到底, 没有"压成 30%"这种中间档。本篇会把这点变成连续旋钮: 每个头分配多少 KV 预算, 是一条连续的敏感度曲线决定的。

** 2025 | KVzip: 让模型复述原文, 复述好的就重要

*** 它看到了什么问题

前面所有方法都是 query 相关的——压完就只服务这一类 query。但实际部署中, 同一份长文档要被反复查询 (代码库、法律文档、医疗记录)。每次重压代价高, 不重压又不准。

*** 它的解法

KVzip 提了个聪明的 *query-agnostic* 评分: 让 LLM 用压缩后的 KV 试着 "复述" 原文 (next-token prediction)。如果某个 KV 对复述贡献大, 它就重要。一次压完, 任何后续 query 都能复用。

*** 留下的坑

KVzip 仍然是"扔 token"——在中等压缩率 (3-4x) 表现很好, NeurIPS oral 收了。但到了高压缩率 (50x、100x), 扔掉这么多, 无论评分多准都顶不住。本篇会展示: 在极高压缩率下, 任何"硬扔"方法都会塌方, 必须走 latent 路线。

** 2025 | Cartridges: 不挑了, 直接学一张小表

*** 它看到了什么问题

Eyuboglu 那组 (HazyResearch, Christopher Ré 实验室) 看到一个根本天花板: 不管你扔得多聪明, 你永远只能在 *原 KV 空间的子集* 里挑。但模型并没有理由相信"原来那几个 KV 向量"就是最优的压缩表示。

*** 它的解法

完全跳出"挑保留谁"的框架。借鉴 prefix-tuning: 把一张小的 KV 表 (比如 100 行) 当成可训练参数, 用合成的"自我学习"(self-study) 对话做 context distillation——让小表的输出去模仿原始长 context 的输出。

结果惊人: 50x 压缩还能基本追平原始性能, 远超所有 eviction 方法。他们管这张训出来的小表叫 *Cartridge* (墨盒, 像打印机墨盒一样, 一份文档配一个墨盒)。

*** 留下的坑

致命的: *慢*。每个 Cartridge 要几个 GPU-hour 训。如果文档相对静态 (一本书、一份代码库) 还能摊销, 但只要 context 经常变 (一次性查询、新到的邮件), 训成本远超推理省下的钱。本篇就是来啃这块硬骨头。

** 2026 | Attention Matching (Zweiger et al.): 解方程, 不训网络

*** 它看到了什么问题

回头看整条线, 本篇作者抓住了一个关键观察: Cartridges 用梯度训练这张小表, 是因为它把目标定成了"最小化下游 loss"。下游 loss 是非凸的, 必须梯度下降。但如果我们换一个 *更直接* 的目标——「让压缩后的注意力 *输出* 和 *质量* 在每一层每一头都尽量贴近原始」——这个目标可以分解成一堆线性最小二乘子问题, 每个都有解析解。

不需要梯度, 不需要训整个网络, 不需要几个 GPU-hour。几秒钟解几个线性方程组就完事。

*** 它的解法

每个 KV head 独立解, 三步:

1. *选保留哪些 keys*: 用 OMP (Orthogonal Matching Pursuit) 贪心选——每次加一个最能减小残差的 key——或者更快的"按注意力均值挑 top-k"。
2. *拟合一个 per-token bias β*: 解一个非负最小二乘 (NNLS)。这个 β 解决了 H2O/KVzip 那个"质量被低估"的老问题——因为扔掉了 T-t 个 token, 留下的 t 个 token 必须 *被加权* 才能在 softmax 分母里代表原来 T 个 token 的总 mass。
3. *拟合压缩后的 values*: 普通最小二乘 OLS 闭式解, 一步搞定。

再加一个"非均匀头预算": 每个头对压缩的敏感度不一样, 离线扫一次曲线, 给敏感的头多分预算, 不敏感的头多砍。这个预算分配是 *输入无关* 的——只跟模型架构有关——所以一次扫完永远复用。

*** 这招为什么管用

最深的洞察在 β 的引入: 之前所有 eviction 方法都默认"剩下的 token 用原来的注意力权重"。但 softmax 是个不可约分的非线性, 你不能简单地从 T 个里挑 t 个还指望分母不变。Attention Matching 显式承认这点——「我必须放大留下的 key 的影响力来弥补丢失的 mass」——并把这个放大系数 β 直接解出来。

结果: 在 QuALITY、LongHealth、QASPER、LongBench v2 上, 50x 压缩跟 Cartridges 同等质量, 但快 *100 倍*。一份 60k token 的病历在单卡 H200 上几秒到几分钟就能压完, Cartridges 要几个小时。

*** 留下的坑

- 解析解的代价是"逐头独立优化"——头之间的协同被忽略。Cartridges 因为是端到端训, 自动捕捉到了。
- self-study 这一步仍然要花时间生成参考 query (60k context 上要 139 秒)。如果想完全实时, 还得用更便宜的 "context-prefill" 模式, 质量会稍降。
- 跟摘要叠加: 文章发现先摘要再 AM-OMP 能做到 ~200x 总压缩。但这又把"摘要丢的关键信息"这个老问题搬回来了。

* 前沿延伸

截至 2026 年 5 月, 这条线还在快速迭代。可以观察的几个方向:

- *训练-免费 vs 训练- amortized 的角力*。Attention Matching 占了"快"的山头, Cartridges 占了"极致质量+可摊销"的山头。中间会不会出现一个"小步训练"——先用 AM 闭式解初始化, 再花几分钟梯度微调?
- *硬件协同*。AM 解出来的压缩 KV 在推理时与 FlashAttention 形式兼容, 而 Cartridges 的训练阶段对硬件要求很重。如果有专门为 "短 KV + per-head 不同预算" 优化的 kernel, AM 的延迟优势会再扩大。
- *online compaction*。本文已经在 AIME 上做了在线 (生成中压缩) 的实验。这条线如果走通, "无限长生成" 就不再只能靠滑窗——而是边生成边把老 KV 压成小表。

* 洞见

这条河里反复出现一个 *看似 vs 真正* 的张力。

KV 压缩最早的直觉是「挑出重要的 token 留下」——重要性这个概念听起来天然成立。但从 H2O 到 KVzip, 评分一代比一代精, 性能曲线在 50x 压缩率以上还是断崖。为什么? 因为「挑」这个动作本身, 隐含了一个错误的本体论假设: *原始 KV 向量是最优的压缩表示*。

Cartridges 第一次挑战了这个假设——别挑了, 训一张全新的表出来。结果做到了 50x 还不塌。但 Cartridges 用的工具 (梯度训练) 是为了对付 "下游 loss 不可解析" 才请出来的——它解决了表征问题, 却把目标函数也搞复杂了。

Attention Matching 干的事是: 既然你已经决定"训一张新表", 那目标函数其实不必是下游 loss。换成"让注意力行为不变", 这个目标可以拆成线性方程组。表征空间的革命 (Cartridges) + 目标函数的退化 (从下游 loss 退到 attention matching) = 几秒钟一张表。

更深一层: *评估口径决定方法选择*。如果你只看 perplexity, H2O 这类方法在中等压缩率上没问题; 一旦评估迁到长文档 QA、retrieval, 极高压缩率成为刚需, 整条 eviction 路线就走不下去。本篇之所以能存在, 是因为社区把评估推进到了 LongHealth、LongBench v2 这种「真正长文档、真正 QA」的场景——这些场景把所有方法的天花板都暴露了出来, 才腾出了 latent + 闭式解的位置。

下一步赌什么? 我赌 *混合栈*: 摘要 (粗压缩 10x) + Attention Matching (再压 20x) + 训一个极小的 Cartridge 给最热的查询 (再压 5x)。每一层用最便宜的工具达到它最擅长的压缩率。这跟存储层级 (寄存器 -> L1 -> L2 -> 内存) 的思路是一样的——没有单一最优, 只有按场景配栈。

* 启发

这条河给「怎么思考压缩问题」三个落地的启发:

1. *先问目标函数能不能换*。当一个领域的所有方法都在用同一个目标函数 (比如下游 loss + 梯度训练), 看看有没有一个 *更直接但更弱* 的代理目标 (比如本篇的"匹配注意力输出"), 它可能解析可解。退一步, 让目标本身变简单, 比死磕原目标的优化器更有杠杆。
2. *承认非线性的副作用*。softmax 的分母把所有 key 牵连在一起, 你不能简单地扔掉一部分还指望剩下的不动。任何"分组操作"前先想一想: *是不是有个守恒量被破坏了, 需要显式补回去?* 本篇的 β 就是补 softmax mass 守恒的。
3. *按场景配栈, 不追单点最优*。KV 压缩没有银弹。短 context 一次性查询 -> 用 Streaming/H2O 够用; 中等 context 多 query -> KVzip; 长 context 反复查询 -> Cartridges; 长 context 一次性 + 要快 -> Attention Matching。先想清楚自己的访问 pattern, 再选方法, 别被"最新论文"绑架。

落到能用: 如果你正在做长文档 RAG 或长 context 服务, 这周可以试一下 GitHub 上 adamzweiger/compaction 仓库——按本篇的报告, 在 Qwen3-4B / Llama-3.1-8B 上几分钟就能跑出 10-50x 压缩的 demo, 直接对比你现在的 baseline。
