* 问题之河

你想让一个大模型读完一本 30 万字的小说再回答问题。问题是：模型每读进一个 token，都要回头看一眼前面所有 token——这个"看一眼"的代价是 token 数的平方。1k token 还行，128k token 就要 30 分钟才能把题目读完。读完之后，每生成一个新 token，还得把前面所有 token 的"Key-Value 缓存"再翻一遍，显存里塞着 100 多 GB 的中间结果。

这就是长上下文的两个瓶颈：*预填阶段*（读题）的平方复杂度，和*解码阶段*（答题）的线性增长 KV 缓存。

为什么这事儿这么难？因为人人都知道注意力大部分是稀疏的——每个 token 真正关心的只有少数几个前文 token。但"少数几个"这件事，*每个 head 不一样*，*每个输入不一样*，*预填和解码两个阶段需求也不一样*。过去六年所有人都在跟这件事较劲：要么静态切一刀（丢精度），要么只管一边（另一边照样慢），要么动态选但要再训练一遍（成本高、不通用）。

TCA-Attention 这条线，本质是一条"如何把稀疏性做得既动态、又免训、又两头通吃"的演化史。

* 演化叙事

** 2017 | Transformer：先种下平方复杂度的根

*** 它看到了什么问题

RNN 太慢了。一个 100 词的句子要 100 步串行才能算完，没法在 GPU 上并行。LSTM 又老忘掉前面的内容。

*** 它的解法

干脆把"按时间步走"这件事彻底扔了。每个 token 一次性看遍所有其他 token，算一个 softmax 加权和。这就是 self-attention。

*** 为什么这招管用（以及哪里埋了新坑）

并行起来快得飞起，长距离依赖也不再衰减。但坑也大：n 个 token 互相看一遍，是 n × n 次计算。512 长度还好，128k 长度直接灾难——A100 单卡上跑 100 万 token 的预填要 30 分钟。这一刀，把后面六年所有"高效注意力"的研究都逼出来了。

** 2023.09 | StreamingLLM：发现"注意力槽"，但只敢扔掉中间

*** 它看到了什么问题

最朴素的想法：既然要省钱，那就只留最近 N 个 token 的 KV，老的扔掉，做成"滑动窗口"。结果发现一旦扔掉最开头的几个 token，模型 perplexity 直接爆炸到 5158，等于胡言乱语。

为什么？因为 softmax 必须把注意力之和分配到 1，如果当前 query 真没什么想看的，模型就把多余注意力"倾倒"到最开头几个 token 上——这几个 token 像情绪垃圾桶，作者管它叫 *attention sink*。一旦扔掉，模型就崩。

*** 它的解法

也很朴素：永远留住开头 4 个 token（attention sink）+ 最近的滑动窗口。剩下中间的全扔。

*** 为什么这招管用（以及哪里埋了新坑）

只要保住情绪垃圾桶不被砸了，模型就稳。LLaMA-2 这套搞下来能稳定跑 400 万 token，比重新计算快 22 倍。

但坑也明显：*中间的内容真的不要了？* 你让它读 128k token 的合同回答中间的条款，对不起，中间被你扔了。这是个 streaming 方案，不是 long-context understanding 方案。而且每个 head 都按同样的 "sink + 窗口" 模式被切，所有 head 一刀切。

** 2023 | H2O：能不能动态地"留住该留的"？

*** 它看到了什么问题

StreamingLLM 那刀切得太硬——位置决定一切，不管这个 token 内容是不是重要。但实际上，*少数 token 占了多数注意力质量*。这些 token 不一定在开头，可能在中间任何位置。

*** 它的解法

边解码边算每个 token 的累积注意力分数，谁分数高谁就是 "Heavy Hitter"（重击者）。预算固定（比如留 20%），每次只留下重击者 + 最近的若干 token。

*** 为什么这招管用（以及哪里埋了新坑）

吞吐量直接比 HuggingFace Accelerate 提了 29 倍，精度也基本保住了。

但 H2O 是个*解码阶段*的优化——它在生成时一边算一边淘汰。预填阶段（你刚把 100k token 题目灌进去的那一刻）它压根没碰，O(n^2) 的计算照旧。所以题目越长，你越是只优化了"答题"，"读题"那一段还是一样慢。

** 2024.04 | SnapKV：在读完题的瞬间，让 prompt 自己告诉你哪些 token 重要

*** 它看到了什么问题

H2O 要等到生成时才知道谁是重击者。可如果 prompt 本身就有 100k token，你*预填都还没做完*，怎么决定 KV cache 留谁？

但作者注意到一个关键观察：在 prompt 的*末端*，attention 已经稳定下来，每个 head 关心的 prefix token 已经成型。也就是说——*提前问一下 prompt 末尾的几个 query，让它们投票选 prefix 里谁重要*，就能在预填一结束、解码开始之前，把 KV 压缩好。

*** 它的解法

在 prompt 末尾开一个小"观察窗"（observation window，几十到上百 token）。用这个窗口里每个 query 对前文的注意力分数做池化、投票，给每个 head 选出它各自关心的"重要 token cluster"。压缩在每个 head 内部独立做。

*** 为什么这招管用（以及哪里埋了新坑）

生成速度提升 3.6 倍，显存效率 8.2 倍，128k token 的合同也能在单张 A100 上跑。在 Mistral-7B 上，SnapKV 用 1024 的 KV 预算就打过了 H2O 用 4096 预算的精度。

但还是有两个坑：第一，SnapKV *依然没加速预填本身*，它只是在预填做完之后压 KV；第二，*每个 head 的预算是统一的*——你给所有 head 都分 1024 个 token，但有些 head 其实只需要 200，有些 head 需要 2000，这种异质性 SnapKV 抓不住。

** 2024.07 | MInference：把预填本身也加速——但只认三种固定模式

*** 它看到了什么问题

SnapKV、H2O 都是"预填完了再压"。MInference 直接质问：*能不能在预填的时候就跳过那些用不上的注意力？* 也就是说，128k × 128k 的 attention 矩阵，能不能只算其中 4k 列？

作者一测：在长上下文里，只算 top 4k 列居然能召回 96.8% 的注意力质量。稀疏性是真的。

*** 它的解法

把所有 head 的稀疏模式归纳成三种典型形状：

- *A-shape*：开头一段 + 对角线（attention sink 的几何形态）
- *Vertical-Slash*：少数全局活跃的列 + 几条斜线
- *Block-Sparse*：成片的稀疏块

每个 head 离线决定它属于哪一类。推理时，先用低秩估计快速预测稀疏位置，然后用专门的 GPU kernel 只算这些位置。

*** 为什么这招管用（以及哪里埋了新坑）

预填阶段提速 10 倍，1M token 的 LLaMA-3-1M 也能跑得动。它是第一个真正动了预填本身的方法。

但坑还是有：*三种模式是手工设计的*，不是从数据里长出来的，遇到"四不像"模式它就没法。而且 MInference 完全不管 KV cache，解码阶段它一点忙都帮不上。*预填快了，解码还是慢；显存还是会爆*。

** 2024.10 | DuoAttention：把 head 分类，两边都加速——但要训

*** 它看到了什么问题

到这里大家发现一个无奈：*预填和解码总是被分开优化*。MInference 管前者，SnapKV / H2O 管后者，没人能两头都管。

DuoAttention 的洞察是：*head 本身就分两类*。一种叫 *retrieval head*（检索头），它真的需要看遍全文才能完成任务（比如做 needle-in-haystack）；另一种叫 *streaming head*（流式头），它的注意力其实就集中在 sink 和最近窗口，跟 StreamingLLM 描述的那种模式一模一样。

如果能把这两类头识别出来——检索头给它全 KV，流式头给它压缩 KV——就同时省了显存和算力。

*** 它的解法

用合成数据 + 一个可学习的门控值给每个 head 打分，让模型自己说哪些头是检索头、哪些是流式头。识别出来之后：
- 检索头：保留完整 KV cache + 完整 attention
- 流式头：只存 sink + 最近窗口的 KV，attention 只算这一小块

这样预填和解码两个阶段都加速：长 prefix 在流式头上根本不展开，KV 也不存。

*** 为什么这招管用（以及哪里埋了新坑）

显存最多省 2.55 倍，预填和解码都提速 1.5-2 倍，配合量化能在单张 A100 上跑 330 万 token。它是第一个真正做到"两阶段 + head-aware"的方案。

但留了一个大坑：*要训*。识别 retrieval head 这一步需要合成数据 + 优化门控值，对每个新模型都得重训一遍。而且训出来之后，每个 head 的"角色"是固定的——这个头到底是检索头还是流式头，跟具体输入无关。换句话说，它做到了 head-aware，但*没做到 context-aware*：同一个 head 处理代码和处理小说应该有不同的稀疏度，DuoAttention 给不出。

而且作者后来发现，DuoAttention 的官方实现里，预填阶段实际上仍然走 FlashAttention-2 全注意力——所谓"预填加速"是因为流式头预存了短 KV，但 attention 计算本身没省下来。

** 2025.12 | TCA-Attention：免训 + 两阶段 + head-aware + context-aware（本文）

*** 它看到了什么问题

到 DuoAttention 这一步，所有维度都被人摸过了，但没人能同时占齐这四样：

| 维度          | StreamingLLM | H2O | SnapKV | MInference | DuoAttention |
|---------------|--------------|-----|--------|------------|--------------|
| 加速预填      | -            | -   | -      | +          | 部分         |
| 加速解码/压KV | +            | +   | +      | -          | +            |
| Head 异质     | -            | -   | 部分   | -          | +            |
| 内容自适应    | -            | +   | +      | +          | -            |
| 免训          | +            | +   | +      | +          | -            |

TCA 想要全占。它看到的核心问题：*head 异质性* 和 *内容自适应* 是分裂在两个阵营里的——SnapKV/MInference 那一系做了内容自适应但 head 一刀切，DuoAttention 做了 head 异质但模式固定。能不能不训练就把两边都占了？

*** 它的解法

两阶段解耦：

1. *离线标定（一次性）*：跑一遍前向。对每个 head，用 log-Gaussian 采样一堆候选稀疏度配置，挑出"保留下来的注意力质量超过阈值 τ"的最稀疏那个。这给每个 head 算出一个*个性化预算*——不用训，只跑一次。

2. *在线核心上下文选择（每次推理）*：用 prompt 的*最后一个 query token* 作为"context summarizer"——只用它一个 query 去衡量 prefix 里每个块的重要性，把代价从 O(k·L) 砍到 O(L)。然后用一个轻量的冗余度指标（注意力质量 + 一个 Herfindahl-Hirschman 集中度项）把这个 head 的预算动态分配到各个 block 上。

预填和解码都用同一套机制。免训。

*** 为什么这招管用（以及哪里埋了新坑）

128k 上下文下，预填提速 2.8 倍，KV cache 减少 61%，精度跟 full attention 持平。在 LongBench-E 和 RULER 上比 XAttention 高 2.6 分。

更狠的是，作者证明了核心上下文选择带来的逼近误差有个明确的上界 2γ·‖V‖∞，γ 是被丢掉那部分注意力的质量，τ 直接控住它。换句话说——稀疏度和误差之间，*有解析关系，不是经验拍脑袋*。

新坑在哪？两个：

一是*离线标定的"一次前向"用的是什么样本？* 如果标定样本和真实推理样本的稀疏模式分布差太多，预算会算偏。论文用的是 LongBench 子集做标定，跨任务泛化性需要更多验证。

二是*用最后一个 query token 做 context summarizer 是个赌注*。它假设最后一个 query 已经能代表"我需要看哪儿"，这在问答场景成立，在多轮对话或多 instruction 场景下未必。

* 前沿延伸

TCA-Attention 之后，*自适应预算分配*这条线正在快速演化：

- *Ada-KV* (NeurIPS 2025) 第一次系统地论证了 head-wise 预算分配的理论价值，被后续大量工作集成（包括 NVIDIA 的 kvpress 框架）。它和 TCA 的差别在：Ada-KV 仍以"给定全局预算 B，怎么分到各 head"为问题，TCA 则是"给定误差容忍度 τ，每个 head 应该用多少预算"——后者把预算决定权也交给了内容。

- *GVote* (ICLR 2026) 进一步挑战"top-down 切蛋糕"的范式：与其先定 B 再切，不如用 Monte Carlo 采样合成 query 投票，让"该留多少"自然涌现。这是 TCA "免训 + 内容自适应"路线的延伸版本。

- *DynamicKV* (2024.12) 在 layer 维度做了类似 TCA 在 head 维度做的事——每个 layer 的预算也是动态的、任务自适应的。下一步合流方向很清楚：layer × head × context 三维联合自适应。

* 洞见

读完整条线，能看到三件事：

*第一，"稀疏注意力"这个问题不是技术问题，是认知问题。* 六年来真正改变的不是算法本身——top-k、滑动窗口、池化打分这些手法 2017 年就有——而是研究者对"哪一种异质性该被建模"的认知不断加深：

#+begin_example
  位置异质 (StreamingLLM) -> 内容异质 (H2O/SnapKV)
   -> head 异质 (DuoAttention) -> head x 内容联合异质 (TCA)
#+end_example

每一步都是把一个"原来被合并的维度"拆开。TCA 是这条线的一个收敛点：它把四个维度的异质性同时放开了。

*第二，"免训"是个非常硬的工程约束，远比看起来重要。* DuoAttention 在 head 异质上做得很彻底，但因为要训，每个新模型都要重训一遍，部署摩擦极大。TCA 用一个"一次前向标定"把训这步绕掉了——这看似是个工程细节，但实际上把这条技术能不能进入生产推理系统的门槛降了一个数量级。社区里 NVIDIA kvpress、vLLM 等正在快速集成这一类免训方法，不是偶然。

*第三，下一步几乎一定往"误差控制"走，而不是往"更巧的模式"走。* TCA 证明了稀疏度和逼近误差有解析关系（2γ·‖V‖∞ 那个界）。一旦这个口子被打开，后续工作必然会沿着"给定下游任务误差容忍度，反推稀疏度"这条路走，而不是再去发明新的固定模式。GVote 已经在做这件事。

* 启发

这意味着你可以：

1. *看到一个"还在用固定预算/固定模式"的注意力优化论文，立刻知道它已经过时一个版本*——下一步必然是 context-adaptive。

2. *在自己做长上下文系统时，优先考虑 TCA 这类"免训 + 双阶段"的方法*——它对生产环境最友好，不会因为换模型就要重训。

3. *把"该留多少"这件事从超参变成可推导的量*。一旦你的 sparsity 方法跟"误差界"挂钩，调参就从黑魔法变成了"给我一个 τ，给你一个保证"——这是工程上质的飞跃。

4. *识别同类问题的演化模式*——任何一个"高效化"的研究线（量化、剪枝、KV 压缩、MoE 路由）都会经历类似的"位置异质 -> 内容异质 -> 多维联合异质 -> 误差可控"的演化。看到一个新方向，先问它现在卡在哪一步，你就知道下一步该做什么。
