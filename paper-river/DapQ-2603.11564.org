* 问题之河

模型读了一百万字的上下文，然后开始一个字一个字往外吐。问题是：这一百万字对应的中间状态（KV cache）塞在显存里，每多读一个字就多一份。怎么扔掉一部分、又不让模型变笨？

这个问题难，是因为「哪些字重要」根本不是一个固定答案——它取决于你接下来要生成什么。可你还没生成呢，你怎么知道生成什么时候会回头看哪个字？这就是这条研究线一代一代人在绕的弯子。

* 演化叙事

这条线的主角不是某一篇论文，是一个反复出现的盘问：「在不知道未来要生成什么之前，怎么知道哪些 token 值得留下」。每一代人都在用越来越接近"未来"的代理去回答它。

** 2023 | StreamingLLM: 第一次承认 "全留" 不可能

*** 它看到了什么问题

最朴素的做法是滑窗 —— 上下文太长就只看最近 N 个 token。但 Xiao 等人发现：一旦把开头的几个 token 也滑出去，模型立刻崩盘，生成出来的东西完全是胡言乱语，困惑度爆炸。

为什么？因为 attention 的 softmax 是归一的，必须把概率分到某些地方。哪怕语义上不重要，开头那几个 token 也被模型当成了"概率的垃圾桶"——它把那些不知道该给谁的注意力都倒在那里。这几个位置叫 *attention sink*。

*** 它的解法

简单粗暴：永远留住开头的 4 个 token，再加一个滑窗。一个 token 是否被保留，看的是它的位置（开头/最近），不是它的内容。

*** 为什么这招管用（以及哪里埋了新坑）

它管用，因为发现了一个真实物理现象（attention sink）。但它的视角是 *位置* 而不是 *内容*。问题是：长文档里那个埋在中段的 needle，按位置算它什么都不是，可一旦你的问题指向它，它就是全场最重要的字。下一代必须能看内容。

** 2023 | H2O: 用注意力分数选 "重头戏"

*** 它看到了什么问题

StreamingLLM 用纯位置选 token，会冤枉中段的关键信息。Zhang 等人想：既然 attention 矩阵本身就告诉我们哪些 token 被关注得最多，为什么不直接用它来选？

*** 它的解法

Heavy Hitter Oracle —— 把整段 prompt 里所有 query 看每个 token 的 attention 全加起来，分数最高的那些叫"重头戏"（heavy hitters），留下；其余的扔。

*** 为什么这招管用（以及哪里埋了新坑）

它管用，因为终于让"重要性"由数据自己说话。但坑在于："所有 query 的累积 attention" 是个混乱的平均量 —— prompt 里有问句、有引用、有铺垫，它们关注的东西彼此抵消。更糟的是，因为 LLM 是单向 mask 的，越靠后的 token 被越少 query 看到，累积分数天然偏低，结果反而是 *最近的 token 被冤枉淘汰*。下一代必须想清楚：到底用谁的 query 来投票？

** 2024 | SnapKV: 只让 "最后几个 token" 投票

*** 它看到了什么问题

H2O 让全 prompt 投票，结果是各种意图被平均掉。Li 等人发现：其实每个注意力头在生成时关注什么，是高度稳定的——这个模式从 prompt 末尾的 16-32 个 token 就能看出来。

换句话说：你不需要全场投票，你只需要让"和未来生成最像"的那一小撮 query 来投票。

*** 它的解法

在 prompt 末尾划出一个 observation window（典型 32 个 token），用这个窗口里的 query 去算和前面所有 KV 的 attention，pooling 之后取 top-k 保留。简洁、有效，立刻把同等预算下的性能拉高一大截。

*** 为什么这招管用（以及哪里埋了新坑）

它管用，因为找到了一个 *代理*：用 prompt 末尾的 query 代理未来 decoding 的 query。但这个代理是个静态快照 —— prompt 末尾的 query 是在描述输入的尾巴，而真正的 decoding query 是在生成回答。两者结构性地不一样，尤其在需要"跨段回忆"的复杂任务上，代理失真。下一代必须问：能不能让代理更像真的？

** 2024 | PyramidKV: 不同层用不同预算

*** 它看到了什么问题

SnapKV 是横向的——选哪些 token。Cai 等人换一个轴想：纵向上，Transformer 的每一层注意力模式差别巨大，浅层广而散，深层窄而锐。给每层分一样多的预算，是浪费。

*** 它的解法

金字塔形预算 —— 浅层多留，深层少留，跨层总预算守恒。observation window 的核心思路不变，只是预算分配换了。

*** 为什么这招管用（以及哪里埋了新坑）

它管用，因为承认了 Transformer 内部的异质性。但它没有动 observation window 本身的对齐问题 —— 用的依旧是 prompt 末尾那几个 query，与真实 decoding 仍然脱节。问题依旧悬在那里：怎么让"投票人"真的像未来的"投票人"？

** 2025 | LAQ++: 干脆先吐一段假回答出来

*** 它看到了什么问题

Wang 等人把矛头直指 SnapKV 的根本缺陷：你的 observation window 在 prompt 这一侧，可你要服务的是 decoding 那一侧。两侧位置不同、上下文不同，没理由相信它们的 query 是对齐的。

*** 它的解法

Lookahead —— 在真正开始生成之前，先让模型自己 *预生成一段假回答*（pseudo response）。这段假回答里的 query，比 prompt 末尾的 query 在分布上离真实 decoding 近得多。用这段假回答的 query 作 observation window，再去选哪些原始 KV 该留。

*** 为什么这招管用（以及哪里埋了新坑）

它管用，因为终于把代理从"输入侧"挪到了"输出侧"。但坑很现实：你为了得到这段假回答，必须先做一遍完整 prefill，于是 KV cache 在被压缩之前 *先膨胀到了最大*。显存峰值反而比不压缩还高，把整个方案在严苛显存约束下的实用价值削掉了一大半。下一代必须问：能不能不真的生成，就拿到一段"和真生成等效"的 query？

** 2026 | DapQ: 假 query 不需要假内容，只需要假位置

*** 它看到了什么问题

DapQ 接住了 CaliDrop 的一个观察：相邻位置的 query 彼此非常相似。Tian 等人顺着这个线索做了个反直觉的拆解实验 ——

把 query 的两个属性拆开看：内容 (content) 和 位置 (positional encoding)。然后做四象限对比：

| 条件                        | 与真 query 的余弦相似度 |
|-----------------------------+-------------------------|
| 内容对+位置对 (SC&SP)       | 1.0000 (上界)           |
| 内容对+位置错 (SC&DP)       | 0.3522                  |
| *内容错+位置对 (DC&SP)*    | *0.7238*               |
| 内容错+位置错 (DC&DP)       | 0.3267                  |

读这张表读三遍 —— 内容完全是胡说八道（"Sorry, I don't know"循环），但位置编码对了，相似度立刻飙到 0.72；反过来内容完美但位置错，相似度只有 0.35。结论惊人：*在 query 长什么样这件事上，"它处在序列的哪里"比"它说了什么"重要得多。*

这意味着 LAQ++ 那一整套"先生成假回答"的复杂工程其实是过度设计——你根本不需要生成真实内容，你只需要 *把正确的未来位置编码注入到一段随便填的 token 上*。

*** 它的解法

在 prompt 后面拼一段长度 N 的 *合成 token*（内容随便，比如重复填充），让它们处在未来 decoding 会用到的位置 [L_p, L_p+N)。

走一遍 prefill。这段合成 token 的 query 就是 *position-aware pseudo queries* —— 内容是假的，但位置编码是真的未来位置。

用这些 pseudo query 与原始 prompt 的 K 算注意力，得到重要性分布，top-k 保留，剩下的（包括所有合成 token 自己）一并扔掉。然后从位置 L_p 开始真正的 decoding。

*** 为什么这招管用（以及哪里埋了新坑）

管用，因为它把"代理"的颗粒度从"造一段假文本"降到了"造一组假位置"，避开了 LAQ++ 的显存峰值——合成 token 的 KV 在 eviction 那一步就被一起丢掉，不占长期空间。在 3% KV 预算这种极端约束下，NIAH 上达到 99.5% 的近无损表现。

埋下的新坑也清晰：第一，pseudo query 长度 N 是个超参数，太短信息不够，太长又会偏离；第二，这一切的前提是 RoPE 这一类位置编码确实主导了 query 的几何结构 —— 换一种位置编码方案（比如某些 ALiBi 变体或学习型位置嵌入），"位置 > 内容"是否还成立，是开放问题。

* 前沿延伸

DapQ 是 2026 年 3 月的论文，截至本文撰写时还没有公开的后续直接批判它的工作。同窗口期可看到的相邻方向：

- *KV 合并 vs KV 淘汰*：Tian 等人自己 2025 年的 KV merging 工作把"扔"换成"合"，跟 DapQ 思路上是互补而非对立——pseudo query 选出"留"的，被淘汰的那些可以试着合并而不是直接丢；
- *Decoding-time 动态淘汰*：一类工作（如 G-KV、SnapKV-D）让 observation window 随生成一起滑动，每生成 w 个 token 就重新选一次。DapQ 是 prefill-time 一次性决策，二者可能融合：先用 pseudo query 做粗筛，再用真实滑窗做细调；
- *与 quantization、low-rank 的正交叠加*：DapQ 减少 token 数量，量化减少每个 token 的字节数，MLA 类方法压缩维度，三者机制独立，预期可乘性叠加。

整条线的"螺旋"很有意思——StreamingLLM 用位置，被嫌粗糙；中间三代越走越往"内容"靠；到 DapQ 一个回马枪，发现 *决定 query 的不是它说什么，而是它在哪儿*，又把核心负担甩回位置编码上。但这次的位置不再是"前 4 个" 这种粗启发，而是"未来 decoding 时该出现的精确位置"——一种被精细化、被内化进 RoPE 几何结构里的位置。

* 洞见

读完整条线，真正在发生的变化不是"找更好的代理"，而是 *对代理是什么的认知在被反复重写*：

1. 第一代以为代理是"位置"——直觉但盲；
2. 第二代以为代理是"被关注的程度"——数据驱动但混乱；
3. 第三代以为代理是"prompt 末尾的 query"——更聚焦但错位；
4. 第四代以为代理是"假生成的 query"——对齐但昂贵；
5. *第五代发现：代理其实只是 query 几何位置的一个投影，内容是噪声项。*

这个反转的意义不止在 KV cache 这一个题目上。它在隐喻一件更大的事：很多 Transformer 现象我们一直归因给"语义"，实际上是 *位置编码的几何结构* 在主导。RoPE 不只是一个旋转矩阵，它在 query/key 空间里塑造了一个隐藏的"位置流形"，注意力模式很大程度上沿着这个流形走。

下一步最可能往哪走？两个方向：
- *理论方向*：把"位置主导 query"形式化。在什么条件下这个等价成立？换 ALiBi、换 learned positional embedding 会怎样？
- *工程方向*：既然位置就够了，那 KV 淘汰、推测解码 (speculative decoding)、prefix sharing 这些场景的"代理 query"是否都能用同一套位置工具搞定？

* 启发

"做研究就是找问题"这句话太空，DapQ 这条线给的具体启发是：*当一个领域陷在工程复杂度爆炸时，回去拆解被默认捆绑在一起的概念*。

LAQ++ 把"未来 query"和"未来内容"绑在一起，因此必须真生成。DapQ 拆开这两个概念，发现内容那部分是冗余，于是工程立刻塌缩成一个非常轻量的方案。

落点在能用：这意味着你可以
- *在面对一个看起来不可避免的工程开销时*，先问"我真的需要这个开销背后的两个量都准确吗，还是其中一个其实不重要？"——做一组消融，看是哪个维度在挑大梁；
- *在 LLM 推理优化场景中*，凡是涉及"模拟未来"的代理（speculative decoding 的 draft model、推测重要性的 oracle、prefill 阶段的预测器），都值得用同样的"内容 vs 位置"四象限拆一遍——很多时候你会发现某一维度可以被替换成零成本的合成信号。
