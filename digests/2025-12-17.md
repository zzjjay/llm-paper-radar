### 1. CTkvr: KV Cache Retrieval for Long-Context LLMs via Centroid then Token Indexing (8/10) 🔁

🧪 kv_cache · centroid-then-token KV retrieval, two-stage dynamic KV selection · Yi-9B · cal: 预填充阶段预计算centroids，无需额外校准数据 · perf: 96K context下Llama-3-8B和Yi-9B分别实现3x和4x吞吐量提升

#### 🧭 为什么推送这篇
**KV cache 压缩** · topic 4/5 · practicality 4/5 · composite 8/10 · 该论文提出CTKVR，一种针对长上下文LLM的KV cache检索方案，采用两阶段centroid-then-token索引策略平衡检索效率与精度。方案利用RoPE后相邻query向量相似性这一关键观察，通过CPU-GPU协同执行优化系统实现。在96K context长度下，Llama-3-8B和Yi-9B分别实现3x/4x吞吐量提升，精度损失<1%，具有清晰的部署价值。但实验模型规模较小（最大9B），且benchmarks名称未具体说明，评估范围有限；算法本身需要自定义CPU-GPU co-execution系统，工程复杂度中等，部署需额外工作。整体属于kv_cache bucket中实用性较强的工作。

#### 摘要
针对长上下文 LLM 推理中 KV cache 内存与访存开销大的问题，提出 CTKVR 两阶段 KV 检索方案。基于观察：RoPE 后位置相邻的 query 向量相似度高、top-k KV 条目大量重叠，prefilling 阶段预计算轻量级 centroid 做粗粒度索引，再做 token 级精化检索，避免 block 级索引的精度损失与 token 级索引的延迟开销。配合 CPU-GPU 协同执行优化索引构建与搜索。在 Llama-3-8B 和 Yi-9B 96K 上下文下分别取得 3× 和 4× 吞吐加速，精度损失 <1%。

- 💡 RoPE 后相邻 query 共享大部分 top-k KV 条目
- 🎯 centroid 粗索引 + token 级精化两阶段检索
- 🔧 CPU-GPU 协同执行索引构建与搜索
- 📊 Llama-3-8B/Yi-9B 96K 下 3×/4× 加速，精度损失 <1%

#### 📎 相关方法 / 对比基线
- block-level KV indexing — 粗粒度检索引入无关项，精度下降
- token-level KV indexing — 检索机制低效，延迟过高

#### 摘要
