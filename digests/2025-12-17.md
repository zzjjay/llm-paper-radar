#### 🧭 为什么推送这篇
**KV cache 压缩** · topic 4/5 · practicality 4/5 · composite 8/10 · 本文提出CTKVR，一种两阶段KV cache检索方案：先用轻量centroid做粗粒度索引，再做token级精细检索，利用RoPE后相邻query相似性减少无关KV读取。目标场景是长上下文推理（96K），在Llama-3-8B和Yi-9B上实现3-4倍吞吐提升，精度损失<1%。方法采用CPU-GPU协同执行优化检索系统，部署路径清晰。不足：最大测试模型为9B，评测benchmark描述模糊，且kv_cache eviction/retrieval在主流serving框架中集成难度中等。整体属于实用性较强的KV cache压缩工作。

#### 摘要
针对长上下文 LLM 推理中 KV cache 内存与访存开销过大的问题，CTKVR 提出 centroid-then-token 两阶段 KV 检索方案，解决 block-level 索引精度差、token-level 索引延迟高的矛盾。基于观察——RoPE 后位置相邻 query 向量高度相似且共享大部分 top-k KV entries，在 prefilling 阶段预计算轻量 centroids 做粗粒度索引，再做 token 级精排。配合 CPU-GPU 协同执行优化索引构建与搜索，在 96K 上下文下 Llama-3-8B 和 Yi-9B 分别取得 3× 和 4× 吞吐加速，精度下降 <1%。

- 💡 利用 RoPE 后相邻 query 共享 top-k KV 的特性
- 🎯 centroid 粗检 + token 精排两阶段索引
- 📊 96K 上下文 Llama-3-8B/Yi-9B 提速 3×/4×
- 🔧 CPU-GPU 协同执行索引构建与搜索

#### 📎 相关方法 / 对比基线
- block-level indexing — 检索无关 KV 导致精度下降
- token-level indexing — 检索机制低效导致高延迟
