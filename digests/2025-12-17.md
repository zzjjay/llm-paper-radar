### 1. CTkvr: KV Cache Retrieval for Long-Context LLMs via Centroid then Token Indexing (8/10)

#### 🧭 为什么推送这篇
**KV cache 压缩** · topic 4/5 · practicality 4/5 · composite 8/10 · 本文提出CTKVR，一种两阶段KV cache动态检索方案，通过centroid粗粒度索引加token级别精细化检索，兼顾检索效率与精度。方法基于RoPE后相邻query向量高度相似的观察，利用CPU-GPU协同执行加速索引构建与搜索，实现长上下文场景下显著的吞吐量提升（3-4x）。评估模型最大为Yi-9B，属于kv_cache类别，准确率损失<1%，具备较强实用性，但最大测试规模偏小（<70B），评估benchmark细节未在摘要中明确列出。

#### 摘要
针对长上下文 LLM 推理中 KV cache 内存开销大和访存延迟高的问题，CTKVR 提出 centroid-then-token 两阶段 KV 检索方案：利用 RoPE 后位置相邻 query 向量高度相似且 top-k KV 重合的特性，在 prefilling 阶段为 KV 预计算轻量级 centroid 做粗粒度索引，再在 token 级做精细检索，避免 block-level 索引的精度损失和 token-level 索引的延迟开销。系统实现采用 CPU-GPU 协同执行优化索引构建与搜索。在多个 benchmark 上精度损失小于 1%，Llama-3-8B 和 Yi-9B 在 96K 上下文下分别取得 3× 和 4× throughput 加速。

- 💡 RoPE 后相邻 query 共享大部分 top-k KV
- 🎯 centroid 粗检索 + token 级精检索两阶段
- 📊 96K 上下文下 Llama-3-8B/Yi-9B 提速 3×/4×
- 📊 精度损失 <1%，CPU-GPU 协同执行

#### 📎 相关方法 / 对比基线
- block-level indexing — 检索无关 KV 导致精度下降
- token-level indexing — 检索机制低效导致高延迟
