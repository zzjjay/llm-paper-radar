🧪 pruning_distill · unstructured pruning, SparseGPT-variant, Hessian-based reordering · LLaMA2-70B · cal: one-shot，需预剪枝估计列/块损失，额外开销较小 · perf: 摘要未报告端到端推理加速数据

#### 🧭 为什么推送这篇
**Pruning / 蒸馏** · topic 4/5 · practicality 3/5 · composite 7/10 · 本文在SparseGPT框架下提出列/块两级重排序策略（ROSE），通过优先剪除潜在误差更大的权重改善非结构化剪枝质量，在LLaMA2-70B等主流LLM上验证了效果。属于标准one-shot非结构化稀疏剪枝，对pruning_distill bucket有相关性。主要限制是：(1) 非结构化稀疏在vLLM/TRT-LLM上缺乏直接加速内核，实际部署路径不明确；(2) 摘要仅报告PPL/zero-shot，缺乏端到端speedup数据；(3) 算法相较SparseGPT增加了预剪枝+两级重排序步骤，实现复杂度有所提升。代码开源，测试规模达70B，topic relevance尚可但practicality受制于非结构化稀疏的部署限制。

#### 摘要

- 💡 指出 SparseGPT 固定剪枝顺序在 columnar 权重下次优
- 🎯 两级重排：块内按 column loss、块间按 block loss
- 🔧 用 block loss relative range 自适应识别 columnar layers
- 📊 在 LLaMA2/3、Mistral 上稳定优于 SparseGPT

#### 🧭 为什么推送这篇
**Survey / 方法论与对比** · topic 4/5 · practicality 3/5 · composite 7/10 · 本文是一项关于prompt compression（输入token压缩）的实证研究，属于survey bucket：提出了instruction survival probability（Psi）和Compression Robustness Index（CRI）两个新的度量框架，并通过5,400次API调用在多个benchmark上量化不同压缩策略下的输出动态和能耗影响。这些发现对部署压缩策略的工程师具有直接指导意义——单一benchmark可能产生误导性结论，结构感知的压缩策略更为可靠。然而，本文针对的是prompt-level（输入侧）压缩而非模型权重/KV cache压缩，与核心LLM量化/剪枝/KV压缩方向略有偏离，且不涉及新的压缩算法，实用性受benchmark多样性要求限制。

#### 摘要
针对 prompt compression 评估只看 input token 压缩比、忽视输出膨胀和真实推理成本的问题，作者在 3 个 benchmark、多 provider 上做了 5400 次 API 调用的复现与扩展研究。提出 instruction survival probability (Psi) 量化关键 prompt 段落在截断后的保留度，发现 benchmark 结构是输出膨胀的主因：在压缩比 r=0.3 下 DeepSeek 在 MBPP 输出膨胀 56 倍 (Psi≈0.15)，但 HumanEval 仅 5 倍 (Psi≈0.72)，GPT-4o-mini 则相对稳定。进一步提出 Compression Robustness Index (CRI) 用于跨 benchmark 评估，并通过 RunPod GPU 的 NVML 直接能耗测量指出 token 节省会高估 joule 节省。

- 📊 r=0.3 时 DeepSeek 在 MBPP 输出膨胀 56x，HumanEval 仅 5x
- 💡 提出 Psi 指标量化任务关键 prompt 段落保留率
- 🎯 提出 CRI 跨 benchmark 压缩鲁棒性指标
- ⚠️ NVML 实测显示 token 节省高估实际能耗节省
