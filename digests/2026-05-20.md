#### 🧭 为什么推送这篇
**PTQ（训练后量化）** · topic 5/5 · practicality 4/5 · composite 9/10 · TORQ是针对MXFP4激活量化的训练后优化方法，通过两层正交旋转解决激活分布与MXFP4块浮点格式的结构不匹配问题。工作在主流LLaMA3和Qwen3上进行了充分评估，精度恢复效果显著（Qwen3-32B MMLU从38.40%→73.63%，接近BF16的74.82%）。该方法无需重新训练，仅需简单的线性变换（Schur-Horn定理和最大熵指导旋转），算法优雅且易于部署。MXFP4是NVIDIA下一代GPU（如Blackwell）的原生格式，具有真实硬件支持和商业部署前景。缺点是未报告具体的端到端推理加速数值和校准成本详情，但作为PTQ方法这已属可接受范围。

#### 摘要

- 🎯 方法：两级正交旋转（inter-block Schur-Horn 旋转 + intra-block 最大熵旋转）解决 MXFP4 activation 量化的两类结构性失配
- 📊 结果：Qwen3-32B 上平均精度从直接 RTN 的 38.40% 提升至 73.63%，接近 BF16 的 74.82%
- 📊 结果：Qwen3-32B WikiText perplexity 降至 8.43，与 BF16 基线 7.61 差距大幅收窄

**KV cache 压缩** · topic 5/5 · practicality 5/5 · composite 10/10 · 这是一篇高质量的KV缓存压缩工作，直接解决长上下文推理中的内存瓶颈。OScaR通过Canalized Rotation和Omni-Token Scaling有效处理Token Norm Imbalance问题，在INT2极端压缩下实现近无损性能。论文具有清晰的理论分析（识别TNI瓶颈）和强劲的实证结果（3.0x加速、5.3x内存减少、4.1x吞吐提升）。方法简洁（无需复杂pipelines如TurboQuant），有优化的CUDA实现和公开代码。跨越文本、多模态、全模态LLM的通用性评估展示了实用价值。虽然最大模型大小和具体精度基准不详，但near-lossless声明和系统性能提升确立了这是一项可部署的高影响工作。

#### 摘要
KV cache 的内存占用已成为长上下文和多模态 LLM 部署的主要瓶颈，现有 per-channel quantization 在极端压缩下效果下降。本文识别出 Token Norm Imbalance（TNI）是 quantization fidelity 的核心瓶颈——当共享 quantization 参数需跨越 norm 差异显著的 token 组时，误差被系统性放大。为此提出 OScaR（Omni-Scaled Canalized Rotation），通过 Canalized Rotation 结合 Omni-Token Scaling 抑制 sequence 维度的方差，配合优化的 CUDA kernels，在 text-only、multi-modal、omni-modal 等 X-LLMs 上实现 INT2 近无损压缩。与 BF16 FlashDecoding-v2 基线相比，OScaR 实现最高 3.0x 解码加速、5.3x 内存压缩和 4.1x 吞吐提升。

- 🎯 方法：Canalized Rotation + Omni-Token Scaling 联合消除 TNI，推进 per-channel paradigm 至 INT2 极端压缩
- 📊 结果：vs BF16 FlashDecoding-v2 基线，解码加速最高 3.0x，内存压缩 5.3x，吞吐提升 4.1x
- 💡 创新：理论与实验证明 Token Norm Imbalance 是 per-channel KV quantization 在极端压缩下失效的根本原因
- 🔧 工程：轻量级设计配合优化 CUDA kernels，覆盖 text-only / multi-modal / omni-modal X-LLMs

### 4. Backtracking When It Strays: Mitigating Dual Exposure Biases in LLM Reasoning Distillation (7/10) 🔁
**arxiv** · `2605.19433` · 2026-05-19
👥 Bing Wang, Shaotian Yan, Chen Shen... · 🏷 cs.CL, cs.AI
🔗 [arXiv](https://arxiv.org/abs/2605.19433) · [HF](https://huggingface.co/papers/2605.19433) · [alphaxiv](https://www.alphaxiv.org/abs/2605.19433)

🧪 pruning_distill · reasoning distillation with trajectory monitoring and backtracking · cal: 未明确说明，涉及动态监控和教师干预的在线蒸馏过程 · perf: 未报告端到端推理性能

#### 🧭 为什么推送这篇
**Pruning / 蒸馏** · topic 4/5 · practicality 3/5 · composite 7/10 · 本文针对LLM推理蒸馏中的暴露偏差问题提出解决方案，属于知识蒸馏范畴。通过动态回溯和教师干预机制改进小模型推理能力，在LIMO-v2和AceReason上取得约3%性能提升。但存在不足：（1）未报告最大模型规模和具体基线模型；（2）评估基准为特定推理数据集，未覆盖MMLU/GSM8K等标准LLM基准；（3）未提供end-to-end推理加速或内存节省的性能数据，主要贡献是蒸馏训练流程的改进而非部署推理优化；（4）涉及动态监控和自适应边界，算法复杂度相对较高，部署时需要额外的监控开销。属于推理蒸馏范畴且有学术贡献，但实用性受限于缺乏标准基准和明确的部署性能收益。

#### 摘要
LLM reasoning distillation 面临双重 exposure bias 问题：off-policy distillation 使用 teacher 生成的 golden trajectories，导致训练与推理分布不匹配（exposure bias）；on-policy distillation 让 student 自主探索，但 teacher 在 student 的次优上下文中也难以提供有效正向引导（reversed exposure bias）。MOTAB（Monitoring Trajectories and Backtracking when it strays）通过动态监控 student 的 on-policy 生成与自适应安全边界的偏离程度，在超出阈值时回退至最近安全状态并引入 teacher 干预纠正轨迹，从而同时缓解两种 bias。在 LIMO-v2 和 AceReason 数据集上，MOTAB 在推理任务中平均提升约 3% 的性能。

- 💡 首次明确指出 on-policy distillation 存在 reversed exposure bias：teacher 在 student 次优上下文中难以给出正向指导
- 🎯 MOTAB 在 on-policy 生成超出自适应安全边界时触发回退+teacher 干预，兼容两类 bias 的缓解
- 📊 在 LIMO-v2 和 AceReason 数据集上平均推理性能提升约 3%
- 🔧 无需修改 teacher/student 架构，作为 distillation pipeline 即插即用

#### 📎 相关方法 / 对比基线
- LIMO-v2 — 作为实验对比数据集/基线 distillation 框架
- AceReason — 作为实验对比数据集/基线 distillation 框架
