#### 🧭 为什么推送这篇
**KV cache 压缩** · topic 4/5 · practicality 3/5 · composite 7/10 · SeKV提出分层GPU-CPU KV缓存方案，通过熵引导语义分段、SVD低秩基和训练好的zoom-in机制实现按需重建，避免token信息丢失。方法评测在4个benchmark上，相比最强语义压缩基线提升5.9%，GPU内存节省53.3%（128K上下文）。代码已开源，base LLM完全冻结，可训练参数极少。不过存在若干实用性顾虑：CPU-GPU之间的低秩重建引入额外延迟，benchmark名称未在摘要中列出（无法确认是否为标准下游任务），最大测试模型规模不明。整体看是一个有趣的KV cache分层压缩方案，有开源代码和明确的内存收益，但CPU重建路径的推理延迟与部署复杂度使实用性打折。

#### 摘要
长上下文LLM推理中KV cache显存随序列长度线性增长，现有压缩方法（token eviction或语义分组）在生成阶段无法从已压缩区间恢复token级细节。SeKV将上下文组织为熵引导的语义span，在GPU上保存轻量summary vector用于粗粒度路由，在CPU上保存低秩SVD基用于按需重建，并通过训练的zoom-in机制在解码时选择性展开相关span。在4个benchmark上，SeKV相比最强语义压缩基线平均提升5.9%，相比128K上下文的全KV caching节省53.3% GPU显存，base LLM完全冻结且可训练参数不足0.05%。

- 🎯 熵引导语义分段 + GPU summary vector粗路由 + CPU SVD低秩基按需重建，构成分层KV cache
- 📊 相比最强语义压缩基线平均提升5.9%，128K上下文下GPU内存节省53.3%
- 💡 base LLM完全冻结，仅新增<0.05%可训练参数（zoom-in机制）
- ⚠️ CPU-GPU低秩重建引入额外延迟，推理延迟未在摘要中量化报告

#### 📎 相关方法 / 对比基线
- token eviction — 直接丢弃token信息，SeKV通过分层存储避免信息损失
- semantic grouping — 在prefill阶段固定压缩决策，无法在生成中恢复细节
